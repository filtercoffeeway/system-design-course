P5 · Write-time vs read-time cost (fan-out & precomputation)

What it is: choosing when you pay for an answer — eagerly at write (precompute) or lazily at read (on demand).
Reach for it when: a single event must reach many consumers, or an expensive query is read far more than its inputs change.
The lever: fan-out on write/materialize (fast reads, expensive writes, storage grows with audience); fan-out on read/compute-on-demand (cheap writes, slow reads); hybrid by node degree or query frequency. Write-time rollup vs read-time aggregation is the same dial for metrics.
New problem it hands you: push → storage blowup and the hot producer (celebrity write triggers millions of synchronous fan-outs). Pull → read latency and repeated recompute. Hybrid → two code paths + a hot-node classifier.
Exemplified by: Social feed, Datadog, URL shortener, Notification.

1 · The problem it solves

Some answers are expensive to compute. You can pay at write time (precompute and store the answer so reads are cheap) or at read time (store the inputs and compute on demand). This is the deepest tension in feeds, notifications, and analytics: write amplification vs read amplification.

2 · The fan-out ladder

Take a social feed (reads ≫ writes). Derive the strategy from the cost of each.

graph TD
  POST[user posts] -->|fan-out on write| F1[follower feed]
  POST --> F2[follower feed]
  POST --> F3[follower feed]
  RP[reader opens feed] -->|fan-out on read| M[merge followees on the fly]

Rung 0 — fan-out on read (compute on demand)

Store each post once; when a user opens their feed, fetch the recent posts of everyone they follow and merge. Writes are O(1) — dead simple.

post -> 1 write read feed (following K=500) -> merge ~500 authors' recent posts, every open system: 5B feed-reads/day x 500 = 2.5T lookups/day <-- read amplification
Bottleneck: reads dominate this workload, and you just made the common operation the expensive one — every feed open re-does a 500-way merge.

Rung 1 — fan-out on write (precompute)

Flip when you pay: on each post, push the item into every follower’s precomputed timeline (in Redis). Now a feed read is a single O(1) cache lookup — the right trade when reads ≫ writes.

avg post -> 200 feed writes (one per follower) read feed -> 1 precomputed lookup (O(1)) <-- the common op is now cheap
Bottleneck: the hot producer. A celebrity with 100M followers turns one post into 100M synchronous writes — a single action that can stall the whole fan-out pipeline, plus storage grows with total audience.

Rung 2 — hybrid by follower count ← the answer

Push for the ordinary majority; for accounts above a follower threshold, don’t push — their posts are pulled at read time and merged into the requesting user’s timeline. You pay O(audience) only where it’s small, and O(merge) only for the handful of celebrities a user follows.

push if followers < ~100,000 (avg user 200 -> push) pull if followers ≥ ~100,000 (celebrity 100M -> pull, merge a few at read)

3 · The new bottleneck it hands you

Push gives you storage blow-up and the hot producer (the celebrity write amplification above). Pull gives you slow reads and repeated recomputation. Hybrid — the practical answer — gives you two code paths plus a classifier deciding who is “hot,” and a merge step that must blend pushed and pulled items in rank order. The same dial appears in metrics as write-time rollup vs read-time aggregation (Datadog).

4 · Real-world example

Twitter timelines

Twitter’s home timeline is the textbook hybrid. For ordinary users it fans out on write: a tweet is pushed into each follower’s precomputed timeline (held in Redis), so reading your timeline is a fast cache read. But for accounts with millions of followers, pushing to every follower is prohibitively expensive, so those tweets are pulled at read time and merged into your timeline when you load it. The system literally chooses the strategy per author based on follower count — the consequence chain (push→hot producer) forced the split.

5 · Applying it in a design

Ask: how lopsided is read vs write, and how skewed is fan-out? Default to push for fast reads, then carve out the heavy hitters to pull. Say: "fan-out on write into a per-user timeline, hybrid pull for celebrity accounts to dodge the write amplification." The same dial appears in metrics as write-time rollup vs read-time aggregation (Datadog).

6 · Pitfalls

  • Pure push with no celebrity carve-out.
  • Precomputing views that are rarely read (wasted writes/storage).
  • Forgetting ranking: an ML-ranked feed can’t be a simple chronological merge.
