5 · Object Store (S3-style)

Hard constraint: durability and scale almost free, but zero transactionality and no in-place mutation — every design that treats it like a database fails.
Primitives exercised: P17 (erasure coding), P3, P8.

Read-after-write consistency guarantees (and the gaps); multipart upload; erasure coding vs 3× replication; presigned URLs as scoped access without proxying; storage classes for access-frequency decay.

1 · Requirements & scale

Functional: store and retrieve arbitrarily large blobs by key (PUT/GET/DELETE), list, with access control. Non-functional: effectively infinite capacity, extreme durability, high throughput — but no in-place edits and no transactions.

Back-of-envelope trillions of objects, exabytes total durability target ~ 99.999999999% (11 nines) object sizes bytes -> terabytes (need multipart) read/write massively parallel, region-local

2 · High-level architecture

graph LR
  C[client] --> FE[API / front-end]
  FE --> IDX[(metadata / index service)]
  FE --> DATA[(storage nodes: erasure-coded chunks)]
  IDX -->|object -> chunk locations| FE

A metadata service maps keys to chunk locations; data is split into chunks, encoded for durability, and spread across many nodes/racks. The API layer enforces auth and assembles objects.

3 · The hard constraint, in depth

You get durability and scale almost free, but zero transactionality and no in-place mutation. An object is replaced whole, not edited; there are no multi-object transactions. Every design that tries to use an object store like a database (frequent small updates, read-modify-write loops) fails. You build around it: write new versions, keep an external index/DB for mutable metadata, and treat objects as immutable.

4 · Component deep dives

Erasure coding vs replication (P17)

Instead of storing 3 full copies, split an object into k data + m parity chunks (e.g. 10+4): any 10 of 14 reconstruct it, surviving 4 losses while storing ~1.4× instead of 3×. The cost is CPU + latency to reconstruct on a degraded read.

Consistency

Historically eventually-consistent listings; modern S3 offers read-after-write consistency for new objects. Know what your store guarantees — it shapes correctness.

Large objects & access

Multipart upload splits a huge object into parts uploaded in parallel and assembled server-side (you can’t stream 5 TB in one request). Presigned URLs grant temporary, scoped access so clients upload/download directly without proxying bytes through your service.

5 · Real-world example

Amazon S3

S3 is designed for 11 nines of durability, achieved by redundantly storing each object across multiple devices and facilities (Availability Zones) using erasure coding rather than naive triple replication — less stored bytes for the same or better durability. It exposes multipart upload for large objects, presigned URLs for scoped direct access, storage classes for tiering (P8), and an explicit, immutable object model (versioning rather than edits). It is the canonical "durable, scalable, but not a database" service that other systems (video, Datadog cold tier) build upon.

6 · Tradeoffs & alternatives

Erasure coding saves storage but slows degraded reads vs replication; pick by read-latency sensitivity. For mutable, transactional, small-record data use a KV store or RDBMS; use the object store for the big immutable blobs and keep pointers in the DB.

7 · Key talking points

  • Lead with: "durable + infinite, but immutable and non-transactional."
  • Erasure coding for durability at lower storage cost (P17).
  • Multipart upload + presigned URLs (don’t proxy big bytes).
  • Separate mutable metadata into an index/DB.
  • Tiering via storage classes (P8).
