13 · Distributed Lock / Leader Election (ZooKeeper / etcd)

Hard constraint: the slow-but-alive holder (GC pause, network blip) — TTL tuning can't make this correct; fencing tokens can.
Primitives exercised: P10 (consensus/fencing).

Raft/Paxos + majority quorum; fencing tokens (monotonic, reject stale); session/heartbeat TTL for death detection; leader election as a lock with metadata; watch/notify instead of polling.

1 · Requirements & scale

Functional: let distributed processes coordinate — acquire a mutual- exclusion lock, or elect a single leader — correctly despite crashes and network faults. Non-functional: safety (never two holders) over liveness, low-ish churn, small data.

2 · High-level architecture

graph TD
  C1[client 1] --> Q[(coordination service: Raft/Paxos quorum)]
  C2[client 2] --> Q
  Q -->|grants lock + fencing token N| C1
  C2 -->|blocked / watch for release| Q

A small, consensus-backed service (ZooKeeper/etcd/Chubby) holds the lock state on a quorum so it survives node loss. Clients acquire via ephemeral keys with sessions, and watch for release instead of polling.

3 · The hard constraint, in depth

A lock holder can be slow but not dead — a GC pause or network blip makes it look dead, its lease expires, another client acquires the lock, and then the original wakes up still believing it holds the lock. Two holders act at once and corrupt state. No TTL value fixes this. The correct fix is a fencing token: the lock service hands out a monotonically increasing number with each grant, and the protected resource rejects any operation carrying a token lower than the highest it has seen (P10).

4 · Component deep dives

Consensus core (P10)

Raft/Paxos majority quorum makes the lock state itself fault tolerant; leader election is "a lock with the leader’s identity as the value."

Sessions, ephemeral nodes & watches

A client holds a session with a heartbeat; its lock is an ephemeral node that auto-releases if the session dies (death detection via TTL). Other clients set a watch to be notified the instant the lock frees — no polling.

Fencing

Monotonic tokens enforce correctness that TTL tuning cannot.

5 · Real-world example

Google Chubby, ZooKeeper, etcd

Google’s Chubby (Paxos) is the original coarse-grained lock + naming service much of Google coordinates through — e.g. electing a master for GFS/Bigtable. ZooKeeper brought the model to the open-source world (Kafka used it for controller election; HBase/Hadoop for coordination), and etcd (Raft) is the coordination store and leader-election backbone of Kubernetes. The cautionary tale is the Redlock debate: Martin Kleppmann showed that a lock without a fencing token isn’t safe under GC pauses — cementing fencing as the non-negotiable piece.

6 · Tradeoffs & alternatives

A consensus service is the safe choice but adds a dependency and quorum latency; a Redis lock is fast but needs fencing to be correct. Coarse locks (one leader) are simpler and usually better than many fine-grained locks. Prefer leader-election + single-writer over distributed mutual exclusion when you can.

7 · Key talking points

  • Use a consensus service (etcd/ZooKeeper/Chubby), don’t DIY (P10).
  • Slow-but-alive holder is the core hazard.
  • Fencing tokens, not TTL tuning, give correctness.
  • Sessions + ephemeral nodes for death detection; watches over polling.
  • Prefer coarse leader election to fine-grained locks.
