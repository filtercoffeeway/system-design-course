15 · Ride-Sharing / Location System (Uber)

Hard constraint: a flood of location writes (~1M/s) meets a variable-radius proximity read — the geo index must absorb both, and the cell size that makes the read cheap makes a downtown cell a hot partition.
Primitives exercised: P15 (geospatial), P1 (batching, write-heavy), P2 (hot-cell partitioning), P13 (live connections), P4 (surge read).

Geohash/H3 partitioning; ~1.25M location updates/s (write-heavy); matching as a supply/demand problem inside a cell; surge as a real-time read of that ratio; routing/ETA as shortest-path on a live-weighted graph.

1 · Requirements

Functional: drivers stream their GPS location continuously; a rider requests a ride and is matched to a nearby available driver; the system prices the trip by local supply/demand (surge); it routes the driver and shows a live ETA. Non-functional:

The defining observation: this is not a storage problem. The working set (where every driver is right now) is small and fits in RAM; what's brutal is the write rate and the geo read. That reframing drives every later decision.

2 · Capacity estimation

5,000,000 active drivers, GPS ping every ~4 s -> 5,000,000 / 4 = 1,250,000 location writes/s (the dominant load) live position record: driver_id 8B + lat 8B + lng 8B + ts 8B = 32 B -> 5,000,000 x 32 B = 160 MB of live state (fits in RAM on ONE box) ride requests ~ 20,000,000/day -> ~231/s avg, ~1,150/s peak (5x) -> reads are ~1000x rarer than writes; optimize the WRITE path first naive: one DB row-write per ping -> 1.25M writes/s a single SQL node does a few k writes/s -> ~250x over budget (§4 kills this)

Two numbers decide the architecture: 1.25M writes/s (so the ingest path cannot touch a durable DB per ping) and 160 MB of live state (so the entire "where is everyone" index fits in memory and can be sharded by geography across a handful of nodes for headroom + HA).

4 · Data model — deriving the geo index (ladder)

The core question is: given a rider's point, return the available drivers within radius R, fast, while 1.25M updates/s rain down. Derive the structure, don't assert "use a quadtree."

Rung 0 — durable DB, one row per driver, WHERE on lat/lng

SELECT * FROM drivers WHERE lat BETWEEN ? AND ? AND lng BETWEEN ? AND ?. A B-tree on lat can range-scan one dimension, but the second dimension isn't covered, so the engine scans a horizontal stripe across the whole map and filters. Breaks on: two things at once — the write rate (1.25M row-updates/s ≈ 250× a single node) and the read (a stripe scan is O(drivers-in-a-latitude-band), not O(drivers-near-you)). Dead on arrival.

Rung 1 — an in-memory uniform grid (fixed lat/lng buckets)

Chop the world into fixed squares (say 0.01° ≈ 1 km). cell_id = (floor(lat/0.01), floor(lng/0.01)) is an O(1) hash. A query reads the rider's cell plus the 8 neighbors (a 3×3 ring, because the nearest driver may be just over a boundary). All in RAM → reads and writes are both O(1)-ish. Breaks on: the Earth isn't flat — a fixed degree-square is ~111 km tall everywhere but shrinks east-west toward the poles (cos(latitude)), so cells are wildly non-uniform; and the grid is flat — there's no cheap way to zoom out when a rural cell has zero drivers and you must widen the radius.

Rung 2 — geohash (hierarchical prefix cells)

Geohash interleaves latitude and longitude bits and Base32-encodes them, so a string prefix is a bounding box, and a longer prefix is a smaller box nested inside the shorter one. That hierarchy is the upgrade over a flat grid: to widen the search you just drop a character.

geohash precision -> approximate cell size len 5 ~ 4.9 km x 4.9 km len 6 ~ 1.2 km x 0.61 km len 7 ~ 153 m x 153 m <- typical dispatch precision len 8 ~ 38 m x 19 m len 9 ~ 4.8 m x 4.8 m index: HASH (geohash7) -> set{ driver_id } (a Redis sorted set / in-mem map per cell) query: drivers in rider's geohash7 + its 8 neighbors; widen to geohash6 if too few

Breaks on: (a) the prefix box is rectangular and its aspect ratio changes with latitude, so "neighbors" aren't equidistant — ring search over-/under-shoots; (b) the edge problem: two points 1 m apart can straddle a prefix boundary and share no common prefix, so you must always probe neighbors; (c) precision jumps by ~32× in area per character, so there's no smooth "widen by 20%."

Rung 3 — H3 hexagonal hierarchical grid (the answer)

H3 (Uber's open grid) tiles the globe with hexagons at 16 resolutions. Hexagons fix geohash's flaws: every neighbor is the same distance away (a hexagon has 6 equidistant neighbors; a square has 4 edge + 4 diagonal at √2), so "expand the search ring" (kRing(cell, k)) grows smoothly and uniformly, and cell areas are near-constant within a resolution.

H3 resolution -> ~edge length res 7 ~ 1.22 km res 8 ~ 461 m <- common dispatch resolution res 9 ~ 174 m res 10 ~ 66 m index: H3index(res8) -> { driver_id -> {lat,lng,ts} } in-memory, sharded by cell match: ring = kRing(riderCell, 1); if |available| < N: k++ and retry
Why this is the canonical answer: uniform neighbor distance makes ring-expansion and supply/demand smoothing across adjacent cells clean, and the hierarchy (a res-8 cell has 7 res-9 children) lets surge and analytics roll up by zoom level for free.

Durable layout vs the in-memory index

The hot index is RAM-only and rebuildable: positions are ephemeral (a 4-second-old GPS ping is worthless), so durability isn't required for the live map — on shard failover the new owner repopulates within one ping interval (~4 s) as drivers send their next update. Trip records, payments, and ratings are durable and live in normal stores (S17-style ledger for fares).

5 · High-level architecture

graph LR
  D[drivers] -->|ping every 4s| GW[location gateway / WebSocket]

  ING --> GEO[(in-mem H3 index, sharded by cell)]
  R[rider request] --> DISP[dispatch / matching]
  DISP -->|kRing query| GEO
  DISP --> SURGE[surge: supply/demand per cell]
  SURGE --> GEO
  DISP -->|offer w/ lease| D
  GEO -. async .-> TRIP[(durable trip store)]

Drivers hold a persistent connection (P13) to a location gateway, which batches (P1) pings and updates the in-memory H3 index sharded by cell. Dispatch queries that index, consults surge, and offers the ride to a driver under a short lease. Trips are written durably out of the hot path.

6 · The write path — absorbing 1.25M pings/s

Each ping does an in-place update of one driver's entry and, if the driver crossed a cell boundary, moves it between cells. The relief ladder for the write rate:

  • Don't persist every ping. The DB rung already died (§4). Writes hit RAM; a sampled/throttled subset (e.g. every Nth ping, or on cell change) is async-journaled for analytics — not on the critical path.
  • Batch + coalesce (P1): the gateway buffers pings for ~200 ms; if a driver pings twice in a window, only the latest matters (last-write-wins on position), so 1.25M pings collapse to ~1 update per driver per window.
  • Shard by cell (P2): the index is partitioned by H3 cell (or a coarse parent cell), so the write load spreads across nodes by geography. A driver's update routes to the node owning its current cell.
New bottleneck — the hot cell. Geography is skewed: midtown Manhattan on Friday night has orders of magnitude more drivers/requests than a suburb. Sharding by cell concentrates that load on one node. Fixes: shard by a coarser parent cell so load averages out; or split the hot cell across replicas with the cell key suffixed by a bucket (cell#0..cell#k) and scatter-gather the buckets on read (P2 hot-key split, P7 merge). The read fan-out over k buckets is cheap because k is small.

7 · The read path — matching (ladder)

A request resolves the pickup point to its H3 cell, queries kRing(cell, k) for available drivers, ranks them, and offers. How you choose and commit the driver is itself a ladder.

Rung 1 — greedy nearest, offer immediately

Pick the closest available driver, offer, done. Simple, low latency. Breaks on: two riders 50 m apart both get offered the same nearest driver (a race), and greedy-nearest is globally poor — assigning the closest driver to rider A may strand rider B with a far one, when a tiny swap would have served both better.

Rung 2 — lease the driver during the offer (fix the race)

When a driver is offered, mark them held with a short lease (a few seconds, the time to accept/decline). A concurrent match skips held drivers. If the offer times out or is declined, the lease expires and the driver returns to the pool. This is the same lease/fencing pattern as P10 / S13 — it prevents double-offering.

Rung 3 — batched window dispatch (fix global quality)

Instead of matching each request the instant it arrives, accumulate requests over a short window (e.g. ~1 s) per area and solve a batch assignment — a min-cost bipartite matching (Hungarian algorithm / min-cost flow) of riders→drivers minimizing total ETA. This is Uber's DISCO insight: a sub-second wait buys a globally better set of matches than greedy-nearest.

greedy: rider A -> driver 1 (80m), rider B -> driver 2 (900m) total 980m batched: rider A -> driver 2 (300m), rider B -> driver 1 (250m) total 550m -> the window trades ~1s latency for shorter combined pickup ETA + fewer cancels

8 · Surge, routing & ETA

Surge is a real-time read over the same location data: per cell, multiplier = f(open_requests / available_drivers) smoothed over neighbors and time. Because H3 cells roll up, surge can be computed at a coarse resolution and interpolated — it's a cheap aggregate read (P4), recomputed every few seconds, not a per-request computation.

ETA / routing is shortest-path on a road graph. The naive Dijkstra over a continent is far too slow per request, so the production ladder is: precompute contraction hierarchies (or CH/ALT landmarks) on the static graph so a query is milliseconds; overlay live edge weights from current traffic (itself derived from driver speeds — the location stream feeds back in); recompute ETAs periodically. Pickup ETA (driver→rider) is the routing cost used to rank candidates in §7.

9 · Scaling & bottlenecks at 10×

At 50M drivers / 12.5M writes/s: the in-memory index no longer fits/serves on a few nodes, so shard finer by H3 parent cell and add read replicas per cell for the (rarer) match reads. The hot-cell problem intensifies during events (a stadium emptying) — pre-split known hot cells and apply P16 backpressure (drop to a coarser ping cadence) when a region saturates. The batch solver becomes the per-cell bottleneck in dense markets; cap the batch size and fall back to greedy+lease under overload (graceful degradation).

10 · Tradeoffs & alternatives

Cell size is the master dial: bigger cells → fewer lookups but more candidates to filter and coarser surge; smaller cells → precise but more boundary handling and more hot-cell risk. In-memory index (fast, ephemeral, fast failover) vs durable geo store (safe but too slow for 1.25M/s). Batched dispatch (better matches) vs greedy (lower latency). Quadtree is the alternative to H3 when density is extremely non-uniform — a quadtree subdivides only where it's dense, so each leaf holds a bounded count, at the cost of rebalancing as drivers move (H3's fixed grid never rebalances, which is why Uber chose it).

11 · Real-world example

Uber

Uber indexes the world with H3, its open-source hexagonal hierarchical grid (P15), and runs matching and surge per cell. Hexagons give uniform neighbor distances, which makes ring-expansion and supply/demand smoothing across adjacent cells clean. Its dispatch system (DISCO) batches ride requests over short time windows and solves a global assignment rather than matching greedily, improving combined ETAs and cutting cancellations. Surge is computed from real-time supply/demand per H3 cell. Live ETAs come from routing over a road graph whose edge weights are continuously updated from the very driver-speed stream the system ingests — the location firehose is both the write load and the input to traffic estimation. The defining trait: a write-heavy, real-time geospatial system where the in-memory location index and its cells are the center of the design.

12 · Interview follow-ups (Q&A)

Q: How do you handle a driver right on a cell boundary? Always query the rider's cell plus its ring of neighbors (kRing(cell,1) = 6 hexes for H3), then filter by true great-circle distance and rank by routing ETA. Never trust the cell alone for the radius decision — the cell is a coarse pre-filter; the exact distance/ETA is the real ranking key.

Q: How do you stop two riders being matched to the same driver? A short lease/hold on the driver during the offer (§7 rung 2), with a fencing token so a stale dispatcher can't re-confirm a driver who already moved on (P10). The lease auto-expires on decline/timeout.

Q: A leader/shard owning a hot cell dies — recovery? Position state is ephemeral, so there's no big rebuild: the standby takes the cell and the in-memory map repopulates within ~one ping interval (~4 s) as drivers send their next update. In-flight offers (held leases) are reconstructed from the durable matching log or simply expire and re-match. This is the "bound the failover cost" check — here the bound is ~4 s, by design.

Q: Why not just persist every location to a database? 1.25M durable writes/s is ~250× a single SQL node and the data is worthless in 4 s — persisting it buys nothing. Keep the live map in RAM; async-journal a sampled subset for analytics/traffic only.

Q: How does the location stream feed traffic/ETA without a loop? Aggregate driver speeds per road segment over a sliding window to estimate live edge weights; routing reads those weights. It's a feedback pipeline, not a synchronous loop — ETAs use the latest published weights, recomputed every few seconds.

Q: What about exactly-once for the match itself? Treat "create trip" as an idempotent operation keyed by request_id (P9): a retried confirm with the same key returns the existing trip rather than creating a second one, so at-least-once delivery of the confirm is safe.

13 · Key talking points

  • It's a write-heavy + geo-query problem (~1.25M pings/s), not a storage problem — the live map is 160 MB and lives in RAM.
  • Geo index derived as a ladder: DB scan → uniform grid → geohash → H3 hexes (uniform neighbors, smooth ring-expansion).
  • Hot downtown cells = hot partition (P2); fix by coarser parent-cell sharding or bucket-split + scatter-gather.
  • Matching ladder: greedy nearest → offer lease (fix race) → batched window assignment (fix global quality, à la DISCO).
  • Surge = real-time supply/demand read per cell; ETA = contraction-hierarchy routing on a live-weighted graph fed by the same stream.
  • Don't persist every ping; positions are ephemeral, so failover is bounded by one ping interval.
