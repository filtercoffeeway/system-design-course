5 · Object Store (S3-style)

Hard constraint: durability and scale almost free, but zero transactionality and no in-place mutation — every design that treats it like a database fails.
Primitives exercised: P17 (erasure coding), P3, P8.

Read-after-write consistency guarantees (and the gaps); multipart upload; erasure coding vs 3× replication; presigned URLs as scoped access without proxying; storage classes for access-frequency decay.

1 · Requirements

Functional: PUT/GET/DELETE/LIST arbitrarily large blobs by key, with access control. Non-functional: effectively infinite capacity, extreme durability (11 nines), high throughput — and explicitly no in-place edits, no transactions.

trillions of objects, exabytes total durability target ~ 99.999999999% (11 nines) = lose ~1 object per 10^11 per year object sizes bytes -> terabytes (need multipart upload) access massively parallel, region-local, read-heavy + immutable

2 · API & data model

PUT /bucket/key (body = bytes) -> 200, version-id GET /bucket/key -> bytes DELETE /bucket/key ; LIST /bucket?prefix=... multipart: InitiateMultipartUpload -> UploadPart(n) x N -> Complete

Two planes: a metadata/index service (key → chunk locations, size, version, ACL — small, mutable, transactional, lives in a KV/DB) and a data plane (the immutable bytes, erasure-coded across nodes). Keeping mutable metadata out of the data plane is the central design move.

3 · High-level architecture

4 · Durability — the ladder

Rung 0 — one copy

Store the object once. A single disk failure loses it. Disks fail constantly at fleet scale, so this is a non-starter.

Bottleneck: no fault tolerance.

Rung 1 — N× replication

Keep 3 full copies on different nodes/AZs. Survives 2 losses, dead simple, and a read just grabs any copy. But it triples storage — at exabyte scale, 200% overhead is enormous money.

Bottleneck: 3× storage cost; you’re paying for whole duplicate copies.

Rung 2 — erasure coding ← the answer

Split the object into k data + m parity chunks; any k of the k+m reconstruct it (Reed-Solomon). The math beats replication on both axes:

overhead survives to store 1 PB 3x replication 3.0x 2 losses 3 PB (10,4) erasure 1.4x 4 losses 1.4 PB -> erasure coding survives MORE failures at LESS than half the storage
Bottleneck (accepted): a read of a missing chunk must fetch k chunks and recompute (CPU + cross-node latency) — a degraded read. So small hot objects sometimes still use replication; big cold blobs use erasure coding.

5 · The write & read paths

PUT object (10,4): front-end auth -> split into 10 data chunks -> compute 4 parity (Reed-Solomon) scatter the 14 chunks across 14 nodes in different racks/AZs record chunk locations in the metadata service -> ACK GET object: metadata -> chunk locations; fetch any 10 of 14 in parallel -> assemble if a node is down: fetch 10 of the surviving 13, recompute the missing chunk (degraded read)

Multipart upload splits a huge object into parts uploaded in parallel and assembled server-side (you can’t stream 5 TB in one request). Presigned URLs grant temporary scoped access so clients upload/download directly to the data plane without proxying terabytes through your API.

The metadata service — its own scaling ladder

The index (key → chunk locations, size, version, ACL) is small per object but there are trillions of them, so it has its own derivation. A single SQL table is the right default — until trillions of rows blow past one node’s index. Since access is a pure keyed lookup (GET /bucket/key), the ladder lands exactly where S14 §4 and S3 land: a sharded KV / wide-column store partitioned by key. LIST is the wrinkle — prefix scans want keys ordered, so the metadata store is usually range-partitioned by key (not hash), which reintroduces the hot-shard risk for a popular prefix (P2).

tiny objects problem: a 1 KB object still needs a metadata row (~hundreds of bytes) -> billions of tiny objects make METADATA, not data, the bottleneck -> mitigations: pack small objects into larger blobs; charge/throttle per-request

6 · The hard constraint, in depth

You get durability and scale almost free, but zero transactionality and no in-place mutation. An object is replaced whole, never edited; there are no multi-object transactions. Every design that tries to use an object store like a database (frequent small updates, read-modify-write loops) fails. You build around it: write new versions, keep mutable state in the external index/DB, treat objects as immutable. Modern S3 gives read-after-write consistency for new objects — know exactly what your store guarantees, because it shapes correctness.

7 · Real-world example

Amazon S3

S3 is designed for 11 nines of durability, achieved by redundantly storing each object across multiple devices and facilities (Availability Zones) using erasure coding rather than naive triple replication — less stored bytes for the same or better durability. It exposes multipart upload for large objects, presigned URLs for scoped direct access, storage classes for tiering (P8), and an explicit, immutable object model (versioning rather than edits). It is the canonical "durable, scalable, but not a database" service that other systems (video, Datadog cold tier) build upon.

8 · Tradeoffs & alternatives

Erasure coding saves storage but slows degraded reads vs replication; pick by read-latency sensitivity (hot small objects → replicate; cold large blobs → erasure-code). For mutable, transactional, small-record data use a KV store or RDBMS; use the object store for the big immutable blobs and keep pointers in the DB. Tiering across storage classes by access-frequency decay is the P8 dial.

9 · Key talking points

  • Lead with: "durable + infinite, but immutable and non-transactional."
  • Erasure coding for durability at lower storage cost (P17).
  • Multipart upload + presigned URLs (don’t proxy big bytes).
  • Separate mutable metadata into an index/DB.
  • Tiering via storage classes (P8).
