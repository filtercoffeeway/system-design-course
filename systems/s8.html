8 · Distributed SQL Engine (Presto / BigQuery-style)

Hard constraint: shuffle (cross-node redistribution for joins/group-by) is the bottleneck; every optimizer decision reduces to minimizing shuffle.
Primitives exercised: P8 (columnar), P7, P12, P2 (pruning).

Columnar storage; predicate/projection pushdown + partition pruning; cost-based planning; hash vs sort-merge vs broadcast join; exchange/shuffle operators; vectorized execution.

1 · Requirements & scale

Functional: ad-hoc analytical SQL (joins, group-by, aggregates) over petabytes, no pre-built indexes. Non-functional: interactive-ish latency on huge scans, elastic parallelism, read-mostly.

Back-of-envelope 1 PB table, query selects 3 of 200 columns row store: read ~all 1 PB -> hopeless columnar: read 3/200 of data ~ 15 TB scanned + partition pruning by date ~ only relevant partitions distribute scan across 1,000 workers -> seconds, if shuffle is small

2 · High-level architecture

graph TD
  SQL[query] --> PL[planner / optimizer]
  PL --> ST1[stage: scan + filter]
  ST1 -->|exchange / shuffle| ST2[stage: join / aggregate]
  ST2 --> R[result]

The planner compiles SQL into a DAG of stages run by many workers. Scans read only projected columns from columnar files; stages exchange data through shuffles for joins and group-bys.

3 · The hard constraint, in depth

Shuffle — redistributing data across the network so that rows that must be joined or grouped land on the same worker — is the bottleneck. The optimizer’s real job is not to compute faster but to avoid reading and moving data: predicate pushdown, projection pushdown, partition pruning, and choosing join strategies that minimize shuffle volume. Every plan decision reduces to "how do I move less data?"

4 · Component deep dives

Columnar scan + pushdown (P8)

Parquet/ORC let you read only needed columns and skip row groups via min/max stats; the planner pushes filters and projections down into the scan (P8).

Join strategies

Broadcast join (ship the small table to every worker — no shuffle of the big one), hash/shuffle join (repartition both tables by key), sort-merge join (for sorted/huge inputs). Cost-based planning estimates row counts to pick.

Execution

Vectorized, column-at-a-time processing (SIMD) instead of row-at-a-time; exchange operators implement the shuffle.

5 · Real-world example

Presto / Trino & Google BigQuery

Presto (built at Facebook, now also Trino) is a distributed MPP SQL engine that queries data in place (S3, HDFS, etc.) by compiling SQL into stages of workers and shuffling between them — Meta runs it over exabyte-scale warehouses for interactive analytics. Google BigQuery (built on Dremel, P8) separates storage from compute and fans a query across thousands of slots over columnar storage; its planner aggressively prunes and pushes down so a multi-TB scan returns in seconds. Both live or die by minimizing shuffle.

6 · Tradeoffs & alternatives

MPP engines are built for big scans, not point lookups or updates — use an OLTP database for those. Pre-aggregation/materialized views trade storage and freshness for query speed. More workers help scans but not a shuffle-bound query.

7 · Key talking points

  • Columnar + pushdown + partition pruning: read less (P8).
  • Shuffle is the bottleneck; the optimizer minimizes data movement.
  • Broadcast vs shuffle vs sort-merge joins by table size.
  • Scatter-gather stages; slowest worker sets latency (P7, P12).
  • Vectorized execution.
