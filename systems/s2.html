2 · Message Queue / Kafka

Hard constraint: a partition is consumed sequentially by one consumer per group → throughput ceiling = partition count × consumer speed.
Primitives exercised: P1, P2, P3 (ISR), P16, P17 (log).

Log-structured storage (append beats random write); partitioning strategies; consumer-group parallelism ceiling; offset management, at-least-once vs EOS; replication factor + ISR; compacted topics for changelogs; retention as a replay buffer.

1 · Requirements & scale

Functional: producers publish messages to topics; consumers read them, possibly much later, possibly re-reading; messages of a related key arrive in order; multiple independent consumers read the same stream. Non-functional: very high write throughput, durability (a committed message survives broker loss), horizontal scalability, and bounded, predictable latency.

Back-of-envelope 1,000,000 messages/sec × 1 KB = 1 GB/s ingest retention 7 days = 1 GB/s × 604,800 s ≈ 600 TB (× replication) replication factor 3 => ~1.8 PB on disk parallelism: consumer group throughput = (#partitions) × (per-consumer rate) -> to consume 1 GB/s at 50 MB/s/consumer => need ≥ 20 partitions/consumers

The estimate exposes the central design fact: your consumer parallelism is capped by partition count, so partition count is the most important upfront decision.

A topic is split into partitions (P2); each partition is an append-only log (P17) replicated across brokers (P3). Producers append (batched, P1); each partition is consumed by exactly one consumer within a group, which is both the parallelism mechanism and the constraint.

3 · The hard constraint, in depth

Within a consumer group, each partition is assigned to exactly one consumer, and that consumer reads the partition strictly in offset order. This is what gives you per-key ordering — but it means you cannot have more active consumers in a group than partitions. If you provision 10 partitions and need more throughput, an 11th consumer sits idle. You cannot increase parallelism without re-partitioning, which is operationally painful (it changes key→partition mapping and breaks ordering during the transition).

So partition count is a capacity decision you must make early and slightly over-provision. Too few = throughput ceiling. Too many = more metadata, more open files, longer leader elections, more end-to-end latency. The "right" number is "enough to hit peak consumer parallelism with headroom."

4 · Component deep dives

The log: why append beats random write (P17)

Each partition is an immutable, append-only sequence of records addressed by a monotonic offset. Appends are sequential disk writes — the single fastest thing a disk does, HDD or SSD — so write throughput is enormous. Reads are also sequential scans from an offset. There is no in-place update and no read-modify-write, which is exactly why a log outruns a B-tree-backed queue. See P17.

Zero-copy and the page cache

Kafka leans on the OS page cache rather than a user-space cache: written data sits in cache and is served to consumers from there. Crucially, broker→consumer reads use zero-copy (sendfile): bytes go from page cache straight to the network socket without being copied into the JVM heap and back. This is a large part of why a single broker can push gigabytes per second.

Partitioning strategy (P2)

The producer chooses a partition: by key hash (all records with the same key land on the same partition → per-key ordering), round-robin (max spread, no ordering), or custom. Key-hash partitioning is where the hot-partition problem bites: a single hot key (one giant tenant) overloads one partition while others idle. Mitigations: a composite key, or splitting the hot key across sub-partitions and merging downstream.

Replication & ISR (P3)

Each partition has a leader and N−1 followers. The leader tracks the in-sync replica set (ISR) — followers caught up within a bound. With acks=all, a write is acknowledged only once all ISR members have it, so it survives the loss of any in-sync broker. The dial: acks=0/1/all and min.insync.replicas trade latency for durability — the P3 consistency ladder in concrete form.

Offsets & delivery semantics

Consumers track their position with a committed offset. Commit after processing → at-least-once (a crash before commit replays the message). Commit before → at-most-once. Exactly-once (EOS) needs idempotent producers (dedup by producer-id + sequence number) plus transactional writes that atomically commit messages and offsets — real but more expensive, so reach for it only when duplicates are unacceptable. Pair with idempotent consumers (P9) and you usually don't need EOS at all.

Retention, replay & compaction

Messages persist for a retention window (time or size) regardless of consumption, so a new or buggy consumer can replay from any offset — the log doubles as a recovery buffer after a bug. Log-compacted topics instead retain the latest value per key forever, which makes a topic a durable changelog/snapshot (the basis for event sourcing and for Kafka's own consumer-offset and Connect state).

Backpressure (P16)

Because storage is durable and consumers pull at their own pace, a slow consumer simply lags — it doesn't push back on producers. The log is the shock absorber. Your job is to monitor consumer lag and scale consumers (up to the partition ceiling). See P16.

5 · Real-world example

LinkedIn

Kafka was built at LinkedIn (~2010) to unify a sprawl of point-to-point data pipelines into one central log: every event — page views, metrics, database changes — published once and consumed by many systems (search, analytics, monitoring, Hadoop). The log-as-backbone idea is laid out in Jay Kreps' essay "The Log". LinkedIn has reported running Kafka at the scale of trillions of messages per day across thousands of brokers, which is the existence proof that an append-only, partitioned, page-cache-served log scales far past what a traditional broker (RabbitMQ, ActiveMQ) reaches.

Modern Kafka can replace ZooKeeper with the built-in KRaft consensus mode for metadata — the controller quorum uses Raft (P10), folding leader election and metadata storage into Kafka itself.

6 · Tradeoffs & alternatives

If you need per-message acknowledgement, complex routing, or priority queues, a traditional broker (RabbitMQ) or a managed queue (SQS) fits better than a log — Kafka deliberately trades those for throughput and replay. If you need millions of independent low-throughput queues, the per-partition overhead hurts. If you need strict global ordering across all keys, you're forced to a single partition and lose all parallelism — a sign the requirement itself should be questioned.

7 · Key talking points

  • Lead with the ceiling: "parallelism = partition count, so I'll size partitions for peak consumer throughput with headroom."
  • Append-only log = sequential writes; page cache + zero-copy reads (P17).
  • Key-hash partitioning for per-key order; call out the hot-partition risk (P2).
  • ISR + acks=all for durability; name the latency/durability dial (P3).
  • At-least-once + idempotent consumers by default; EOS only when required (P9).
  • Retention = replay buffer; compaction = changelog.
