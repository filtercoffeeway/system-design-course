4 · Relational Database Internals + Scaling

Hard constraint: cross-shard joins are prohibitively expensive — denormalize to avoid them, or accept fan-out-to-all-shards + merge.
Primitives exercised: P17, P2, P3 (replica lag), P7.

MVCC; index types (B-tree/hash/GIN/GiST) and why you can't index freely; read replicas + replication lag; sharding strategies + the cross-shard join problem; connection pooling as a DB resource; 2PC and why it's rarely right.

1 · Requirements & scale

Functional: structured data with ACID transactions, joins, secondary indexes. Non-functional: keep those guarantees as data and traffic outgrow a single node. The whole story is "how far can one box go, and what breaks first."

Scaling ladder 1) one node, tuned -> vertical limit 2) + read replicas -> read scale, write still single-node 3) + caching / connection pooling -> relieve read + connection pressure 4) + functional partitioning (by table) -> independent services 5) + horizontal sharding (by key) -> write scale, lose cross-shard joins

3 · The hard constraint, in depth

Once you shard, joins across shards are prohibitively expensive — a join that used to be local now fans out to every shard and merges over the network. You respond by denormalizing so related rows share a shard, choosing a shard key that co-locates what you read together, and accepting that some queries scatter-gather (P7). Cross-shard transactions need 2PC, which blocks and rarely scales.

4 · Component deep dives

MVCC & the WAL

Multi-Version Concurrency Control lets readers see a consistent snapshot without blocking writers (each write makes a new version). The WAL (P17) gives durability and feeds replication.

Read replicas & lag (P3)

Replicas scale reads but lag the primary; reading your own write from a replica can return stale data — route critical reads to the primary or wait for a position.

Sharding (P2)

Range, hash, or directory-based. The shard key is the most important decision; get it wrong and you get hot shards or unavoidable cross-shard queries.

Connection pooling

Each Postgres connection costs real memory/process; thousands of app workers can exhaust the DB. A pooler (PgBouncer) multiplexes many clients onto few server connections — connection count is a database resource, not just a network one.

5 · Real-world example

YouTube / Vitess

YouTube outgrew a single MySQL and built Vitess (now a CNCF project, also used by Slack, GitHub, and others) to shard MySQL horizontally while keeping it looking like one database to the app. Vitess sits in front of many MySQL shards: a VTGate proxy parses SQL, routes each query to the right shard(s) by the sharding key, scatter-gathers when a query spans shards, and manages resharding online. It is the productized answer to "how do you keep relational semantics past one node" — and it leans on exactly the cross-shard-join constraint above, steering apps toward shard-local queries.

6 · Tradeoffs & alternatives

Replicas first (cheap read scale), shard only when writes force it. If you need horizontal writes and distributed transactions, a NewSQL store (CockroachDB/Spanner, Raft-replicated ranges) buys that at higher latency. If you need massive write scale and can drop joins/ACID, a KV store fits better.

7 · Key talking points

  • Climb the ladder: tune → replicas → cache/pool → partition → shard.
  • Shard key co-locates data; cross-shard joins are the wall (P2, P7).
  • Replica lag → read-after-write care (P3).
  • MVCC for concurrency; connection pooling as a finite resource.
  • Avoid 2PC; denormalize instead.
