8 · Distributed SQL Engine (Presto / BigQuery-style)

Hard constraint: shuffle — moving data across the network so rows that must join or group land on the same worker — dominates. Every optimizer decision reduces to "how do I read and move less data?"
Primitives exercised: P8 (columnar), P7 (scatter-gather), P12 (tail/stragglers), P2 (pruning/partitioning), P18 (distributed sort).

Columnar storage; predicate/projection pushdown + partition pruning; cost-based planning; broadcast vs shuffle-hash vs sort-merge joins; exchange/shuffle operators; vectorized execution; stragglers.

1 · Requirements

Functional: run ad-hoc analytical SQL — multi-table joins, group-by, aggregates, window functions — over petabytes, with no pre-built secondary indexes. Non-functional:

PropertyTargetConsequence
Latencyinteractive-ish (seconds–minutes) on huge scansmassive read parallelism; minimize shuffle
Workloadread-mostly, scan-heavy, few rows out of manycolumnar, not row store (P8)
Elasticityfan a query across 100s–1000s of workersstateless workers, separated storage/compute
Not requiredpoint lookups, row updates, transactionsuse OLTP (S4) for those

This is OLAP, the mirror image of S4 (OLTP). OLTP touches a few rows by key and must be transactional; OLAP touches billions of rows but a few columns, and reads dominate. That single difference — many rows, few columns, no updates — justifies columnar storage and an MPP (massively parallel processing) engine.

2 · Capacity estimation

1 PB table, 200 columns, query selects 3 of them row store: must read ~all 1 PB -> hopeless columnar: read 3/200 of the bytes = ~15 TB scanned + partition pruning to 7 of 1095 daily partitions = ~96 GB actually read + row-group min/max skipping -> often far less distribute the 15 TB (pre-prune) scan over 1,000 workers -> ~15 GB/worker -> seconds, IF the shuffle between stages is small shuffle-hash join: repartition 15 TB by join key across 1,000 workers -> ~15 GB/worker crosses the network <- this is the cost that matters

The numbers say it all: the engine's leverage is reading less (columnar + pruning takes 1 PB down to ~96 GB) and moving less (the shuffle volume, not the scan, is what makes a query slow). Everything below is in service of those two.

3 · API & query lifecycle

submit(sql) -> query_id status(query_id) -> {state, stages[], rows_processed, bytes_scanned} results(query_id) -> stream of rows lifecycle: SQL -> parse -> logical plan -> cost-based optimize -> physical plan -> DAG of STAGES -> schedule TASKS on workers -> exchange between stages -> result

4 · Storage & the read path — read less (ladder)

The first lever is reading fewer bytes off storage. Derive it.

Rung 0 — row store, full scan

Rows stored together; a 3-column query still drags all 200 columns off disk. 1 PB read. Breaks on: I/O — you read 197 columns you don't need.

Rung 1 — columnar (Parquet/ORC) + projection pushdown

Store each column contiguously (P8); the scan reads only the 3 projected columns → ~15 TB instead of 1 PB. Same-type values compress hard (RLE, dictionary, delta). Breaks on: you still scan every row of those 3 columns even when a filter rejects most.

Rung 2 — predicate pushdown + row-group min/max skipping

Columnar files store per-row-group min/max stats. Push the WHERE filter into the scan: if a row group's max < the predicate's lower bound, skip the whole group without decoding it. A WHERE amount > 1000 over a file where a group's max is 500 reads zero bytes of that group. Breaks on: data not laid out to make stats selective (random order → every group's min/max spans the predicate).

Rung 3 — partition pruning + clustering/sort

Physically partition by a common filter column (e.g. date) so WHERE date BETWEEN touches only matching directories — 7 of 1095 partitions → ~96 GB. Cluster/sort within partitions on a second column so min/max skipping becomes highly selective. This is P2 applied to a warehouse.

5 · High-level architecture

graph TD
  SQL[query] --> CO[coordinator: parse + cost-based optimize + schedule]
  CO --> ST1[stage 1: scan + filter + partial agg]
  ST1 -->|exchange / shuffle by key| ST2[stage 2: join / final agg]
  ST2 -->|exchange| ST3[stage 3: sort / limit]
  ST3 --> CO
  CO --> R[result]

A coordinator compiles SQL into a DAG of stages; each stage runs as many parallel tasks (one per worker per data split). Stages are separated by exchange operators that move (shuffle) data between them. Workers are stateless and read columnar files from shared storage (S3/HDFS/GCS) — the modern "separation of storage and compute," which lets compute scale elastically and independently.

6 · The execution model: stages, tasks, exchanges

A query becomes a tree of stages. A leaf stage scans a table; its tasks each handle a split (a chunk of files/row groups). An intermediate stage consumes the output of its children through an exchange. The exchange is the only place data crosses the network, and it comes in three flavors:

  • Repartition (hash) exchange — route each row to a worker by hash(key) % N so all rows with the same key co-locate (for joins/group-by). This is the shuffle.
  • Broadcast exchange — send the entire (small) input to every worker.
  • Gather exchange — collect all partial results to one node for the final result/limit (P7).

Partial aggregation is the cheapest shuffle-reducer: for GROUP BY country, COUNT(*), each scan task first aggregates locally (one partial count per country it saw), then shuffles only the small partial-aggregate rows, and a final stage merges them. A billion input rows shuffle as a few hundred partials (P7 partial aggregation). This is why COUNT/SUM/AVG are nearly free while COUNT(DISTINCT) (which needs the raw values) is expensive — the latter often uses HyperLogLog sketches to stay cheap.

7 · Join strategies — minimizing shuffle (ladder)

The bottleneck is shuffle. Joining two big tables the naive way repartitions both by the join key across the network — tens of TB in flight. The optimizer's real job isn't computing faster; it's moving less data. The choice of join algorithm is the single biggest lever.

Rung 1 — broadcast join (one side is small)

If one table is small (a dimension table, say 50 MB), broadcast it to all N workers and stream the big table past it locally — the huge table is never shuffled.

broadcast 50 MB dim table to 1,000 workers = 50 GB total network vs shuffling a 15 TB fact table = 15,000 GB -> broadcast wins by ~300x WHEN the small side fits in worker memory

Breaks on: the "small" side isn't small — broadcasting a 100 GB table to 1,000 workers is 100 TB of network and blows worker memory.

Rung 2 — shuffle (partitioned) hash join (both sides large)

Repartition both tables by hash(join_key) so matching keys land on the same worker; each worker builds a hash table on its slice of the smaller side and probes with the larger. Cost = shuffling both tables once (~15 GB/worker in our example). Breaks on: data skew — one hot key (e.g. user_id = guest) sends a huge fraction of rows to one worker, which becomes a straggler. Fix with skew/salt handling: split the hot key across several partitions and replicate its match rows.

Rung 3 — sort-merge join (inputs already sorted / too big to hash)

If both sides are sorted on the key (or too large for a hash table to fit in memory), sort each side (a distributed external sort, P18) and merge in a single linear pass. Robust to inputs larger than RAM; the cost is the sort. Used when hash tables would spill heavily.

JoinUse whenNetwork costWeakness
Broadcastone side fits in worker RAMsmall side × N workersfails if "small" side grows
Shuffle hashboth sides largeboth sides shuffled onceskewed keys → straggler
Sort-mergesorted/huge inputsshuffle + sortsort cost; needs spill-to-disk

The optimizer picks among these using cost-based planning: it estimates each table's row count and size from statistics (and runtime sampling), then chooses the plan with the least estimated data movement. A wrong estimate (stale stats) is the usual cause of a query that "should be fast" running for an hour — it broadcast a table that turned out to be huge.

8 · Vectorized execution & join order

Inside a worker, processing is vectorized: operators work on batches of ~1024 column values at a time rather than row-at-a-time, so the CPU runs tight loops over contiguous arrays that the compiler auto-SIMDs and that stay in cache — often 10×+ over the classic row-at-a-time "Volcano" model. Some engines (e.g. Spark) go further and JIT-compile a whole stage into one fused loop (whole-stage codegen). Separately, for multi-join queries the optimizer chooses join order to keep intermediate results small — join the most selective pair first so later joins see fewer rows; a bad order can produce an intermediate result orders of magnitude larger than any input.

9 · Scaling, stragglers & failure

New bottleneck — the straggler. A query's latency is the slowest worker in each stage (P12). With 1,000 tasks each 99% likely to be fast, the chance all finish promptly is 0.991000 ≈ 0.004% — a slow task is virtually guaranteed. Fixes: speculative execution (launch a duplicate of a lagging task on another worker, take whichever finishes first), even split sizing so no task is much bigger, and skew handling on the shuffle (above). Adding workers helps a scan-bound query but not a shuffle-bound or skewed one — know which you have.

Failure model: classic MPP (Presto/Trino) keeps intermediate data in memory and streams between stages — fast, but a worker failure fails the whole query (re-run it). Batch engines (Spark, MapReduce, P18) materialize shuffle output to disk between stages, so a failed task re-runs from the last checkpoint, not from scratch — slower but survivable for hours-long jobs. The tradeoff is interactive-fragile vs batch-resilient; some engines (Trino fault-tolerant execution) now offer both modes.

10 · Tradeoffs & alternatives

MPP engines are built for big scans, not point lookups or updates — use OLTP (S4) for those, or an HTAP system if you truly need both. Pre-aggregation / materialized views / cubes trade storage and freshness for query speed (compute the rollup once, read it many times) — the same lever as S6's metric rollups. Separation of storage and compute (BigQuery, Snowflake, Trino-on-S3) lets you scale compute elastically and pay per query, at the cost of network latency to object storage (mitigated by caching hot files on worker SSD). More workers help scans, never a shuffle-bound query.

11 · Real-world example

Presto / Trino & Google BigQuery

Presto (built at Facebook, forked as Trino) is an MPP SQL engine that queries data in place (S3, HDFS, Hive, many connectors) by compiling SQL into stages of stateless workers and shuffling between them in memory — Meta runs it for interactive analytics over exabyte-scale warehouses. Google BigQuery descends from Dremel (P8): it separates storage (Colossus, columnar Capacitor format) from compute (a shared pool of "slots"), fans a query across thousands of slots, and shuffles through a distributed in-memory shuffle tier; its planner aggressively prunes and pushes down so a multi-TB scan returns in seconds. Both live or die by minimizing shuffle — BigQuery's disaggregated shuffle was a landmark precisely because the shuffle is the constraint.

12 · Interview follow-ups (Q&A)

Q: Why is shuffle the bottleneck and not the scan? Columnar + pruning already cut the scan to a small fraction of the table and it parallelizes linearly with workers. Shuffle moves data across the network between stages and doesn't shrink by adding workers — a 15 TB repartition is 15 TB no matter how many nodes. So the optimizer optimizes data movement, not CPU.

Q: How do you join a huge fact table to a small dimension without shuffling the fact? Broadcast the small side to every worker and probe locally — the fact table never moves. Only valid while the "small" side fits in worker memory; the optimizer falls back to a shuffle-hash join when it doesn't.

Q: One join key is wildly hot (skew) — what happens and how do you fix it? All rows for that key hash to one worker, which becomes a straggler that sets the whole stage's latency. Fix by salting: split the hot key into key#0..key#k across workers and replicate its matching rows on the other side, then union. Some engines detect skew at runtime and do this automatically.

Q: Why does COUNT(*) scale but COUNT(DISTINCT x) hurt? COUNT(*) partial-aggregates locally and shuffles tiny partials; COUNT(DISTINCT) needs the actual distinct values, which can't be merged from counts — so use an approximate sketch (HyperLogLog) that is mergeable when exactness isn't required (same trick as S22/S6).

Q: A worker dies 9 minutes into a 10-minute query — what happens? In a streaming MPP engine, the query fails and re-runs (intermediate state was in memory). In a batch engine that materializes shuffle to disk, only the lost task re-runs from its checkpoint. Choosing between them is the interactive-vs-resilient tradeoff.

Q: Why is this read-only / no point lookups? Columnar layout makes reading one full row expensive (gather 200 separate column files) and makes updates a rewrite; there are no secondary indexes for key lookups. OLAP optimizes scans of many rows / few columns; OLTP (S4) optimizes few rows / all columns with transactions. Different layouts, different systems.

13 · Key talking points

  • OLAP is the mirror of OLTP: many rows, few columns, read-mostly → columnar (P8).
  • Two levers: read less (columnar + projection + partition pruning + min/max skipping: 1 PB → <100 GB) and move less (minimize shuffle).
  • Coordinator compiles SQL into a DAG of stages; exchanges (shuffle/broadcast/gather) are the only network hops.
  • Join ladder: broadcast (small side) → shuffle-hash (both large) → sort-merge (sorted/huge), chosen by cost-based planning.
  • Partial aggregation shrinks group-by shuffles; skew creates stragglers (salt the hot key).
  • Latency = slowest worker (P12) → speculative execution; vectorized/SIMD execution inside workers.
