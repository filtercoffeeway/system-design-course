12 · Distributed Task Scheduler (Cadence / SQS+Lambda)

Hard constraint: you can't have exactly-once execution and fault-tolerance without a too-expensive distributed transaction — accept at-least-once + idempotent workers.
Primitives exercised: P9, P10 (cron firer), P16 (leasing).

Idempotency as first-class; visibility timeout; dead-letter queue; the "who fires the cron?" problem (single scheduler + distributed lock); task leasing with renewal.

1 · Requirements

Functional: run a unit of work reliably — now, later (at a future time), or on a schedule (cron) — with retries and no duplicate effect. Non-functional:

PropertyTargetConsequence
Deliveryat-least-onceworkers must be idempotent (P9)
Durabilityan accepted task is never lostpersist before ack; queue is durable
Fault tolerancea dead worker’s task re-runsleases + visibility timeout
Scaleworkers scale horizontallypull-based dispatch (§8)
Visibilityfailures are inspectableretries + dead-letter queue

2 · Capacity estimation

10,000,000 tasks/day / 86,400 s ~ 116/s average, bursty -> peak ~1,000/s delayed tasks pending at any time ~ 100,000,000 (scheduled minutes–months out) -> "find due tasks" cannot be a full scan of 100M rows every second (§7) task record ~ a few hundred bytes -> 100M x ~300 B ~ 30 GB pending state

3 · API & data model

enqueue(task, run_at?, idempotency_key, context) -> task_id poll(worker_id, capacity) -> leased task(s) (pull model, §8) ack(task_id, lease_token) / nack(task_id)

Two tables, because two access patterns pull in opposite directions:

tasks (partition key = task_id) -- point access: ack, status, idempotency task_id PK payload state ENUM(pending, leased, done, dead) lease_until TIMESTAMP NULL -- visibility-timeout expiry lease_token UUID NULL -- fences a stale worker's ack (P10) attempts INT context JSON -- W3C trace context: traceparent/tracestate (§ tracing) concurrency_policy ENUM(allow, forbid, replace) -- for recurring jobs schedule (partition key = time_bucket, e.g. "2026-06-20T14:32") -- the OUTBOX time_bucket PK part -- minute (or 5-min) the task is due in run_at TIMESTAMP task_id ref -- row is DELETED once the task is moved to the worker queue
Why two tables (the partitioning paradox): if the single tasks table is partitioned by task_id, the due-time query WHERE run_at ≤ now() has no partition key — the engine must scatter-gather the run_at index across every partition (P7), which at 100M pending tasks cripples the DB. The fix is a separate time-partitioned schedule/outbox table: the scheduler reads only the partition for the current minute bucket, and deletes the row once the task is enqueued for a worker. Point operations stay on tasks (by task_id); time scans stay on schedule (by time_bucket). Same data, two layouts for two access patterns.

The lease_token is a fencing token (P10): an ack is only honoured if its token matches the current lease, so a slow worker whose lease already expired and was reassigned can’t ack someone else’s run.

5 · Delivery semantics — the ladder

Rung 0 — at-most-once (fire and forget)

Hand the task to a worker, delete it immediately. If the worker crashes, the task is lost. Fine for a best-effort metric, unacceptable for real work.

Bottleneck: a crash silently drops the task.

Rung 1 — at-least-once (delete only after ack)

Keep the task; delete it only when the worker acks success. A crash before ack means the task re-runs — never lost, but possibly run twice.

Bottleneck: duplicate execution (worker finished, then crashed before acking).

Rung 2 — at-least-once + idempotent workers ← the answer

Accept that re-runs happen and make running twice harmless: workers carry an idempotency key (P9) so a duplicate effect is deduped. True exactly-once execution would need a distributed transaction across the queue and the worker’s side effects — too expensive — so you buy exactly-once effect cheaply with idempotency instead. (See the hard constraint, §9.)

6 · The task lifecycle & lease (worked)

When a worker leases a task it becomes invisible to others for T seconds (the visibility timeout). Ack deletes it; a crash lets the lease expire and another worker retakes it. Long tasks renew the lease before it lapses:

visibility timeout T = 30s t=0 worker A leases task (lease_until=30, token=k1) -> invisible to others t=20 A still running -> renews lease (lease_until=50) [heartbeat] t=35 A finishes -> ack(token=k1) -> task deleted [happy path] crash case: t=0 A leases (lease_until=30, token=k1) t=12 A crashes (no renew) t=30 lease expires -> task visible again t=31 worker B leases (token=k2) t=40 A's ghost returns, ack(token=k1) -> REJECTED (current token is k2) <- fencing

Tuning T is a tradeoff: short T recovers from dead workers fast but risks double-running a slow-but-alive one; long T is safe against false death but slow to recover. Retries use exponential backoff (1s, 2s, 4s, …); after N attempts the task moves to a dead-letter queue so a poison task can’t loop forever. Always have a DLQ.

7 · Delayed & scheduled tasks — the ladder

Rung 0 — poll a due-time query

SELECT * FROM tasks WHERE run_at <= now() AND state='pending' every second. Simple; fine at thousands of pending tasks. With 100M pending it re-scans a huge index every second — and (see §3) if tasks is partitioned by task_id this scan scatter-gathers across every partition. The scan dominates.

Bottleneck: repeated (and cross-partition) scan over a large future-task set.

Rung 1 — time-partitioned schedule table

Put due tasks in the separate schedule table partitioned by time_bucket (the minute they’re due). Each tick reads only the current bucket’s partition — a single-partition range read, not a cluster-wide scan — and deletes rows as it drains them. Reads now scale with due tasks, not total pending.

Rung 2 — in-memory min-heap (priority queue by run_at)

Hold near-term tasks in a binary min-heap ordered by due time. The scheduler peeks the top (earliest task) in O(1), sleeps until exactly that time — or wakes early if a sooner task is inserted — then pops and enqueues it. Event-driven: no polling, no scanning, and fires at the right instant.

heap ops: insert O(log n), peek-min O(1), pop O(log n) loop: t = heap.peek().run_at; sleep(t - now) (or until a sooner insert); dispatch heap.pop()

It is a cache, not the source of truth: the heap lives on the elected scheduler (P10) and holds only a near-term horizon (e.g. tasks due in the next few minutes), refilled from the schedule table as time advances — you can’t fit 100M far-future tasks in RAM. Standard in-process timer structure (Java DelayQueue, Go timers).

Failover storm — and the fix: when the leader dies, the new one must rebuild this heap, but it must not scan 100M rows (a 30 s rebuild = a 30 s spike in delivery latency). Because Rung 1 already bucketed by time, the new leader loads only the current and next bucket(s) — the next few minutes of work — so recovery is a single-partition read in milliseconds, not a table scan. Coupling the time-bucket index to the heap is what makes failover cheap.

Rung 3 — timing wheel (at extreme churn)

The heap’s O(log n) per op only bites with millions of timers and very high insert/cancel rates (think per-request timeouts). A timing wheel gets O(1) insert and expiry. Picture a clock face: an array of N slots used as a ring, each slot one tick of time, and a hand that advances one slot per tick and fires whatever tasks sit there.

8-slot wheel, tick=1s, hand at slot 0: schedule A due in 3s -> slot (0+3)=3 insert = O(1) schedule B due in 5s -> slot 5 hand ticks 1,2,3... -> at slot 3 fire A, at slot 5 fire B expiry = O(1)/tick due > N slots? store rounds = d / N and fire when the counter hits 0

A single wheel only reaches N ticks ahead, so hierarchical wheels stack resolutions like clock hands — second, minute, hour. A far-future task lives in a coarse wheel and cascades down to a finer one as its time nears (a 90 s task waits in the minute wheel, then cascades into the second wheel at slot 30). This gives O(1) at unbounded range; the cost is resolution — a task fires at the next tick boundary, so you accept bounded jitter. Same durable-backed, near-term-horizon idea as the heap; it’s how Linux kernel timers, Kafka’s SystemTimer, and Netty’s HashedWheelTimer work. Reach for it when the heap is the bottleneck, not before.

  HW["hour wheel (coarse)"] -->|hand advances → cascade| MW["minute wheel"]

Cron: who fires it? (P10)

If every scheduler node fires a cron, it runs N times. Elect a single scheduler via a distributed lock / leader election (P10, S13) that enqueues due tasks for the worker pool; a fencing token stops a stale ex-leader from also firing.

Cron overrun: what if a run overlaps the next tick?

A job scheduled every 1 min but taking 5 min is still running when its next tick arrives. Don’t leave this implicit — make it a concurrency_policy on the job (this is exactly Kubernetes CronJob’s concurrencyPolicy):

Allow -> start the new run anyway (concurrent runs; default for independent jobs) Forbid -> skip this tick if the previous run is still in flight Replace -> cancel/kill the in-flight run and start the new one

The firer checks the previous run’s state (via its task_id/lease) before enqueuing the next, and applies the policy.

8 · Picking the right worker by capacity

A classic follow-up: with a pool of workers of differing load, how does a task reach one that can actually handle it? Derive it.

Rung 0 — random / round-robin push

Scheduler assigns each task to a worker by round-robin. Ignores load, so a slow or already-busy worker keeps getting piled on while others idle — tail latency explodes.

Bottleneck: dispatch is blind to capacity.

Rung 1 — pull-based (workers pull when free) ← best default

Flip the direction: workers long-poll the queue and take the next task only when they have a free slot. Capacity tracking becomes implicit — a busy worker simply doesn’t pull — so load self-balances with zero global state. This is the SQS+Lambda model, and the right default for homogeneous tasks.

Prefetch trap (buffer bloat): for throughput, workers often prefetch a batch (say 10) rather than one at a time. But the lease/visibility timeout starts when a task is leased, not when it starts running. If task #1 runs long, #9 and #10 sit idle in the worker’s local buffer until their lease expires — and another worker re-leases them, causing accidental concurrent execution while the first worker is perfectly healthy. Fixes: cap prefetch to what a worker can start ~immediately, and heartbeat/renew the lease for every task in the buffer, not just the one on a thread. (Kafka’s max.poll.records + max.poll.interval is this exact knob.)
Bottleneck: you can’t target a worker (no affinity/locality), and a single shared queue can be a contention point at very high rates.

Rung 2 — Join-Idle-Queue (JIQ)

A scalable middle ground between pull and push. Each dispatcher keeps a small list of idle workers; when a worker finishes and has a free slot it advertises itself to a (random) dispatcher’s idle-list. The dispatcher routes each task to a worker from that list and removes it; if the list is empty it falls back (random worker / another dispatcher). The load-balancing work moves off the request path — no per-task polling of worker load — so it scales to huge fleets, and because a worker only advertises when it actually has spare capacity, JIQ is capacity-aware by construction (an underpowered or busy worker simply advertises less often). It keeps the self-balancing property of pull while letting the dispatcher push to a known-free worker. (Lu et al., Microsoft, 2011.)

Bottleneck: still doesn’t honour task-specific affinity; under sustained overload the idle-lists are empty and it degrades to the fallback.

Rung 3 — least-loaded push (when you must target)

If tasks need a specific worker (data locality, a warm cache, a GPU), push to the worker reporting the lowest load via heartbeats. But the global load view is stale, so every dispatcher piles onto whoever currently looks least-loaded — a herd that overshoots.

Rung 4 — power-of-two-choices ← robust targeted push

Pick two workers at random and send to the less loaded of the two. With stale information this gives near-optimal balance and avoids the herd — a tiny change with an outsized effect (max load ~log n → ~log log n).

But doesn’t random P2C pick an underpowered worker? Yes — naive P2C compares raw queue length, so it can sample two weak/busy workers and route to the “less bad” one, ignoring real capacity. Three fixes: (1) compare utilization (in-flight ÷ capacity), not raw count — a small worker at 2/2 correctly loses to a big one at 4/10; (2) filter feasible workers first, then P2C among survivors (the K8s filter-then-score shape), so an underpowered worker is never a candidate; (3) P2C + peak-EWMA (Finagle’s default) — latency-based load implicitly reflects capacity, since a weak worker runs slower and looks more loaded.

On top of any targeted rung, layer capability routing (GPU tasks → GPU workers) and affinity (consistent-hash a task’s entity to the same worker for cache locality, P2).

9 · The hard constraint, in depth

You cannot get both exactly-once execution and fault tolerance without an expensive distributed transaction spanning the queue and the worker’s side effects. A worker can finish the work and crash in the instant before acking — so the task will run again. The accepted answer is at-least-once + idempotent workers (P9): make a second run produce the same result, and for any external side effect carry an idempotency key so the effect happens once even when the execution happens twice. Stop chasing true exactly-once.

10 · Scaling, observability & bottlenecks

Workers scale out trivially under the pull model. The queue is the thing that bottlenecks: partition it (P2) for throughput, accepting that global ordering is lost (usually fine for independent tasks). The scheduler/cron firer is a single elected node — cheap, but a failover must be clean (fencing) and cheap to recover (the bucketed-rebuild fix, §7). A poison task that always fails would loop forever without the DLQ + max-attempts cap. A hot tenant can starve others — add per-tenant queues or weighted fair scheduling.

Observability: don’t lose the trace

An async task scheduled for 3 days out is a debugging black hole: when it fails, how do you tie it back to the HTTP request that created it? Capture the W3C Trace Context (traceparent/tracestate) into the task’s context field at enqueue time, and have the worker re-establish that span when it runs days later. Now the worker’s logs stitch to the original request in your APM (Datadog, Honeycomb), so a 3-day-delayed failure is traceable to its origin. Carry the idempotency key in the same field.

11 · Real-world example

AWS SQS + Lambda & Uber Cadence

The simple case is SQS + Lambda: SQS provides at-least-once delivery, a visibility timeout so two consumers don’t process the same message at once, and a built-in dead-letter queue after a max receive count; Lambda workers must be idempotent because redelivery happens. For complex multi-step jobs, Uber built Cadence (now Temporal), which persists workflow state so a long-running task resumes exactly where it left off after a crash and handles retries/timeouts as first-class concepts — the same at-least-once + idempotency philosophy, scaled to durable workflows (and the engine behind sagas, P14).

12 · Tradeoffs & alternatives

A plain queue (SQS) is simple but you build orchestration yourself; a workflow engine (Temporal/Cadence) handles state and retries but is heavier. Shorter visibility timeouts recover from dead workers faster but risk double-processing live-but-slow ones.

13 · Interview follow-ups (Q&A)

How do you pick a worker by capacity? Prefer pull-based dispatch (or JIQ — idle workers advertise to dispatchers) so capacity is signalled by having a free slot, with no per-task load polling. When you must push, use power-of-two-choices — but compare utilization (load ÷ capacity), not raw queue length, or it routes to an underpowered worker; equivalently, filter feasible workers, then P2C. Layer capability/affinity routing on top (§8).

How do you guarantee a task isn’t run twice for a money-side effect? You can’t prevent double execution; you make the effect idempotent — the worker carries an idempotency key and the side effect is deduped on it (P9).

A slow worker’s lease expired and the task was reassigned — how do you stop its late ack from corrupting state? The lease carries a fencing token; the queue rejects an ack whose token isn’t the current one (P10).

How do you schedule 100M future tasks without scanning constantly? Don’t poll a big WHERE run_at≤now; bucket by due-time and read only the current bucket, with a timing wheel for near-term tasks (§7).

How do you stop one tenant from starving others? Per-tenant queues or weighted fair queueing, so a flood from one tenant can’t monopolise the worker pool.

What if a task must run exactly at its scheduled second? Be honest: a distributed scheduler gives at-least the requested time with small jitter, not hard real-time; tighten the wheel resolution if needed, but don’t promise wall-clock precision.

Why two tables instead of one? Point ops (ack/status) want partitioning by task_id; the due-scan wants partitioning by time. One table can’t serve both — a task_id-partitioned table forces the due query to scatter-gather every partition. So tasks (by id) + a schedule outbox (by time_bucket), drained and deleted as tasks fire (§3).

A cron run overruns its next tick — concurrent, skip, or replace? Make it explicit with a concurrency_policy: Allow (run concurrently), Forbid (skip while the prior run is in flight), or Replace (kill the old, start the new) — the firer checks the previous run’s state before enqueuing (§7).

How do you debug a task that fails 3 days after it was scheduled? Propagate W3C trace context (traceparent) in the task’s context field and re-open the span in the worker, so its logs stitch back to the original request in your APM (§10).

How does prefetching cause double execution even with healthy workers? Buffered tasks keep aging against their lease; if the buffer drains slowly they expire and get re-leased elsewhere. Cap prefetch and heartbeat every buffered task, not just the running one (§8).

14 · Key talking points

  • At-least-once + idempotent workers; drop the exactly-once dream (P9).
  • Visibility timeout / lease prevents double-running.
  • Always a dead-letter queue for poison tasks.
  • Single elected scheduler for cron (P10).
  • Pull-based dispatch self-balances load; power-of-two-choices when you must push.
  • Exponential backoff on retries.
