3 · Key-Value Store (Redis / DynamoDB-style)

Hard constraint: CAP felt viscerally — under partition, return stale (AP) or error (CP). The choice is the application's, and you must reason about it explicitly.
Primitives exercised: P17 (LSM/WAL/Bloom), P2 (consistent hashing), P3, P4.

LSM vs B-tree; WAL for crash recovery; consistent hashing; single-/multi-leader/leaderless quorum; eventual consistency + vector clocks; Bloom filters to skip disk on misses.

1 · Requirements

Functional: get(key) and put(key, value) (optionally delete, TTL) at planet scale. No queries, no joins — pure keyed access.

2 · Capacity estimation

100,000,000,000 keys x 1 KB = 100 TB logical replication factor N=3 = 300 TB on disk 1,000,000 ops/s / ~50,000 per node ~ 20+ nodes for throughput alone -> 100 TB / node SSD budget also forces ~tens of nodes; partitioning is mandatory

3 · API & data model

get(key) -> value, version put(key, value, [version]) -> ok (version enables compare-and-set) delete(key) -> ok

The data model is the access pattern: an opaque key → value map, no secondary indexes, no scans. That’s exactly why a KV store — not a relational DB — fits (the store-selection ladder from S14 §4 lands here whenever access is pure keyed lookup). The key is the partition key.

4 · High-level architecture

graph LR
  C[client] --> CO[coordinator node]
  CO -->|consistent hash ring| N1[replica 1]
  CO --> N2[replica 2]
  CO --> N3[replica 3]

Any node can act as coordinator for a request — there is no leader — and writes/reads use quorums. Two sub-decisions deserve their own derivation: how keys map to nodes, and how each node stores bytes.

How keys map to nodes — the partitioning ladder

Naive node = hash(key) % N spreads evenly but reshuffles ~80% of keys when you add a node (measured in P2) — fatal for an elastic cluster. Consistent hashing places nodes and keys on a ring so adding a node moves only ~1/N of keys; virtual nodes (each physical node owns ~100–256 ring segments) even out the load and scatter a departing node’s range across many neighbours:

grow 4 -> 5 nodes, 10,000 keys: hash % N -> ~80% of keys move (cluster-wide stampede) consistent hash -> ~18% of keys move (only the new node's arc) + vnodes -> per-node load within ~10% of even

So the ring + vnodes is what makes the store incrementally scalable — the property the requirements demanded. Each key’s value is then replicated to the next N nodes clockwise on the ring.

5 · The write path & storage engine

Storage engine ladder — B-tree vs LSM

Each node needs an on-disk structure. A B-tree (the OLTP default) updates pages in place: great for reads, but a KV store is write-heavy and in-place updates are random I/O that caps write throughput. An LSM engine buffers writes in a memtable + WAL, flushes sorted SSTables, and compacts in the background — turning writes into sequential appends and trading in some read amplification (cut back by Bloom filters that skip SSTables for absent keys). For a write-heavy KV store the LSM wins; full mechanics and the amplification trade are in P17.

The write itself

The coordinator routes a put to the N replicas and waits for W acks:

put(k,v), N=3 W=2: coordinator hashes k -> replicas {A,B,C} sends v to A,B,C; waits for 2 acks (W=2) A ok, B ok -> ACK client (C may still be catching up) if C is DOWN -> hinted handoff: park C's copy on a neighbor D, tagged "for C", replay when C returns

6 · The read path

get(k), N=3 R=2: coordinator reads from 2 replicas (R=2) A returns v2 (version 7), C returns v1 (version 5) newest wins -> return v2 read repair: asynchronously push v2 to C so it converges

Because R + W > N (2 + 2 > 3) the read set and write set always overlap, so a quorum read is guaranteed to see the latest quorum write — the mechanism worked in P3. Read repair piggybacks on reads to heal stale replicas; the background Merkle-tree anti-entropy repairs replicas that reads never touch (each replica keeps a hash tree of its key ranges; two replicas compare roots, then descend only into subtrees whose hashes differ, so they exchange just the diverged keys).

7 · Conflicts & the hard constraint

Leaderless writes mean two clients can update k concurrently on different replicas. A version vector (a per-replica counter) detects whether two versions are causally ordered or genuinely concurrent:

cart, vv = [A:0, B:0] client adds milk via replica A -> {milk}, vv=[A:1, B:0] client adds eggs via replica B -> {eggs}, vv=[A:0, B:1] (B hadn't seen A's write) compare [A:1,B:0] vs [A:0,B:1]: neither dominates -> CONCURRENT -> conflict

How you resolve it is its own small ladder: last-writer-wins (pick by timestamp — simple, but silently drops the other update, so the cart loses an item) → version vectors + app merge (keep both, let the app reconcile — Dynamo returns both siblings) → CRDT (a type like a set-union cart that merges automatically and always converges). Pick by whether a dropped write is acceptable; for a cart it isn’t, so you merge. Full mechanics in P3.

CAP, felt viscerally. Under a network partition the quorum can’t be formed for some keys, and you must choose — another ladder:

ChoiceUnder partitionUse when
AP (Dynamo-style, tunable quorum)serve possibly-stale / accept writes via sloppy quorum + hinted handoffcart, sessions, likes — availability > freshness
CP (Raft-replicated KV, e.g. etcd)minority partition refuses writes/reads → errorsconfig, locks, anything that must be correct

Dynamo chose AP so “add to cart” is never rejected; etcd chose CP so cluster state is never wrong. Stating this choice and justifying it from the use case is the whole answer.

8 · Membership, failure detection & deletes

With no leader, nodes learn the ring and each other's liveness via gossip: each node periodically exchanges membership + heartbeat state with a few random peers, so knowledge of a join/leave/failure spreads epidemically in O(log n) rounds without a central registry. A node is marked down by a φ-accrual failure detector (a suspicion level that rises with missed heartbeats) rather than a hard timeout, which adapts to variable latency. Deletes are subtle in an LSM + replicated store: you can't just remove a key, because a replica that missed the delete would resurrect it on the next anti-entropy. So a delete writes a tombstone (a versioned "deleted" marker) that propagates like any write and is only physically purged after a grace period (> the longest possible replica downtime) — the same resurrection hazard as CRDTs.

9 · Scaling & bottlenecks

Adding a node steals a fair set of vnodes from existing nodes, moving only ~1/N of keys (P2). The classic failure is the hot key / wide partition — one key (or a low-cardinality partition key like country=US) lands all its load on one replica set, recreating the single-node ceiling; mitigate by splitting the key or fronting it with a cache (P4). Stronger consistency (higher R/W, or CP) costs latency and availability; that’s the dial you tune per workload. Sloppy quorum + hinted handoff keep an AP store writable even when N's natural replicas are partitioned away: the write goes to the next N reachable nodes on the ring, with hints to replay onto the rightful owners once they return — trading a wider staleness window for never rejecting a write.

9 · Real-world example

Amazon Dynamo / DynamoDB

Amazon’s 2007 Dynamo paper is the blueprint: consistent hashing + virtual nodes, quorum reads/writes (R + W > N), vector clocks for conflict detection, hinted handoff for transient failures, and Merkle trees for anti-entropy — all chosen to keep the shopping cart writable even during failures (an "add to cart" must never be rejected). The managed descendant, DynamoDB, keeps the partitioned + replicated core (three AZs) and exposes the consistency dial as eventually- vs strongly-consistent reads. Cassandra and Riak are open-source embodiments of the same paper.

10 · Tradeoffs & alternatives

Leaderless AP maximizes availability but pushes conflict handling onto you. A Raft/Paxos-replicated CP KV (etcd-like) gives linearizable reads but sacrifices availability in a partition and write throughput. A simpler single-leader KV (Redis + replicas) is great until the write rate or dataset exceeds one primary. Pick by the consistency and scale the workload actually needs.

11 · Interview follow-ups (Q&A)

Q: With R+W>N, do I always read the latest value? You're guaranteed the read set overlaps the last successful quorum write, so a quorum read sees it. But it isn't full linearizability: concurrent writes can produce siblings, in-flight writes may be partially applied, and sloppy quorums widen the window. R+W>N gives "read-your-quorum-writes," not a total order.

Q: Why not just use last-writer-wins for conflicts? LWW silently discards a concurrent update — fine for "user status," data-losing for a cart (an added item vanishes). Detect concurrency with version vectors and either return siblings for app merge or use a CRDT (P19) that merges deterministically. Choose by whether a dropped write is acceptable.

Q: How does a node recover after being down — full re-sync? No: hinted handoff replays the writes it missed during a short outage, and Merkle-tree anti-entropy repairs the rest by comparing range hashes and exchanging only the diverged keys — bounded by the diff, not a full scan of the dataset.

Q: AP vs CP — how do you decide in the interview? From the use case: if a rejected write is worse than a stale read (cart, sessions, likes) choose AP; if a wrong value is worse than an error (config, locks, balances) choose CP. State it explicitly and justify it — that is the answer.

Q: Why LSM and not a B-tree? A KV store is write-heavy, and B-tree in-place updates are random I/O that caps write throughput. LSM turns writes into sequential appends (memtable+WAL → SSTable → compaction), trading read amplification that Bloom filters largely claw back (P17).

Q: A single hot key saturates one replica set — fix? Split the key into sub-keys spread across the ring, or front it with a cache (P4); a low-cardinality partition key (e.g. country=US) is the wide-partition version of the same bug — choose a higher-cardinality key.

Q: How do deletes not come back from the dead? Tombstones: a delete is a versioned marker that propagates like a write and is purged only after a grace period longer than any replica's expected downtime, so anti-entropy can't resurrect the key.

12 · Key talking points

  • Open with the CAP choice and justify AP vs CP from the use case.
  • Consistent hashing + vnodes for elastic partitioning (P2).
  • Quorums (R + W > N) as the consistency dial (P3).
  • LSM + WAL + Bloom internals (P17).
  • Vector clocks, hinted handoff, Merkle anti-entropy, gossip + tombstones for AP correctness.
