10 · Chat System (WhatsApp / Slack)

Hard constraint: routing — when A messages B, A's gateway must find which of hundreds of stateful connection servers currently holds B's live socket, against millions of connect/disconnect churn events. The scaling problem is connection state, not the message table.
Primitives exercised: P13 (live connections), P4 (routing registry), P1 (offline store-and-forward), P9 (dedup/idempotency), P5 (group fan-out).

WebSocket gateways; routing registry (user→gateway); per-conversation sequence numbers (global ordering impossible and unnecessary); receipts as a separate event stream; presence as high-frequency aggregation; client-side dedup on retry.

1 · Requirements

Functional: 1:1 and group messaging; delivery and read receipts; presence (online/last-seen/typing); message history and multi-device sync; offline delivery. Non-functional:

The reframing that drives everything: in chat the scarce resource is open sockets and the routing between them, not message storage (messages are tiny and often deleted after delivery). So the design centers on the connection tier and the registry, not a clever database.

2 · Capacity estimation

2,000,000,000 users, ~400,000,000 concurrently online connections per gateway (efficient async stack) ~ 1,000,000+ -> ~400 gateway servers just to HOLD the sockets conn memory ~10 KB/socket -> 1M conns ~ 10 GB RAM/server (buffers + TLS state) messages ~ 100,000,000,000/day = ~1.2M/s average (bursty higher) message ~ 200 B -> if stored a day ~ 20 TB/day (often deleted post-delivery) presence: 400M online x heartbeat/30s = ~13,000,000 heartbeats/s <- aggregation problem group fan-out: one send to a 256-member group = up to 256 deliveries (P5)

3 · API & data model

// over the persistent socket send(conversation_id, client_msg_id, body) -> ack{server_seq} // client_msg_id = dedup key recv -> {conversation_id, server_seq, sender, body, ts} receipt(conversation_id, up_to_seq, state) // delivered | read presence(subscribe conversation_id|user_id) -> stream of {user, online, last_seen} messages: (conversation_id, seq) -> {sender, body, ts} // partition key = conversation_id inbox: user_id -> queue of undelivered {conversation_id, seq} // offline (§8) routing: user_id -> {gateway_id, session, last_seen} // the registry (§5)

Partitioning by conversation_id keeps a conversation's messages and its monotonic seq together — the natural unit for ordering and history reads.

4 · Transport (ladder)

Rung 1 — short polling

Client asks "anything new?" every few seconds. Breaks on: latency (up to the poll interval) and waste — 400M clients polling every 5 s is 80M req/s of mostly-empty responses.

Rung 2 — long polling / SSE

Client holds a request open until the server has something (long poll), or uses Server-Sent Events for a one-way push stream. Better latency, but still HTTP overhead per message and awkward for the client→server direction. Breaks on: chat is bidirectional and chatty (typing, receipts) — one-way or request-scoped transports fight the workload.

Rung 3 — WebSocket (the answer)

One persistent, full-duplex TCP connection per client to a gateway (P13). Messages flow both ways with minimal framing overhead; the server can push instantly. The cost is statefulness: the gateway now holds live connection state, which is the entire scaling challenge below.

5 · The hard constraint: routing (in depth)

A connects to gateway G1; B connects to G2. When A sends to B, A's gateway must discover that B's socket lives on G2, among hundreds of gateways, and forward the message there — while connect/disconnect churn changes the answer millions of times a minute.

The mechanism is a routing registry: user_id → {gateway_id, session}, written on every connect and deleted on disconnect, kept in a fast shared store (Redis / a gossip-backed table). The send path:

A -> G1: send(conv, msg) G1: persist message (durable, assign seq) // never ack before persist G1: registry.lookup(B) -> G2 G1 -> G2: forward(msg) ; G2 -> B over B's socket B online? push now. B offline? enqueue to B's inbox (store-and-forward, §8)
New bottleneck — registry churn & failover. The registry takes millions of writes/min from connect/disconnect, and a gateway crash orphans ~1M entries at once. Fixes: a heartbeat/TTL on each registry entry so stale routes self-expire (don't rely on a clean disconnect); on gateway failure, its clients reconnect (to a new gateway) and re-register — the registry rebuilds from the clients, not from a scan. Bound the reconnect storm with jittered backoff. This is the "bound the failover cost of in-memory state" check: recovery time is one reconnect interval, and undelivered messages are safe because they were persisted before ack.

An alternative to a central registry is consistent-hash routing: gateway = hash(user_id) so anyone can compute where a user should be without a lookup — but real deployments still need a registry because a user's actual gateway is wherever they happened to connect (load balancers, reconnects), so the registry remains the source of truth, optionally with hashing as a hint.

7 · Ordering & delivery semantics

Per-conversation ordering, not global. A single global order across all messages would require a global sequencer — a throughput bottleneck and totally unnecessary (no one perceives the order of two unrelated chats). Instead each conversation has a monotonic seq assigned by the message service for that conversation; clients order by it and detect gaps (and request a resync if a seq is missing).

Delivery state is a separate event stream, not a field mutated on the message: "sent → delivered → read" are receipts that flow as their own small messages (often coalesced: "read up to seq N"). Embedding state in the message would mean rewriting it on every status change and fighting the immutable, ordered log; a separate receipt stream keeps the message log append-only.

Exactly-once is unaffordable; use at-least-once + dedup (P9): the client stamps each send with a client_msg_id; a retried send (after a lost ack) carries the same id, and the server (and receiving client) dedup on it. So a flaky network causes at-most-once effect despite at-least-once attempts.

8 · Offline delivery, groups & history

Store-and-forward: if B is offline, the message sits in B's durable inbox queue and is pushed when B reconnects (the client sends its last-seen seq; the server streams everything after it). Some systems (WhatsApp historically) delete the server copy once delivered to all recipients, minimizing storage — the message store is a transient buffer, not an archive. Others (Slack/Discord) keep full history for search and multi-device sync, trading storage for features.

Groups are a P5 fan-out: one send expands to up to N member deliveries, each routed individually (and inboxed if offline). For small groups (chat caps are typically hundreds–~1024) fan-out-on-write is fine; the celebrity-style explosion of S9 doesn't arise because groups are bounded. Each member still gets the same per-conversation seq, so everyone sees one consistent order.

Multi-device turns one user into several sockets/inboxes that must converge: deliver to all of a user's devices and sync read state across them (a per-(user,device) cursor over the conversation seq).

9 · Presence (the hidden scaling trap)

New bottleneck — presence is a firehose. 400M online users heartbeating every 30 s is ~13M updates/s, and naive presence means every status change fans out to everyone who has that user's chat open — an O(users × contacts) broadcast that can dwarf actual messaging. Relief: make presence approximate and pull-on-demand (compute "online" lazily from last-heartbeat when a contact opens the chat, rather than pushing every change), debounce/batch updates, and only publish presence to currently-interested subscribers (people viewing that conversation), not the whole contact graph. "Last seen" is just now − last_heartbeat < threshold.

10 · Scaling, tradeoffs & alternatives

Gateways scale horizontally (sockets are the unit); the registry shards by user. WebSocket (bidirectional, low overhead) vs SSE/long-poll (simpler, one-way, better through old proxies). Keep vs delete server history trades storage/cost for multi-device sync and search. Erlang/Go-style lightweight concurrency (one cheap process/goroutine per connection) vs thread-per-connection (doesn't reach 1M/box). Strong per-conversation ordering is cheap; global ordering is expensive and pointless. End-to-end encryption (Signal protocol) is orthogonal to routing — the server routes ciphertext it can't read, which also forces "delete after delivery" since the server can't index content anyway.

11 · Real-world example

WhatsApp

WhatsApp famously served tens of millions of users with a tiny engineering team by building on Erlang/OTP on FreeBSD — a runtime designed for millions of lightweight concurrent processes, one per connection — and reported driving a single server to very high simultaneous connection counts (on the order of a million-plus). That is the whole game for chat: the scarce resource is open sockets and the routing between them, not message storage. Messages were store-and-forward and deleted from servers after delivery. Slack and Discord solve the same routing/presence problem with a stateful gateway tier and a registry (P13), keeping history for search and multi-device sync.

12 · Interview follow-ups (Q&A)

Q: How does A's server find B among hundreds of gateways? A routing registry (user_id → gateway) written on connect, TTL'd by heartbeat, deleted on disconnect. A's gateway looks up B's gateway and forwards. Consistent hashing can hint where B should be, but the registry is the source of truth because users connect wherever the LB places them.

Q: A gateway holding 1M sockets crashes — what happens? Its registry entries TTL out (don't rely on clean disconnects); its clients detect the drop and reconnect to other gateways, re-registering as they go — recovery is one reconnect interval, bounded by jittered backoff. No messages are lost because they were persisted before ack.

Q: Why per-conversation sequence numbers instead of a global order? Global ordering needs a single sequencer (a bottleneck) and is imperceptible/unnecessary across unrelated chats. A per-conversation monotonic seq gives the only ordering users actually observe, and lets clients detect gaps and resync.

Q: How do read receipts work without rewriting messages? Receipts are a separate event stream ("delivered/read up to seq N"), coalesced, so the message log stays append-only and immutable instead of being mutated on every status change.

Q: A message is sent twice due to a retry — how do you avoid a duplicate? The client stamps a client_msg_id; server and receiving client dedup on it (P9). At-least-once delivery with idempotent effect, since exactly-once over a flaky network is unaffordable.

Q: Why is presence so expensive and how do you tame it? 13M heartbeats/s plus O(users×contacts) change broadcasts can exceed messaging load. Make presence approximate and pull-on-demand (derive from last heartbeat when a chat opens), debounce, and publish only to active subscribers.

Q: How do groups differ from the social-feed celebrity problem? Group size is bounded (hundreds to ~1024), so fan-out-on-write per message is fine — there's no 100M-follower explosion (S9). Everyone shares the conversation's seq for one consistent order.

13 · Key talking points

  • The scarce resource is sockets, not storage; stateful WebSocket gateways hold ~1M conns each (P13).
  • The wall is routing: a registry (user→gateway), TTL'd by heartbeat, rebuilt from clients on failover.
  • Order per conversation with a monotonic seq, never globally; clients detect gaps and resync.
  • Delivery state (sent/delivered/read) is a separate receipt stream, keeping the message log append-only.
  • At-least-once + client_msg_id dedup (P9); store-and-forward for offline, delete-after-delivery to save storage.
  • Presence is a firehose — make it approximate, debounced, pull-on-demand to interested subscribers only.
