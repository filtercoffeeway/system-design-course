P20 · Distributed tracing (spans, propagation, sampling)

What it is: stitch one user request's path across many services into a single causal tree of timed spans, by propagating a shared trace id through every hop.
Reach for it when: debugging latency/errors in a microservice or async system — "which of the 30 services made this request slow?" — and any deferred work (queues, cron) you need to tie back to its origin.
The lever: a small, propagated trace context (trace id + parent span id) carried on every call, plus a sampling decision so you keep the useful traces without storing 100% of them.
New problem it hands you: context must cross process / thread-pool / queue boundaries or the trace breaks; high-cardinality span storage is expensive (P8); naive head sampling drops the rare error trace you actually wanted.
Exemplified by: Datadog APM / observability, and the async-context check in S12 / S11.

1 · The problem it solves

One click can fan through 30 services, a queue, and three databases. When it's slow or errors, a per-service log tells you that service was slow but not why the request was — you can't see the causal chain. Distributed tracing reconstructs the whole request as a tree of spans (one timed unit of work per service/operation), linked by a shared trace id, so you can see exactly where the time and the error went. It's the third pillar of observability alongside metrics and logs (S6) — promoted here to a standalone primitive because the propagation + sampling machinery is reused across many systems (S12, S11, P9).

2 · The data model: trace, span, context

A trace is a tree of spans. Each span records: trace_id (shared by the whole tree), its own span_id, its parent_span_id (who called it), an operation name, start/end timestamps, and tags (service, status, attributes). The parent links make the tree; the timestamps make the waterfall.

trace_id = T1 span A (gateway) parent=none 0ms-120ms |- span B (auth) parent=A 2ms-18ms |- span C (orders svc) parent=A 20ms-118ms |- span D (db query) parent=C 25ms-110ms <- 85ms here = the culprit

The waterfall instantly shows span D (the DB) dominates, something no single service's logs reveal.

3 · Context propagation (the core mechanism)

For child spans to link to their parent, each service must pass the trace context to the next. The W3C Trace Context standard carries it in a traceparent HTTP header:

traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01 (55 chars) ^version ^trace-id (128 bit / 32 hex) ^span-id (64b) ^flags flags bit 0 = "sampled" -> downstream services honor the same keep/drop decision (plus tracestate: vendor-specific key=value pairs, e.g. for multi-vendor trace info)

On each hop: read the incoming traceparent, start a new span whose parent_span_id = the incoming span id, and inject a fresh traceparent (same trace id, new span id) on outbound calls. The hard part — and the most common cause of broken traces — is propagation across non-HTTP boundaries:

  • Across a queue: stamp the trace context into the message headers so a consumer minutes later continues the trace (the S12/S11 async-context check). Use a span link rather than a parent when the producer doesn't wait.
  • Across threads / async / thread-pools: the context lives in thread-local storage; hand it off explicitly when work jumps threads, or the child span loses its parent.
  • Baggage: optional propagated key–value pairs (e.g. user_tier=premium) that ride alongside for cross-service correlation — powerful but adds to every header, so keep it tiny.

4 · How it works (diagram)

  ORD -->|T1 in message headers| Q[(queue)]

  GW --> COL[span collector / agent]
  ORD --> COL
  WK --> COL
  COL --> STORE[(trace store: query by trace_id / service / latency)]

Each service emits its spans to a local agent/collector (out of the request path), which forwards to a backend that assembles spans sharing a trace id into the tree and indexes them.

5 · The sampling ladder

Tracing every request at full fidelity is too expensive to store — so you sample. But naive sampling throws away the rare slow/error trace you most wanted. Derive the sampling strategy.

Rung 1 — head sampling (decide at the start)

At the first service, flip a coin: keep this trace with probability p (e.g. 1%), and propagate the decision in the sampled flag so every downstream service agrees (you never get half a trace). Cheap and simple.

1,000,000,000 requests/day, head sample 1% -> 10,000,000 traces/day stored per trace ~ 20 spans x 500 B ~ 10 KB -> ~0.1 TB/day kept (vs ~10 TB at 100%)

Breaks on: the decision is made before you know the outcome — so a 1% sample keeps mostly boring fast traces and drops 99% of the errors and slow tails, exactly the ones worth keeping.

Rung 2 — tail sampling (decide after the trace completes)

Buffer all spans for a short decision window, then keep a trace if it's interesting (errored, slow, hit a rare endpoint) and down-sample the boring ones. Now you keep the traces that matter.

buffer 100% for ~10 s; route all spans of a trace to the SAME buffer node by hash(trace_id)%N so one node sees the whole trace to decide in-flight: ~11,500 traces/s x 10 s x 10 KB ~ ~1.1 GB buffered (bounded, not stored) keep: all errors + slow (p99) + p% of normal

Breaks on: cost and complexity — you must buffer everything briefly and shard by trace id so one node sees a whole trace. Often combined: tail-sample for keep/drop, head-sample a baseline for RED metrics.

Rung 3 — metrics from 100%, sample only what's stored

Compute RED metrics (Rate, Errors, Duration) from every span before sampling, so your dashboards/alerts are accurate even though you only store a sample of full traces. You get exact aggregates and representative exemplars without 100% storage.

6 · Storage: high-cardinality spans (P8)

Spans are wide, high-cardinality records (every request has unique ids/timestamps), queried by trace id (point lookup), by service/operation, and by latency range (scan). That's the same two-access-pattern split as S6: a point-lookup layout keyed by trace_id to assemble one trace, plus a columnar store (P8) for analytical queries ("p99 latency of the orders service by endpoint"). Retention is short (traces age out fast) and the store is write-heavy, so it leans on the same columnar/LSM machinery as a metrics/logs backend.

7 · Service maps & correlation

Aggregating parent→child edges across many traces yields a service dependency map (who calls whom, with latencies/error rates per edge) — built for free from the span tree. And because the trace_id is propagated everywhere, it can be stamped into logs and attached to metrics exemplars, so the three pillars cross-link: from a latency spike (metric) → an exemplar trace → the logs of the slow span. That correlation is the payoff of having one propagated id.

8 · Real-world example

Dapper · OpenTelemetry · Datadog

Google's Dapper paper established the model — low-overhead, always-on tracing with sampling as the key to making it affordable, and propagated context as the key to making it work. OpenTelemetry standardized the SDKs and the W3C Trace Context header so traces flow across vendors and languages. Datadog APM (and Jaeger, Zipkin, Honeycomb) implement spans → agent → backend with both head and tail sampling, RED metrics computed before sampling, columnar span storage for analytics, and trace↔log↔metric correlation on the shared trace id. The recurring lessons match the constraints: propagate context across every boundary (including queues and threads), and sample by outcome (tail) so you keep the errors, not just the happy path.

9 · Applying it in a design

Whenever a request crosses services or defers work, say: "propagate a W3C trace context on every hop — including into queue message headers and across thread handoffs — emit spans to a local collector, and tail-sample so I keep all errors/slow traces while storing a small fraction of normal ones; compute RED metrics from 100% before sampling." It's the concrete answer to "how would you debug this in production?" and the async-observability requirement in S12/S11.

10 · Pitfalls

  • Dropping context across a queue or thread-pool → the trace silently breaks into disconnected fragments.
  • Head-sampling only → you throw away the rare error/slow traces you needed.
  • Sampling decisions that differ per service → partial traces (always propagate the sampled flag).
  • Storing 100% of spans → runaway cost; or computing metrics after sampling → wrong rates.
  • Unbounded baggage on every header → bloated requests.
