P8 · Tiered & columnar storage

What it is: matching the storage format and medium to the access pattern and the data's age.
Reach for it when: storage cost grows unbounded, or analytical scans read few columns over many rows.
The lever: columnar layout (Parquet/ORC) to read only projected columns; compression (Gorilla for time series); immutable time-partitioned blocks; lifecycle tiers hot→warm→cold by access-frequency decay.
New problem it hands you: cold-tier read latency; (de)compression CPU; columnar is great for scans but poor for point writes/updates; immutability turns updates into append + compaction.
Exemplified by: Datadog, Distributed SQL, Object store, Video.

1 · The problem it solves

Two different cost problems: analytical queries that read a few columns over billions of rows waste enormous I/O in a row store, and data that is rarely read shouldn’t sit on expensive hot storage. Columnar layout fixes the first; tiering fixes the second.

2 · How it works

Columnar (Parquet, ORC): store all values of a column together. A query that selects 3 of 200 columns reads only those 3, and same-type runs compress beautifully (dictionary, run-length, delta). Tiering: keep recent/hot data on fast storage, roll older data to cheaper tiers on an access-frequency decay (hot → warm → cold/archive). Time-series systems combine both: immutable, time-partitioned, compressed blocks that age into object storage.

3 · The new bottleneck it hands you

Columnar is superb for scans but poor for point writes/updates — you rewrite column files, so it’s append/batch oriented, not OLTP. Compression trades CPU for I/O. Tiering trades cold-tier latency (a query over archived data is slow) and, for object storage, request cost. Immutability means updates become append + compaction.

4 · Real-world example

Google Dremel / BigQuery

Dremel (the engine behind BigQuery) stores data in a columnar format and scans only the projected columns across thousands of nodes, which is how it scans terabytes in seconds without indexes — the columnar read is the optimization. Its open-source descendant Parquet is now the default analytical file format (used by Spark, Presto, Snowflake). On the tiering side, Amazon S3 storage classes (Standard → Infrequent Access → Glacier) implement the access-frequency decay model: cheaper per-GB, slower/per-request-pricier to read.

5 · Applying it in a design

For analytics: "columnar storage with predicate/projection pushdown so we only read needed columns." For cost: "hot data fine-grained on SSD, roll up and tier to object storage after N days." See Distributed SQL and Datadog.

6 · Pitfalls

  • Using columnar for a write-heavy OLTP workload.
  • Tiering data to cold storage that’s still queried interactively.
