9 · Social Feed (Twitter / Instagram)

Hard constraint: the celebrity / hot-user problem — one write from 100M followers triggers 100M synchronous fan-outs; naive push deadlocks.
Primitives exercised: P5 (fan-out), P2 (hot-key), P4 (sorted set).

Fan-out on write vs read; hybrid (push for normal, pull for celebrities) + hot-node detection; ranked vs chronological (ML breaks simple merge); Redis sorted set as the natural timeline structure.

1 · Requirements & scale

Functional: users follow others; posting distributes to followers; each user reads a merged, recent (often ranked) timeline. Non-functional: very fast reads (timeline open is the most common action), reasonable write latency, handle wildly uneven follower graphs.

Back-of-envelope 300,000,000 DAU, each opens timeline ~10x/day = 3x10^9 reads/day (~35k/s avg, higher peak) avg user posts a few times/day -> reads >> writes by orders of magnitude follower distribution power-law: most tiny, a few at 100M+

3 · The hard constraint, in depth

The celebrity / hot-user problem. With fan-out on write, one post by an account with 100M followers triggers 100M timeline writes — synchronously, that deadlocks the system; even async it’s a massive, bursty amplification. So you cannot push for everyone. The standard answer is hybrid: push for ordinary authors, pull-and-merge for the few huge ones, detected by follower count (P5, P2 hot-key).

4 · Component deep dives

Fan-out strategy (P5)

Push (fan-out on write) precomputes each follower’s timeline for O(1) reads; pull (fan-out on read) merges followees’ recent posts at query time. Hybrid mixes them by author degree — see P5.

Timeline storage (P4)

A per-user timeline is naturally a Redis sorted set keyed by score = timestamp (or rank score), capped to the most recent N entries — a cache (P4) that can be rebuilt from source.

Ranking

A chronological feed is a simple merge; an ML-ranked feed scores candidates (engagement, recency, affinity) and can no longer be a plain timeline merge — it becomes candidate-generation + ranking.

5 · Real-world example

Twitter / Instagram

Twitter’s home timeline pushes tweets into per-follower timelines held in Redis for fast reads, but switches to pulling at read time for very-high-follower accounts to avoid the fan-out explosion — the canonical hybrid. Instagram (heavily Django + Cassandra + Redis) similarly precomputes feeds and caps stored entries, leaning on caching and careful denormalization. In both, the architecture is a direct consequence of the celebrity constraint: there is no single fan-out strategy that works for both the long tail and the megastars.

6 · Tradeoffs & alternatives

Pure push: fastest reads, breaks on celebrities and wastes writes on inactive followers. Pure pull: cheap writes, slow reads at scale. Hybrid: best overall, at the cost of two paths and a hot-author classifier. Ranking adds latency and infra but drives engagement.

7 · Key talking points

  • Reads >> writes; optimize the timeline read.
  • Hybrid fan-out; push for the tail, pull for celebrities (P5).
  • Redis sorted set as the timeline structure (P4).
  • Hot-user detection (P2).
  • Chronological merge vs ML ranking.
