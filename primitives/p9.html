P9 · Idempotency & "exactly-once-ish"

What it is: making a repeated operation safe so retries don't double-apply.
Reach for it when: any at-least-once delivery, retry, or external side effect (charge a card, send a push).
The lever: idempotency key carried by the client; a dedup table written transactionally with the effect (key, result); idempotent worker logic so re-execution converges.
New problem it hands you: no true exactly-once across systems you don't control (FCM, Stripe, the bank) — settle for at-least-once + idempotent + reconciliation. The dedup table is itself state to scale and expire.
Exemplified by: Task scheduler, Notification, Payment.

1 · The problem it solves

Networks retry. A client times out and resends; a queue redelivers; a worker restarts mid-task. If the operation has a side effect (charge a card, send a push, decrement inventory), a retry must not apply it twice. Idempotency makes “do it again” safe — and as usual the right design is reached by ruling out the cheaper answers.

2 · The ladder to safe retries

Rung 0 — at-least-once, no dedup

Just retry on timeout. Guarantees the effect happens at least once — and double-charges when the first attempt actually succeeded but its ack was lost.

Bottleneck: duplicate side effects.

Rung 1 — at-most-once (fire and forget)

Never retry. No duplicates — but now a lost request means the charge silently never happens. You’ve traded a double-apply for a drop.

Bottleneck: lost operations; unacceptable for money.

Rung 2 — client-supplied idempotency key + dedup table ← the answer

The client generates a unique idempotency key per logical operation and sends it on every retry. The server records (key → result) and, on a repeat, returns the stored result instead of re-running. The make-or-break detail is how the record is written — do it wrong and you reintroduce duplicates under concurrency (next section).

3 · Getting the dedup write right (the race)

The naive version — “check if key seen; if not, execute; then record it” — has a fatal gap under concurrent retries:

two retries of the SAME key arrive together: req A: SELECT key -> not found req B: SELECT key -> not found (A hasn't recorded yet) req A: charge card $$$ req B: charge card $$$ <-- double charge

The fix is to make the key’s first write atomic via a unique constraint, and to commit the dedup record in the same transaction as the effect. A small state machine on the key row makes it precise:

idempotency_keys key VARCHAR PRIMARY KEY -- UNIQUE is the whole trick status ENUM(in_progress, done) response JSON -- the stored result created_at TIMESTAMP -- for expiry per request: BEGIN INSERT (key, in_progress) -- if key exists -> unique violation on violation: read row; status=done -> return stored response (true duplicate) status=in_progress -> 409, "retry shortly" (original still running) <perform the side effect> UPDATE key SET status=done, response=... COMMIT -- record + effect commit together
graph TD
  C[request + idempotency key] --> I{INSERT key in_progress}
  I -->|unique violation, done| D[return stored response]
  I -->|unique violation, in_progress| W[409 retry shortly]
  I -->|inserted| E[do effect, same txn]
  E --> U[UPDATE status=done, store response] --> COMMIT[commit together]

Now exactly one of two concurrent retries wins the INSERT; the other is deflected to the stored result or told to wait. The unique constraint, not the SELECT, is what makes it correct.

Choosing the key — and idempotent-by-construction

The key must be stable across retries of the same logical request but unique across different ones. Two sources: (1) client-generated — a UUID minted once per user action and reused on every retry (Stripe’s model); a fresh UUID per network attempt would defeat the whole scheme. (2) content-derived — a hash of the meaningful fields (user, amount, order_id), which also dedups two independently submitted identical requests, sometimes wanted, sometimes not.

Better still, make the operation idempotent by construction so no dedup table is needed: a PUT that sets a value is naturally repeat-safe where a POST that appends is not; a conditional update (UPDATE ... WHERE status='pending') applies once even if run twice; using a natural unique id (order_id) as the primary key turns a duplicate insert into a harmless constraint violation. Reach for the dedup table only when the effect can’t be made naturally idempotent.

4 · The new bottleneck it hands you

You cannot get true exactly-once across systems you don’t control (a payment processor, FCM, a bank) — if your call to them times out, you genuinely don’t know whether their side committed. The honest target is at-least-once + idempotent + reconciliation: retry safely using their idempotency key too, and run a later batch that compares your records against theirs to catch any divergence. The dedup table is itself state to scale and expire (Stripe keeps keys ~24h), and the key must be scoped so genuinely-distinct requests don’t collide nor genuine retries diverge. Idempotency also pairs with fencing tokens (P10): a fencing token stops a stale writer, an idempotency key stops a duplicate one — different failures, both needed for an exactly-once-ish effect.

5 · Real-world example

Stripe

Stripe’s API takes an Idempotency-Key header on every mutating request. Stripe stores the first response keyed by it and replays that exact response for any retry with the same key, so a network blip during a charge can be retried safely without double-charging. Keys are scoped and expire after 24 hours. Stripe’s own engineering writing frames idempotency keys as the foundation of "robust and predictable APIs," and it’s the same mechanism that underpins their payment correctness.

6 · Applying it in a design

Anywhere you have retries or external effects: "client-supplied idempotency key, dedup record written in the same transaction as the effect via a unique constraint, at-least-once delivery, reconcile against the provider." Pair it with at-least-once queues (Kafka, task scheduler) so you never need true EOS.

7 · Pitfalls

  • Storing the dedup record in a separate transaction from the effect (race window).
  • Keys that are too broad (collisions) or too narrow (retries treated as new).
  • Assuming a downstream "succeeded" because your call returned — reconcile.
  • Relying on SELECT-then-INSERT instead of a unique constraint — it races under concurrent retries.
