Curriculum / Part III · Systems / Cluster A — Foundations

1 · Rate Limiter

Hard constraint: must add near-zero latency to the hot path. If checking the limit costs more than the request, you solved the wrong problem.
Primitives exercised: P16 backpressure, P3 consistency (counters), P4 caching (Redis).

Token bucket vs sliding vs fixed window (and their failure modes); distributed counter coordination (local accuracy vs global latency); Redis as fast-but-not-perfectly-consistent state; fail-open vs fail-closed when the store is down.

1 · Requirements & scale

Functional: allow N requests per window per identity (user, IP, API key); reject or queue the rest; ideally tell the client when to retry. Non-functional: the check must be near-instant and add negligible latency, work across many app servers, and fail sensibly when its own store is down.

Back-of-envelope 1,000,000 req/s across 1,000 app servers = 1,000 req/s/server limit check budget < ~1 ms (must be cheaper than the request) state per identity ~ a counter + timestamp (tens of bytes) 10,000,000 active identities x ~50 B ~ 500 MB -> fits in Redis memory

2 · High-level architecture

graph LR
  C[client] --> MW[rate-limit middleware]
  MW -->|atomic incr+expire| R[(Redis)]
  MW -->|allow| APP[service]
  MW -->|deny 429 + Retry-After| C

Every app server runs the same check against a shared fast store (Redis) using an atomic operation so concurrent requests don’t race. Denied requests get 429 with Retry-After.

3 · The hard constraint, in depth

The limiter sits on the hot path of every request, so the check must cost less than the work it protects. That rules out anything chatty or strongly-coordinated per request; you want one O(1) atomic op against memory, and you must decide what happens if Redis is unreachable — fail-open (allow, protect availability) or fail-closed (deny, protect the backend).

4 · The algorithm ladder

Rung 0 — fixed window

One counter per clock window: INCR limit:user:12:00, reject past N, reset each minute. Dead simple — but it allows a 2× burst across the boundary:

limit 100/min: 12:00:59 -> 100 requests (fills the 12:00 window) 12:01:00 -> 100 requests (fresh 12:01 window) => 200 requests in ~1 second, twice the intended rate
Bottleneck: edge bursts double the real rate.

Rung 1 — sliding window log

Store the timestamp of every request; count those within the last 60 s. Exact — no edge burst — but it stores one entry per request, so a heavy client costs a lot of memory.

Bottleneck: memory grows with request volume.

Rung 2 — sliding window counter

Approximate the sliding window by weighting the current and previous fixed windows by overlap — O(1) memory, no hard edge burst. The usual sweet spot for a pure rate cap.

Rung 3 — token bucket ← the default

Store (tokens, last_refill); each request refills tokens += rate × elapsed (capped at b) and spends one. It bounds the long-run rate and permits a controlled burst up to b — good UX:

bucket b=10, refill r=1/s: t=0 tokens=10 -> 10 requests burst -> tokens=0 (burst allowed, up to b) t=0..5 no requests -> tokens refill to 5 t=5 request -> tokens=4 long-run rate bounded to r, bursts to b

(Leaky bucket is the queueing variant — drains at a fixed rate, smoothing output at the cost of added delay.)

5 · Making it distributed — the counter ladder

1M req/s spread over 1,000 servers means the counter is shared state. Another ladder:

RungMechanismBreaks on
Per-node local counterseach server limits its own sliceloose global limit (N× servers can each allow the full quota)
Global counter in Redisone atomic INCR+expire (or a Lua script for token-bucket) per requesta round trip on the hot path; Redis is now a dependency
Redis + local pre-check (← typical)cheap local guard, periodic sync to Redisslight over-admit, acceptable

The atomicity matters: read-then-write races under concurrency, so the increment-and-check must be one atomic op — a single INCR or a Lua script that Redis runs atomically. And decide the failure mode up front: if Redis is unreachable, fail-open (allow — protect availability) or fail-closed (deny — protect the backend).

6 · Real-world example

Stripe

Stripe documented running multiple layers of rate limiters in front of its API rather than one. They distinguish a request-rate limiter (token-bucket style, the steady-state cap), a concurrency limiter (cap simultaneous in-flight requests so one client can’t hog workers), and load-shedding tiers that prioritize critical traffic (e.g. live charges) over less-critical traffic (e.g. test or batch calls) when the system is under stress. The limiters are implemented in Redis with atomic operations. The lesson: at scale "rate limiting" is several different limiters defending different resources, not one counter.

7 · Tradeoffs & alternatives

Local-only counters scale infinitely but enforce limits loosely; a global Redis counter is accurate but a dependency on the hot path (mitigate with fail-open). Exact sliding windows cost memory; the sliding-window-counter approximation is usually the sweet spot.

8 · Key talking points

  • Lead with: "the check must be cheaper than the request — one atomic Redis op."
  • Token bucket by default; name the burst-vs-smoothness tradeoff.
  • Global vs local counter = accuracy vs latency (P3).
  • Fail-open vs fail-closed when the store is down.
  • Return 429 + Retry-After; consider per-tier limits.
