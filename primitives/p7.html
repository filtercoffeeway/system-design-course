P7 · Scatter-gather & partial aggregation

What it is: fan a query to all shards, compute partials locally, merge centrally.
Reach for it when: the answer spans partitions (search ranking, group-by, cross-shard query).
The lever: push computation to the data (partial aggregates, top-K per shard), then combine; minimize what crosses the network.
New problem it hands you: the slowest shard sets the latency (tail latency, see P12); shuffle (redistribution for joins/group-by) becomes the dominant cost; partials can be approximate (local-only stats).
Exemplified by: Distributed SQL, Search, Datadog.

1 · The problem it solves

When data is partitioned (P2), no single node can answer a query that spans the whole keyspace — a search ranking, a global aggregate, a group-by. Scatter-gather fans the query out to every shard, has each compute a partial answer locally, and merges the partials centrally.

2 · How it works

graph TD
  Q[coordinator] --> S1[shard 1: partial agg]
  Q --> S2[shard 2: partial agg]
  Q --> S3[shard 3: partial agg]
  S1 --> M[merge partials]
  S2 --> M
  S3 --> M
  M --> R[result]

The art is pushing computation to the data: each shard returns a small partial instead of raw rows. Worked — global top-3 across 3 shards: each shard sends only its own top-3, the coordinator merges:

S1 top-3: a(90) b(70) c(30) S2 top-3: d(85) e(60) f(20) -> coordinator merges 9 candidates -> S3 top-3: g(95) h(50) i(10) global top-3 = g(95), a(90), d(85)

This is correct because a global top-K member must be top-K on its own shard — so each shard need only ship K rows, not all of them. Aggregations that are associative/commutative (sum, count, max, mergeable sketches) parallelize the same way; ones that aren’t — exact median, exact distinct-count — can’t merge from local partials and need approximations (e.g. a HyperLogLog or t-digest sketch per shard, merged centrally).

3 · The new bottleneck it hands you

The slowest shard sets the latency. A query is as slow as the worst shard it touches, and that compounds brutally with fan-out width:

each shard is "fast" (under its p99) 99% of the time: 1 shard -> 99.0% of queries fast 10 shards -> 90.4% fast (9.6% hit a slow shard) 100 shards -> 36.6% fast -> 63% of queries hit at least one slow shard!

So a wide scatter turns a rare per-shard slowdown into the common case — which is why fan-out systems lean on hedging and per-shard timeouts (P12). Shuffle — redistributing data between shards for a join or group-by — becomes the dominant cost in analytics (distributed SQL), and partials from local-only stats can be approximate (distributed IDF, P6).

4 · Real-world example

Elasticsearch (query-then-fetch)

An Elasticsearch search runs in two scatter-gather phases. In the query phase the coordinator scatters the query to every shard; each returns just the top-K document IDs and scores (tiny). The coordinator merges these into a global top-K. Then in the fetch phase it gathers the full documents for only those K winners from the relevant shards. Returning IDs first, not whole documents, is the "push the cheap partial, gather the expensive part last" pattern — and the query is bounded by the slowest shard in the scatter.

5 · Applying it in a design

Use it for any cross-partition read. Say: "scatter to all shards, each returns a partial top-K / partial aggregate, coordinator merges — and I’ll bound the slowest shard with a timeout or hedge."

6 · Pitfalls

  • Shipping raw rows to the coordinator instead of partials.
  • Ignoring tail latency / one slow shard.
  • Assuming non-mergeable aggregates parallelize.
