14 · URL Shortener (bit.ly style)

Hard constraint: sub-millisecond lookup — a CDN/Redis cache at the edge is the read architecture; the storage system barely matters.
Primitives exercised: P4, P11, P5 (analytics).

Base62 encoding; read-heavy optimization (1000× redirects); collision avoidance (pre-generate vs hash-and-check); 301 vs 302 tradeoff; click analytics as the same ingestion pattern as Datadog.

1 · Requirements

Functional. (1) shorten(longUrl) returns a short code; (2) GET /{code} redirects to the original URL; (3) optional custom/vanity aliases; (4) optional expiry; (5) click analytics. Out of scope for the core design: editing a link’s target, user accounts — mention them, then set aside.

Non-functional — these are what actually shape the design:

PropertyTargetWhy it matters
Read latencyp99 < ~10 ms end-to-end; lookup itself sub-msA redirect sits in the user’s click path; every ms is felt.
Read:write ratio~100:1 to 1000:1Reads dominate by orders of magnitude → optimise the read path above all.
AvailabilityRedirects ~99.99%; shorten can be lowerA dead redirect breaks every link ever shared. Reads > writes in priority.
UniquenessCodes globally unique, never reissuedA collision silently sends a user to the wrong site.
DurabilityMapping must never be lostLinks live for years; losing one is a dead link forever.

2 · Capacity estimation

Numbers first — they justify every later choice. Assume 100M new links/day.

WRITES 100,000,000 new links/day / 86,400 s ≈ 1,160 writes/s (peak ~3-5x → ~5k/s) READS (redirects) read:write 100:1 ≈ 116,000 redirects/s read:write 1000:1 (viral) ≈ 1,160,000 redirects/s STORAGE row ≈ 500 B (code, long URL, owner, created_at, expiry) 100M/day x 365 x 500 B ≈ 18.2 TB/yr (sharded KV / partitioned SQL) CODE SPACE (base62 = [0-9a-zA-Z]) 62^6 = 56,800,235,584 (~57 B) → ~1.5 yr of links at 100M/day 62^7 = 3,521,614,606,208 (~3.5 T) → ~96 yr at 100M/day ← pick 7 chars 62^8 = 218,340,105,584,896 CACHE hot set ≈ 0.1% of a day's links = ~100,000 links x ~100 B ≈ 10 MB fits in RAM even the full day (100M x 100 B = 10 GB) fits a small Redis cluster

The takeaways that drive the rest: reads dwarf writes (so this is a caching problem), 7 base62 chars is the sweet spot, and the hot working set is tiny enough to live entirely in memory.

3 · API design

POST /api/shorten body : { "longUrl": "https://...", "customAlias"?: "go/launch", "expiry"?: ts } 200 : { "shortUrl": "https://sho.rt/15FTGg", "code": "15FTGg" } 409 : custom alias already taken GET /{code} → 302 Location: <longUrl> (or 301; see tradeoffs) 404 if unknown/expired

The shorten endpoint is a normal JSON POST; the redirect is a bare GET /{code} so the short link is just https://sho.rt/15FTGg with nothing extra in the path.

4 · Data model & store selection

The schema is trivial — one row per link, looked up by its code:

links (primary key = code) code VARCHAR(7) PK "15FTGg" long_url TEXT "https://example.com/very/long/path?x=1" owner_id BIGINT created_at TIMESTAMP expiry_at TIMESTAMP NULL -- custom aliases live in the same table; the alias IS the code

The interesting question is which store — and the honest answer is “start with SQL and escalate only when it actually breaks,” not “NoSQL because it’s web-scale.”

Rung 0 — a single SQL table (right default; fine for ages)

One Postgres/MySQL table, code as primary key. The access pattern is a point lookup on the PK — the single thing every database is fastest at; SQL does it as well as anything. At thousands of writes/s and tens of millions of rows this is completely adequate. Don’t reach further yet.

Bottleneck (where SQL breaks — and note why): it’s not the query shape, it’s volume on one node. At 100M links/day that’s ~36 B rows and ~18 TB/yr — past one machine’s disk and, more importantly, past the RAM that holds the index. The PK is a random 7-char code, so the B-tree index has no insert locality: once it exceeds memory, each insert and each lookup risks a random disk seek, and the single primary caps write throughput. Vertical scaling buys time, then stops.

Rung 1 — read replicas (reads scale, writes don't)

Redirects are 99%+ of traffic, so fan reads out to replicas. This is real headroom — but it does nothing for write volume or the fact that the whole dataset still must fit on the primary. It’s a patch, not the fix.

Bottleneck: one primary still owns every write and the full 18 TB. You have to split the data itself.

Rung 2 — shard by code → and now SQL earns you nothing

Partition the rows across N nodes by hash(code). That solves volume and write throughput. But look at what SQL is now giving you: no joins (one table), no multi-row transactions (every op is one key), no range scans (codes are random). You’ve reduced a relational database to a distributed hash map — and you’re hand-rolling the sharding, rebalancing, and cross-node replication yourself.

Rung 3 — a KV / wide-column store (← pick)

A purpose-built KV / wide-column store (DynamoDB / Cassandra) is a sharded hash map: code as the partition key gives single-partition O(1) lookups, with horizontal partitioning, replication, and rebalancing built in (P2). You arrive here not because “URL shorteners use NoSQL” but because the access pattern is pure keyed lookup and the only thing SQL still offered was features this workload never uses.

RungGood untilBreaks on
Single SQL table~thousands w/s, index fits RAMIndex > memory → random-seek inserts; one-node write cap
+ read replicasread-heavy growthWrites + full dataset still on one primary
Sharded SQLvolume & writes scaleYou’re running a hash map by hand; SQL features unused
KV / wide-column (← pick)scales horizontally by key(accepted) no joins/scans — but we never needed them
Different access pattern, different landing rung. This ladder ends at KV because the access is keyed point-lookup. If instead the workload were analytical scans over a few columns of many rows (e.g. the click-analytics table, or metrics), a row store dies on scan I/O — it reads whole rows to touch two columns — and the ladder lands on a columnar store (P8) instead. The store follows the access pattern, not fashion.

So the mapping table is KV; analytics live elsewhere — click events go to a separate pipeline (below), keeping the hot redirect table small and read-mostly.

5 · High-level architecture

graph LR
  U[user click] --> CDN[CDN / edge cache]
  CDN -->|hit ~edge| U
  CDN -->|miss| SVC[redirect service]
  SVC --> CACHE[(Redis: code->URL)]
  CACHE -->|miss| KV[(sharded KV: code->URL)]
  W[shorten API] --> KGS[key-gen service] --> KV
  SVC -.click event.-> Q[[queue]] --> AGG[analytics]

Two independent paths share the KV store. The write path (shorten) mints a code and persists the mapping. The read path (redirect) is a cache cascade: CDN → Redis → KV, where almost every request is answered before it reaches the store. A dotted async path peels click events off to analytics without touching the redirect’s latency budget.

6 · Code generation — the escalation ladder

This is the one genuinely interesting decision, and the right way to reach the answer is not to memorise “use a key-gen service.” Start with the simplest thing, find the load that kills it, and let each bottleneck push you to the next rung. Whatever produces the number, it’s base62-encoded into the short code — [0-9a-zA-Z], ~5.95 bits/char, so 7 chars address 3.5 trillion links:

counter base62 code chars 125 -> "21" 2 1,000,000 -> "4c92" 4 1,000,000,000 -> "15FTGg" 6 56,800,235,583 -> "ZZZZZZ" 6 ← last 6-char code (= 62^6 - 1) 56,800,235,584 -> "1000000" 7 ← rolls over to 7 chars cleanly 3,500,000,000,000 -> "ZCpdjxe" 7

Rung 0 — DB auto-increment (good to ~a few k writes/s)

INSERT the link, let the database hand back an auto-increment BIGINT, base62 it. At 10 rps this is completely fine — ship it. Unique, sortable, zero extra infrastructure.

Bottleneck: one node’s write path caps at roughly a few thousand inserts/s (WAL + fsync bound), and it’s a single point of failure. Worse, the moment you shard the database to scale writes, each shard’s counter starts at 1, so two shards mint the same id — auto-increment doesn’t survive sharding. And every shorten now waits on a durable DB write.

Rung 1 — Redis INCR (good to ~100k/s)

Move the counter to a single in-memory atomic counter: INCR url:seq. ~100k+ ops/s, decoupled from the durable write. Plenty for 1,160 writes/s with headroom.

Bottleneck: it’s now one Redis node — a single point of failure and a single throughput ceiling — and every shorten pays a network round-trip to it. You can replicate Redis, but a counter can’t be naively replicated (two replicas would re-issue numbers), so you’re back to one hot node.

Rung 2 — Range / batch leasing (scales ~linearly)

Stop asking for one id at a time. Each app server leases a block of, say, 1,000 counter values (8,000,000–8,000,999) from a coordinator (P10 — a DB row, ZooKeeper, or etcd) and serves codes from RAM with no coordination until the block runs out. That turns a per-write lock into a per-1,000-writes lock — a 1000× drop in coordination. At our load the coordinator sees ~1 lease/sec, not 1,160 ops/sec. Add app servers and throughput scales linearly; collisions are impossible by construction because ranges are disjoint, so the redirect path never checks uniqueness. This is the right answer for a URL shortener — and notice we derived the “central key-gen service,” we didn’t assume it.

Residual costs (acceptable here): a server that crashes mid-block leaves a gap in the id space (harmless — we have trillions), and ids are only roughly time-ordered across servers, not strictly. For a URL shortener neither matters.

When the requirement is different — Snowflake

If you didn’t need short codes but did need distributed, time-sortable ids with zero per-id coordination (order ids, event ids), the ladder ends one rung further at a Snowflake id: [41-bit ms timestamp][10-bit machine id][12-bit seq] = a 64-bit int generated entirely locally, ~4,096 ids/ms/node (~4M/s/node), sortable by time. Its cost is the bit you trade for going fully coordination-free: machine-id assignment and clock-skew handling. It’s a poor fit here only because 64 bits is ~11 base62 chars — longer than we want for a human-facing link.

The ladder, at a glance

RungGood untilBreaks on

Alternative family — hash-and-check: instead of a counter, hash the long URL (SHA/MD5), take the first 7 base62 chars, retry with a salt on the rare collision. Stateless and gives the same URL the same code (natural dedup), but every write does a read-before-write to detect collisions, and birthday math makes those non-trivial as the table grows — which is why counter+ranges is usually preferred unless content-deduplication is itself a requirement.

Production failure modes & failover. Each rung has a failure mode as well as a throughput ceiling; the containing mechanisms are detailed in the primitives, so they’re not repeated here:
  • Counter: must never fail over to a lagging replica — a promoted replica re-issues already-used ids. Needs a consensus-backed counter → P10.
  • Range leasing: the high-water mark must be allocated through consensus or a coordinator failover double-issues ranges; and a server keeps minting from its local block while the coordinator is briefly down (graceful degradation) → P10, P16.
  • Snowflake: backward clock jumps (refuse-to-issue + leap smearing) and machine-id collisions (lease worker ids from ephemeral nodes) → P10.
  • Retries minting duplicate ids are deduped with an idempotency key → P9.

Scenario — what happens when the KGS is down

“The key-gen service is down” means the coordinator that hands out blocks is unreachable — not the app servers. Because each server already holds a leased block of ~1,000 values in RAM and mints from it with no coordination, writes keep succeeding through the outage. You only feel it when a server drains its block and tries to lease the next one. So the design degrades by rung:

  • Short outage — ride it out for free. Size blocks generously and pre-fetch the next block before the current one drains. The local buffer absorbs any realistic coordinator-recovery window and no write ever stalls. At ~1.16k writes/s a 1,000-value block lasts ~0.86s per server, so pre-fetching (or leasing 100k-value blocks → ~86s of runway) covers a normal failover. This is the simplest fix and costs no new code.
  • Long outage — namespaced hash-and-check fallback. If an outage could outlast the buffer, fall back to the stateless hash(long_url) family (needs only the store, not the coordinator). The trap: hash codes scatter uniformly across the same 7-char space, and the counter path never checks uniqueness — so a fallback code can collide with a range the counter leases later, and the counter won’t catch it. Fix by making the spaces disjoint: give fallback codes a reserved prefix or 8th char the counter never emits. Collision-free-by-construction survives the switch; read-before-write lives only on the fallback path.
The real danger isn’t “down,” it’s bad failover. If the coordinator fails over without durably committing its high-water mark, the new coordinator re-issues a range that’s already out → duplicate codes. That is why the high-water mark must move through consensus (P10), not just “make the coordinator HA.”

7 · The write path (shorten)

Client POSTs the long URL → app server takes the next code from its leased range (no coordination) → writes code → long_url to the KV store (idempotency key from the client so a retried POST doesn’t mint a second code) → returns https://sho.rt/{code}. For a custom alias, attempt an insert if-not-exists on the alias; a conflict returns 409. Writes are ~1k/s — trivially handled.

8 · The read path — deriving the cache cascade

The whole system lives or dies on redirect latency at ~106 reads/s. But don’t assert “put a CDN and Redis in front” — derive it the same way, rung by rung.

Rung 0 — read straight from the store

Every redirect does a GET code against the KV store. Correct, and at low traffic completely fine.

Bottleneck: at ~1.16M reads/s the store needs enormous provisioning, and each read pays its full lookup + network latency on the user’s click path. We’re paying disk/replica cost for data that barely changes.

Rung 1 — an in-memory cache (Redis) in front (P4)

Clicks follow a Zipfian distribution — a few viral links dominate — so the hot working set is tiny (~10 MB, see capacity). Front the store with Redis and >99% of reads are served from RAM; the KV store sees only a cold-link trickle (~5,800/s).

Bottleneck: a single viral link is one key — all its traffic lands on the one Redis shard that owns it (the P2 hot-key problem), and every read still crosses the network to your region.

Rung 2 — push to the edge / CDN (P11) ← the architecture

Cache the 302 itself at the CDN edge, keyed by code, near the user. The truly viral links are absorbed at hundreds of edge POPs — no single node is a hotspot — and the redirect is answered geographically close, shaving the round-trip. The result is the three-tier cascade CDN → Redis → KV, which we arrived at by killing the hot-key and latency bottlenecks in turn.

301 vs 302 — caching vs analytics

A 301 (permanent) is cached by the browser and intermediaries indefinitely, so subsequent clicks never reach you — cheapest load, but you lose per-click analytics and can never change the target. A 302 (found/temporary) comes back to your server every time — more load, but you count every click and retain control. Pick by whether analytics matter; most shorteners use 302 for exactly that reason.

Analytics pipeline

Each 302 emits a click event (code, timestamp, geo, referrer, UA) onto a queue, aggregated asynchronously — the same ingestion shape as Datadog (S6): batch (P1) → queue (S2) → roll-up. This is deliberately off the redirect hot path: the user is redirected immediately; counting happens behind the queue, so analytics load never inflates redirect latency.

9 · Scaling & bottlenecks

The mapping table shards cleanly by code (the key is the shard key, so every lookup is single-partition — no scatter-gather). Redis scales by sharding the keyspace and adding read replicas for hot keys; the CDN absorbs the truly viral links so no single Redis node is a hotspot (P2 hot-key problem). The key-gen counter is the one global resource, and range-leasing is what keeps it from becoming a bottleneck. The analytics pipeline scales independently and can fall behind without affecting redirects.

10 · Tradeoffs & alternatives

Counter vs hash: ranges remove hot-path collisions at the cost of a key-gen service; hashing is stateless but pays a uniqueness read and leaks that the same URL maps to one code. 301 vs 302: load vs analytics. Vanity aliases share the code namespace, so they need an if-not-exists insert and can’t collide with generated codes (one trick: generate from a counter range that vanity codes are excluded from, or keep them in the same uniqueness check). Expiry adds a TTL and a sweeper, and the redirect must treat an expired row as a 404. Predictability: sequential counter codes are guessable/enumerable; if that matters, encrypt the counter (e.g. a Feistel permutation) before base62 so codes look random while staying collision-free.

11 · Real-world example

Bitly / TinyURL

Bitly serves on the order of billions of redirects a month and is built around exactly this shape: a tiny key→URL mapping fronted by aggressive caching/CDN so redirects are effectively cache hits, short base62-style codes, and a separate asynchronous analytics pipeline that counts clicks without slowing the redirect. The interesting engineering is all on the read/cache and analytics sides; storing the mapping itself is nearly trivial — which is the whole lesson of this system.

12 · Key talking points

  • It’s a read/caching problem, not a storage problem — CDN + Redis cascade is the architecture (P4, P11); >99% of reads never touch the store.
  • Counter + base62 with range leasing makes codes collision-free by construction and removes hot-path coordination; 7 chars = 3.5T codes = ~96 years of headroom.
  • The key is the shard key, so every lookup is single-partition — no scatter-gather.
  • 301 vs 302 is the caching-vs-analytics tradeoff.
  • Analytics ride an async pipeline off the hot path (S6 shape), so counting never slows a redirect.
  • Capacity sanity: ~1k writes/s, ~1M reads/s, ~18 TB/yr, ~10 MB hot set.
