P17 · Storage durability & write internals

What it is: how a single storage node doesn't lose data on crash, and the fundamental read-vs-write structure tradeoff.
Reach for it when: designing or reasoning about inside any database or durable store.
The lever: WAL (write-ahead log) — append intent before the mutation, replay on recovery (universal crash recovery). LSM-tree (writes sequential/fast, reads merge levels, compaction is the hidden cost) vs B-tree (predictable reads, random-I/O writes at scale). Erasure coding vs replication (erasure stores less for the same durability, costs reconstruction). Bloom filters to skip disk for absent keys.
New problem it hands you: LSM → read amplification + compaction load; B-tree → write amplification; WAL → fsync cost on the hot path; erasure coding → CPU + latency to reconstruct on read.
Exemplified by: KV store, RDBMS, Object store, Kafka (the log).

1 · The problem it solves

Underneath every database is the same pair of questions: how do you not lose data when the process crashes mid-write, and do you optimise the on-disk structure for writes or for reads? Getting this right is what separates a toy store from a durable one. This page is the write half of storage; P8 is the read/layout half — together they are how a real storage engine (and a TSDB) is built.

2 · Durability: the write-ahead log

A mutation isn’t durable until it survives a crash. The WAL makes it so: before touching the main data structure, append a record of the intent to a sequential log file and fsync it. Only then acknowledge the write. If the process dies mid-operation, recovery replays the log from the last checkpoint and re-applies anything that hadn’t been persisted to the main structure. Because the log is append-only it’s the fastest possible disk pattern — sequential writes, no seeks.

The cost is the fsync on the hot path: a durable write can’t return until the disk confirms. Group commit amortises this — the engine batches the WAL records of, say, 100 concurrent writes into one fsync, turning 100 disk syncs into 1. That is the same batching lever as P1, applied to durability.

3 · The structure tradeoff: LSM vs B-tree

Every engine then picks an on-disk structure, and the choice is fundamentally writes-optimised (LSM) vs reads-optimised (B-tree).

LSM-tree, traced step by step

An LSM buffers writes in a sorted in-memory structure (the memtable), and only ever writes to disk in large sequential SSTable files — never in place. Trace a few operations on keys:

1. PUT k=42 v=A → append to WAL, insert into memtable {42:A} 2. PUT k=17 v=B → WAL, memtable {17:B, 42:A} (kept sorted) 3. PUT k=42 v=C → WAL, memtable {17:B, 42:C} (overwrites in memory) 4. DELETE k=17 → WAL, memtable {17:<tombstone>, 42:C} (delete = a marker) ... memtable fills (e.g. 64 MB) ... 5. FLUSH → write memtable out as immutable SSTable L0-a (sorted), start fresh memtable On disk now: L0-a = [17:tombstone, 42:C] Later writes build L0-b, L0-c ... then compaction merges them into L1.

Two things make this work. Writes are always sequential (memtable flush + WAL append), so an LSM ingests far faster than random in-place updates. And updates/deletes don’t modify old files — a new value shadows the old, and a delete writes a tombstone; the dead data is reclaimed later by compaction.

Compaction

The catch: a key can now live in the memtable and several SSTables. Left alone, reads slow down and dead/overwritten data piles up. Compaction merges SSTables in the background, keeping only the newest value per key and dropping tombstoned keys. Leveled compaction (RocksDB default) keeps each level ~10× the previous and merges a file up into the next level; tiered compaction merges same-size runs together. They sit at different points of the amplification triangle:

graph TD
  W[write] --> WAL[(WAL: sequential, fsync)]
  W --> MT[memtable in RAM, sorted]
  MT -->|flush when full| L0[SSTables L0]
  L0 -->|compaction| L1["L1 (~10x L0)"]
  L1 -->|compaction| L2["L2 (~10x L1)"]
  R[read] --> BF{bloom filter per SSTable}
  BF -->|maybe present| L0
  BF -->|definitely absent| SKIP[skip file, no disk I/O]

B-tree, for contrast

A B-tree updates pages in place: a write finds the leaf page and rewrites it. Reads are predictable (a key is in exactly one place, ~log₃ pages deep) and range scans are excellent — which is why OLTP databases (Postgres, MySQL/InnoDB) use B-trees + WAL. But in-place updates are random I/O, and a small change dirties a whole page (often 8–16 KB), so under a write-saturated load a B-tree pays in random writes where an LSM pays in background compaction.

4 · The three amplifications (the numbers)

Every storage structure trades along three axes; you can’t minimise all three.

AmplificationMeansLSM (leveled)B-tree
Writebytes written to disk per byte of dataHigh: a byte is rewritten once per level — ~10× fanout × levels, e.g. ~40–50× for 4–5 levels~2–3× (page rewrite + WAL)
Readdisk reads per lookupUp to one per level — but bloom filters cut it to ~1~1 (one root-to-leaf path)
Spacedisk used per byte of live dataLeveled ~1.1×; tiered up to ~fanout× mid-mergeFragmentation (~1.3×) from partly-full pages

This is the whole tuning story of an LSM: leveled compaction buys low space and read amplification by paying high write amplification; tiered does the reverse. You pick the corner that matches the workload.

Bloom filters: skipping disk for absent keys

Without help, a point lookup for a key that isn’t in level i still costs a disk read to find that out — once per level. A bloom filter per SSTable answers “is this key definitely not here?” from memory, so absent keys skip the file entirely. The cost is a small false-positive rate (it never says “absent” wrongly, only “maybe present”), tunable by bits-per-key:

bits/key hashes false-positive rate 8 6 ~2.2% 10 7 ~0.8% ← RocksDB default 16 11 ~0.05%

At 10 bits/key (~0.8% FP), >99% of reads for absent keys are answered without touching disk — which is what collapses LSM read amplification from “one I/O per level” back to ~1.

5 · Durability across nodes: replication vs erasure coding

A single node’s WAL survives a crash; surviving disk loss needs redundancy across nodes. Replication keeps N full copies (3× replication tolerates 2 losses, costs 200% overhead). Erasure coding splits an object into k data shards and computes m parity shards; any k of the k+m reconstruct the object. A (10,4) code tolerates 4 losses at only 40% overhead instead of 300% for the equivalent-durability 4× replication — the same reason object stores (S5) use it. The cost: a read of a missing shard must fetch k shards and recompute (CPU + cross-node latency), so erasure coding suits cold/large objects, replication suits hot small ones.

6 · The new bottleneck it hands you

LSM → read amplification + compaction load: a read may check memtable + several SSTables (bloom filters fix the absent-key case, not the present-key merge), and compaction competes with foreground traffic for I/O — a write burst that outruns compaction causes write stalls. B-tree → write amplification from in-place random-I/O page updates. WAL → fsync cost on the hot path (group-commit amortises it, at the cost of a few ms of batching latency). Erasure coding → reconstruction cost on every degraded read. There is no free structure; you are always picking which amplification to pay.

7 · Real-world example

RocksDB / LevelDB

Google’s LevelDB and Facebook’s RocksDB are the LSM engines under a huge swath of modern infrastructure (it’s the storage engine inside CockroachDB, TiKV, Kafka Streams state stores, and many others). They embody the pattern exactly: a WAL for durability, a memtable for fast writes, leveled SSTable compaction, and bloom filters to avoid disk reads for absent keys. The recurring operational theme is the LSM tradeoff — tuning compaction to balance write throughput, read latency, and space amplification. Contrast with Postgres/MySQL-InnoDB, which use B-trees + WAL for read-predictable OLTP. See KV store and RDBMS.

8 · Applying it in a design

Pick the structure from the workload: "write-heavy ingest → LSM (sequential writes, accept compaction and write amplification); read-heavy with range scans → B-tree; WAL either way for crash recovery." The sentence that shows you understand it: "I’ll take an LSM here because ingest dominates — I’m trading write amplification and compaction I/O for sequential writes, and leaning on bloom filters to keep reads near one I/O."

9 · Pitfalls

  • Choosing a B-tree store for a write-saturated workload.
  • Ignoring compaction’s I/O cost when sizing an LSM store — a write burst that outruns compaction causes write stalls.
  • Skipping fsync (or trusting an OS buffer) and calling it “durable.”
  • Under-sizing bloom filters, so absent-key reads still hit disk once per level.
