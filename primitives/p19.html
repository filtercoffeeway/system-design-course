P19 · CRDTs (conflict-free replicated data types)

What it is: data structures whose replicas can be updated independently and always merge to the same value with no coordination — concurrent writes resolve automatically and deterministically.
Reach for it when: multi-master / multi-region writes, offline-first apps, collaborative editing, shopping carts — anywhere you want availability under partition (AP) without lost updates.
The lever: design the merge to be commutative, associative, and idempotent, so order and duplication of updates don't matter — convergence is mathematically guaranteed.
New problem it hands you: metadata growth (tombstones, version tags); the merge semantics may not match business intent (a "removed" item can resurrect); not every type has a sensible CRDT (you can't CRDT an account balance with a hard floor).
Exemplified by: Dynamo-style KV conflict resolution (the "→ CRDT" rung), Redis CRDT, Riak, collaborative editors.

1 · The problem it solves

In a multi-leader / AP system (P3), two replicas accept writes to the same key during a network partition. When they reconcile, whose write wins? Last-write-wins silently discards a concurrent update (lost data); requiring coordination to avoid conflicts sacrifices availability (the CP side of CAP). A CRDT sidesteps the dilemma: it's a type whose concurrent updates merge deterministically with no coordination, so every replica that has seen the same set of updates — in any order, with any duplicates — computes the same value. This is the concrete mechanism behind S3's conflict-resolution ladder ending "→ CRDT," unpacked here.

2 · The one rule: merge must be commutative, associative, idempotent

Convergence is guaranteed if (and essentially because) the merge function forms a join-semilattice — three properties:

  • Commutative: merge(a,b) = merge(b,a) — order of arrival doesn't matter (networks reorder).
  • Associative: merge(merge(a,b),c) = merge(a,merge(b,c)) — grouping doesn't matter (you can merge in any batches).
  • Idempotent: merge(a,a) = a — receiving the same update twice changes nothing (networks redeliver).

Together these mean updates can flow over an unreliable, reordering, duplicating gossip network and every replica still converges. The cautionary counter-example: a "just add the deltas" counter is not a CRDT — redelivery double-counts and replicas diverge. The whole craft is finding a representation whose merge has these three properties.

BROKEN: counter = counter + delta on each received update replica gossips "+1"; the message is redelivered -> counted twice -> diverges (NOT idempotent) FIXED (G-counter, below): track per-node totals, merge = elementwise MAX (idempotent)

3 · The catalog, worked

G-Counter (grow-only counter)

Represent the counter as a vector of per-node sub-counters; a node only increments its own entry; the value is the sum; merge is elementwise max.

replica A: {a:3, b:1, c:0} value = 4 replica B: {a:2, b:1, c:5} value = 8 merge (elementwise max): {a:3, b:1, c:5} value = 9 re-merge with A again -> still {a:3,b:1,c:5} (idempotent: max(3,3)=3)

Max is commutative, associative, and idempotent — so this converges. Why per-node? Because a node's own count only ever grows monotonically, so max correctly recovers the latest count each node has reached, immune to reorder/duplication. (A PN-counter that also decrements is just two G-counters, P and N; value = sumP − sumN.)

LWW-Register (last-write-wins register)

Store (value, timestamp); merge keeps the higher timestamp; ties broken by a deterministic rule (e.g. node id) so the result is the same everywhere.

(x, t=5) merge (y, t=7) -> (y, t=7) // higher timestamp wins concurrent (x, t=5)@nodeA vs (y, t=5)@nodeB -> tiebreak by node id (deterministic)

Converges, but by design discards the losing write — acceptable for "user's current status," lossy for "set of items." LWW is what S3's conflict ladder starts with, and its weakness (silent lost updates, dependence on clock skew — a fast clock makes one node always win) is exactly what pushes you up the ladder to version vectors and then true CRDTs.

OR-Set (observed-remove set)

The hard case: a set where you can add and remove concurrently without "remove wins forever" or "add resurrects deleted items." Trick: each add(x) tags the element with a unique id; remove(x) only removes the (x, id) tags it has observed; an element is present if it has at least one un-removed tag.

A: add(x) -> {(x, id1)} B: add(x) -> {(x, id2)} // concurrent, different unique ids A: remove(x) -> removes only id1 (the tag A has seen) merge -> {(x, id2)} still present -> x is in the set -> a concurrent add "wins" over a remove that never saw it (add-wins semantics)

This makes add/remove commute. The cost is tombstones: removed tags must be remembered (or garbage-collected carefully) so a late-arriving stale add doesn't resurrect a deleted element.

4 · State-based vs op-based

State-based (CvRDT)Op-based (CmRDT)
Gossipsthe whole (merged) stateindividual operations
Needs merge to becommutative+associative+idempotentoperations to commute
Network requirementtolerates loss/dup/reorder (just resend state)exactly-once, causal delivery of ops
Costbandwidth (ships full state)cheap ops, but needs reliable delivery

State-based is robust over flaky gossip (idempotent merge makes redelivery free) but heavy; op-based is lean but pushes the reliability/ordering burden onto the transport. Delta-state CRDTs are the practical middle: ship only the changed part of the state, keeping idempotent-merge robustness without resending everything.

6 · The new bottleneck it hands you

Metadata grows. OR-Sets accumulate tombstones; version vectors grow with the number of replicas; LWW needs synchronized-ish clocks for "last" to mean anything. Garbage-collecting tombstones safely (without resurrecting deleted items when a stale update arrives late) is genuinely hard.

The merge may not match intent. CRDTs guarantee convergence, not correctness for your business rule. An OR-Set's add-wins means a "removed" item can stay if a concurrent add didn't see the remove. You cannot CRDT an invariant like "balance ≥ 0" — that needs coordination (a reservation/lock, S21) because two replicas can each independently approve a withdrawal that's jointly invalid. CRDTs trade a global invariant for availability.

8 · Real-world example

Amazon Dynamo · Riak · Redis

Amazon's Dynamo shopping cart is the classic motivating case: to stay writable during partitions, concurrent cart updates are merged rather than rejected — historically by merging the carts (a union), which is why a deleted item could occasionally reappear (an OR-Set-style add-wins artifact). Riak shipped a library of CRDTs (counters, sets, maps, registers) for exactly this AP reconciliation, and Redis Enterprise's active-active geo-replication is built on CRDTs so writes in multiple regions converge without a coordinator. Collaborative editors (and Automerge/Yjs) use sequence CRDTs so two people typing offline merge to one document. The recurring lesson: CRDTs buy availability and automatic merge, and you pay in metadata and in accepting the merge's semantics.

9 · Applying it in a design

Reach for a CRDT when you need multi-writer availability without lost updates and the data has a sensible merge. Say: "replicas update independently and merge with a commutative/associative/idempotent function, so they converge with no coordination — I'll use a PN-counter for the counts and an OR-Set for the cart, and accept add-wins; for the balance invariant I can't use a CRDT, so that path needs coordination." It's the principled answer to S3's conflict-resolution rung beyond version vectors.

10 · Pitfalls

  • Assuming a "just add deltas" counter converges — it isn't idempotent and diverges on redelivery.
  • Using LWW where lost concurrent writes matter (it silently discards them).
  • Forgetting tombstone growth / GC in remove-capable sets.
  • Trying to enforce a hard invariant (balance ≥ 0, unique seat) with a CRDT — impossible without coordination.
  • Op-based CRDTs without exactly-once, causal delivery — the operations stop commuting safely.
