P6 · Inverted index

What it is: a map term → documents containing it, so you find matches without scanning the corpus.
Reach for it when: full-text or faceted search, or lookups over a high-cardinality attribute space (tags).
The lever: tokenize/stem → posting lists with term frequency + position; TF-IDF/BM25 for ranking; immutable segments merged in background rather than in-place updates.
New problem it hands you: near-real-time gap (newly written data isn't searchable until refreshed) and distributed IDF (each shard knows only local stats, so global relevance needs a stats pass or approximation). Plus index storage/cardinality cost.
Exemplified by: Search engine, Datadog.

1 · The problem it solves

To find which documents contain "system design" you do not want to scan every document. An inverted index flips the data around: instead of document → words, it stores word → list of documents (a posting list). Lookups become a set intersection instead of a full scan.

2 · Building the index — worked

Tokenize each document, normalize/stem, and for every term store a posting list of (doc id, term frequency, positions). Three tiny docs:

d1: "the quick brown fox" d2: "the lazy brown dog sleeps all day in the brown grass" d3: "quick brown fox jumps" inverted index (term -> postings with positions): brown -> [ (d1, tf1, @2), (d2, tf2, @2,9), (d3, tf1, @1) ] fox -> [ (d1, tf1, @3), (d3, tf1, @2) ] quick -> [ (d1, tf1, @1), (d3, tf1, @0) ]
graph LR
  D[docs] --> T[tokenize + stem]
  T --> I[("term → posting list")]
  Q["query: brown fox"] --> I
  I --> R[doc ids + positions + tf]
  R --> S[score & rank]

A query brown fox is now an intersection of two short posting lists ({d1,d2,d3} ∩ {d1,d3} = {d1,d3}) plus a position check for the phrase — never a corpus scan. Positions are what let you answer “brown immediately before fox” (a phrase query).

3 · Ranking with BM25 — worked

Boolean match isn’t enough; you need the best docs first. BM25 scores each match by three intuitions: a term is worth more if it’s rare across the corpus (IDF), more if it appears often in the doc (TF, with diminishing returns), and less if the doc is long (length normalization). For brown fox over the corpus above (k1=1.2, b=0.75, avgdl=6.33):

idf(brown)=0.134 (in all 3 docs → common → low) idf(fox) =0.470 (in 2 of 3 → rarer → higher) doc1 "the quick brown fox" (len 4) -> BM25 = 0.711 doc3 "quick brown fox jumps" (len 4) -> BM25 = 0.711 doc2 "...brown...brown grass" (len 11) -> BM25 = 0.152

Notice the mechanics fall out: d1 and d3 win because they contain the rare term fox; d2 has brown twice but no fox and is long, so length normalization pushes it far down. The rare term dominates the ranking — exactly what you want.

4 · Compression & immutable segments

Posting lists are huge (every term → every doc that has it), so they’re compressed. Doc ids in a posting list are sorted, so store gaps (delta) + varint:

doc ids: 3, 7, 8, 9, 40, 41, 200 deltas: 3, 4, 1, 1, 31, 1, 159 -> small ints -> ~1 byte each (varint) vs 4-byte ids

And the index is written as immutable segments, merged in the background, never updated in place — rewriting a posting list on every doc would be far too slow. This is the same LSM-style write pattern as P17: append small segments, compact later.

5 · The new bottleneck it hands you

Near-real-time gap: a freshly written document isn’t searchable until the next refresh creates a new segment (seconds, not milliseconds). Distributed IDF: when the index is sharded, each shard knows only its local term statistics, so a globally correct relevance score needs a separate stats pass or an approximation — which is why relevance can look wrong on small/skewed indexes. Plus the index itself costs storage proportional to vocabulary × postings.

6 · Real-world example

Apache Lucene (Elasticsearch / Solr)

Lucene — the engine inside Elasticsearch and Solr — stores the index as immutable segments. New documents go into an in-memory buffer and are flushed to a new small segment on a refresh (Elasticsearch defaults to ~1 second, which is the near-real-time gap made concrete). Background merges combine small segments into bigger ones to keep query fan-out bounded and to actually purge deleted docs (deletes are just tombstones until a merge). This immutable-segment design is the direct ancestor of how Datadog indexes its tag space and how Elasticsearch scales search.

7 · Applying it in a design

Reach for an inverted index for full-text, faceted search, or high-cardinality tag lookups. Mention the gap: "writes are searchable after a ~1s refresh, and I’ll account for distributed IDF when ranking across shards."

8 · Pitfalls

  • Expecting in-place updates / instant searchability.
  • Ignoring merge cost (write amplification from constant merging).
  • Assuming local shard scores are globally comparable.
