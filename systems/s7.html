7 · Search Engine (Elasticsearch / Lucene)

Hard constraint: distributed relevance — each shard has only local term stats, so global IDF needs a separate pass or approximation.
Primitives exercised: P6, P2, P7, P12.

Inverted index construction (tokenize/stem/posting lists); TF-IDF/BM25; immutable segments + merge; near-real-time gap; sharded scatter-gather ranking with fetch-then-enrich.

1 · Requirements & scale

Functional: full-text and faceted search over a large corpus, ranked by relevance, near-real-time. Non-functional: low query latency, high indexing throughput, horizontal scale.

Back-of-envelope 1,000,000,000 docs x 5 KB = 5 TB corpus (text) inverted index ~ fraction of corpus, vocabulary-bound shards corpus / per-shard capacity (e.g. 20-50 GB/shard) query scatter to all shards, merge top-K

3 · The hard constraint, in depth

Relevance in a distributed index is the deep problem: each shard holds only local term statistics, but a correct TF-IDF/BM25 score needs global document frequency (IDF). So scores from different shards aren’t directly comparable — you either run a distributed-frequency pass, accept an approximation, or use enough docs per shard that local stats approximate global. This is why relevance looks off on small or skewed indexes.

4 · Component deep dives

Inverted index + segments (P6)

Tokenize/stem → posting lists with tf + positions; written as immutable segments, merged in the background; new docs searchable after a ~1s refresh — see P6.

Distributed query: query-then-fetch (P7)

The coordinator scatters to all shards; each returns its top-K IDs + scores; the coordinator merges to a global top-K, then fetches full documents for only those winners (P7). The slowest shard sets latency (P12).

Scoring

BM25 by default; relevance is not keyword matching but a tuned ranking of term importance, document length, and field boosts.

5 · Real-world example

Elasticsearch (Lucene)

Elasticsearch shards an index across nodes; each shard is a full Lucene index of immutable segments. A search runs Lucene’s query-then-fetch across shards as above. To tame distributed IDF, Elasticsearch defaults to per-shard scoring (good enough when shards are large and uniform) but offers a dfs_query_then_fetch mode that first gathers global term statistics across shards for accurate scoring at the cost of an extra round trip — the exact tradeoff the hard constraint predicts. Near-real-time comes from the ~1s refresh creating new segments.

6 · Tradeoffs & alternatives

More shards = more parallelism but worse tail latency and worse local-IDF approximation; fewer shards = better relevance but less scale. For pure analytics over structured data a columnar SQL engine beats search; for vector similarity you’d add an ANN index alongside the inverted index.

7 · Key talking points

  • Inverted index + immutable segments + ~1s refresh (P6).
  • Query-then-fetch: ship top-K IDs, fetch docs last (P7).
  • Distributed IDF is the relevance wall; name the dfs tradeoff.
  • BM25 ranking, not keyword matching.
  • Shard count trades scale vs relevance vs tail latency.
