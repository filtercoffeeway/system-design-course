21 · Booking / Reservation System

Hard constraint: never double-book a finite resource (a seat, a room, an inventory unit) under high concurrency — while keeping availability search fast. The split that makes it tractable: availability reads may be stale; the booking write must be linearizable.
Primitives exercised: P3 (consistency), P10 (locks/leases), P9 (idempotent confirm), P2 (inventory sharding), P4 (availability cache), P14 (saga).

"Design hotel booking / ticket-seat reservation." Inventory contention under concurrency: overbooking-prevention ladder; hold/expire TTL state machine; the double-booking race; search-vs-book consistency split.

1 · Requirements

Functional: search availability for a resource over a date/time range; place a temporary hold while the user pays; confirm the booking; cancel/modify; never sell the same unit twice. Non-functional:

The load-bearing realization: reads and writes have opposite consistency needs. Browsing "is this hotel available?" can be slightly stale (worst case the user finds it's gone at checkout). But the actual booking write is a correctness operation — two confirms for the last seat must not both succeed. Designing those two paths differently is the whole problem.

2 · Capacity estimation

hotel inventory: 1M hotels x ~5 room-types x 365 days ~ 1.8B inventory rows read:write ratio ~ 1000:1 (many browse, few book) -> cache the reads (P4) flash sale (concert): 50,000 seats, 1,000,000 users hit "buy" in the first minute -> ~833 successful bookings/s, BUT ~20:1 contention on the SAME hot rows -> the problem is not throughput; it's concurrent contention on scarce units hold TTL: 5-15 min payment window; an expirer returns un-paid holds to the pool

3 · API & data model

searchAvailability(resource, date_range, qty) -> [units] // cacheable, may be stale hold(unit_id, user, qty, idem_key) -> {hold_id, expires_at} confirm(hold_id, payment_token, idem_key) -> {booking_id} release(hold_id) / cancel(booking_id) inventory: (resource_id, unit_id, date) -> {state: available|held|booked, version, hold_id?, hold_expires?} partition key = resource_id (or unit) so a booking hits ONE partition (P2) bookings: booking_id -> {unit_id, user, dates, status, idem_key}

Partitioning by resource/unit (P2) is deliberate: a booking touches one unit, so it lands on one partition and can be made atomic there — you never need a cross-partition transaction to sell a seat. This is the "every write needs a partition key" check.

4 · The overbooking-prevention ladder (the heart)

Two users click "book the last room" within milliseconds. Both read "available," both write "booked." Now it's double-booked. Every rung below is a way to make that write sequence safe under concurrency.

Rung 0 — read-then-write (no concurrency control)

if available: set booked. Breaks immediately: the classic check-then-act race — both transactions read "available" before either writes. Double-booked.

Rung 1 — optimistic concurrency (version / compare-and-set)

Read the row with its version; write conditionally: UPDATE ... SET state='booked', version=version+1 WHERE unit_id=? AND version=? AND state='available'. Exactly one update matches; the loser sees 0 rows affected and retries/fails. No locks held. Good until: high contention on a hot unit — under a flash sale, most writers lose the CAS and retry in a storm, wasting work (though correctness holds).

A: UPDATE seat SET state=booked,version=6 WHERE id=42 AND version=5 AND state=available -> 1 row (wins) B: UPDATE seat SET state=booked,version=6 WHERE id=42 AND version=5 AND state=available -> 0 rows (version moved; loses) -> B retries, sees state=booked, gives up cleanly. No double-book.

Rung 2 — pessimistic row lock (SELECT ... FOR UPDATE)

Lock the row inside a transaction so the second writer waits rather than failing: BEGIN; SELECT ... FOR UPDATE; if available UPDATE; COMMIT. Better when contention is high and you'd rather queue than retry-storm. Breaks on: a hot row serializes all contenders (a lock convoy), lock hold time includes app latency, and it doesn't span databases — if inventory is sharded across nodes, a single DB lock can't coordinate them.

Rung 3 — distributed lock / lease (cross-node)

When inventory spans nodes, take a short lease on the unit via a consensus store (P10 / S13), with a fencing token so a slow holder can't confirm after its lease expired. Use this only when a single-DB transaction can't (genuinely distributed inventory). Breaks on: the lock service is a dependency and adds latency; over-using distributed locks where a row lock would do is a common over-engineering smell.

Rung 4 — partition the contention away

For extreme contention (a 50k-seat on-sale), reduce contention structurally: shard inventory so different units are independent rows/partitions (P2) — a million users contend across 50k independent rows, not one counter. For pooled/fungible inventory (N identical rooms), avoid a single hot counter by splitting it into per-shard sub-counters and decrementing a random shard (the inverse of S1's distributed counter); a virtual waiting room throttles the herd before it reaches inventory at all.

5 · The double-booking race, traced

Why the naive path fails, concretely, and why rung 1 fixes it:

t0 user A reads seat 42 -> available (version 5) t0 user B reads seat 42 -> available (version 5) // both saw "available" t1 A: UPDATE ... WHERE version=5 AND available -> OK, version=6 // A booked t1 B: UPDATE ... WHERE version=5 AND available -> 0 rows // version moved -> B fails t2 B retries read -> state=booked -> show "just taken", offer alternatives RESULT: exactly one booking. The conditional WHERE is the atomic check-and-set.

The principle generalizes: make the inventory mutation a single atomic conditional write (CAS, row lock, or a Redis DECR that can't go below zero), never a separate read and write — that closes the window where two readers both see availability.

6 · The search/book consistency split

Search must be fast and is read-heavy (~1000:1); booking must be correct. Forcing search to be linearizable would make browsing slow and hammer the inventory store; allowing booking to be eventually consistent would double-sell. So split them.

Availability search reads from a cached/replicated, possibly-stale view (P4, read replicas) — it's fine if a room shown as available was just taken; the user simply discovers it at the hold step. The hold/confirm write goes to the authoritative inventory partition with one of the concurrency-control rungs above — linearizable on that unit (P3). The hold step is the consistency boundary: search is best-effort; the moment money/commitment is involved, you switch to the strongly-consistent path. This is P3's "tunable consistency per operation" applied within one system.

7 · The hold / expire state machine

Users need time to pay, but a held-forever unit that's never paid for is lost inventory. So a hold is a time-boxed lease:

graph LR
  AVAIL[available] -->|hold, set TTL| HELD[held]
  HELD -->|confirm + payment| BOOKED[booked]
  HELD -->|TTL expires / release| AVAIL
  BOOKED -->|cancel| AVAIL

A hold writes state=held, hold_expires=now+15min atomically (same CAS as §4). Confirm transitions held→booked only if the hold is still valid and matches the holder. An expirer returns lapsed holds to the pool. Crucially, the expiry must be checked at confirm time, not trusted from a background sweep alone — the confirm's conditional write includes AND hold_expires > now AND hold_id = ?, so a confirm arriving after expiry fails cleanly even if the sweeper hasn't run yet (this is the P10/S12 "lease ages from grant time" check — never act on a lease you only assume is still valid).

8 · Idempotent confirm & the payment saga

Confirm involves an external payment, so it's a distributed action that can fail or be retried halfway. Two protections: (1) idempotency key (P9) on confirm so a retried request (user double-clicks, network retries after a charge succeeded) returns the existing booking rather than booking/charging twice. (2) A saga (P14) for the multi-step "reserve → charge → confirm" flow: if payment fails after the hold, release the hold (compensation); if confirm fails after charge, refund. There's no cheap distributed transaction across the inventory DB and the payment provider, so you use at-least-once steps + idempotency + compensation, exactly as S17 does for money movement.

9 · Scaling, deliberate overbooking & failure

At flash-sale scale, the levers are: a virtual waiting room that admits users to the inventory at a controlled rate (so 1M users don't hit the DB at once — P16 backpressure), inventory sharding so contention spreads across independent rows, and an in-memory authoritative inventory (Redis with atomic DECR) backed by durable writes for the hottest events. Deliberate overbooking is a twist worth naming: airlines and hotels intentionally oversell by a few percent because a predictable fraction no-shows — so the "inventory" is a business-tuned number above physical capacity, with a bump/compensation policy for the rare over-capacity event. That's a product decision layered on the correctness machinery, not a bug. On failover, the authoritative inventory must be rebuilt from durable state, and in-flight holds either persisted or allowed to expire (bounded recovery, S12 check).

10 · Tradeoffs & alternatives

Optimistic (no locks, retries on conflict) suits low contention and scales reads; pessimistic locks suit high single-node contention but serialize and don't cross shards; distributed leases handle cross-node inventory at the cost of a dependency. Longer hold TTLs improve UX but lock up inventory (and enable hold-then-flip scalping); shorter TTLs free inventory but rush users. Strong consistency on the booking path costs latency but is non-negotiable for correctness; the win is confining it to the write while serving search from cheap stale reads. Pooled/fungible inventory (rooms) uses counters; assigned inventory (seat 14C) uses per-unit rows — different contention shapes.

11 · Real-world example

Ticketmaster · Booking.com · OpenTable

High-demand ticketing (Ticketmaster-style on-sales) uses virtual waiting rooms to throttle the herd, short seat holds with TTLs during checkout, and per-seat atomic state transitions so a seat is sold exactly once — the assigned-inventory pattern. Hotel/lodging platforms (Booking.com, Expedia) keep a huge, mostly-read availability index served from caches/replicas (stale-tolerant search), and switch to a strongly-consistent reservation write at booking time, often with overbooking allowances tuned to no-show rates. Restaurant reservations (OpenTable) are the same hold→confirm machinery on table-time slots. Across all of them the spine is identical: cheap stale availability reads, one atomic conditional write per unit, a TTL'd hold, and an idempotent confirm wrapped in a payment saga.

12 · Interview follow-ups (Q&A)

Q: Two users book the last seat at the same instant — how do you guarantee one wins? Make the booking a single atomic conditional write (version CAS, SELECT FOR UPDATE, or a bounded DECR), never a separate read then write. Exactly one conditional update matches; the other gets 0 rows and is told it's taken.

Q: Optimistic vs pessimistic locking — which? Optimistic (version CAS) for low/moderate contention — no locks, just retry on conflict. Pessimistic row locks for sustained high contention on a single DB where you'd rather queue than retry-storm. Distributed leases only when inventory genuinely spans nodes.

Q: Why can search be eventually consistent but booking can't? Search is read-heavy and a stale "available" only costs a failed hold attempt; double-selling a real unit is a correctness violation. So search reads from caches/replicas while the booking write is linearizable on the unit (P3).

Q: How do holds expire without leaking inventory or double-confirming? A TTL'd hold plus a confirm whose conditional write requires hold_expires > now and a matching hold_id — so an expired hold can't be confirmed even before the background sweeper reclaims it (never trust a lease you only assume is valid).

Q: A user's payment retried and they got charged/booked twice — prevention? Idempotency key on confirm (P9): the retry returns the existing booking. The reserve→charge→confirm flow is a saga (P14) with compensations (release hold / refund) since there's no cheap cross-system transaction.

Q: 1M users for 50k seats — how do you not melt the DB? A virtual waiting room admits users at a controlled rate (P16), inventory is sharded so contention spreads across 50k independent rows, and the hottest on-sales use an in-memory atomic counter/row backed by durable writes.

Q: Is overbooking always a bug? No — airlines/hotels deliberately oversell by a few percent expecting no-shows; "capacity" is a business-tuned number with a bump/compensation policy. That's a product layer over the same correctness machinery, distinct from an accidental double-book.

13 · Key talking points

  • Core split: availability reads stale-OK; booking write linearizable on the unit (P3).
  • Overbooking ladder: read-then-write → optimistic version CAS → row lock → distributed lease+fence → partition the contention (P2, P10).
  • Make the mutation one atomic conditional write; never check-then-act.
  • Hold → TTL → confirm/expire state machine; confirm re-checks expiry (lease ages from grant time).
  • Idempotent confirm (P9) + payment saga (P14) so retries don't double-book/charge.
  • Flash sales: waiting room (P16) + sharded inventory; deliberate overbooking is a tuned business policy, not a bug.
