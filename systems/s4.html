4 · Relational Database Internals + Scaling

Hard constraint: cross-shard joins are prohibitively expensive — denormalize to avoid them, or accept fan-out-to-all-shards + merge.
Primitives exercised: P17, P2, P3 (replica lag), P7.

MVCC; index types (B-tree/hash/GIN/GiST) and why you can't index freely; read replicas + replication lag; sharding strategies + the cross-shard join problem; connection pooling as a DB resource; 2PC and why it's rarely right.

1 · Requirements

Functional: structured data with ACID transactions, joins, and secondary indexes. Non-functional: keep those guarantees as data and traffic outgrow a single node. The whole story is “how far does one box go, and what breaks first.”

say: 50,000 writes/s and 500,000 reads/s, 10 TB, growing one tuned primary: ~tens of k writes/s, low-TB working set -> we're past it -> the question is which scaling lever, in which order, and what each costs

2 · Internals you scale: MVCC, indexes, the WAL

Before scaling out, know what a single node is doing. MVCC (multi-version concurrency control) lets readers see a consistent snapshot without blocking writers: each update writes a new row version stamped with a transaction id, and a reader sees the newest version committed before its snapshot.

row "balance" updated by txns 100, 105: v1 (xmin=100) balance=50 <- a txn with snapshot <105 still reads this v2 (xmin=105) balance=70 <- a txn with snapshot ≥105 reads this old versions kept until no snapshot needs them, then VACUUM reclaims them

The WAL (P17) gives crash durability and feeds replication. Indexes (B-tree for ranges, hash for equality, GIN/GiST for text/geo) speed reads but each one is a structure every write must also update — so you can’t index every column; indexes trade write throughput and space for read speed.

3 · The scaling ladder

Rung 0 — one tuned primary

Right indexes, enough RAM for the working set. Goes a long way — tens of thousands of writes/s. Always start here.

Bottleneck: reads and writes share one node; the box has a ceiling.

Rung 1 — read replicas

Stream the WAL to replicas and send reads there. Reads scale out; the 10:1 read:write ratio means this buys a lot. But writes still go to the one primary, and replicas lag — a read-your-write off a replica can miss your own update (P3); route critical reads to the primary or wait for a WAL position.

Bottleneck: write throughput and dataset still bound by one primary.

Rung 2 — connection pooling & caching

Each Postgres connection is a real process (~MBs); thousands of app workers exhaust the DB before CPU does. A pooler (PgBouncer) multiplexes many clients onto few server connections — connection count is a DB resource. A cache (P4) absorbs hot reads. Relieves pressure, but still one primary for writes.

Bottleneck: write ceiling unmoved.

Rung 3 — functional partitioning (split by table/domain)

Move whole tables (e.g. events, users) to separate DB instances. Each gets its own write capacity — but a single table that’s too big still can’t fit, and cross-domain joins now cross a network.

Bottleneck: one hot table can still exceed a node.

Rung 4 — horizontal sharding (split rows by key) ← the wall

Partition a table’s rows across N shards by a shard key (P2). Write throughput finally scales linearly — at the cost of the hard constraint below.

5 · The hard constraint: cross-shard joins

Once you shard, a join across shards is prohibitively expensive. A join that was a local index lookup now fans out to every shard and merges over the network (P7), and is bounded by the slowest shard:

SELECT * FROM orders JOIN users ON ... , sharded by different keys: local join (1 node): 1 index lookup cross-shard over 50 shards: scatter to 50, gather + merge -> 50x the work, latency = slowest of 50 shards (P7 tail math)

You respond by denormalizing so related rows share a shard, and by choosing a shard key that co-locates what you read together (orders sharded by user_id so a user’s orders + profile live together). Some queries still scatter-gather; accept it. Cross-shard transactions need 2PC, which holds locks across shards and rarely scales — another reason to keep a transaction inside one shard.

6 · Real-world example

YouTube / Vitess

YouTube outgrew a single MySQL and built Vitess (now a CNCF project, also used by Slack, GitHub, and others) to shard MySQL horizontally while keeping it looking like one database to the app. Vitess sits in front of many MySQL shards: a VTGate proxy parses SQL, routes each query to the right shard(s) by the sharding key, scatter-gathers when a query spans shards, and manages resharding online. It is the productized answer to "how do you keep relational semantics past one node" — and it leans on exactly the cross-shard-join constraint above, steering apps toward shard-local queries.

7 · Tradeoffs & alternatives

Replicas first (cheap read scale), shard only when writes force it. If you need horizontal writes and distributed transactions, a NewSQL store (CockroachDB/Spanner, Raft-replicated ranges) buys that at higher latency. If you need massive write scale and can drop joins/ACID, a KV store fits better.

8 · Key talking points

  • Climb the ladder: tune → replicas → cache/pool → partition → shard.
  • Shard key co-locates data; cross-shard joins are the wall (P2, P7).
  • Replica lag → read-after-write care (P3).
  • MVCC for concurrency; connection pooling as a finite resource.
  • Avoid 2PC; denormalize instead.
