15 · Ride-Sharing / Location System (Uber)

Hard constraint: geo-search must be fast with a variable radius; geohash precision trades search radius against result-set size.
Primitives exercised: P15, P1 (write-heavy), P2 (hot-cell).

Geohash/quad-tree partitioning; ~1M location updates/min (write-heavy); matching as supply-demand in a cell; surge as a real-time read on that ratio; routing vs ETA (static graph shortest-path + dynamic traffic).

1 · Requirements & scale

Functional: track driver locations in real time, match a rider to the nearest available driver, price by supply/demand, route and estimate ETA. Non-functional: absorb a huge write rate of location updates and answer proximity queries fast.

Back-of-envelope 5,000,000 active drivers, update every ~4s ~ 1,250,000 location writes/s matching: nearest available within radius -> geo index lookup per request this is a WRITE-heavy + geo-query problem, not a storage problem

2 · High-level architecture

graph LR
  D[drivers] -->|location updates| LOC[location service]
  LOC --> GEO[(geo index: H3 cells -> drivers)]
  R[rider request] --> DISCO[matching / dispatch]
  DISCO --> GEO
  DISCO --> SURGE[surge: supply/demand per cell]

3 · The hard constraint, in depth

Geo-search must be fast and the radius is variable (expand until you find enough drivers). Indexing by geohash/H3 gives approximate range search, where the cell precision trades search radius against result-set size, and you must also search neighboring cells to catch drivers just across a boundary (P15). Layer on a ~1M/s update rate and a hot cell downtown and it’s a write-heavy, skew-prone problem.

4 · Component deep dives

Geospatial index (P15)

Driver positions are bucketed into H3/geohash cells; a match queries the rider’s cell plus its ring of neighbors — see P15.

Location ingest (P1/P2)

High-frequency updates are batched and written to an in-memory geo store; partition by cell, but watch for hot downtown cells (P2).

Matching / dispatch

A supply-demand problem within a cell: find nearby available drivers, rank by ETA, and offer. Uber’s dispatch (DISCO) batches requests over short windows to make globally better matches rather than greedy nearest-first.

Surge & routing

Surge is a real-time read of the supply:demand ratio per cell over the same location data. ETA/routing uses shortest-path (Dijkstra/A*) on a road graph adjusted by live traffic.

5 · Real-world example

Uber

Uber indexes the world with H3, its open-source hexagonal hierarchical grid (P15), and runs matching/surge per cell. Hexagons give uniform neighbor distances, which makes "expand the search ring" and smoothing supply/demand across adjacent cells clean. Its dispatch system batches ride requests over short time windows to optimize matches city-wide instead of greedily, and surge pricing is computed from real-time supply/demand per H3 cell. The defining trait is that it is a write-heavy real-time geospatial system, so the location index and its cells are the center of the design.

6 · Tradeoffs & alternatives

Bigger cells = fewer lookups but more candidates to filter; smaller cells = precise but more boundary handling and more hot-cell risk. Batched dispatch yields better matches at the cost of a little latency vs greedy nearest. In-memory geo store for speed vs durable store for safety.

7 · Key talking points

  • Write-heavy + geo-query, not storage-heavy.
  • H3/geohash cells; search cell + neighbor ring (P15).
  • Hot downtown cells = hot partition (P2).
  • Batched dispatch for better global matching.
  • Surge = real-time supply/demand read per cell.
