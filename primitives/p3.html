P3 · Replication & the consistency ladder

What it is: keeping copies of data for durability, availability, and read scale — and choosing how consistent those copies must be.
Reach for it when: you need to survive node loss, serve more reads, or place data near users.
The lever: single-leader (simple, follower lag), multi-leader (write-anywhere, conflicts), leaderless/quorum (Dynamo R+W>N). The consistency ladder is the dial: strong → bounded-staleness → read-your-writes → eventual. CAP: under partition pick availability (stale, AP) or consistency (error, CP).
New problem it hands you: replication lag → read-after-write anomalies. Multi-leader/leaderless → write conflicts needing version vectors or last-writer-wins. Stronger consistency → higher latency, lower availability under partition.
Exemplified by: KV store, RDBMS, Kafka (ISR), Object store.

1 · The problem it solves

A single copy of data dies with its node and serves only as fast as one machine. Replication keeps multiple copies so you survive failures, scale reads, and place data near users. The catch is that copies drift apart, so you must also choose how consistent they have to be.

2 · The replication ladder

Rung 0 — one copy

Simplest possible. Dies with its node, and serves only as fast as one machine.

Bottleneck: no durability, no read scale.

Rung 1 — single-leader, async replication

All writes go to one leader; it streams the change log to read-only followers. Buys durability (copies survive node loss) and read scale (fan reads across followers). The leader is still the single writer, and followers lag.

Bottleneck: replication lag → a read of a follower right after a write may not see it (read-after-write anomaly); writes still cap at one node.

Rung 2 — synchronous / quorum replication

Wait for replicas to acknowledge before confirming a write. No stale reads if you read a quorum — at the cost of write latency. The leaderless form (Dynamo-style) drops the leader entirely: write to W replicas, read from R, and choose R + W > N so the read and write sets must overlap. Worked, N=3:

graph LR
  W["write v2 (W=2)"] --> A["A: v2"]
  W --> B["B: v2"]
  C["C: v1 (missed it)"]
  R["read (R=2) from B,C"] --> B
  R --> C
  R --> M["see v2 & v1 → newest wins → v2"]
N=3, W=2, R=2 → W + R = 4 > 3 → write set {A,B} and read set always overlap write v2 to {A,B}; C still holds v1 read from {B,C}: gets v2 (B) and v1 (C) → version says v2 newer → return v2 ✓ if instead R=1 and you read {C}: returns stale v1 ✗ (W+R = 3, not > N)

So the quorum inequality isn’t a slogan — it’s the exact condition that the nodes you read are guaranteed to include one that saw the latest write.

Bottleneck: higher latency (wait for a majority), and the leaderless/multi-leader forms admit write conflicts (next section).

3 · The consistency ladder (the dial)

Orthogonal to where writes go is how consistent reads must be — a dial, not a switch, from most expensive to loosest: strict serializable → linearizable → bounded staleness → read-your-writes → eventual. Pick the loosest rung the requirement tolerates. CAP: under a network partition you must drop either consistency (serve stale, AP) or availability (return an error, CP). PACELC adds the part everyone forgets — even with no partition (E, “else”) you trade latency vs consistency on every request.

4 · Conflicts & version vectors

The moment two replicas can both accept a write (multi-leader or leaderless), they can diverge. A version vector — a per-replica counter — detects whether two versions are causally ordered or genuinely concurrent:

start: cart = {} , vv = [A:0, B:0] A adds milk: value={milk}, vv=[A:1, B:0] B adds eggs: value={eggs}, vv=[A:0, B:1] (B didn't see A's write) compare [A:1,B:0] vs [A:0,B:1]: neither dominates → CONCURRENT → conflict resolution: merge ({milk,eggs}) or last-writer-wins (one silently lost)

Last-writer-wins is simplest but silently drops one update; a mergeable type (a CRDT, like a set-union cart) keeps both. The point is you can’t avoid the choice once two writers exist.

5 · The new bottleneck it hands you

Replication lag is the headline cost: read a follower right after writing the leader and you may not see your own write (read-after-write anomaly). Fixes — route a user’s reads to the leader for a few seconds, or track a version and wait for a replica to catch up. Multi-leader/leaderless introduce write conflicts needing version vectors or last-writer-wins (which silently drops data). Stronger consistency costs latency and availability: a quorum write waits for a majority, and a linearizable system can’t serve a minority partition.

6 · Real-world example

Amazon DynamoDB & Google Spanner

DynamoDB exposes the dial directly: a read can be eventually consistent (cheaper, may be stale, hits any replica) or strongly consistent (reads the leader replica, costs more). Its writes are quorum-replicated across three Availability Zones. At the opposite end, Google Spanner buys external consistency (linearizable, globally) by using TrueTime — GPS- and atomic-clock-synchronized time with a bounded uncertainty interval — and deliberately waiting out that uncertainty on commit. The contrast is the whole ladder: DynamoDB lets you pay less for staleness; Spanner pays latency for global strong consistency.

7 · Applying it in a design

State the requirement first, then pick the rung: "this counter can be eventually consistent, so I’ll read from any replica; this balance must be read-your-writes, so I’ll pin the user to the leader or wait for a version."

8 · Pitfalls

  • Defaulting to "strong consistency everywhere" and eating needless latency.
  • Last-writer-wins silently losing concurrent updates.
  • Reading your own writes from a lagging replica.
