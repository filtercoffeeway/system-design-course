10 · Chat System (WhatsApp / Slack)

Hard constraint: routing — when A messages B, how does A's server find which of 10k connection servers holds B's socket?
Primitives exercised: P13, P4 (routing lookup), P1 (offline delivery).

WebSocket/SSE; stateful connection servers; per-conversation sequence numbers (global ordering impossible and unnecessary); delivery receipts as a separate event stream; presence as high-frequency aggregation; client-side dedup on retry.

1 · Requirements & scale

Functional: 1:1 and group messaging, delivery + read receipts, presence, history, offline delivery. Non-functional: real-time push, ordered per conversation, durable, at huge connection counts.

Back-of-envelope 2,000,000,000 users, say 400M concurrent online connections per gateway (efficient stack) ~ on the order of 1M+ per server => hundreds of gateway servers just to hold sockets presence heartbeats hundreds of millions/min -> aggregation problem

3 · The hard constraint, in depth

Routing. When A sends to B, A’s gateway must find which of thousands of gateway servers currently holds B’s socket. That requires a fast, always-current registry (user_id → gateway) updated on every connect and disconnect — millions of churn events. If B is offline, the message must be stored and delivered on reconnect. Connection state, not the message table, is the scaling problem (P13).

4 · Component deep dives

Connections & routing (P13)

Persistent WebSockets on stateful gateways; a Redis registry for routing; rebalance and re-register on gateway failure — see P13.

Ordering & delivery

Per-conversation sequence numbers (global ordering is impossible and unnecessary). Delivery state (sent/delivered/read) is a separate event stream, not embedded in the message. Client-side dedup handles retried sends (P9).

Storage & offline

Messages persisted (store-and-forward) so offline users get them on reconnect; some systems (WhatsApp historically) delete server copies once delivered.

Presence

Online/last-seen is a high-frequency aggregation of heartbeats — expensive at scale, often approximate.

5 · Real-world example

WhatsApp

WhatsApp famously served tens of millions of users with a tiny team by building on Erlang/OTP running on FreeBSD — a runtime designed for millions of lightweight concurrent processes, one per connection. They reported pushing a single server to very high simultaneous connection counts (on the order of a million-plus), which is the whole game for chat: the scarce resource is open sockets and the routing between them, not message storage. Messages were store-and-forward and deleted from servers after delivery. Slack and Discord solve the same routing/presence problem with a gateway tier (P13).

6 · Tradeoffs & alternatives

WebSocket (bidirectional) vs SSE/long-poll (simpler, one-way). Keeping vs deleting server-side history trades storage/cost vs multi-device sync. Strong per-conversation ordering is cheap; global ordering is not and isn’t needed.

7 · Key talking points

  • Persistent connections; the scarce resource is sockets (P13).
  • Routing registry updated on every connect/disconnect.
  • Per-conversation sequence numbers; receipts as a separate stream.
  • Store-and-forward for offline; client dedup (P9).
  • Presence as high-frequency aggregation.
