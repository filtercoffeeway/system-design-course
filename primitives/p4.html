P4 · Caching strategies

What it is: a fast tier in front of a slow one, exploiting locality.
Reach for it when: reads dominate writes and read latency is the complaint (the URL shortener is 1000:1 reads:writes — a caching problem in a storage costume).
The lever: placement (client, edge/CDN, app-tier, DB buffer pool); write policy — cache-aside (lazy), write-through (consistent, slower writes), write-back (fast writes, crash risk); eviction (LRU/LFU/TTL).
New problem it hands you: invalidation (stale data when the source changes) and stampede / thundering herd (a hot key expires, N requests hit origin at once — mitigate with coalescing, early refresh, locks). Plus cold-start misses and memory cost.
Exemplified by: URL shortener, Video (CDN), Social feed, Chat, KV (Bloom filter).

1 · The problem it solves

The cheapest request is the one you don’t make. A cache puts a fast tier in front of a slow one (memory in front of disk, an edge node in front of an origin) and exploits locality: a small set of hot items serves most reads. The URL shortener is 1000:1 reads:writes — it is a caching problem wearing a storage costume.

2 · How it works (cache-aside)

graph LR
  A[app] -->|1 get| C[(cache)]
  C -->|hit| A
  A -->|2 miss -> read| DB[(database)]
  DB --> A
  A -->|3 set| C

The dominant pattern is cache-aside: app checks cache, on miss reads the DB and populates the cache. Write-through writes cache and DB together (consistent, slower writes). Write-back writes cache now and DB later (fast, risky on crash). Eviction is LRU/LFU/TTL.

3 · The new bottleneck it hands you

Invalidation — one of the two hard problems in CS — is keeping the cache honest when the source changes; a stale entry serves wrong data until it expires or is purged. Stampede / thundering herd: a hot key expires and thousands of requests miss simultaneously and crush the origin. Defenses: request coalescing (one miss fills, others wait), probabilistic early expiration, and leases. Plus cold-start misses and the memory cost of the cache itself.

4 · Real-world example

Facebook (Memcache)

Facebook’s "Scaling Memcache at Facebook" (NSDI 2013) is the canonical caching war story. Two of their fixes map straight to the bottlenecks above. Leases attack both stampedes and stale sets: on a miss, memcache hands exactly one client a lease token to recompute the value; other clients wait briefly rather than all hitting the DB, and a stale set whose lease was invalidated is rejected. They also run pools (separating high-churn keys from stable ones so eviction patterns don’t interfere) and regional replication with a careful invalidation pipeline driven off the database commit log. The lesson: at scale, the cache’s hard problems are invalidation and stampede, not hit rate.

5 · Applying it in a design

Reach for a cache the moment reads dominate and a hot set exists. Name the tradeoff: "cache-aside with a short TTL; I’ll coalesce misses to prevent stampede and accept bounded staleness." If you can’t tolerate staleness, move to write-through and invalidate on write.

6 · Pitfalls

  • No stampede protection on hot keys.
  • Caching with no invalidation plan ("we’ll just TTL it" hides bugs).
  • Caching user-specific data with a shared key (leaks across users).
