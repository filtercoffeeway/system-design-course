System Design Curriculum

A bottleneck-first, composable toolkit

The goal is not to memorize 50 problems. It is to build a mix-and-match toolkit: given any system — including one you've never seen — you locate the bottleneck, reach for the right primitive, and reason about the new bottleneck that primitive introduces. The 18 named systems are case studies that prove each primitive in a real domain.

A system is (dominant bottleneck) + (the primitives you compose to relieve it) + (the secondary bottlenecks that composition creates).

How to use this site

Three parts, indexed in the order you actually work through a design: Part I (the catalog below) is the diagnosis layer; Part II is one page per composable primitive; Part III is one page per system case study. Each primitive and system page has a Full write-up placeholder for you to expand.

Part I — Diagnosis

The Bottleneck Catalog

Read it as: you observe a symptom → you have candidate levers → each lever hands you a new problem to defend. No lever is free. Naming the cost you're accepting is the move that matters.

Symptom (the bottleneck)Candidate leversNew bottleneck each lever introduces
Write throughput too highBatch/async buffer; log-structured (LSM) writes; partition the write key; pre-aggregate at sourceLost ordering; partition skew (hot partition); compaction cost; loss of raw detail
Read latency too highCache; read replica; denormalize; precompute (fan-out on write); CDN/edgeInvalidation + stampede; replication lag; write amplification + storage blowup; edge staleness
One write fans out to many readersFan-out on write (push); fan-out on read (pull); hybrid by node degreeStorage linear in followers + hot-user (push); slow reads (pull); two code paths (hybrid)
A single key/partition is hotSplit the key; request coalescing/dedup; dedicated cache replica; hybrid fan-outRecombination logic; cache consistency; detecting hot keys in real time
Cross-shard joins / queriesDenormalize; co-locate by shard key; scatter-gather then mergeWrite amplification; rigid access patterns; fan-out reads + tail latency
Full scan over huge datasetColumnar storage; predicate/projection pushdown; partition pruning; MPPShuffle becomes the bottleneck; planning complexity; bad plans on bad stats
Search/lookup over a large corpusInverted index; secondary indexes; materialized viewsNear-real-time gap; distributed IDF problem; index storage cost
Storage cost growing unboundedTiered storage; erasure coding vs replication; compression; retention/TTLCold-tier latency; reconstruction cost; (de)compression CPU; data gone after TTL
Need coordination / a single leaderConsensus (Raft/Paxos); distributed lock; leader electionQuorum-majority latency; slow-but-alive holder → need fencing tokens
Need exactly-once / no double effectIdempotency keys; transactional dedup table; saga + compensationCan't guarantee across external systems; reconciliation cost
Real-time push to clientsPersistent connections (WebSocket/SSE); stateful connection serversConnection-state routing; reconnection/dedup
Tail latency (p99/p999) too highHedged requests; aggressive timeouts; load shedding; backpressureWasted duplicate work; dropped requests; upstream must handle "slow down"
Geo / proximity queriesGeohash / quad-tree bucketingRadius-vs-result-set precision tradeoff; boundary cells
Producer outpaces consumerBackpressure; bounded buffers; drop/sample; durable queueData loss (drop); unbounded queue lag; latency grows

How to drive it: state the dominant bottleneck → pick a lever → name the new bottleneck it creates → decide whether that one is acceptable or needs its own lever. That chain is the design.

Recommended study sequence

Learn the primitives in the order that maximizes transfer; pull in each system as the case study where that primitive is the star. After steps 1–9 you hold every transferable primitive.

  1. Rate limiter → P16, P3: the gentle on-ramp.
  2. Kafka → P1, P2, P16, P17: the decoupling buffer at the center of everything.
  3. Key-value store → P17, P2, P3: storage internals.
  4. RDBMS scaling → P17, P2, P3, P7: sharding, MVCC, replica lag.
  5. Object store → P17, P3, P8: blob storage + erasure coding.
  6. Datadog → P1, P6, P7, P8, P5, P16: the full ingestion pipeline + TSDB.
  7. Social feed → P5, P2, P4: fan-out and the write/read cost dial.
  8. Search engine → P6, P7, P12: inverted index + distributed ranking.
  9. Distributed SQL → P8, P7, P12: MPP and shuffle minimization.
  10. Chat → P13, P4: real-time connections and routing.
  11. Notification → P9, P5, P13: multi-channel delivery + idempotency.
  12. Task scheduler → P9, P10: at-least-once + distributed work.
  13. Distributed lock → P10: consensus and fencing.
  14. URL shortener → P4, P11: read-heavy caching.
  15. Ride-sharing → P15, P2: geospatial indexing + hot cells.
  16. Video streaming → P11, P8, P1: CDN + encoding pipeline.
  17. Payment → P14, P9, P3: saga + distributed correctness.
  18. Web crawler → P2, P1, P4: distributed graph traversal.
