P12 · Tail latency: hedging & load shedding

What it is: defending p99/p999 when the average is fine but the worst case isn't — especially in any scatter-gather where the slowest of N dominates.
Reach for it when: fan-out reads, SLO-bound APIs, or "median is fast but users complain."
The lever: hedged/backup requests (send a second after a delay, take the first back); tight timeouts + retry budgets; load shedding (reject early when overloaded); request prioritization.
New problem it hands you: hedging wastes duplicate work and can amplify load under stress; shedding drops real requests; aggressive timeouts can turn a slow success into a failure.
Exemplified by: Distributed SQL, Search, Chat — anywhere P7 is in play.

1 · The problem it solves

In a request that fans out to many services or shards, the average is a lie: if each backend is usually fast but occasionally slow, the overall request is as slow as the slowest of N. At fan-out 100, a 1-in-100 slow event hits almost every request. Tail latency (p99, p999) is the real user experience.

2 · How it works

Levers: hedged requests (send the request, and if no reply within, say, the p95, send a duplicate to another replica and take whichever returns first); tied requests (let replicas cancel each other once one starts); aggressive timeouts + retry budgets; load shedding (reject early when overloaded so you fail fast instead of degrading everyone); and request prioritization.

3 · The new bottleneck it hands you

Hedging wastes work (duplicate requests) and, if naive, amplifies load exactly when the system is already stressed — so hedge only above a high percentile and cap the extra load. Shedding drops real requests. Aggressive timeouts can turn a slow success into an outright failure.

4 · Real-world example

Google

Dean & Barroso’s "The Tail at Scale" (CACM 2013) is the foundational account and comes from Google’s own serving systems. Their hedged requests technique — send to a second replica only after the first has missed the 95th percentile — reportedly cut the 99.9th-percentile latency of a benchmark read service dramatically while adding only a couple percent more total requests. The companion idea, tied requests, lets the two copies cancel each other so you rarely pay for both. It’s the canonical proof that you fight the tail with redundancy, not just faster code.

5 · Applying it in a design

Whenever you scatter-gather (P7) or have an SLO: "I’ll hedge above p95 to a second replica and shed load past capacity, accepting a few percent extra requests to protect p99.9."

6 · Pitfalls

  • Hedging from the first millisecond (doubles load).
  • Retrying into an already-overloaded system (retry storms) — use budgets + backoff.
  • Optimizing the median while p99 quietly kills UX.
