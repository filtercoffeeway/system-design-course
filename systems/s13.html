13 · Distributed Lock / Leader Election (ZooKeeper / etcd)

Hard constraint: the slow-but-alive holder (GC pause, network blip) — no TTL value can make a bare lock correct, because the holder can wake up after expiry still believing it holds the lock. Fencing tokens can.
Primitives exercised: P10 (consensus/fencing), P3 (quorum), P9 (idempotency).

Raft/Paxos + majority quorum for the lock state; fencing tokens (monotonic, reject stale) for correctness; session/heartbeat TTL for death detection; leader election as "a lock whose value is the leader's identity"; watch/notify instead of polling.

1 · Requirements

Functional: let independent processes coordinate — acquire a mutual-exclusion lock, release it, and elect a single leader — correctly despite crashes, GC pauses, and network partitions. Support a watch so a waiter learns of release without polling. Non-functional:

The defining stance: safety over liveness. It is acceptable for the lock to be briefly unavailable (no one can acquire during a leader election); it is never acceptable for two clients to believe they hold it.

2 · Capacity & the shape of the problem

data: a handful of small keys (lock name -> {owner, token, lease_until}); KB, not GB load: acquires/releases are rare vs reads; the bottleneck is CORRECTNESS, not QPS quorum: cluster 3 -> majority 2, tolerates 1 failure cluster 5 -> majority 3, tolerates 2 failures <- common cluster 7 -> majority 4, tolerates 3 failures acquire: ~1 RTT to a majority (~1-10 ms same region; cross-region is painful)

This system is unusual: throughput is almost irrelevant; the entire design exists to be correct under the worst partial-failure timing. So we derive the lock by escalating against failure modes, not against load.

3 · API design

acquire(lock_name, ttl) -> { granted: bool, token: int } // token = fencing token release(lock_name, token) -> ok | rejected(stale token) renew(lock_name, token, ttl) -> ok | lost // heartbeat the lease watch(lock_name) -> stream of {released, acquired_by} // leader election is the same primitive: campaign(election, candidate_id) -> { leader: bool, term: int } // term IS the fence

4 · Deriving the lock (ladder over failure modes)

Rung 0 — a row in a single database (SELECT ... FOR UPDATE)

Acquire = insert a row (lock_name, owner) with a unique constraint; release = delete it. Correct as long as the DB is up. Breaks on: the DB is a single point of failure — if it dies, all coordination stops; and if the holder crashes without deleting the row, the lock is held forever (no expiry).

Rung 1 — a single Redis key with SET NX PX (TTL)

SET lock owner NX PX 10000 — set only if absent, auto-expire in 10 s, so a crashed holder's lock self-frees. Fast (one node, sub-ms). Breaks on two things: (a) single Redis is a SPOF, and a replica failover can lose the key (async replication) → two holders; (b) the slow-but-alive holder — the core hazard below — which no TTL value fixes.

The central hazard: slow-but-alive. Client A acquires with a 10 s TTL and a token. A then suffers a 12 s GC pause (or its network drops). The lease expires; client B legitimately acquires. Then A wakes up still believing it holds the lock and writes to the protected resource. Two writers, corruption. Making the TTL longer just widens the window where a truly-dead holder blocks everyone; making it shorter makes false expiry more likely. TTL tuning trades one failure for another — it cannot be correct.

Rung 2 — fencing tokens (the correctness fix)

The lock service hands out a monotonically increasing token with every grant. The protected resource (the DB, the file store) records the highest token it has seen and rejects any operation carrying a lower token. Now the slow holder is harmless:

A acquires -> token 33 ; A pauses (GC) ; lease expires B acquires -> token 34 ; B writes (resource records highest = 34) A wakes, writes with token 33 -> 33 < 34 -> REJECTED

This is P10's fencing token. Crucially, fencing requires the resource to cooperate (check the token). A lock the resource doesn't fence is only an optimization, never a safety guarantee. In Raft-based systems the term number serves as the fence; in ZooKeeper the znode's zxid/version does.

Rung 3 — quorum-replicated lock state (the availability fix)

Put the lock state behind a consensus group (Raft/Paxos) of 3 or 5 nodes so it survives minority failure and never loses a committed grant on failover. An acquire is a consensus write committed by a majority; because a majority must agree, two clients cannot both be granted. Combined with fencing, this is the production answer: ZooKeeper, etcd, Chubby.

5 · High-level architecture

graph TD
  C1[client 1] -->|acquire| Q[(coordination service: Raft/Paxos 3 or 5 nodes)]
  C2[client 2] -->|acquire / watch| Q
  Q -->|grant + fencing token 34| C1
  Q -->|blocked; watch fires on release| C2
  C1 -->|op carries token 34| R[(protected resource: checks token)]
  R -->|reject token < highest_seen| C1

A small consensus-backed service holds the lock state on a quorum. Clients acquire via ephemeral keys tied to a session, carry the returned fencing token to the protected resource, and watch for release rather than polling.

6 · Sessions, ephemeral nodes & watches

Death detection is a lease: a client holds a session with a TTL (say 10 s) and sends heartbeats (every ~3 s); miss enough and the service declares the session dead and releases its locks. The lock is an ephemeral node — it exists only while the session lives, so a crashed holder is auto-released without anyone polling.

Watches over polling: thousands of waiters polling "is it free yet?" would hammer the cluster. Instead each waiter registers a one-shot watch and is notified the instant the lock frees. To avoid a thundering herd (all waiters wake and race), ZooKeeper's recipe has each waiter create a sequential ephemeral node and watch only the node just ahead of it — release wakes exactly one successor (a fair queue), not everyone.

New bottleneck — the lease/heartbeat is itself the slow-but-alive trap. Session expiry is just a distributed TTL, so it has the same window as rung 1. That's why fencing is not optional even with a consensus service: the session TTL gives liveness (eventually free a dead client); the fencing token gives safety (a late client can't corrupt). You need both.

7 · Leader election = a lock with a value

Leader election is the same primitive: candidates contend for one key; the winner writes its identity as the value and holds it via session. Followers watch the key and re-campaign when it frees. The Raft term (or ZAB epoch) is the fencing token: every action the leader takes is stamped with its term, and replicas reject messages from an older term, so a deposed-but-unaware leader can't act. This is exactly how Kafka's controller, HBase's master, and Kubernetes components elect via etcd.

8 · The Redlock controversy (in depth)

Redlock is a proposed algorithm to make Redis locks safe by acquiring the same lock on N independent Redis nodes (e.g. N=5) and considering it held only if a majority (≥3) grant it within the TTL. Martin Kleppmann's critique — the canonical interview talking point — is that this still doesn't give safety, for two reasons:

  • It relies on bounded clock drift and bounded pauses. Redlock's correctness argument assumes each node's clock advances at roughly the same rate and that a process can't pause longer than the TTL. A GC pause or VM freeze longer than the TTL reintroduces the exact slow-but-alive double-holder — the same flaw as a single lock.
  • It has no fencing token. Without the resource rejecting stale tokens, no lock algorithm can be safe against a paused holder. So the fix isn't "more Redis nodes"; it's fencing.

Antirez (Redis's author) countered that Redlock is fine for its intended use (efficiency locks, where occasional double-execution is merely wasteful, not corrupting). The settled interview answer: for efficiency, a single Redis lock is fine; for correctness, you need a consensus store and fencing tokens at the resource. Redlock cemented fencing as the non-negotiable piece.

9 · Scaling & bottlenecks

A consensus group does not scale writes by adding nodes — more nodes mean a larger quorum and slower commits, so you size it for fault tolerance (5 nodes), not throughput. To handle more locks, shard the lock namespace across multiple independent consensus groups (each lock name hashes to a group). Keep the cluster in one region; cross-region quorum latency (tens of ms per acquire) is usually unacceptable, so prefer one region as the coordination authority. Reads of lock state can be served from a local follower (with a staleness caveat) or via a linearizable read (ReadIndex) when correctness needs it.

10 · Tradeoffs & alternatives

A consensus service is the safe choice but adds an operational dependency and quorum latency; a single Redis lock is fast but only an efficiency lock unless the resource fences. Coarse-grained locks (one leader owns a whole shard) are simpler and usually better than many fine-grained locks — prefer leader-election + single-writer over distributed mutual exclusion whenever you can, because then there is no contention to coordinate at all. The deepest tradeoff is philosophical: a distributed lock is a liveness/safety bargain, and the only way to win it is to push the final safety check (the fence) down to the resource that actually matters.

11 · Real-world example

Google Chubby · ZooKeeper · etcd

Google's Chubby (Paxos) is the original coarse-grained lock + naming service much of Google coordinates through — e.g. electing a master for GFS/Bigtable; its designers note it's used far more for election and naming than fine-grained locking. ZooKeeper (ZAB) brought the model to open source: Kafka historically used it for controller election, HBase/Hadoop for coordination, via ephemeral sequential nodes + watches. etcd (Raft) is the coordination store and leader-election backbone of Kubernetes — component leaders renew a lease key, and the Raft term fences a deposed leader. The cautionary tale across all three is the Redlock debate: a lock without a fencing token isn't safe under GC pauses, which is why every serious system pairs the consensus grant with a monotonic fence at the resource.

12 · Interview follow-ups (Q&A)

Q: Why isn't a longer TTL enough to fix the slow holder? A longer TTL just widens the window where a genuinely-dead holder blocks everyone (hurts liveness), and the paused-holder race still exists for any pause longer than the TTL. The race is inherent to time-based leases; only a fencing token checked at the resource eliminates it.

Q: Who actually enforces the fence? The protected resource must record the highest token it has seen and reject lower ones. If the resource can't be modified to do that, you cannot have a safe lock — you only have an efficiency lock. This is the most-missed point.

Q: Why an odd number of nodes (3, 5)? Majority quorum needs a strict majority; an even cluster (e.g. 4) tolerates the same failures as the next-lower odd (1) while costing an extra node and being more prone to split votes. 5 nodes tolerate 2 failures — the usual sweet spot.

Q: What happens during a leader election — can two leaders exist? Briefly there may be no leader (unavailable), but never two that can act: a new leader gets a higher term, and the old leader's writes are rejected by replicas as stale-term. Availability dips; safety holds.

Q: A client's session expires while it's mid-operation — now what? It must stop and re-acquire; any in-flight op still carries the old (now-stale) token and will be fenced out at the resource. Make operations idempotent (P9) so a re-acquire-and-retry is safe.

Q: Lock vs lease vs leader election — when each? Use leader election when one process should own a whole responsibility (a single-writer shard) — cheapest and contention-free. Use a lock/lease for short critical sections over shared state. Avoid many fine-grained locks; they multiply the coordination you have to get right.

13 · Key talking points

  • Use a consensus service (etcd/ZooKeeper/Chubby), don't DIY (P10); safety over liveness.
  • The core hazard is the slow-but-alive holder; no TTL value fixes it.
  • Fencing tokens (monotonic, rejected-if-stale at the resource), not TTL tuning, give correctness.
  • Quorum replication makes the lock state fault-tolerant; the fence makes use of the lock safe — you need both.
  • Sessions + ephemeral nodes for death detection; watches (with sequential-node queueing) over polling to avoid thundering herds.
  • Redlock: fine as an efficiency lock; not safe without fencing. Prefer coarse leader election to fine-grained locks.
