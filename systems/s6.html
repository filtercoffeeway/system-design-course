6 · Distributed Log + Metrics Platform (Datadog)

Hard constraint: cardinality — the number of unique tag combinations, not the point rate, drives cost and scalability.
Primitives exercised: P1, P6, P7, P8, P5, P16.

Agent-side batching + pre-aggregation; Gorilla compression; inverted index on tag space; write-time rollup vs read-time aggregation; index-everything vs archive-and-index-selectively; immutable time-partitioned blocks; scatter-gather MPP with pushdown.

1 · Requirements & scale

Functional: ingest metrics, logs, and traces from millions of hosts; let users query and aggregate them ad hoc ("p99 latency of checkout by region over the last 4 hours"), build dashboards, and fire alerts on thresholds. Non-functional: ingest must never block or drop under load more than gracefully; queries over recent data should feel interactive (sub-second to seconds); retention spans from high-resolution recent data to cheap long-term rollups.

Back-of-envelope (metrics path) 2,000,000 hosts × 200 metrics/host = 4×10^8 active series each series: 1 point / 10s = 4×10^7 points/sec ingested naive: 1 UDP packet per app metric call = 10^9+ calls/sec at the source (!!) -> source pre-aggregation collapses this to ~ points/sec, not calls/sec storage: 4×10^7 pts/s × 16 B/pt (naive) ≈ 640 MB/s, ~55 TB/day raw -> Gorilla compression ~ 1.3 bytes/point => ~10× smaller on disk

Two numbers decide the architecture. The first is that raw event rate at the source is ~1000× the point rate — so you must aggregate before the wire (P1). The second is that what actually blows up cost is not points but cardinality: every distinct combination of tags is a separate series to index and store.

2 · High-level architecture

  KQ --> RU[Rollup + index workers]

  Q[Query service] --> HOT
  Q --> COLD
  U[User / dashboard / alert] --> Q

Data is aggregated twice at the source, buffered durably in a log, rolled up and indexed by tag, written to a hot time-series store for recent data and tiered to cheap object storage for history. Queries scatter across the storage shards, aggregate partials, and gather.

3 · The hard constraint, in depth: cardinality

A metric like request.latency isn't one series — it's one series per unique tag combination: {service, host, endpoint, status, region, version, ...}. Ten tags with ten values each is 10^10 potential series. The point rate might be modest, but each new series needs an index entry, its own compressed block, and memory in the write path. High-cardinality tags (user_id, request_id, full URL) are the classic way to accidentally turn a cheap metric into a multi-million-series monster.

This is why the design separates metrics (low-cardinality, indexed, pre-aggregated) from logs/traces (high-cardinality, archived, indexed selectively). It's also why naive "just put it in a SQL table with a tag column" fails: the index on the tag cross-product is the cost, and it grows combinatorially.

4 · Component deep dives

Ingestion: aggregate at the source (P1)

The DogStatsD client aggregates in-process (counters summed, gauges last-write, distributions accumulated) and flushes every ~2 s. The Agent's embedded DogStatsD server aggregates across all processes on the host over a 10 s flush interval, then compresses and batches to intake. See P1 for the full mechanism — this is the layer that makes the 1000× event-to-point reduction real, and it's the part of the Datadog story you specifically wanted captured.

Percentiles you can merge: DDSketch

Averages merge trivially; percentiles don't. To compute a global p99 from per-host data you'd normally need the raw samples. Datadog builds a DDSketch per series at the source — a fixed-size, fully mergeable quantile sketch with a relative-error guarantee (e.g. 1%). Per-host sketches merge on the backend into a global sketch, so distribution metrics cost a bounded amount regardless of sample count.

Compression: Gorilla for time series

Consecutive points in a series have near-identical timestamps and slowly-changing values. Gorilla-style compression (delta-of-delta on timestamps, XOR on float values) shrinks a 16-byte point to roughly 1–2 bytes amortized — an order of magnitude — which is what makes keeping months of data economical. This is the storage half of P8.

Indexing the tag space: inverted index (P6)

To answer "all series where service=checkout AND region=eu", you keep an inverted index from each tag value to the set of series IDs that carry it. A query intersects posting lists (service=checkout ∩ region=eu) to find matching series, then reads their compressed blocks. Cardinality is precisely the size of this index.

Write-time vs read-time rollup (P5)

You can roll up at write time (precompute 1-min, 1-hour aggregates as data arrives — fast reads, fixed granularities, more storage) or at read time (store fine points, aggregate on query — flexible, slower, cheaper to store). Real systems do both: hot recent data stays fine-grained for flexibility; older data is rolled up to coarser resolution and tiered to object storage. That's the P5 dial plus P8 tiering.

Query: scatter-gather MPP (P7)

A query like "p99 by region, last 4h" fans out to every storage shard holding relevant series, each shard decodes its blocks, applies the time filter and a partial aggregation (merging DDSketches, summing counters), and returns small partials that the coordinator merges. Pushing the aggregation down to the data — not shipping raw points to a central node — is the P7 pattern, and the slowest shard sets the latency (P12).

Backpressure (P16)

When intake or downstream lags, the durable Kafka buffer absorbs the burst; if it keeps growing, the Agent must shed — sample, drop, or slow down — rather than fall over. The pipeline is built to degrade, not collapse. See P16.

5 · Real-world example

Datadog

The two-tier source aggregation above (DogStatsD client → Agent, 10 s flush, DDSketch for distributions) is exactly how Datadog keeps cost proportional to unique series instead of raw event volume. The deliberate split between metrics (pre-aggregated, low-cardinality, always indexed) and logs (high-cardinality, archived to object storage, with their "Logging without Limits" model of decoupling ingestion from indexing so you choose what to index after the fact) is the productized form of the cardinality constraint: it lets customers send everything cheaply and pay the index cost only where they need search.

Other telemetry stacks make the same moves under different names: Prometheus pulls and stores locally with its own TSDB and warns hard about high-cardinality labels; Facebook's original Gorilla paper is the source of the compression scheme; M3 (Uber) and Mimir/Thanos scale the Prometheus model with object-storage tiering.

6 · Tradeoffs & alternatives

If you needed arbitrary high-cardinality slicing (per-user metrics), you'd stop pre-aggregating and move toward a columnar analytics engine over raw events — far more flexible, far more expensive, slower queries. If you needed exact percentiles rather than 1%-relative, you'd ship more data or accept higher cost. If retention mattered more than query flexibility, you'd roll up harder at write time and tier sooner. The whole design is a set of dials between cost, query flexibility, and resolution.

7 · Key talking points

  • Lead with the hard constraint: "the cost driver is cardinality — unique tag combinations — not point rate."
  • Aggregate at the source in two tiers; ship summaries, not events (P1).
  • Use mergeable sketches (DDSketch) so percentiles combine across hosts.
  • Time-series compression (Gorilla) + immutable time-partitioned blocks (P8).
  • Inverted index on tags; intersect posting lists to resolve queries (P6).
  • Scatter-gather with pushdown aggregation; merge partials (P7).
  • Separate metrics from logs; index logs selectively to contain cardinality.
