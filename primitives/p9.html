P9 · Idempotency & "exactly-once-ish"

What it is: making a repeated operation safe so retries don't double-apply.
Reach for it when: any at-least-once delivery, retry, or external side effect (charge a card, send a push).
The lever: idempotency key carried by the client; a dedup table written transactionally with the effect (key, result); idempotent worker logic so re-execution converges.
New problem it hands you: no true exactly-once across systems you don't control (FCM, Stripe, the bank) — settle for at-least-once + idempotent + reconciliation. The dedup table is itself state to scale and expire.
Exemplified by: Task scheduler, Notification, Payment.

1 · The problem it solves

Networks retry. A client times out and resends; a queue redelivers; a worker restarts mid-task. If the operation has a side effect (charge a card, send a push, decrement inventory), a retry must not apply it twice. Idempotency makes "do it again" safe.

2 · How it works

graph TD
  C[client sends op + idempotency key] --> S{key seen?}
  S -->|no| E[execute + store key,result in same txn]
  S -->|yes| R[return stored result, do not re-execute]
  E --> R

The client supplies an idempotency key. The server records (key → result) in a dedup table, written transactionally with the effect so the record and the side effect commit together. A repeat of the same key returns the stored result instead of re-running. Workers are written so that re-execution converges to the same state.

3 · The new bottleneck it hands you

You cannot get true exactly-once across systems you don’t control (a payment processor, FCM, a bank) — you can’t know if their side committed. The honest target is at-least-once + idempotent + reconciliation (a later batch compares your records with theirs). The dedup table is itself state you must scale and expire, and the key must be chosen so genuinely-distinct requests don’t collide nor genuine retries diverge.

4 · Real-world example

Stripe

Stripe’s API takes an Idempotency-Key header on every mutating request. Stripe stores the first response keyed by it and replays that exact response for any retry with the same key, so a network blip during a charge can be retried safely without double-charging. Keys are scoped and expire after 24 hours. Stripe’s own engineering writing frames idempotency keys as the foundation of "robust and predictable APIs," and it’s the same mechanism that underpins their payment correctness.

5 · Applying it in a design

Anywhere you have retries or external effects: "client-supplied idempotency key, dedup record written in the same transaction as the effect, at-least-once delivery, reconcile against the provider." Pair it with at-least-once queues (Kafka, task scheduler) so you never need true EOS.

6 · Pitfalls

  • Storing the dedup record in a separate transaction from the effect (race window).
  • Keys that are too broad (collisions) or too narrow (retries treated as new).
  • Assuming a downstream "succeeded" because your call returned — reconcile.
