P15 · Geospatial indexing

What it is: making "what's near me" fast over a 2D space.
Reach for it when: proximity matching or coordinate range queries under high update rates.
The lever: geohash or quad-tree bucketing turns 2D proximity into prefix/bucket lookups; precision level controls bucket size.
New problem it hands you: radius vs result-set precision — coarse cells over-return, fine cells miss neighbors across boundaries; high-frequency location updates make it write-heavy (P1/P2 apply); hot cells (downtown at rush hour) recreate the hot-partition problem.
Exemplified by: Ride-sharing / location systems.

1 · The problem it solves

"Find the drivers within 2 km of this rider" is a 2-D range query. A B-tree on latitude alone is useless (it can’t bound both dimensions). You need to turn 2-D proximity into something you can index and look up cheaply.

2 · How it works

Geohashing interleaves latitude/longitude bits into a single string where a shared prefix means physical nearness, so a proximity search becomes a prefix scan. Worked, for San Francisco:

(lat 37.7749, lon -122.4194) interleave lon/lat bits: 0 1 0 0 1 1 0 1 1 0 ... -> base32 -> "9q8yyk8" prefix length = cell size: "9q8" ~ 156 km "9q8yy" ~ 5 km "9q8yyk" ~ 1.2 km "9q8yyk8" ~ 150 m two points sharing a longer prefix are physically closer

A “within 2 km” query picks the prefix length whose cell ≈ 2 km and scans that prefix. Quad-trees instead recursively subdivide space into four cells, splitting dense areas finer (adaptive to density, where geohash is a fixed grid). Either way you must also search the neighboring cells, because the nearest driver may be just across a cell boundary — a single-cell scan silently misses them.

3 · The new bottleneck it hands you

Radius vs result-set precision: coarse cells over-return (you scan too many candidates), fine cells miss neighbors across boundaries (the nearest driver is just over the cell line), so you must search adjacent cells too. High-frequency location updates make this write-heavy (P1/P2 apply), and a hot cell (downtown at rush hour) recreates the hot-partition problem.

4 · Real-world example

Uber H3

Uber built and open-sourced H3, a hierarchical hexagonal geospatial index. Hexagons are used because every neighbor is equidistant from the center (unlike squares, which have both edge and corner neighbors at different distances), which makes "expand the search ring" uniform and makes smoothing supply/demand across adjacent cells cleaner. Uber indexes driver locations and demand into H3 cells to run matching and to compute surge pricing per cell in real time. The hierarchy lets them zoom resolution up or down depending on density. See Ride-sharing.

5 · Applying it in a design

"Index locations into geohash/H3 cells; search the target cell plus its ring of neighbors; tune cell size for the radius; shard by cell but watch for hot downtown cells."

6 · Pitfalls

  • Searching only the home cell and missing boundary neighbors.
  • One global cell size for wildly different densities.
