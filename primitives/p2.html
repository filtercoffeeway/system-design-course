P2 · Partitioning / sharding (and the hot-partition problem)

What it is: splitting data or load across nodes so no single node is the ceiling.
Reach for it when: a dataset or write rate exceeds one node; or you need parallelism.
The lever: partition by key (co-locates related data, ordering per key), hash (even spread), range (range scans), or directory. Consistent hashing minimizes redistribution on topology change.
New problem it hands you: skew — the hot partition/hot key. Even average spread says nothing about the worst key. A celebrity, viral URL, hot geo-cell, or high-traffic tenant lands on one partition and recreates the single-node bottleneck. Mitigate: split the hot key, coalesce requests, dedicated cache, or go hybrid. Detecting hot keys in real time is its own problem. Cross-partition joins also get expensive (see P7).
Exemplified by: Kafka, KV store, RDBMS, Search, Social feed, Ride-sharing.

1 · The problem it solves

One machine has a ceiling: finite disk, RAM, CPU, and network. When your data or your request rate exceeds what one node can hold or serve, you must spread it across many nodes. Partitioning (a.k.a. sharding) is how: split the keyspace so each node owns a slice, and route every request to the node that owns its key.

2 · How to map keys to nodes — the ladder

You pick a partition key and a function mapping key → node. Don’t jump to “consistent hashing” — derive why it exists.

Rung 0 — modulo hashing: node = hash(key) % N

Spreads keys evenly across N nodes, O(1) routing, dead simple. Perfect — until the cluster size N changes. Because every node index is computed % N, changing N reshuffles almost the entire keyspace:

10,000 keys, grow cluster 4 -> 5 nodes: modulo hash %N -> 80.3% of keys move to a different node (theory: ~(1 - 1/N) = 4/5 of keys remap)
Bottleneck: adding one node moves ~80% of the data and invalidates ~80% of every cache — a scaling event becomes a cluster-wide stampede. Elastic scaling is effectively impossible.

Rung 1 — consistent hashing: a ring

Place both nodes and keys on a hash ring (0 … 232); a key is owned by the next node clockwise. Now adding/removing a node only re-homes the keys in one arc — the rest don’t move:

same 10,000 keys, 4 -> 5 nodes on a ring: consistent hashing -> ~18% of keys move (theory ~1/(N+1) = 1/5) 4x less data movement than modulo, and it shrinks as the cluster grows
Bottleneck: with only one point per node, the ring arcs are uneven — some nodes own much larger arcs and get more load; and removing a node dumps its entire arc onto its single clockwise neighbour.

Rung 2 — consistent hashing + virtual nodes ← the standard

Give each physical node many small ring segments (e.g. 100–256 vnodes). Load averages out across many small arcs, and when a node leaves, its segments scatter to many neighbours instead of one. Measured, 4 nodes × 100 vnodes over 10,000 keys:

per-node load: min 2,126 max 2,755 (ideal 2,500) — within ~10% node removal: its ~100 segments redistribute across all remaining nodes, not onto one

4 · The new bottleneck it hands you

Skew — the hot partition. Even spread on average says nothing about the worst key. A single celebrity, a viral video, a giant tenant, or a monotonically increasing key (timestamp, auto-increment ID) lands disproportionate load on one partition, recreating the single-node bottleneck you tried to escape.

The math is unforgiving — a hot key is one key, so no amount of hashing helps; it has exactly one owning partition:

1M rps total, one celebrity key = 30% of traffic -> 300,000 rps slam the single partition that owns that key the other 999,999 keys share the rest; this one node is the ceiling again mitigation — split the hot key into sub-keys: "celeb" -> "celeb#0" .. "celeb#9" (write to a random sub-key, read all 10 and merge) -> ~30,000 rps each, spread over up to 10 partitions

Other mitigations: a dedicated cache or replica for the hot key, or request coalescing (collapse N concurrent reads of the same key into one). The hard part is that you must detect hot keys in real time to react — a problem of its own. Cross-partition queries are the other tax: anything spanning keys now fans out to all partitions and merges (P7), and joins across shards get expensive (RDBMS scaling).

5 · Real-world example

Amazon Dynamo / Cassandra

Amazon's Dynamo paper (2007) popularized consistent hashing with virtual nodes for partitioning, and Cassandra and Riak adopted it directly. Each node is assigned many tokens (vnodes) around a ring; a key is hashed to a point and stored on the next N nodes clockwise (N = replication factor). Adding a node steals a fair slice of tokens from existing nodes rather than triggering a global reshuffle, which is what makes these clusters elastic. The classic operational footgun is exactly the hot-partition problem: a wide partition (e.g. a partition key like country with billions of rows under US) overwhelms one node — Cassandra docs explicitly warn to choose a high-cardinality, evenly distributed partition key.

6 · Applying it in a design

Choose the partition key for two things at once: even distribution and query locality (data you read together should live together). Say the tradeoff out loud: "I'll hash on user_id for spread, accept that cross-user queries fan out, and watch for hot tenants — which I'd handle by sub-sharding the hot key."

7 · Pitfalls

  • Low-cardinality or skewed partition keys (status, country, boolean).
  • Monotonic keys (timestamps, sequential IDs) → all writes hit the last partition.
  • Forgetting that resharding changes key→node mapping and breaks ordering mid-move.
