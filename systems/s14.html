14 · URL Shortener (bit.ly style)

Hard constraint: sub-millisecond lookup — a CDN/Redis cache at the edge is the read architecture; the storage system barely matters.
Primitives exercised: P4, P11, P5 (analytics).

Base62 encoding; read-heavy optimization (1000× redirects); collision avoidance (pre-generate vs hash-and-check); 301 vs 302 tradeoff; click analytics as the same ingestion pattern as Datadog.

1 · Requirements & scale

Functional: shorten a long URL to a short code; redirect the code to the original; optional analytics. Non-functional: redirects are read-heavy and must be sub-millisecond; codes are globally unique and never collide.

Back-of-envelope 100,000,000 new links/day ~ 1,200 writes/s read:write ~ 1000:1 ~ 1,200,000 redirects/s code space: 62^7 ~ 3.5x10^12 (7 base62 chars is plenty) storage: 100M/day x 365 x ~500 B ~ 18 TB/yr (fits sharded KV)

2 · High-level architecture

graph LR
  U[user] --> CDN[CDN / edge cache]
  CDN -->|hit| U
  CDN -->|miss| SVC[redirect service]
  SVC --> CACHE[(Redis)]
  CACHE -->|miss| KV[(sharded KV: code -> URL)]
  SHORTEN[shorten API] --> KGS[key-gen service] --> KV

3 · The hard constraint, in depth

The lookup must be sub-millisecond at enormous read volume, so a CDN/Redis cache at the edge is the read architecture (P4, P11). The backing store barely matters because almost every read is a cache hit on a small set of hot links. This is fundamentally a caching problem, not a storage problem.

4 · Component deep dives

Code generation

Two approaches: hash-and-check (hash the URL, take N chars, retry on collision) or a key-generation service that pre-generates unique base62 codes and hands them out in batches (no collision checks on the hot path). Base62 (a-z,A-Z,0-9) keeps codes short and URL-safe; 7 chars covers trillions.

Read path (P4/P11)

Edge CDN → Redis → sharded KV, in that order; the vast majority of redirects never reach the KV store.

301 vs 302

301 (permanent) is cached by browsers forever — cheapest for you but you lose per-click analytics; 302 (temporary) always returns to your server — more load but you count every click. Choose by whether analytics matter.

Analytics

Each redirect emits a click event into the same ingestion pattern as Datadog (batch → queue → aggregate), kept off the redirect hot path.

5 · Real-world example

Bitly / TinyURL

URL shorteners like Bitly serve billions of redirects and are built around exactly this shape: a tiny key→URL mapping fronted by aggressive caching/CDN so redirects are effectively cache hits, base62-style short codes, and a separate asynchronous analytics pipeline that counts clicks without slowing the redirect. The interesting engineering is all on the read/cache and analytics sides; the storage of the mapping is almost trivial.

6 · Tradeoffs & alternatives

Hash-and-check is stateless but needs collision handling; a key-gen service removes hot-path collisions but adds a component. 301 saves load but kills analytics. Custom vanity codes complicate uniqueness.

7 · Key talking points

  • It’s a caching/read problem — CDN + Redis is the architecture (P4, P11).
  • Base62 codes; key-gen service vs hash-and-check.
  • 301 vs 302 = caching vs analytics.
  • Async click pipeline off the hot path (S6 pattern).
