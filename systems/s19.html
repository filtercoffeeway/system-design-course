19 · Search Autocomplete / Typeahead

Hard constraint: return ranked completions on every keystroke in a few ms at ~106 QPS — so the answer is a precomputed top-k at each prefix served from RAM, with the hot head absorbed at the edge. The matching structure is secondary; the precomputation and the skew are the system.
Primitives exercised: P2 (partitioning / hot-key), P4 (caching), P11 (CDN / edge), P1 (batch + stream), P6 (search retrieval, sibling system S7).

This is what “how does YouTube/Google search work” usually means in an interview: the as-you-type suggestion box, not full retrieval (S7). Trie vs FST; materialize top-k at the node; first-letter sharding is the trap; hash + replicate + edge-cache the head fixes the hot key; a streaming speed layer keeps trending queries fresh.

1 · Requirements

Functional. (1) suggest(prefix) returns the top k (≈5–10) completions for a prefix, ranked by popularity/relevance; (2) it fires as the user types each character; (3) tolerate minor typos (“netflox” → “netflix”); (4) reflect trending/breaking queries within minutes, not a day. Out of scope for the core design: executing the actual search and ranking result documents — that’s the search engine (S7); mention the boundary, then set it aside.

Non-functional — these shape everything:

PropertyTargetWhy it matters
Latencyp99 < ~100 ms end-to-end; lookup itself a few msIt fires on every keystroke; a slow suggestion is worse than none and is discarded by the client.
Read:write ratiooverwhelmingly readsPopularity is aggregated offline/streaming, never per request → optimise the read path above all.
Traffic shapeextreme Zipfian skewA tiny set of short, popular prefixes is most of the load → caching the head is the architecture, not an add-on.
Freshnesstrending visible in minutesA daily batch can’t surface a term that started spiking an hour ago.
Consistencyeventually-consistent ranking is fineIt’s a hint list; being a few minutes stale on the tail is invisible to users.

2 · Capacity estimation

Numbers first — they decide where the design lands. Assume Google scale: ~5 billion searches/day, and the suggest endpoint fires on (debounced) keystrokes, ≈6 calls per search.

QUERY VOLUME 5,000,000,000 searches/day x 6 suggest calls = 3.0e10 suggest q/day 3.0e10 / 86,400 s ≈ 347,000 QPS average peak ~3x ≈ 1,040,000 QPS INDEX CORPUS (head + torso of distinct queries we index) ~5e9 distinct query strings x ~20 chars trie that materializes top-k at EVERY node ≈ 2 TB -> too big for one box; fits across a small fleet in RAM -> an FST (suffix-shared, bit-packed) cuts this several-fold TRAFFIC SKEW (why caching dominates) top ~10,000 prefixes ≈ 50-70% of ALL suggest traffic -> a tiny edge cache absorbs the majority of requests -> the sharded fleet only serves the well-distributed long tail SERVING ~1.04M peak QPS / ~50k QPS per RAM box ≈ ~21 boxes for raw QPS -> so the real constraint is MEMORY FOOTPRINT + SKEW, not aggregate QPS LATENCY BUDGET p99 < ~100 ms end-to-end; serving lookup a few ms (walk + return cached list)

The takeaways that drive the rest: it’s a read/caching problem with a 2 TB in-memory index; aggregate QPS is small (~21 boxes), so what actually hurts is fitting the index and surviving the skew; and freshness can’t come from batch alone.

3 · API design

GET /suggest?q=net&k=8&lang=en&ctx=... 200 : { "suggestions": [ {"text":"netflix","score":0.99}, {"text":"netflix login","score":0.71}, {"text":"net worth","score":0.55}, ... ] } Client rules (offload work to the edge of the system): - debounce ~50-150 ms so not every keystroke hits the server - cancel the in-flight request when a newer character is typed - cache per-prefix results locally for the session

The endpoint is a bare keyed GET so it is trivially cacheable by a CDN. Debounce + request-cancellation on the client cut a large fraction of calls before they leave the browser — the cheapest rung of all.

4 · The matching structure — derive it, don’t assert “trie”

The serve-time job: prefix → top-k completions in a few ms. Enumerate the family and the axis that picks between them, rather than naming one and stopping:

StructureHow it does prefix→top-kCost / when it wins
Sorted term array + binary searchPrefix defines a contiguous range; scan it, sort by scoreDead simple (rung-0 baseline); range is huge for short prefixes and you re-rank per query.
Trie (prefix tree), top-k cached at each nodeWalk len(prefix) nodes, return the node’s precomputed listO(prefix length), no serve-time sort. The canonical answer. Cost: memory + rebuild of cached lists.
Ternary search treeTrie with 3-way nodesMore memory-efficient than a fat 256-way trie; slightly slower walk.
FST / FSA (finite-state transducer)Shares suffixes too; maps string→weight compactly5–10× smaller than a pointer trie — how you actually fit ~2 TB in RAM. This is what Lucene uses. Harder to mutate in place.
Succinct trie (LOUDS)Same traversal, bit-packedSmallest footprint; read-mostly, rebuilt rather than edited.

The decisive move is independent of which structure you pick: materialize the top-k completions at each prefix node ahead of time. That converts serve-time work from “range-scan + sort” into “walk to the node, return a cached list of 5” — which is the only way to hit a few-ms p99 at this QPS. Trie is the right thing to say; an FST is the right thing to ship (it makes the 2 TB fit); the precomputed-top-k idea is the part that actually matters.

node "net" -> precomputed top-k (sorted by score, computed offline) ["netflix", "net worth", "netflix login", "netgear", "net a porter", ...] walk: n -> e -> t (3 hops) then return the list. No sort at serve time.

Typo tolerance, when the interviewer asks: run the prefix walk against a Levenshtein automaton (edit-distance ≤1–2) intersected with the FST — the same structure traversed fuzzily, no separate index.

Two read tiers feed one merge. The edge answers the hot head outright. On a miss, the suggest service queries the base trie/FST fleet (stable, batch-built) and the small fresh layer (streaming), then merges the two ranked lists. Underneath, the query-log stream forks into a batch path that rebuilds the base and a speed path that maintains approximate trending counts. The user’s keystroke never waits on either pipeline.

6 · Partitioning — one box → sharding → the hot-key problem

This is the decision Chandra-style answers get wrong by stopping at “shard by first letter.” Derive it rung by rung.

Rung 0 — single in-memory trie/FST

One process holds the whole structure; every suggest walks it. Correct, and fine at small scale.

Bottleneck: the materialized index (~2 TB) won’t fit in one machine’s RAM, and one box can’t absorb ~1M QPS. Both force a partition.

Rung 1 — shard by first letter (the trap — raise it, then reject it)

Route by the leading character: an a… shard, a b… shard, and so on. Simple, but it breaks two ways — both of which a sharp interviewer will name:

Bottleneck — catastrophic skew (the hot-key problem, P2): letters and short prefixes are wildly non-uniform. The shard owning s/t/a, or hot prefixes like how…, yo…, the…, takes a huge multiple of an even share. With ~26 buckets you cannot rebalance, and you can’t split a hot letter without abandoning the alphabet rule. Worse, short prefixes are simultaneously the hottest and the most expensive (largest candidate sets), so the skew compounds.

Rung 2 — shard by hash(prefix)

Map each distinct prefix string to a shard via consistent hashing (P2). Now load spreads evenly across distinct prefixes and you can add shards smoothly.

Bottleneck: hashing fixes structural skew (the alphabet) but not popularity skew. Every request for yout still hashes to the same single shard — one viral prefix is still one hot key. Hashing balances the many, not the one.

Rung 3 — attack popularity skew directly (← the real fix)

Two complementary tools, and the second is the big one:

  • Replicate hot shards / consistent hashing with bounded loads: detect the heavy prefixes and serve them from N replicas so no single box is the ceiling.
  • Edge/CDN cache of the head (P11, P4): the top ~10k prefixes are ~50–70% of all traffic. Cache their completion lists at the edge with a short TTL (seconds–minutes). The hot keys are absorbed before they ever reach a shard, so the sharded fleet only sees the long tail — which is exactly the traffic that is well distributed. The hot-key problem you correctly smelled is mostly solved by caching the head, not by a cleverer shard key.

Rung 4 — two-tier trie

Keep the top of the tree (short, super-hot prefixes) in a small, heavily-replicated front layer; shard the deep subtrees (long, rare prefixes) across the fleet. Short-prefix queries — the hot ones — never touch the sharded tail at all.

7 · Freshness — pulling from a live stream

Where do the popularity scores come from, and how does a trending term appear in minutes instead of tomorrow? This is a lambda / kappa split (P1): a stable batch base plus a fast approximate delta, merged at serve time.

Batch layer — the stable base

A daily (or hourly) job over the query logs computes per-prefix top-k and rebuilds the serving trie/FST. Accurate and complete, but stale by up to its rebuild interval — it cannot surface a term that started spiking an hour ago.

Bottleneck: rebuild latency. By itself the batch base is always “yesterday’s popularity,” so breaking/trending queries are invisible until the next rebuild. That gap is what the speed layer fills.

Speed layer — the fresh delta

Query events stream into Kafka; a stream processor (e.g. Flink) maintains approximate popularity online, because keeping an exact count per distinct query at ~1M QPS is infeasible in memory:

  • Count-Min Sketch for frequency in fixed memory — a few hash-indexed counter arrays; it can over-count a little but never under-counts, in O(1) space per query instead of one counter per distinct string.
  • Space-Saving / heavy-hitters to keep the top-k per prefix without storing every query that was ever typed.
  • Exponential time decay on the counts, so “trending” means rising now, not popular all-time — a term that spikes can outrank an evergreen one within minutes.

Merge at serve time

The suggest service consults the stable base and the fresh top-k layer and merges the two ranked lists before returning. A query that started trending minutes ago shows up via the speed layer; later the batch layer folds it into the base and the speed layer’s decayed entry ages out. That two-layer merge is exactly “results are close to real-time and pull from a stream to stay fresh.”

8 · Ranking — beyond raw frequency

Popularity is the baseline signal, not the ranker. Production systems re-rank with a model over: query frequency + recency/trend, the user’s own history, location/language, session context (the previous query), and — for YouTube specifically — video-side signals like watch-time, freshness, and channel authority. Mechanically it stays cheap: candidate generation produces the ~50–100 completions for a prefix (frequency-ordered, from the trie/FST), then the model re-ranks only that small candidate set. You never run the model over the whole corpus — only over what a prefix already narrowed it to. Personalization is the same shape: candidate-gen stays global, personalization is a re-rank of the small set, so it never blows up the index.

9 · Interview follow-ups (pre-empted)

  • “Why not LIKE 'pre%' on an indexed column?” A B-tree gives you the prefix range, but you still scan + sort by score per query, at ~350k+ QPS, under a few-ms budget. The DB falls over on both QPS and re-ranking cost. Precomputed top-k in RAM is what hits the latency target.
  • “How do you update one node’s top-k when a count changes?” You don’t, per event. Counts flow through the speed layer; the materialized top-k is refreshed by batch rebuild plus the merged fresh layer. In-place trie edits at write-rate are exactly what you avoid.
  • “Why an FST over a plain trie?” Suffix sharing + bit-packing makes ~2 TB fit in RAM across a small fleet; a pointer trie is several times larger. The cost — it’s build-then-serve, not easily mutated — is fine, because freshness lives in the speed layer, not in mutating the base.
  • “Consistent hashing — what about a single viral prefix?” Hashing balances distinct prefixes, not a single hot one. That’s handled by replicating the hot shard + edge-caching the head (§6, rung 3).
  • “Cache hit rate and TTL?” The head is ~50–70% of traffic in ~10k entries → very high hit rate on a tiny cache; a TTL of seconds–minutes keeps trending fresh while still absorbing the hot keys.
  • “Personalized suggestions?” Candidate generation stays global (prefix → top-k); personalization is a re-rank of that small candidate set with per-user signals — it doesn’t touch the index size.
  • “Multi-language / Unicode?” Index per language/locale (separate FSTs) and route by the lang param; normalize (case-fold, strip accents) before the walk so “cafe” and “café” share a path.

10 · Real-world example

Google / YouTube suggest · Elasticsearch/Lucene

Google’s suggest endpoint returns ranked completions on every keystroke at planet scale, blending all-time popularity with trending and personalized signals — the batch-base-plus-fresh-delta shape above. Lucene/Elasticsearch ship this as concrete machinery: the FST-backed completion suggester stores weighted terms compactly in memory and returns top-k by weight in a prefix walk, with a context suggester for per-category/locale filtering — a production instance of “materialize top-k, serve from RAM.” The lesson mirrors S14: the interesting engineering is the read/cache and freshness sides; the prefix structure itself is nearly a library call.

11 · Key talking points

  • It’s a read/caching problem: precompute top-k at each prefix node, serve from RAM; aggregate QPS is small (~21 boxes) — memory footprint and skew are the real constraints.
  • Trie to teach, FST to ship (fits ~2 TB in RAM); the load-bearing idea is materialized top-k, not the structure name.
  • First-letter sharding is the trap — raise it, reject it for catastrophic skew → hash(prefix) for structural balance → replicate + edge-cache the head for popularity skew (P2/P11) → two-tier trie.
  • The head is tiny and huge: ~10k prefixes ≈ 50–70% of traffic, so a small edge cache solves most of the hot-key problem.
  • Freshness = lambda/kappa: daily batch base + streaming speed layer (Count-Min Sketch + heavy-hitters + time decay), merged at serve time.
  • Ranking is candidate-gen then re-rank the small set — never score the whole corpus.
  • Boundary: this is suggestion, not retrieval — the actual search is S7.
