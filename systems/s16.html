16 · Video Streaming (YouTube / Netflix)

Hard constraint: you never stream origin→user at scale (~250 Tbps at peak) — every byte must come from a CDN edge near the user. The real question is always "what populates the edge, and when?"
Primitives exercised: P11 (CDN/edge), S5 (object store), P1 (chunked upload + async transcode), P18 (parallel transcode/stitch), P12 (startup latency).

Encoding pipeline (one upload → N renditions via async, chunk-parallel workers); adaptive bitrate (HLS/DASH, the player switches on static segments); the CDN as the entire read path; resumable chunked upload; manifest files.

1 · Requirements

Functional: upload a source video; transcode it to many qualities/devices; store and stream it to huge concurrent audiences with smooth, adaptive playback; support seeking and resume. Non-functional:

The defining split: encoding is a batch problem; serving is a CDN problem. They share almost nothing — one is write-once compute fan-out, the other is read-mostly bandwidth at planetary scale. Designing the system is designing those two pipelines and the object store between them.

2 · Capacity estimation

SERVING (the wall): 50,000,000 concurrent streams x ~5 Mbps avg = 250,000,000 Mbps = ~250 Tbps peak egress no origin and no commercial backbone serves 250 Tbps -> push it ALL to the edge CDN 95% hit -> origin sees ~5% = ~12.5 Tbps (still huge -> tiered + pre-position) STORAGE per 2-hour title (all renditions): 240p 0.3 / 360p 0.7 / 480p 1.5 / 720p 3 / 1080p 6 / 4K 16 Mbps sum over a 2 h (7200 s) movie ~ 25 GB per title (one codec family) x multiple codecs (H.264 / HEVC / AV1) x audio tracks x subtitles -> 50-100+ GB/title SEGMENTS: 4 s segments -> 7200/4 = 1800 segments per rendition; thousands of small files per title

250 Tbps is the number that dictates everything: there is no "serve from the database" rung here. The only viable design caches every byte at the edge, so the entire read path is "fill the CDN intelligently."

4 · The upload & transcode pipeline (write path)

Resumable chunked upload (P1)

A multi-GB upload over a flaky connection can't be one PUT. Split into fixed chunks (e.g. 5–10 MB) uploaded by index; the server acks each, and a dropped connection resumes from the first un-acked index rather than restarting. Chunks land in a staging area of the object store (S5); completeUpload assembles them and enqueues the transcode job.

Transcoding — one upload → N renditions

The source is transcoded into a matrix of renditions (resolution × bitrate × codec) so any device/bandwidth has a fit. This is embarrassingly parallel batch compute. The key trick is to not transcode a 2-hour file serially:

  • Chunk-parallel transcode (P18 map/shuffle shape): split the source at GOP/keyframe boundaries into independent chunks, transcode each chunk on a different worker in parallel, then stitch the outputs back in order. A 2-hour encode that would take hours serially finishes in minutes across a fleet. Splitting on keyframe boundaries is essential so each chunk decodes independently.
  • Per-title / per-scene encoding: instead of one fixed bitrate ladder for all content, analyze the title's complexity and assign bitrates per title (a cartoon needs far fewer bits than an action scene for the same quality). Netflix's per-title and per-shot encoding saves large amounts of bandwidth at equal quality.

Each rendition is then segmented into short (2–10 s) files plus a manifest, and written to the object store. The packaging step also produces multiple container/codec variants (fMP4/CMAF for DASH, TS/CMAF for HLS) so one set of media serves both protocols.

New bottleneck — transcode cost & storage explosion. Every extra rendition/codec multiplies both compute and storage (50–100+ GB/title). The relief: encode the full matrix only for popular titles; for the long tail, encode a smaller ladder and add renditions on demand the first time they're requested. AV1 saves ~30% bandwidth over H.264 but costs far more to encode — a classic compute-now vs bandwidth-later tradeoff, worth it only for high-view titles.

6 · The read path: adaptive bitrate (HLS/DASH)

The genius of streaming at scale is that the server serves dumb static files; all intelligence is in the player. The server publishes a manifest (HLS .m3u8 / DASH .mpd) listing, per rendition, the URL of each segment. The player:

  1. Fetches the manifest, picks a starting rendition (usually low, for fast start).
  2. Downloads segments into a buffer, measuring throughput on each.
  3. If measured bandwidth > the next-higher rendition's bitrate (with headroom) and the buffer is healthy, it requests the next segment at higher quality; if bandwidth drops or the buffer drains, it steps down.

Because segments are aligned across renditions (same start times, keyframe-aligned), the player can switch quality at any segment boundary seamlessly. The server never decides quality — it just serves whichever segment URL the player asks for, which is why it scales: it's pure static-file delivery over HTTP, ideal for caching.

ABR control loop (per segment): est_bw = segment_bytes / download_time (EWMA-smoothed) if buffer > target and est_bw > bitrate(next_up)*1.2: level up if buffer < min or est_bw < bitrate(current): level down -> the SAME 4s wall-clock segment exists at every bitrate; switch at boundaries

7 · The CDN: what fills the edge (ladder)

At 250 Tbps, origin→user is impossible. Every byte is served from a CDN edge near the user; origin only fills the edge. The whole design question is "what populates the CDN, and when?" (P11).

Rung 1 — on-demand (pull) fill

Edge caches a segment on first request (a miss fetches from origin, then serves all later viewers locally). Simple, storage-efficient. Breaks on: a brand-new blockbuster release — the first viewers in every region all miss simultaneously, hammering origin (a thundering herd of misses), and the cold-start viewers see poor startup. Also a viral spike can saturate origin before the edge warms.

Rung 2 — pre-position (push) popular content

Predict tomorrow's popular titles and copy their segments to edge appliances during off-peak hours (overnight, when the network is idle). At peak, those titles are already at the edge → ~100% hit, origin serves almost nothing. Breaks on: wasted storage for titles that flop, and you can't pre-position everything (the long tail). So combine: pre-position the predicted-popular head, pull-fill the tail.

Rung 3 — tiered CDN + ISP-embedded edges

Put edge appliances inside ISP networks (Netflix Open Connect) so the bytes never even traverse the public internet backbone — the user streams from a box on their own ISP. Add a mid-tier cache between edge and origin to absorb misses, so origin sees only the misses-of-misses. This is how ~12.5 Tbps of origin load (the 5% miss rate) is further cut by orders of magnitude.

8 · Startup latency & seeking

Time-to-first-frame is a conversion metric, so the player starts at a low bitrate (small first segment downloads fast) and ramps up once the buffer is healthy — better to start instantly at 480p and climb than to stall at 4K. Some systems ship a tiny dedicated first segment. Seeking works because segments are independently decodable (each starts on a keyframe): jumping to 1:23:00 just requests the segment covering that time, no need to decode from the start. Rebuffering is the enemy metric; the ABR algorithm trades quality for buffer health to avoid it.

9 · Scaling, live, & DRM

Live streaming changes the math: you can't pre-encode, so the pipeline must transcode and package in near-real-time with a small latency budget; low-latency HLS/DASH shrink segment size (or use chunked-transfer) to cut glass-to-glass delay. DRM (Widevine/FairPlay/PlayReady) encrypts segments; the player fetches a license/key separately — this is orthogonal to delivery (encrypted segments still cache fine at the edge). Thumbnails/scrubbing use a sprite sheet of low-res frames. At 10× scale the levers are: more edge capacity, better popularity prediction for pre-positioning, and more aggressive per-title encoding to shave bitrate.

10 · Tradeoffs & alternatives

More renditions/codecs = better quality and bandwidth efficiency per device but more transcode cost and storage. Pre-positioning wastes storage on flops but guarantees peak performance; on-demand fill is cheaper but risks origin load — so the real systems do both. Own CDN (Netflix Open Connect: capex, control, ISP embedding) vs commercial CDN (Akamai/Cloudfront: simplicity, pay-as-you-go) is the classic build-vs-buy at the scale where bandwidth is your dominant cost. AV1/HEVC vs H.264 trades encode compute for delivery bandwidth.

11 · Real-world example

Netflix

Netflix transcodes each title into a large matrix of renditions ahead of time, using per-title and per-shot encoding to assign bitrates by content complexity (saving significant bandwidth at equal quality), cuts them into segments, and serves everything through its own CDN, Open Connect (P11), whose appliances sit inside ISP networks. During off-peak it pre-positions predicted-popular content onto those boxes, so at peak a user streams from an edge appliance on their own ISP and Netflix's origin serves almost nothing. The player runs adaptive bitrate over the segment manifest. Encoding is a batch problem; serving is a CDN problem — the two halves of this system.

12 · Interview follow-ups (Q&A)

Q: Why is all the adaptive-quality logic in the player, not the server? So the server serves only static, cacheable files. If the server picked quality it would need per-viewer state and dynamic responses — uncacheable, and impossible at 250 Tbps. Dumb static segments + a manifest let the CDN do all the heavy lifting.

Q: How do you transcode a 2-hour 4K source fast? Split it at keyframe boundaries into independent chunks, transcode them in parallel across a worker fleet, then stitch in order (P18 map/merge shape). Keyframe alignment is what makes the chunks independently decodable.

Q: A new title launches globally at midnight — how do you avoid an origin meltdown? Pre-position its segments to the edge during the preceding off-peak window so peak demand is ~100% edge hits. For unexpected virality, a tiered cache absorbs the miss storm and request coalescing (single-flight) collapses concurrent misses for the same segment into one origin fetch (P4).

Q: How does seeking to the middle work without downloading the whole video? Segments are independently decodable (each begins on a keyframe), so the player just requests the segment covering the target timestamp. The manifest maps time ranges to segment URLs.

Q: How do you keep startup fast? Start at a low bitrate (tiny first segment), play immediately, and ramp quality once the buffer fills — instant 480p beats a stall at 4K. The metric to protect is rebuffering, not peak resolution.

Q: Where do encryption/DRM and live streaming fit? DRM encrypts segments and delivers keys via a separate license server — orthogonal to delivery, since encrypted segments still cache at the edge. Live can't pre-encode, so it transcodes/packages in near-real-time with small/low-latency segments; everything else (ABR, CDN) is the same.

13 · Key talking points

  • The CDN is the system; origin only fills it (P11). 250 Tbps peak makes origin serving impossible.
  • Two pipelines: encoding = batch (one upload → N renditions, chunk-parallel transcode, per-title bitrates) and serving = CDN.
  • ABR (HLS/DASH): static keyframe-aligned segments + a manifest; the player measures bandwidth and switches quality per segment.
  • Fill ladder: on-demand pull → pre-position popular off-peak → tiered + ISP-embedded edges (origin idle at peak).
  • Startup latency: low-bitrate fast start + ramp; seeking works because segments are independently decodable.
  • Renditions/codecs trade transcode+storage cost for quality and delivery bandwidth (AV1/HEVC vs H.264).
