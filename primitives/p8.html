P8 · Tiered & columnar storage

What it is: matching the storage format and medium to the access pattern and the data's age.
Reach for it when: storage cost grows unbounded, or analytical scans read few columns over many rows.
The lever: columnar layout (Parquet/ORC) to read only projected columns; compression (Gorilla for time series); immutable time-partitioned blocks; lifecycle tiers hot→warm→cold by access-frequency decay.
New problem it hands you: cold-tier read latency; (de)compression CPU; columnar is great for scans but poor for point writes/updates; immutability turns updates into append + compaction.
Exemplified by: Datadog, Distributed SQL, Object store, Video.

1 · The problem it solves

This primitive answers two different cost questions that look unrelated but share one fix — match the physical layout to the access pattern instead of storing everything the same way.

The first is scan I/O. An analytical query like SELECT avg(latency_ms) FROM requests WHERE day = '2026-06-19' touches two columns out of maybe two hundred, but a row store interleaves all columns on disk: to read those two values per row it must drag every other column through the I/O path and the CPU cache. On a billion-row table that is the difference between reading ~16 GB and reading ~200 GB. Columnar layout fixes this.

The second is storage cost over time. A metric scraped every 10 seconds is read constantly for the first hour, occasionally for the first week, and almost never after a month — yet it costs the same per byte on hot SSD whether it is read a million times or never. Tiering fixes this by letting the storage medium decay in cost as the data’s access frequency decays. Compression sits underneath both: it is what makes a column cheap to scan and cheap to keep.

2 · How it works

Columnar layout

A row store writes one record contiguously: (t0,host0,200)(t1,host1,201)…. A column store (Parquet, ORC) transposes that — it writes all timestamps together, then all hosts, then all status codes:

row store : [t0,h0,200][t1,h1,201][t2,h2,200][t3,h3,500] … column store: [t0,t1,t2,t3 …] [h0,h1,h2,h3 …] [200,201,200,500 …] ↑ one column, one contiguous run, one data type

Two wins fall out of this transpose. Projection pushdown: a query reads only the column files it names, skipping the rest at the I/O layer. Predicate pushdown: each column chunk carries min/max statistics (and often a Bloom filter) in its footer, so a chunk whose [min,max] can’t satisfy WHERE day = … is skipped without being read at all. And because a column is one data type with low cardinality or slow-changing values, it compresses an order of magnitude better than a mixed row.

The encodings that make a column small

Compression here is not one algorithm but a small toolbox, picked per-column-chunk by the writer based on the data it sees:

EncodingBest forSample column → encoded form
DictionaryLow-cardinality strings/categories [us-east,us-east,eu-west,us-east] → dict {0:us-east,1:eu-west} + codes [0,0,1,0] (2 bits each)
Run-length (RLE)Long repeats (often after sorting) [200,200,200,200,500] → (200×4)(500×1)
Delta + bit-packSorted/monotonic ints, IDs, timestamps [1000,1004,1008,1012] → base 1000 + deltas [4,4,4] packed in 3 bits
Frame-of-referenceClustered numeric ranges [5012,5009,5021] → ref 5000 + [12,9,21]

These stack: dictionary codes are then RLE’d, deltas are then bit-packed, and the whole chunk is finally run through a general compressor (Snappy, Zstd). The combination is why a sorted low-cardinality column can shrink 20× or more.

Compression for time series: Gorilla

Time-series points have a brutal amount of structure the generic encodings above don’t fully exploit: consecutive points have near-identical timestamps and slowly-changing float values. A raw point is 16 bytes — an 8-byte timestamp and an 8-byte IEEE-754 double. Facebook’s Gorilla scheme compresses a stream of such points to ~1.37 bytes per point on production data — better than an order of magnitude — using two tricks, one per field.

Timestamps → delta-of-delta

Metrics are scraped on a fixed interval, so the gap between gaps is almost always zero. Gorilla stores the delta of the delta (how much the interval changed) with a variable-length prefix code — the more unusual the value, the more bits it costs:

delta-of-delta D = (t_n − t_n−1) − (t_n−1 − t_n−2) D == 0 → '0' (1 bit) ← the common case D in [-63, 64] → '10' + 7-bit value D in [-255, 256] → '110' + 9-bit value D in [-2047,2048] → '1110'+ 12-bit value otherwise → '1111'+ 32-bit value

Worked on a 60-second scrape that hiccups by 8 seconds once:

t1=1609459260 delta=60 (first delta, 14 bits) t2=1609459320 delta=60 dod= 0 → '0' 1 bit t3=1609459380 delta=60 dod= 0 → '0' 1 bit t4=1609459448 delta=68 dod=+8 → '10' + 0001000 9 bits t5=1609459508 delta=60 dod=-8 → '10' + 1111000 9 bits

On a clean stream essentially every timestamp after the first costs a single bit (the paper reports the vast majority of real timestamps land in the D==0 case), versus 64 bits stored raw.

Values → XOR of consecutive doubles

Two successive readings of the same metric usually share most of their bits, so Gorilla stores v_n XOR v_n−1. If the values are identical the XOR is all zeros — one bit. If not, the set bits cluster in a contiguous middle block, so it stores just that block: a count of leading zeros, the block length, then the meaningful bits.

v0 = 15.5 → store full 64 bits (block seed) v1 = 15.5 XOR = 0 → '0' 1 bit v2 = 14.0625 XOR = 0000000000000011001000…0 lead0=14 meaningful=11001 (len 5) → '1' '1' [14][5][11001] 18 bits v3 = 14.25 XOR = 0000000000000000101000…0 lead0=16 trail0=45 — fits inside v2's window → '1' '0' [00101] 7 bits

Note v3: because its meaningful bits sit within the leading/trailing zero window already established by v2, Gorilla skips re-sending the lengths — control bit '0' says “reuse the previous window” and it spends only the 5 window bits. That window-reuse is most of the value-side savings. The first 64-bit seed amortizes to nothing over a two-hour block of hundreds of points.

Per-point budget, raw vs Gorilla (steady metric, 2-hour block): raw : 64-bit ts + 64-bit value = 128 bits = 16.0 bytes Gorilla : ~1-bit ts + ~1-2 bit value (amort.) ≈ 11 bits ≈ 1.37 bytes → ~12× smaller; months of metrics fit in RAM, which is Gorilla's whole point

Tiering

Compression shrinks each point; tiering decides where the compressed blocks live as they age. Keep recent data fine-grained on fast storage, roll older data up (downsample 1s → 1m → 1h) and migrate it to cheaper media on an access-frequency decay. Time-series systems combine all three moves: immutable, time-partitioned, Gorilla- compressed blocks that downsample and age into object storage.

  H["hot: SSD, raw resolution, last hours"] --> W["warm: rolled up 1m, last weeks"]
  W --> C["cold: object store / archive, 1h, months+"]

3 · Variants & when to use each

LeverUse whenCost it trades
Columnar (Parquet/ORC)Wide tables, scans touch few columnsBad at point updates; rewrite whole column chunks
Gorilla / time-series codecRegular-interval float metrics(De)compress CPU; needs append-only, in-block decoding
Downsampling / roll-upsOld data queried at coarse resolutionLossy — raw resolution is gone
Lifecycle tiers (S3 classes)Access frequency decays with ageCold reads are slow + per-request priced

4 · The new bottleneck it hands you

Columnar is superb for scans but poor for point writes/updates — a single-row change means rewriting column chunks, so it is append/batch oriented, not OLTP. Compression trades CPU for I/O: every read now pays a decode, and Gorilla’s in-block scheme means you can’t random-access point #500 without walking the block from its seed. Tiering trades cold-tier latency — a query over archived data is slow and, on object storage, per-request priced — and downsampling is lossy, so a roll-up you regret can’t be un-rolled. Immutability turns every update into append + compaction, which is the write-amplification problem P17 (LSM storage) exists to manage.

5 · Real-world example

Facebook Gorilla

Gorilla is Facebook’s in-memory time-series store (the engine under Beringei and the inspiration for Prometheus’s TSDB). The delta-of-delta + XOR scheme above took their production metric stream from 16 bytes/point to ~1.37 bytes/point — the ~12× that let them hold 26 hours of data for ~10 billion time series in RAM, turning queries that hit disk into in-memory reads. The compression is the architecture: it is what makes the in-memory tier economically possible.

Google Dremel / BigQuery · Parquet · S3

Dremel (the engine behind BigQuery) stores data columnar and scans only projected columns across thousands of nodes — how it scans terabytes in seconds without indexes. Its open-source descendant Parquet is now the default analytical file format (Spark, Presto, Snowflake), carrying the same dictionary/RLE/delta encodings and per-chunk min/max stats for predicate pushdown. On the tiering side, Amazon S3 storage classes (Standard → Infrequent Access → Glacier) implement the access-frequency decay model directly: cheaper per-GB, slower and per-request-pricier to read. See Datadog (S6) and Object store (S5).

6 · Applying it in a design

For analytics: “columnar storage with predicate and projection pushdown so we read only the columns and chunks a query needs.” For metrics: “Gorilla-style delta-of-delta on timestamps and XOR on values, so a month of series fits in RAM.” For cost: “raw resolution on SSD for N days, then downsample and tier to object storage.” The sentence that shows you understand the tradeoff: “columnar and compressed buys scan throughput and cheap retention, but I’m giving up point updates and paying CPU on every read — so this tier is append-only.” See Distributed SQL (S8) and Datadog (S6).

7 · Pitfalls

  • Using columnar for a write-heavy OLTP workload — every update rewrites chunks.
  • Tiering or downsampling data that is still queried interactively at fine resolution.
  • Leaving columns unsorted: RLE and delta encodings collapse on unsorted data, so the clustering/sort order is half the compression win.
  • Forgetting Gorilla blocks are sequential-decode-only — you can’t seek to one point without replaying the block from its 64-bit seed.
