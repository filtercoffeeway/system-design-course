7 · Search Engine (Elasticsearch / Lucene)

Hard constraint: distributed relevance — each shard sees only local term statistics, so a globally-correct BM25 score (which needs global document frequency) isn't directly comparable across shards without an extra pass.
Primitives exercised: P6 (inverted index), P2 (sharding), P7 (scatter-gather), P12 (tail latency), P1 (refresh/segment batching).

Inverted-index construction (tokenize/stem/posting lists); BM25 ranking; immutable segments + background merge; the near-real-time refresh gap; sharded query-then-fetch with global top-K merge.

1 · Requirements

Functional: full-text search over a large corpus, ranked by relevance (not just matching), with phrase queries, filters/facets, and near-real-time visibility of new documents. Non-functional:

Search is the retrieval sibling of S19 autocomplete (prefix prediction) and shares the inverted-index machinery with S6's log search. The thing that makes it hard isn't finding matches — it's ranking them correctly when the index is split across many machines.

2 · Capacity estimation

1,000,000,000 docs x ~5 KB text = ~5 TB raw corpus inverted index (compressed postings) ~ 20-30% of text = ~1.3 TB shard target ~30 GB/shard (Lucene works well in tens of GB) -> ~1.3 TB / 30 GB ~ 43 primary shards query target ~50,000 QPS -> each shard replicated for read fan-out + HA a query scatters to all ~43 shards -> tail latency is the enemy (§9)

Two facts shape everything: the index is big enough to need ~40+ shards (so every query is a scatter-gather, and the slowest shard sets latency), and posting lists are large enough that the read path must avoid touching whole documents until the very end (query-then-fetch).

3 · API & data model

index(doc_id, fields{title, body, tags, ts}) -> ok (async; visible after refresh) search(q, filters, from, size, sort) -> [{doc_id, score, highlights}] suggest(prefix) -> typeahead (see S19) inverted index (P6): term -> posting list: [ (doc_id, tf, [positions]), ... ] sorted by doc_id, delta+varint compressed per term: document frequency df ; per doc: length dl

4 · The inverted index & scoring (built up)

4.1 Building the index (P6)

Indexing each document: tokenize (split into terms) → normalize (lowercase, fold accents) → stem/lemmatize ("running"→"run") → drop or keep stop-words → append (doc_id, tf, positions) to each term's posting list. Positions enable phrase queries ("new york" = york at position+1 after new). See P6 for the full byte-level treatment (delta + varint compression, BM25 worked example).

4.2 Relevance: BM25, derived

Why not just count keyword matches? Because a term that appears in every document (e.g. "the") is worthless for ranking, and a 10,000-word document will out-match a focused 300-word one purely by length. BM25 fixes both with two ideas:

  • IDF (inverse document frequency) — rare terms are worth more. idf = ln(1 + (N − df + 0.5)/(df + 0.5)). With N = 1e9 docs: a term in 10M docs has idf ≈ 4.6; a term in 1,000 docs has idf ≈ 13.8 — the rare term carries ~3× the weight.
  • TF saturation + length normalization — more occurrences help, but with diminishing returns (the 10th "jaguar" adds little over the 3rd), and long documents are penalized so they can't win by sheer size.
BM25 term score = idf * ( tf*(k1+1) ) / ( tf + k1*(1 - b + b*dl/avgdl) ) k1=1.2, b=0.75 rare term (df=1000, idf=13.8), tf=3, avgdl=500: short doc (dl=300): score ~ 23.7 long doc (dl=2000): score ~ 13.2 <- same tf, but length-penalized

A query's score is the sum of its terms' BM25 contributions, plus optional field boosts (a match in title counts more than in body). This is the default Lucene/Elasticsearch ranker.

4.3 Immutable segments + the refresh gap (P1)

Posting lists can't be edited in place cheaply, so Lucene writes immutable segments: new/updated docs accumulate in an in-memory buffer and are flushed as a new small segment on refresh (default ~1 s) — that's the near-real-time gap, a document isn't searchable until the next refresh. A delete is a tombstone marking the doc id dead; an update is delete + re-add. Background merge compacts many small segments into fewer large ones (and physically drops tombstoned docs), keeping the per-query segment count low. This is P1 batching (buffer writes, flush periodically) and mirrors the LSM compaction story (P17).

New bottleneck — segment proliferation. A 1-second refresh creates ~86,400 tiny segments/day; a query must consult every segment, so search slows as segments pile up. The merge policy is the relief valve: too-aggressive merging burns I/O and write-amplifies; too-lazy merging slows queries. Tune refresh interval up (e.g. 30 s) for write-heavy/log workloads where 1 s freshness isn't needed.

6 · The query path — query-then-fetch (ladder)

Rung 1 — ask each shard for full top-K documents

Coordinator scatters the query; each shard returns its top-K full documents; coordinator merges. Breaks on: shipping full documents from every shard for results you'll throw away — if K=10 and there are 43 shards, you move 430 full docs to keep 10. Huge waste of bandwidth.

Rung 2 — query-then-fetch (the standard)

Query phase: each shard scores its matches locally and returns only the top-K (doc_id, score) — tiny. Merge: coordinator merges the 43×K candidates into one global top-K (P7). Fetch phase: coordinator fetches the full documents only for the K global winners, from the shards that own them, and builds highlights. Two round trips, but only K full docs move regardless of shard count.

query phase: 43 shards x K=10 -> 430 (id,score) pairs ~ KBs merge: pick global top-10 fetch phase: 10 full docs fetched -> vs 430 in the naive rung

7 · The hard constraint: distributed relevance (the wall)

BM25 needs global document frequency (df = how many of all N docs contain the term). But each shard only knows its local df. So the idf a shard uses to score is wrong, and scores from different shards aren't directly comparable — a rare-overall term might look common on one shard and rare on another, skewing the merged ranking. This is why relevance looks off on small or unevenly-filled indexes.

The relief ladder:

  • Approximate (default): if shards are large and documents are randomly distributed, local df ≈ global df / shard_count, so local scoring ranks almost right. Cheap, one round trip. This is Elasticsearch's default query_then_fetch.
  • Exact (dfs_query_then_fetch): add a preliminary DFS phase that gathers term statistics (df, total docs) from all shards, computes global idf, and distributes it so every shard scores with the same global stats. Correct, at the cost of an extra round trip — the exact tradeoff the constraint predicts.
  • Route to avoid the problem: route documents so each shard is a representative random sample (don't shard by a field that correlates with the query), which keeps the approximation honest.

8 · Sharding & routing

Shard by hash(doc_id) % shards (P2) so documents spread evenly — this both balances storage and keeps local-df a fair sample of global-df (§7). Each shard has replicas for read fan-out and HA; a search can hit any replica. Custom routing (route all of one tenant's docs to one shard via a routing key) makes that tenant's queries hit a single shard instead of scattering — great for multi-tenant latency, but it concentrates load and worsens the df approximation, so it's a deliberate tradeoff.

9 · Scaling, tail latency & two-phase ranking

New bottleneck — tail amplification. A query is only as fast as its slowest shard (P12). With 43 shards each 99% likely to be fast, the chance all 43 respond promptly is 0.9943 ≈ 66% — so ~1 in 3 queries waits on a straggler. More shards = more parallelism per shard but worse tail. Fixes: hedged requests (query a second replica after p95 and take the first answer), adaptive replica selection (route to the least-loaded replica), and capping shard count.

Two-phase ranking for quality at scale: BM25 retrieves a few hundred cheap candidates per shard; a second, expensive re-ranker (learning-to-rank model, or a neural/semantic model using click signals) re-scores only the top candidates the coordinator gathered. This is the same candidate-generation → re-rank shape as S19 and S9 — never run the expensive model over the whole corpus, only over a recalled shortlist.

10 · Tradeoffs & alternatives

More shards trade scale and per-shard speed against worse tail latency and worse local-IDF approximation; fewer shards trade scale for better relevance. For pure analytics over structured data a columnar SQL engine (S8) beats search (scans, no relevance). For semantic / "find similar" search you add a vector ANN index (HNSW) alongside the inverted index and combine lexical BM25 with vector similarity (hybrid search) — BM25 is exact-term, vectors capture meaning; modern engines run both and blend the scores. For freshness-vs-throughput, raise the refresh interval on log/append workloads where 1 s visibility isn't needed.

11 · Real-world example

Elasticsearch (Apache Lucene)

Elasticsearch shards an index across nodes; each shard is a full Lucene index of immutable segments. A search runs Lucene's query-then-fetch across shards exactly as in §6. To tame distributed IDF it defaults to per-shard scoring (good enough when shards are large and uniform) but offers dfs_query_then_fetch, which first gathers global term statistics across shards for accurate scoring at the cost of an extra round trip — the exact tradeoff §7 predicts. Near-real-time comes from the ~1 s refresh flushing the in-memory buffer to a new segment; background merge compacts segments and purges tombstoned deletes. Recent versions add native vector (kNN/HNSW) search for hybrid lexical+semantic ranking.

12 · Interview follow-ups (Q&A)

Q: Why query-then-fetch instead of fetching docs in one pass? To avoid shipping full documents from every shard for results you'll discard. The query phase moves tiny (id, score) pairs; only the K global winners' full docs are fetched. Bandwidth is independent of shard count.

Q: Why are scores inconsistent across a small index? Local per-shard df diverges from global df when shards are small or skewed, so idf (hence BM25) is computed on wrong statistics. Use dfs_query_then_fetch for global stats, or ensure shards are large random samples.

Q: A new document isn't showing up immediately — why? The refresh interval (default ~1 s) controls when buffered writes become a searchable segment. It's a deliberate batching tradeoff (P1); lower it for freshness, raise it for write throughput.

Q: How do you do a phrase query like "new york"? Posting lists store term positions; a phrase match requires the terms to appear at consecutive positions in the same doc. It's more expensive than a bag-of-words match, so engines often use a faster term match to recall candidates and verify positions only on those.

Q: How do you keep relevance good at huge corpus size? Two-phase ranking: cheap BM25 recall of a few hundred candidates, then an expensive learning-to-rank / semantic re-ranker over only that shortlist (same shape as S9/S19). Never score the whole corpus with the expensive model.

Q: Pagination to page 1000 (from=10000) is slow — why? Deep pagination forces each shard to return from+size candidates so the coordinator can skip correctly — cost grows with the offset. Use search-after (a cursor on the last sort value) instead of large from offsets.

13 · Key talking points

  • Inverted index + immutable segments + ~1 s refresh + background merge (P6, P1).
  • Relevance is BM25 (IDF + TF-saturation + length norm + field boosts), not keyword matching.
  • Query-then-fetch: scatter top-K (id, score), merge global top-K, fetch only winners' docs (P7).
  • The wall is distributed IDF: local df ≠ global df → approximate (default) vs exact DFS pass (extra round trip).
  • Latency = slowest shard (P12); more shards = more parallelism but worse tail → hedged requests.
  • Two-phase ranking (BM25 recall → LTR/semantic re-rank); add a vector ANN index for hybrid semantic search.
