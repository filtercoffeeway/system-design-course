16 · Video Streaming (YouTube / Netflix)

Hard constraint: you never stream origin→user at scale; every byte must be cached at the edge — the question is always "what populates the CDN?"
Primitives exercised: P11, P8, P1 (chunked upload + async transcode).

Encoding pipeline (one upload → N renditions via async workers); adaptive bitrate (HLS/DASH, player switches on static segments); CDN as the entire read path; resumable chunked upload; manifest files.

1 · Requirements & scale

Functional: upload, transcode, store, and stream video to huge concurrent audiences across varying devices and bandwidths. Non-functional: smooth playback (adaptive quality), low startup latency, and origin bandwidth that doesn’t melt.

Back-of-envelope peak: tens of millions of concurrent streams x several Mbps = many Tbps one upload -> N renditions (240p..4K) x codecs = large transcode fan-out => you cannot serve this from origin; the CDN is the system

2 · High-level architecture

graph LR
  UP[upload] --> CH[chunked store]
  CH --> TR[transcode workers: N renditions]
  TR --> OBJ[(object store: segments + manifests)]
  OBJ --> CDN[CDN edge]
  CDN --> PLY[player: adaptive bitrate]

3 · The hard constraint, in depth

You never stream origin→user at scale. Every video byte must be served from a CDN edge near the user; the origin only fills the edge. So the real design question is always "what populates the CDN, and when?" — not "how do I serve the video?" (P11).

4 · Component deep dives

Encoding pipeline

One upload becomes many renditions (resolutions/bitrates/codecs) produced by async transcode workers — an embarrassingly parallel batch job. Each rendition is cut into short segments (2–10s).

Adaptive bitrate (ABR)

HLS/DASH: the server publishes a manifest listing segment URLs per rendition; the player measures bandwidth and switches quality segment-by-segment. The server just serves static files — intelligence lives in the client.

CDN & fill (P11)

Segments live in an object store and are cached at the edge; popular titles are pre-positioned (P11). Origin sees only misses.

Resumable upload

Large files upload in chunks by index so a dropped connection resumes mid-file.

5 · Real-world example

Netflix

Netflix transcodes each title into a large matrix of renditions (per resolution, bitrate, codec, and even per-title/per-scene optimized encodes) ahead of time, cuts them into segments, and serves everything via its own CDN, Open Connect (P11), whose appliances sit inside ISPs. During off-peak it pre-positions predicted-popular content onto those boxes, so at peak the user streams from an edge appliance on their own ISP and Netflix’s origin serves almost nothing. The player runs adaptive bitrate over the segment manifest. Encoding is a batch problem; serving is a CDN problem.

6 · Tradeoffs & alternatives

More renditions = better quality/efficiency per device but more transcode cost and storage. Pre-positioning wastes storage on titles that flop but guarantees peak performance; on-demand fill is cheaper but risks origin load. Own-CDN (Netflix) vs commercial CDN (most others) trades capex/control for simplicity.

7 · Key talking points

  • The CDN is the system; origin only fills it (P11).
  • One upload → many renditions via async transcode.
  • ABR (HLS/DASH): static segments + manifest, smarts in the player.
  • Pre-position popular content; segments in object store (S5, P8).
  • Chunked resumable upload.
