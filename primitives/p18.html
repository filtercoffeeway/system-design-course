P18 · Distributed sort & external merge (MapReduce)

What it is: sort (or group/join) a dataset far larger than one machine's RAM by spilling sorted runs to disk and merging them — and, at cluster scale, by shuffle-partitioning data across nodes.
Reach for it when: "sort/group/join a dataset bigger than memory," batch ETL, building an index, coordinating a cluster over a huge input.
The lever: never need all data in memory at once — produce sorted runs that fit in RAM, then do a streaming k-way merge; across nodes, range/hash-partition so each node sorts only its slice.
New problem it hands you: the shuffle (moving data so co-sorted/co-grouped rows land together) becomes the dominant cost (P7); data skew makes one partition a straggler (P12); intermediate spill needs disk + fault tolerance.
Exemplified by: Distributed SQL (sort-merge join), Video transcode (map/stitch), MapReduce/Spark.

1 · The problem it solves

You must sort, group, deduplicate, or join a dataset that doesn't fit in RAM — a terabyte of logs, a petabyte of clickstream, the input to an index build. An in-memory sort (quicksort) needs the whole array in memory and dies the moment the data exceeds RAM. The fix is a family of techniques that process the data in memory-sized pieces, the foundation under sort-merge joins, group-by, and the entire MapReduce/Spark batch model.

2 · The ladder: in-memory → external → distributed

Rung 0 — in-memory sort

sort(array) in RAM. O(n log n), simple, fast. Breaks on: data > RAM — you can't even load it.

Rung 1 — external merge sort (one machine, data > RAM)

Two phases. Run-generation (the "spill"): read a RAM-sized chunk, sort it in memory, write it to disk as a sorted run; repeat until the input is exhausted. Merge: open all runs at once and do a k-way merge — a min-heap holds the current head of each run; pop the smallest, write it to output, advance that run, repeat. Memory only ever holds one chunk (phase 1) or k small read-buffers + one output buffer (phase 2), never the whole dataset.

1 TB to sort, 8 GB RAM: run generation: 1 TB / 8 GB = 125 sorted runs of 8 GB each merge fan-in: RAM / per-run buffer = 8 GB / 8 MB = ~1000 runs mergeable in one pass 125 runs < 1000 -> ONE merge pass suffices passes = ceil(log_1000(125)) = 1 total disk IO ~ read+write data once per phase ~ 3 x 1 TB = 3 TB

The k-way merge, stepped: heads of 3 runs = [run1:2, run2:5, run3:1]; heap min = 1 → emit 1, advance run3 to its next value; heap is now [2,5,9] → emit 2; and so on. Each output record costs one O(log k) heap operation. Breaks on: too many runs for one merge pass (k > fan-in) forces multiple passes (each re-reads the whole dataset), and a single disk's I/O caps throughput.

Two optimizations that matter in practice. (1) Replacement selection generates longer runs than a plain sort-a-chunk approach: keep a min-heap of size = RAM, and as you emit the smallest record, pull in the next input record — if it's ≥ the one just emitted it can still join the current run, otherwise it's held for the next. On random data this yields runs of ~2× RAM on average (and a single run if the input is already nearly sorted), halving the number of runs the merge must handle. (2) Double-buffering / overlap: while the CPU merges from one set of read buffers, the next blocks are prefetched and the output buffer is flushed asynchronously, so disk I/O and compute overlap instead of stalling each other — the difference between being I/O-bound at disk speed and being twice as slow. Both are why a real external sort is closer to "read once, write once, merge once" than the naive pass count suggests.

Rung 2 — distributed sort / shuffle (data » one machine)

Spread the data and the sorting across a cluster. Two patterns:

  • Range-partition then local sort (TeraSort style): sample the data to pick range boundaries; route each record to the node owning its range (the shuffle); each node sorts its range locally (external merge sort, rung 1); concatenating nodes' outputs in range order yields a globally sorted result — no final global merge needed.
  • MapReduce shuffle (sort as a side effect): map emits (key, value); the framework partitions by key, sorts each partition, and groups equal keys; reduce sees keys in sorted order with their values grouped. Sorting is how grouping happens.
map -> partition(key) -> sort within partition -> reduce (keys arrive grouped & sorted) 1 PB over 1000 nodes -> ~1 TB/node; each node range-sorts its slice; shuffle moves each record once to its target partition <- the dominant cost

4 · The new bottleneck: shuffle & skew

The shuffle dominates. Sorting locally is cheap; moving every record across the network to its target partition is the expensive part — an all-to-all transfer of the entire dataset (the same cost that dominates distributed SQL joins, P7). Minimizing and overlapping shuffle with compute is the real engineering.

Skew makes one partition a straggler. If range boundaries are poorly chosen (or one key is hot), one node gets far more data than the rest and the whole job waits on it (P12). Fixes: sample the data to pick balanced range boundaries; salt a hot key across sub-partitions; speculative execution for the laggard.

The fault-tolerance dial differs too: batch frameworks (MapReduce, Spark) materialize the shuffle output to disk so a failed task re-runs from the last stage, not from scratch — the reason a 6-hour job survives a node death (contrast streaming MPP in S8, which re-runs the whole query).

6 · Real-world example

Google MapReduce / Hadoop / Spark

Google's MapReduce made distributed sort the backbone of batch computing: the framework's guaranteed "keys arrive at the reducer sorted and grouped" is a distributed external sort, implemented as map-side spill-and-sort + a shuffle + a reduce-side merge. The TeraSort benchmark (sorting a terabyte, later petabytes) is the canonical demonstration: sample to pick range boundaries, range-partition in the shuffle, sort each partition locally, and the concatenation is globally ordered. Spark generalizes this with in-memory shuffles and spill-to-disk when partitions exceed RAM — and a sort-merge join is literally "sort both sides (externally if needed), then merge." The recurring lesson: the sort is easy; the shuffle and the skew are the system-design problem.

7 · Applying it in a design

Reach for it whenever an operation needs all the data ordered or grouped but the data exceeds memory. Say: "this is bigger than RAM, so I'll generate sorted runs and k-way merge them; at cluster scale I'll range-partition in a shuffle and sort each partition locally — and I'll sample to avoid skew and materialize the shuffle for fault tolerance." It's the engine under group-by, sort-merge joins (S8), index builds (P6), dedup at scale (S18), and parallel transcode/stitch (S16).

8 · Pitfalls

  • Assuming an in-memory sort will "just" work on data near or over RAM (OOM / thrash).
  • Ignoring shuffle cost — it usually dwarfs the sort itself.
  • Picking range boundaries blindly instead of sampling → skewed partitions and a straggler.
  • Too many runs for a single merge pass → extra full passes over the data.
  • Not materializing intermediate shuffle output in long jobs → a single failure restarts everything.
