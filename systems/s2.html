2 · Message Queue / Kafka

Hard constraint: a partition is consumed sequentially by one consumer per group → throughput ceiling = partition count × per-consumer speed. Partition count is a capacity decision you must make early.
Primitives exercised: P1 (batching), P2 (partitioning), P3 (ISR/replication), P16 (backpressure), P17 (log storage), P9 (idempotency).

Log-structured storage (append beats random write); partitioning strategies; consumer-group parallelism ceiling; offset management, at-least-once vs EOS; replication factor + ISR; compacted topics for changelogs; retention as a replay buffer; rebalancing.

1 · Requirements

Functional: producers publish messages to topics; consumers read them, possibly much later, possibly re-reading; messages of a related key arrive in order; multiple independent consumer groups read the same stream at their own pace. Non-functional:

2 · Capacity estimation

1,000,000 messages/s x 1 KB = 1 GB/s ingest retention 7 days = 1 GB/s x 604,800 s ~ 600 TB x replication factor 3 ~ 1.8 PB on disk consumer-group throughput = (#partitions) x (per-consumer rate) to consume 1 GB/s at 50 MB/s/consumer -> need >= 20 partitions/consumers

The estimate exposes the central design fact: your consumer parallelism is capped by partition count, so partition count is the most important upfront decision (§3 ladder).

4 · The hard constraint, in depth

Within a consumer group, each partition is assigned to exactly one consumer, and that consumer reads the partition strictly in offset order. This is what gives per-key ordering — but it means you cannot have more active consumers in a group than partitions. Provision 10 partitions and an 11th consumer sits idle. You cannot raise parallelism without re-partitioning, which is operationally painful (it remaps keys and breaks ordering during the transition).

So partition count is a capacity decision made early and slightly over-provisioned. Too few = throughput ceiling. Too many = more metadata, more open files, longer leader elections, more end-to-end latency. The constraint is also the feature: it's exactly what makes per-key ordering cheap.

A topic is split into partitions (P2); each partition is an append-only log (P17) replicated across brokers (P3). Producers append, batched (P1); each partition is consumed by exactly one consumer within a group — both the parallelism mechanism and the constraint.

Why append beats random write (P17)

Each partition is an immutable, append-only sequence of records addressed by a monotonic offset. Appends are sequential disk writes — the single fastest thing a disk does, HDD or SSD — so write throughput is enormous. Reads are sequential scans from an offset. No in-place update, no read-modify-write, which is exactly why a log outruns a B-tree-backed queue. See P17.

Zero-copy and the page cache

Kafka leans on the OS page cache rather than a user-space cache: written data sits in cache and is served to consumers from there. Broker→consumer reads use zero-copy (sendfile): bytes go from page cache straight to the network socket without being copied into the JVM heap and back. This is a large part of why a single broker can push gigabytes per second — the data path never enters application memory.

6 · Partitioning strategy (P2)

The producer chooses a partition: by key hash (all records with the same key land on the same partition → per-key ordering), round-robin (max spread, no ordering), or custom. Key-hash partitioning is where the hot-partition problem bites: a single hot key (one giant tenant) overloads one partition while others idle, and that partition's single consumer becomes the bottleneck for the whole group.

New bottleneck — the hot partition. Mitigations escalate: a composite key (tenant + sub-id) to spread one tenant across partitions; or split the hot key into key#0..key#k and merge downstream (P7), accepting that strict per-key order is relaxed to per-sub-key. The general lesson: per-key ordering and even load are in tension, and the key design is where you trade between them.

7 · Replication, ISR & delivery semantics (P3)

Each partition has a leader and N−1 followers. The leader tracks the in-sync replica set (ISR) — followers caught up within a bound. With acks=all, a write is acknowledged only once all ISR members have it, so it survives the loss of any in-sync broker. The dial: acks=0/1/all and min.insync.replicas trade latency for durability — the P3 consistency ladder in concrete form. Setting min.insync.replicas=2 with RF=3 means a write needs 2 copies, tolerating one broker loss without blocking.

Offsets & delivery semantics

Consumers track their position with a committed offset. Commit after processing → at-least-once (a crash before commit replays the message). Commit before → at-most-once. Exactly-once (EOS) needs idempotent producers (dedup by producer-id + sequence number) plus transactional writes that atomically commit messages and offsets — real but more expensive, so reach for it only when duplicates are unacceptable. Pair at-least-once with idempotent consumers (P9) and you usually don't need EOS at all — the dedup happens at the effect, not the transport.

8 · Retention, replay, compaction & backpressure

Messages persist for a retention window (time or size) regardless of consumption, so a new or buggy consumer can replay from any offset — the log doubles as a recovery buffer after a bug. Log-compacted topics instead retain the latest value per key forever, making a topic a durable changelog/snapshot (the basis for event sourcing and for Kafka's own consumer-offset and Connect state).

Backpressure (P16): because storage is durable and consumers pull at their own pace, a slow consumer simply lags — it doesn't push back on producers. The log is the shock absorber. Your job is to monitor consumer lag and scale consumers (up to the partition ceiling). See P16.

9 · Consumer-group rebalancing

New bottleneck — rebalancing pauses. When a consumer joins/leaves/crashes, the group must reassign partitions. The classic eager protocol is stop-the-world: all consumers revoke all partitions, then re-acquire — the whole group pauses, and at scale (hundreds of consumers) that's a noticeable outage on every deploy. Relief: cooperative/incremental rebalancing (only the moved partitions are revoked, the rest keep processing) and static membership (a consumer keeps its identity across a brief restart so a rolling deploy doesn't trigger a reassignment at all). This is the real-world reason a naive "just add consumers" answer is incomplete — the rebalance cost is part of the design.

10 · Tradeoffs & alternatives

If you need per-message acknowledgement, complex routing, or priority queues, a traditional broker (RabbitMQ) or a managed queue (SQS) fits better than a log — Kafka deliberately trades those for throughput and replay. If you need millions of independent low-throughput queues, the per-partition overhead hurts. If you need strict global ordering across all keys, you're forced to a single partition and lose all parallelism — a sign the requirement itself should be questioned. Pull (consumer-paced, replayable, simple backpressure) vs push (lower latency, harder flow control) is the deep transport choice; Kafka chose pull, which is why the log can be the shock absorber.

11 · Real-world example

LinkedIn

Kafka was built at LinkedIn (~2010) to unify a sprawl of point-to-point data pipelines into one central log: every event — page views, metrics, database changes — published once and consumed by many systems (search, analytics, monitoring, Hadoop). The log-as-backbone idea is laid out in Jay Kreps' essay "The Log." LinkedIn has reported running Kafka at the scale of trillions of messages per day across thousands of brokers — the existence proof that an append-only, partitioned, page-cache-served log scales far past what a traditional broker (RabbitMQ, ActiveMQ) reaches.

Modern Kafka can replace ZooKeeper with the built-in KRaft consensus mode for metadata — the controller quorum uses Raft (P10), folding leader election and metadata storage into Kafka itself.

12 · Interview follow-ups (Q&A)

Q: How do you add consumer parallelism, and what's the limit? Add consumers to the group up to the number of partitions; beyond that they idle. To go higher you must add partitions, which remaps keys and disrupts ordering during the change — so size partitions for peak with headroom from the start.

Q: How does Kafka guarantee ordering? Only per partition: one consumer reads a partition in strict offset order, and a key always hashes to the same partition. There is no global order across partitions — and you almost never need one.

Q: A committed message must never be lost — how? Replication factor 3 with acks=all and min.insync.replicas=2: a write isn't acked until 2 in-sync replicas have it, surviving one broker loss. It trades a little latency for durability (P3 dial).

Q: At-least-once vs exactly-once — which and why? Default to at-least-once (commit offset after processing) plus an idempotent consumer (P9), so replays are harmless. Use EOS (idempotent producer + transactions) only when duplicates are truly unacceptable, because it costs throughput and complexity.

Q: One key is far hotter than the rest — what happens? Its partition (and that partition's single consumer) becomes the bottleneck while others idle. Fix with a composite key or by splitting the hot key across sub-partitions and merging downstream — trading strict per-key order for balance.

Q: Why does every deploy cause a latency blip? Eager rebalancing stops the whole group to reassign partitions when membership changes. Use cooperative/incremental rebalancing and static membership so a rolling restart moves only affected partitions (or none).

Q: A consumer was down for hours — does it lose messages? No — retention is decoupled from consumption, so it resumes from its committed offset and replays the backlog (consumer lag), as long as the data is still within the retention window.

13 · Key talking points

  • Lead with the ceiling: "parallelism = partition count, so I'll size partitions for peak consumer throughput with headroom."
  • Append-only log = sequential writes; page cache + zero-copy reads (P17).
  • Key-hash partitioning for per-key order; call out the hot-partition risk (P2).
  • ISR + acks=all + min.insync.replicas for durability; name the latency/durability dial (P3).
  • At-least-once + idempotent consumers by default; EOS only when required (P9).
  • Retention = replay buffer; compaction = changelog; rebalancing cost is part of the design.
