17 · Payment System

Hard constraint: you can't span a distributed transaction across your DB and an external processor — only your side is controllable. Saga + idempotency + reconciliation is the standard answer.
Primitives exercised: P14, P9, P3 (strong consistency).

Double-entry bookkeeping (entries sum to zero → bug detection); idempotency keys on payment APIs; saga with compensations; reconciliation as the correctness net; exactly-once via a transactional idempotency table.

1 · Requirements

Functional: move money correctly — charge, transfer, refund — integrating external processors/banks. Non-functional: correctness above all (no money created or destroyed), full auditability, and the one domain where eventual consistency is unacceptable. Throughput is secondary to being right.

correctness: reconcile to the cent — zero money created/destroyed external deps: card networks, banks, Stripe — you control ONLY your side every state change auditable & immutable (regulators, disputes)

2 · API & data model

POST /charges Idempotency-Key: <uuid> (retry-safe — P9) body: { amount: 5000, currency: usd, source, merchant } -> { id, status: succeeded|pending|failed }

The source of truth is a double-entry ledger: every money movement is recorded as balanced entries that sum to zero. Append-only and immutable — a correction is a new compensating entry, never an edit. A $50 charge with a $2 fee:

txn T1 "charge $50, $2 platform fee": account amount customer_funds -50.00 (money leaves customer) merchant_balance +48.00 platform_revenue +2.00 ---------------------------------- SUM 0.00 <-- the invariant: if it isn't 0, you have a bug

The zero-sum rule isn’t accounting nostalgia — it’s a continuous self-check that makes a whole class of bugs (lost or duplicated money) impossible to commit silently.

3 · High-level architecture

graph LR
  API[payment API + idempotency] --> LEDGER[(double-entry ledger)]
  API --> SAGA[saga: authorize -> capture -> notify]
  SAGA --> EXT[external processor / bank]
  RECON[reconciliation job] --> LEDGER
  RECON --> EXT

4 · The charge write path

POST /charges (Idempotency-Key K): 1. INSERT idempotency key K (in_progress) — unique constraint (P9) if K already done -> return stored result (no double charge) 2. authorize with processor (saga step 1) [external] 3. on auth OK: write balanced ledger entries + mark K done IN ONE TXN 4. capture (saga step 2); notify (step 3) if capture fails -> compensation: void/refund the authorization (P14)

The two correctness primitives carry the load: idempotency (P9) so a retried POST /charges can never double-charge (the key + ledger write commit together), and the saga (P14) so a multi-step flow that fails midway is unwound by compensations rather than left half-done.

5 · The hard constraint, in depth

You cannot wrap a distributed transaction across your database and an external processor (Stripe, a bank) — you only control your side and can’t two-phase-commit with VISA. So the standard answer is saga + idempotency + reconciliation: drive the steps with compensations (P14), make every call idempotent (P9), and run a batch that compares your ledger to the processor’s statement line by line to catch and fix any drift. Reconciliation is the real correctness guarantee — an after-the-fact net, not a real-time one, because real-time agreement with an external party is impossible.

6 · Real-world example

Stripe / Uber ledger

Stripe’s API is built on idempotency keys (P9) precisely so charge retries are safe, and internally money movement is modeled as immutable double-entry ledger entries with reconciliation against banks/card networks. Uber built LedgerStore, a purpose-built immutable double-entry ledger handling tens of billions of entries to record every financial event (fares, payouts, refunds) with strong auditability. Both show the pattern: idempotent APIs, an append-only double-entry ledger as the source of truth, sagas for multi-step flows, and reconciliation as the final net.

7 · Tradeoffs & alternatives

Strong consistency and auditability are non-negotiable, so you accept lower throughput and more process than other domains. 2PC across your own services is possible but blocks resources; a saga is preferred. Real-time guarantees are impossible against external parties — reconciliation is how you make it right after the fact.

8 · Key talking points

  • Double-entry ledger: entries sum to zero = built-in bug detector.
  • Idempotency keys to prevent double-charge (P9).
  • Saga + compensation for multi-step flows (P14).
  • Reconciliation is the real correctness mechanism, not real-time.
  • You only control your side; no distributed txn with the bank.
