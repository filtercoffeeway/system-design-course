12 · Distributed Task Scheduler (Cadence / SQS+Lambda)

Hard constraint: you can't have exactly-once execution and fault-tolerance without a too-expensive distributed transaction — accept at-least-once + idempotent workers.
Primitives exercised: P9, P10 (cron firer), P16 (leasing).

Idempotency as first-class; visibility timeout; dead-letter queue; the "who fires the cron?" problem (single scheduler + distributed lock); task leasing with renewal.

1 · Requirements & scale

Functional: run a unit of work reliably — now, later, or on a schedule — with retries and no duplicate effect. Non-functional: at-least -once execution, fault tolerance, horizontal worker scale, visibility into failures.

Back-of-envelope 10,000,000 tasks/day ~ 115/s avg, bursty each task: lease -> run -> ack, with retries delayed tasks a future-time index scanned continuously

2 · High-level architecture

graph LR
  P[producer] --> Q[(task queue)]
  Q -->|lease w/ visibility timeout| W1[worker]
  Q --> W2[worker]
  W1 -->|ack on success| Q
  W1 -->|fail x N| DLQ[(dead-letter queue)]
  SCH[scheduler + lock] -->|enqueue due tasks| Q

3 · The hard constraint, in depth

You cannot get both exactly-once execution and fault tolerance without an expensive distributed transaction. A worker can finish a task and crash before acking — so the task will run again. The accepted answer is at-least-once delivery + idempotent workers (P9): make running twice produce the same result, and stop chasing true exactly-once.

4 · Component deep dives

Visibility timeout & leasing

When a worker picks up a task it becomes invisible to others for T seconds (the visibility timeout / lease). If the worker finishes and acks, the task is deleted; if it dies, the lease expires and another worker retakes it. Long tasks renew the lease.

Retries & dead-letter queue

Transient failures retry with exponential backoff; after N failures the task moves to a DLQ for inspection so a poison message can’t loop forever. You always want a DLQ.

Cron in a distributed system (P10)

"Who fires the scheduled job?" If every node fires it, it runs N times. The standard answer: a single scheduler elected via a distributed lock / leader election (P10, S13), which enqueues due tasks for the worker pool.

5 · Real-world example

AWS SQS + Lambda & Uber Cadence

The simple case is SQS + Lambda: SQS provides at-least-once delivery, a visibility timeout so two consumers don’t process the same message at once, and a built-in dead-letter queue after a max receive count; Lambda workers must be idempotent because redelivery happens. For complex multi-step jobs, Uber built Cadence (now Temporal), which persists workflow state so a long-running task resumes exactly where it left off after a crash and handles retries/timeouts as first-class concepts — the same at-least-once + idempotency philosophy, scaled to durable workflows (and the engine behind sagas, P14).

6 · Tradeoffs & alternatives

A plain queue (SQS) is simple but you build orchestration yourself; a workflow engine (Temporal/Cadence) handles state and retries but is heavier. Shorter visibility timeouts recover from dead workers faster but risk double-processing live-but-slow ones.

7 · Key talking points

  • At-least-once + idempotent workers; drop the exactly-once dream (P9).
  • Visibility timeout / lease prevents double-running.
  • Always a dead-letter queue for poison tasks.
  • Single elected scheduler for cron (P10).
  • Exponential backoff on retries.
