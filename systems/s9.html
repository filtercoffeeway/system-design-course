9 · Social Feed (Twitter / Instagram)

Hard constraint: the celebrity / hot-user problem — one post by an account with 100M followers triggers 100M timeline writes. Naive fan-out-on-write can't absorb that, and naive fan-out-on-read can't serve the timeline fast. The answer is a hybrid driven by author degree.
Primitives exercised: P5 (fan-out), P2 (hot-key), P4 (timeline cache / sorted set), P7 (merge), P12 (tail).

Fan-out on write vs read; hybrid (push for the tail, pull for celebrities) + hot-author detection; Redis sorted set as the timeline; chronological merge vs ML ranking (candidate-gen + re-rank).

1 · Requirements

Functional: users follow others; a post distributes to followers; each user opens a merged, recent (often ML-ranked) home timeline; support likes/reposts/replies counts. Non-functional:

PropertyTargetConsequence
Read latencytimeline opens in < ~200 msprecompute timelines; cache (P4)
Read:write ratioreads » writes (~20:1+)do work at write time, not read time
Graph skewpower-law: most tiny, a few at 100M+no single fan-out works (§4, the wall)
Freshnessnew posts appear in secondspush for most; pull-merge for the few huge
Availabilitytimeline always loads (stale OK)cache is rebuildable from source posts

The whole design follows from one asymmetry: reads vastly outnumber writes, so you want to pay at write time to make reads cheap — except the celebrity case breaks that, which is the entire interview.

2 · Capacity estimation

300,000,000 DAU, each opens timeline ~10x/day = 3,000,000,000 reads/day -> ~35,000 reads/s average, ~100,000/s peak posts: ~0.5/user/day = 150,000,000/day -> ~1,700/s -> read:write ~ 20:1 -> optimize the READ path fan-out-on-write cost (avg 200 followers): 150M posts x 200 = 30,000,000,000 timeline writes/day celebrity (100M followers): ONE post = 100,000,000 timeline writes at 1M writes/s that's ~100 s of pure fan-out for a single tweet <- the wall timeline cache: 300M users x ~800 recent entries x 16 B (id+score) ~ 3.8 TB in Redis

3 · API & data model

post(user_id, content) -> post_id getTimeline(user_id, cursor, limit) -> [posts] // the hot path follow(user_id, target) / unfollow(...) posts: post_id (Snowflake, time-sortable) -> {author, content, ts, counts} graph: followers(user_id) -> [user] / following(user_id) -> [user] timeline: user_id -> sorted set { post_id : score } (Redis, capped to recent N) (P4)

Post IDs are Snowflake-style (time-ordered, see S14/P10) so a timeline sorts by ID without a separate timestamp and merges are trivial.

4 · The fan-out ladder (the heart of the design)

Rung 1 — fan-out on read (pull)

Store nothing per-follower. At read time, fetch the recent posts of everyone the user follows and merge-sort by time (P7). Good until: users follow many accounts — merging hundreds of followees' posts on every timeline open, 100k times/s, is enormous read-time work and slow. Reads dominate, so paying at read time is exactly backwards.

Rung 2 — fan-out on write (push)

When a user posts, immediately append the post id to every follower's precomputed timeline (a Redis sorted set). A read is then O(1): just slice the user's own timeline. Perfect for the read-heavy workload. Breaks on: the celebrity — a 100M-follower post is 100M writes per tweet (~100 s of fan-out), a massive bursty amplification; and it wastes writes pushing to inactive followers who'll never open the app.

Rung 3 — hybrid (push for the tail, pull for the head)

Classify authors by follower count. Ordinary authors (below a threshold T) are pushed at write time into followers' timelines. Celebrities (followers ≥ T) are not pushed; instead, at read time, the reader's timeline is the merge of (a) their precomputed pushed timeline and (b) a pull of the handful of celebrities they follow (whose recent posts are cached once, globally, and read by all their followers).

read(user): base = timeline_cache[user] // pushed posts from normal authors (O(1)) celebs = [c for c in following(user) if is_celebrity(c)] recent = union(celeb_recent_cache[c] for c in celebs) // few, globally cached return rank(merge(base, recent))[:limit]

Why this works: pushing a celebrity to 100M timelines is the expensive thing, and there are few celebrities, so pulling them is cheap (one cached read per celebrity, shared by all followers). Pushing the long tail is cheap per author and makes the common read O(1). You pay the right cost on the right side for each population. The threshold T (~tens of thousands of followers) is tuned where push cost (followers per post) exceeds pull cost (followers-who-open × merge work).

5 · High-level architecture

graph TD
  POST[post] --> CL{author degree?}
  CL -->|normal| FO[fan-out workers] -->|append id| TL[(per-user timeline: Redis sorted set)]
  CL -->|celebrity| CC[(celeb recent-posts cache, global)]
  READ[open timeline] --> TL
  READ --> CC
  TL --> MG[merge + rank] --> READ
  POST --> PS[(durable post store)]

Fan-out is async off the post path: post() writes the durable post + enqueues a fan-out job; workers drain it into follower timelines. The timeline cache is capped to the most recent N ids and is rebuildable from the durable post store + graph, so losing it is a latency event, not data loss (P4).

6 · The hot-author / celebrity handling (P2)

The celebrity is a hot key (P2): the author node, and the read fan-out of their post, concentrate load. Detection is by follower count (or recent posting velocity), maintained as the graph changes. A user can cross the threshold (a viral moment), so the classifier must update and the system must handle the transition (stop pushing, start being pulled).

Two subtleties the sharp interviewer raises: (1) The fan-out itself must be fanned out — even a 200-follower push, multiplied by 1,700 posts/s, is millions of writes/s, so fan-out workers are sharded and the timeline store is partitioned by user. (2) Bursty celebrity reads — when a celebrity posts, all their followers' next reads pull the same cached post; request coalescing / a hot replica (P4) prevents a stampede on that one cache key.

7 · Ranking: chronological vs ML (ladder)

Chronological is a pure time-sorted merge — trivial, but lower engagement and vulnerable to spam volume. ML-ranked feeds change the shape: the timeline is no longer "the most recent posts" but "the best posts," scored by predicted engagement, affinity to the author, recency, and media type. That turns the read into a two-stage pipeline (same shape as S7/S19):

  • Candidate generation: gather a few hundred candidates (pushed timeline + pulled celebrity posts + "out-of-network" recommended posts).
  • Ranking: a model scores each candidate; the top ones are returned. The expensive model runs only over the recalled shortlist, never the whole post corpus.

This is why a ranked feed can't just be a Redis slice: the sorted set gives candidates, and an inference service produces the final order. The counts (likes/reposts) feeding ranking are themselves an approximate-counting problem at scale (S22).

8 · Scaling & bottlenecks at 10×

At 1M reads/s: the timeline cache shards by user; celebrity post caches get extra replicas to absorb read bursts. Fan-out becomes the write-side bottleneck during mega-events (a global celebrity posts during the Super Bowl) — the fan-out queue absorbs the burst and drains over seconds (followers see the post within seconds, which is fine). Inactive-follower waste grows: optimize by only pushing to recently active users and lazily rebuilding a returning user's timeline from source (pull) on login. The graph store (followers/following) is itself huge and is sharded; "who follows X" for a celebrity is a hot, large list, often stored/streamed rather than loaded wholesale.

9 · Tradeoffs & alternatives

Pure push: fastest reads, but the celebrity storm and wasted writes to dormant followers. Pure pull: cheap writes, slow reads — wrong for a read-heavy product. Hybrid: best overall, at the cost of two code paths and a hot-author classifier. Chronological (simple, transparent, "real-time") vs ML-ranked (more engagement and infra, less predictable, harder to debug "why did I see this?"). Cap stored timeline length (you don't need a user's 10,000th-oldest post in cache) to bound memory — deep scrollback falls back to a pull from the durable store.

10 · Real-world example

Twitter / Instagram

Twitter's home-timeline architecture pushes tweets into per-follower timelines held in Redis for fast O(1) reads, but switches to pulling at read time for very-high-follower accounts to avoid the fan-out explosion — the canonical hybrid, and the source of the "celebrity problem" framing. Instagram (Django + Cassandra + Redis) similarly precomputes feeds, caps stored entries, and leans heavily on caching and denormalization. Both moved from chronological to ranked feeds, turning the read into candidate-generation + ML ranking. In every case the architecture is a direct consequence of the celebrity constraint: there is no single fan-out strategy that serves both the long tail and the megastars.

11 · Interview follow-ups (Q&A)

Q: Why not just fan-out on write for everyone? A 100M-follower post is 100M writes (~100 s of fan-out for one tweet) and wastes writes on dormant followers. The expensive population (celebrities) is small, so pull them at read time instead; push only the cheap long tail.

Q: How do you pick the push/pull threshold? Where push cost (followers written per post) exceeds pull cost (active followers × per-read merge of that author). Empirically tens of thousands of followers; it's tuned, and authors crossing it switch strategy.

Q: A user follows 50 celebrities — their read now merges 50 pulled lists. Is that slow? Each celebrity's recent posts are cached once globally and read by all followers, so it's 50 cheap cache reads + a small merge, not 50 expensive computations. Coalesce concurrent reads of a hot celebrity post.

Q: How do you avoid a stampede when a celebrity posts and millions refresh? The celebrity post lands in one global cache key; request coalescing / hot replicas serve the burst (P4), rather than each reader recomputing.

Q: How does an ML-ranked feed change the design? The timeline sorted set becomes a candidate source, not the final order; a ranking service scores candidates and reorders. Reads go from a Redis slice to candidate-gen + inference, adding latency and infra (same pattern as S7).

Q: What happens when the timeline cache is lost? It's a rebuildable cache, not the source of truth — rebuild a user's timeline from the durable post store + their follow graph on next read. Availability over precomputation; the cost is a slower first load.

Q: How do you handle counts (likes/reposts) at this scale? Approximate, mergeable counters rather than a hot row per post (S22/P2 hot-key) — exact-to-the-like isn't required for ranking or display.

12 · Key talking points

  • Reads » writes (~20:1) → do the work at write time to make reads O(1).
  • Fan-out ladder: pull → push → hybrid (push the tail, pull celebrities) — the celebrity is a hot key (P5, P2).
  • Per-user timeline = a capped Redis sorted set, a rebuildable cache (P4); Snowflake ids make merges time-ordered.
  • Push only to active followers; lazily rebuild a returning user's timeline from source.
  • Ranked feed = candidate generation + ML re-rank, not a plain merge (same shape as S7/S19).
  • Celebrity post = a hot cache key; coalesce reads to avoid a stampede.
