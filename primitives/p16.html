P16 · Backpressure / flow control

What it is: letting a slow consumer tell a fast producer to slow down, rather than collapsing.
Reach for it when: any producer can outpace any consumer (ingestion, queues, streaming).
The lever: bounded buffers; credit/ack-based flow control; a durable queue as a shock absorber; explicit drop/sample policies when losing data beats falling over.
New problem it hands you: if you block, latency propagates upstream; if you drop/sample, you lose data; an unbounded buffer just defers the collapse and grows lag.
Exemplified by: Kafka, Datadog ingest, Rate limiter.

1 · The problem it solves

A fast producer pointed at a slower consumer eventually overruns it: queues grow without bound, memory fills, and the system collapses — often catastrophically and all at once. Backpressure is the mechanism by which a consumer tells a producer "slow down" so the system degrades gracefully instead.

2 · Why an unbounded buffer is a trap (Little's Law)

The instinct — “just add a bigger queue” — defers collapse, it doesn’t prevent it. Whenever arrival rate exceeds service rate, the queue grows without bound, and Little’s Law (L = λW) turns that backlog directly into ever-growing latency:

producer 10,000/s -> consumer 8,000/s (deficit 2,000/s) unbounded queue grows 2,000 items/s; after 60s -> 120,000 backlog Little's Law: wait W = L / throughput = 120,000 / 8,000 = 15 s lag, and rising -> the data is stale before it's processed; memory eventually OOMs

A bounded buffer forces the decision instead of hiding it. With a cap of 10,000 the buffer fills in 5 s, then a policy must fire.

3 · The options, lossless to lossy

graph LR
  P[producer] -->|bounded buffer| C[consumer]
  C -.->|credits / acks: I can take N more| P
  P -.->|buffer full: block / drop / sample| X[policy]

Block the producer (lossless, but propagates slowness upstream); credit/ack flow control (consumer grants the producer permission for N more items, so the producer self-limits to the consumer’s rate and the queue never overflows); a durable queue as a shock absorber (Kafka — absorbs bursts on disk); or explicit drop/sample when losing data beats falling over (metrics, logs). Worked credit flow:

consumer advertises credit = 1,000 (its free buffer slots) producer may send at most 1,000 in-flight; as consumer drains, it returns credits -> producer's effective rate is pinned to the consumer's drain rate (8,000/s) -> queue depth stays ≤ the advertised window; no overflow, no OOM

4 · The new bottleneck it hands you

Every choice has a cost: blocking pushes latency back up the chain (and can deadlock if it reaches the user); dropping/sampling loses data; an unbounded buffer just defers the collapse while growing lag until the data is useless. There is no free option — you are choosing which failure mode you prefer.

5 · Real-world example

TCP & Netflix concurrency-limits

The oldest example is TCP flow control: the receiver advertises a window ("I have room for N more bytes") and the sender must not exceed it — credit-based backpressure built into the internet. At the application layer, Netflix open-sourced concurrency-limits, an adaptive load-shedder that treats a service like TCP: it infers the right concurrency limit from observed latency (TCP-Vegas-style) and sheds excess requests rather than letting queues build and latency explode. The Reactive Streams standard (RxJava, Project Reactor) bakes credit-based backpressure into stream APIs for the same reason.

6 · Applying it in a design

Any producer/consumer boundary: "bounded buffer with explicit policy — block if we can’t lose data, sample if we can; a durable log absorbs bursts; shed load past capacity." It’s the natural follow-on to batching (P1).

7 · Pitfalls

  • Unbounded in-memory queues (OOM waiting to happen).
  • No defined policy for "buffer full."
  • Retry storms that ignore backpressure signals.
