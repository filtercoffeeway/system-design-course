1 · Rate Limiter

Hard constraint: the check sits on the hot path of every request, so it must add near-zero latency. If checking the limit costs more than the request it protects, you solved the wrong problem — that rules out anything chatty or strongly-coordinated per call.
Primitives exercised: P16 (backpressure), P3 (counter consistency), P4 (Redis state), P2 (sharded counters).

Fixed window / sliding log / sliding-window counter / token bucket / GCRA (and their failure modes); distributed counter coordination (local accuracy vs global latency); Redis as fast-but-not-perfectly-consistent state; fail-open vs fail-closed when the store is down; multi-dimensional limiting.

1 · Requirements

Functional: allow N requests per window per identity (user, IP, API key, endpoint), reject or queue the rest, and tell the client when to retry (429 + Retry-After). Support multiple, simultaneous limit dimensions and tiers. Non-functional:

PropertyTargetConsequence
Latencycheck < ~1 ms (cheaper than the request)one O(1) atomic op against memory (§4–5)
Distributionone limit across 1000s of app serversshared counter or coordinated local (§5)
Accuracyenforce the rate without big over/under-admitalgorithm choice (§4); accuracy/latency dial
Resiliencesane behavior if the limiter's store is downfail-open vs fail-closed (§6)
Memorystate per identity stays tinyO(1)-per-identity algorithms

The defining tension: a rate limiter is pure overhead on the happy path, yet it must enforce a global limit across a fleet — and global state is exactly what's slow. The whole design is reconciling "near-zero latency" with "one shared count."

2 · Capacity estimation

1,000,000 req/s across 1,000 app servers = 1,000 req/s/server limit-check budget < ~1 ms (must be cheaper than the work it guards) state per identity ~ a counter + timestamp (tens of bytes) 10,000,000 active identities x ~50 B ~ 500 MB -> fits in Redis memory on one node (shard by identity across Redis nodes if hotter; see P2)

3 · API design

allow(key, limit, window) -> { allowed: bool, remaining: int, retry_after: sec } key = "user:123" | "ip:1.2.3.4" | "apikey:abc:endpoint:/charge" on deny: HTTP 429 + headers Retry-After: 3 X-RateLimit-Limit / X-RateLimit-Remaining / X-RateLimit-Reset

Returning remaining and reset lets well-behaved clients self-throttle before hitting the wall — cheaper for everyone than reactive 429s.

4 · The algorithm ladder (full family)

Rung 0 — fixed window

One counter per clock window: INCR limit:user:12:00, reject past N, key expires each minute. Dead simple, O(1) memory. Breaks on the boundary burst — it allows 2× the rate across a window edge:

limit 100/min: 12:00:59 -> 100 requests (fills the 12:00 window) 12:01:00 -> 100 requests (fresh 12:01 window) => 200 requests in ~1 second, twice the intended rate

Rung 1 — sliding window log

Store the timestamp of every request (a sorted set); count those within the last 60 s, evicting older. Exact, no edge burst. Breaks on memory: it stores one entry per request, so a client doing 1,000 req/min costs 1,000 timestamps — expensive, and the eviction work grows with volume.

Rung 2 — sliding window counter (the approximation sweet spot)

Keep just two fixed-window counters (current + previous) and weight the previous one by how much of it still falls inside the rolling window. O(1) memory, no hard edge burst:

limit 100/min; at 12:01:15 (15 s into current window): prev (12:00) count = 80, current (12:01) count = 30 weight_prev = (60 - 15)/60 = 0.75 estimate = 30 + 80*0.75 = 90 < 100 -> ALLOW (smoothly decays the previous window instead of dropping it at the boundary)

The cost is a small approximation (it assumes the previous window's requests were uniform), which is fine for almost all rate-cap use cases — the usual default for a pure cap.

Rung 3 — token bucket (default when bursts should be allowed)

Store (tokens, last_refill); each request refills tokens = min(b, tokens + rate × elapsed) and spends one. Bounds the long-run rate and permits a controlled burst up to bucket size b — the best UX for APIs (a client idle for a while can spike briefly):

bucket b=10, refill r=1/s: t=0 tokens=10 -> 10-request burst -> tokens=0 (burst up to b allowed) t=0..5 idle -> tokens refill to 5 t=5 request -> tokens=4 long-run rate bounded to r, bursts to b

Leaky bucket is the queueing dual: requests enter a fixed-size queue that drains at rate r, smoothing bursty input into a steady output at the cost of added delay (and drops when the queue is full). Use token bucket to allow bursts, leaky bucket to absorb and smooth them.

Rung 4 — GCRA (generic cell rate algorithm)

The token bucket's elegant equivalent that stores a single number: a "theoretical arrival time" (TAT). Each request checks whether now is past TAT minus the allowed burst tolerance, and advances TAT by the inter-request interval. Same behavior as token bucket (rate + burst) with one timestamp and no separate refill step — popular in libraries (e.g. Redis rate-limit modules) precisely because it's O(1) state and exact.

5 · Making it distributed — the counter ladder

1M req/s over 1,000 servers means the count is shared state. Another ladder, trading accuracy against hot-path latency:

RungMechanismBreaks on
Per-node local counterseach server limits its own slice (limit/N each)loose global limit; uneven load means some nodes reject while others have headroom
Global counter in Redisone atomic INCR+expire, or a Lua script for token-bucket/GCRA, per requesta round trip on the hot path; Redis is now a dependency & a hotspot
Redis + local pre-check (typical)cheap local guard admits up to a local lease, syncs to Redis periodicallyslight over-admit (bounded by the lease) — acceptable
Atomicity is non-negotiable. A read-then-write (GET; check; SET) races under concurrency — two requests both read 99, both allow, count hits 101. The increment-and-check must be one atomic operation: a single INCR (fixed/sliding window) or a Lua script Redis runs atomically (token bucket / GCRA need read-modify-write of tokens+last_refill). This is P3 consistency on the counter.
The hot counter. A single global key for a popular tenant is itself a hotspot. Relief: shard the counter into key#0..key#k sub-counters across Redis nodes (P2), each holding 1/k of the budget, and sum/borrow as needed — the inverse of S22's counting problem. The local-pre-check rung is the common production answer: each node holds a small lease of the budget and only talks to Redis when its lease is exhausted, bounding both latency and over-admit.

6 · Failure mode: fail-open vs fail-closed

The limiter has its own dependency (Redis). When it's unreachable you must have already decided: fail-open (allow the request — protect availability, accept that the backend is briefly unprotected) or fail-closed (deny — protect the backend, accept that you're rejecting legitimate traffic). The right choice depends on what the limiter guards: a public read API usually fails open (don't take the site down because Redis blipped); a fragile or abuse-prone endpoint (login, expensive write) fails closed. A local in-memory fallback that enforces a conservative per-node limit during the outage is a good middle ground.

7 · Multi-dimensional & layered limiting

Real APIs don't have one limit; they enforce several at once, and a request must pass all: per-user (fairness), per-IP (abuse), per-API-key (billing tier), per-endpoint (protect an expensive route), and a global limit (protect the whole backend). Each is its own counter keyed differently; the check evaluates the relevant set and denies if any is exhausted. Beyond rate, two other limiter kinds matter: a concurrency limiter (cap simultaneous in-flight requests so one slow client can't hog all workers — a semaphore, not a counter) and load shedding (under stress, reject low-priority traffic to keep capacity for critical traffic). "Rate limiting" at scale is really a small portfolio of limiters defending different resources.

8 · Scaling & where the limiter lives

The check runs at the edge — API gateway / reverse proxy / middleware — so it rejects before a request consumes app resources. At higher scale the Redis tier shards by identity (P2) and the local-pre-check rung cuts round trips. The limiter is also a P16 backpressure mechanism: it converts "too much load" into explicit, early, cheap rejections (with Retry-After) rather than letting the backend collapse. It's a building block of larger systems — login rate-limiting in S20, provider pacing in S11, the flash-sale waiting room in S21.

9 · Tradeoffs & alternatives

Local-only counters scale infinitely but enforce limits loosely; a global Redis counter is accurate but a hot-path dependency (mitigate with local pre-check + fail-open). Exact sliding-log costs memory; the sliding-window-counter approximation is the usual sweet spot; token bucket/GCRA win when bursts are desirable. Reject (429) is cheap and honest; queueing (leaky bucket) smooths bursts but adds latency and hides overload. The deepest call is fail-open vs fail-closed — it encodes whether availability or backend protection matters more for the thing being guarded.

10 · Real-world example

Stripe

Stripe documented running multiple layers of rate limiters in front of its API rather than one. They distinguish a request-rate limiter (token-bucket style, the steady-state cap), a concurrency limiter (cap simultaneous in-flight requests so one client can't hog workers), and load-shedding tiers that prioritize critical traffic (e.g. live charges) over less-critical traffic (e.g. test or batch calls) when the system is under stress. The limiters are implemented in Redis with atomic operations (Lua scripts for the read-modify-write algorithms). The lesson matches §7: at scale "rate limiting" is several different limiters defending different resources, not one counter.

11 · Interview follow-ups (Q&A)

Q: Why not just use a fixed window — it's simplest? The boundary burst: a client can send the full quota at the end of one window and again at the start of the next, doubling the real rate for ~1 s. Sliding-window-counter or token bucket fixes it with the same O(1) memory.

Q: Token bucket vs leaky bucket? Token bucket allows a burst up to b and bounds the average — good API UX. Leaky bucket smooths bursty input into a steady drain (queue), adding latency and dropping on overflow. Choose by whether you want to permit bursts or absorb them.

Q: How do you enforce one global limit across 1,000 servers cheaply? A shared atomic counter in Redis (single INCR or a Lua script for token bucket), fronted by a per-node local pre-check that holds a small lease of the budget so most requests never make a network call — bounding both latency and over-admit.

Q: Why does the increment have to be atomic? Read-then-write races: two concurrent requests both read 99 and both allow, exceeding the limit. A single atomic INCR or a server-side Lua script makes increment-and-check indivisible.

Q: Redis is down — allow or deny? A pre-decided policy per use case: fail-open for availability-critical read paths (with a conservative local fallback), fail-closed for abuse-prone or fragile endpoints. There's no universal right answer; it encodes which risk you'd rather take.

Q: A single hot key (one huge tenant) overloads one Redis node — fix? Shard the counter into k sub-counters across nodes (P2), each with 1/k of the budget, and aggregate/borrow — the same hot-key split used elsewhere in the course.

Q: How do you handle bursts from many keys vs one key? Per-key token buckets bound each identity's burst independently; a separate global limiter and a concurrency limiter protect the shared backend from the aggregate even when no single key misbehaves.

12 · Key talking points

  • Lead with the constraint: "the check must be cheaper than the request — one atomic Redis op."
  • Algorithm ladder: fixed window (edge burst) → sliding log (memory) → sliding counter → token bucket/GCRA (burst + O(1) state).
  • Distributed ladder: local-only (loose) → global Redis (accurate, a dependency) → Redis + local pre-check (the sweet spot); increment must be atomic (P3).
  • Fail-open vs fail-closed when the store is down — decided per endpoint.
  • Multi-dimensional (user/IP/key/endpoint/global) + concurrency limiter + load shedding; it's a portfolio, not one counter.
  • Return 429 + Retry-After/remaining so clients self-throttle; it's a P16 backpressure tool.
