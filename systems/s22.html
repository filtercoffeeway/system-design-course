22 · Top-K Trending Topics (Twitter trends)

Hard constraint: over a firehose of ~105–106 token-events/s, continuously surface the top ~K (10–50) topics for a sliding window — and rank them by surge, not raw volume — using memory that does not grow with the number of distinct topics. So the spine is approximate counting (Count-Min Sketch + heavy-hitter summary) + per-shard top-K with a scatter-gather merge + time-decay/windowing + a baseline-deviation score.
Primitives exercised: P7 (scatter-gather merge of per-shard top-K), P2 (partition the count stream / hot-key), P1 (batch base + streaming speed layer), P4 (cache the tiny result), P8 (columnar rollup store for windows), P16 (backpressure on the firehose).

This is the “Twitter trending” / “top-K viewed” / “bestseller” / “most-shared-articles-in-the-last-hour” question. The trap is to model it as SELECT topic, COUNT(*) ... ORDER BY 2 DESC LIMIT k. At firehose scale you cannot keep an exact counter per distinct topic, you cannot sort millions of topics per refresh, and — crucially — raw count is the wrong ranking: an evergreen term beats a breaking one every time. We derive each of those three away.

1 · Requirements

Functional. (1) getTrending(window, region, k) returns the top k trending topics (hashtags / phrases / entities) for a time window; (2) windows are multiple and rolling — “now / last 5 min”, “last hour”, “last 24 h”; (3) trending means rising — a topic that is surging relative to its own baseline, not merely high-volume; (4) optionally sliced by region/language and lightly personalized; (5) updated in near-real-time (seconds-to-a-minute lag is fine). Out of scope: the full-text search behind a trend (that’s S7) and the as-you-type suggestions (S19) — mention the boundary, then set it aside.

Non-functional — these decide the architecture:

The asymmetry that shapes everything: this is a write-heavy, read-trivial system — the exact mirror image of S19 autocomplete (read-heavy). All the engineering is on the ingest/counting side; the serving side is “return a cached list of 50.”

2 · Capacity estimation

Numbers first — they decide where the design lands. Assume Twitter-ish scale: ~500 million tweets/day, each exploded into ~15 candidate topics (hashtags + a few n-gram phrases + named entities).

INGEST VOLUME 500,000,000 tweets/day / 86,400 s ≈ 5,800 tweets/s average big-event peak ~10x ≈ 58,000 tweets/s x ~15 candidate topics per tweet avg token-events/s ≈ 87,000 /s peak token-events/s ≈ 870,000 /s token-events/day = 7.5e9 /day DISTINCT TOPICS (the cardinality that kills exact counting) ~5e7 distinct tokens/day (hashtags + phrases + entities, long tail) EXACT hashmap counter: key(~20B)+count(8B)+overhead(~48B) = 76 B/entry 5e7 x 76 B ≈ 3.8 GB per node, AND it grows with every new distinct topic -> unbounded. <-- this is the rung-0 killer APPROXIMATE counter (Count-Min Sketch), eps=0.001, delta=0.001 width w = ceil(e/eps) = 2,719 depth d = ceil(ln 1/delta) = 7 counters = w x d = 19,033 x 4 B ≈ 74 KB (per window, per shard) -> CONSTANT in cardinality. 3.8 GB -> 74 KB. That is the whole trick. SERVING result = ~50 topics x ~30 B ≈ 1.5 KB, identical per region -> one cached object; read QPS is irrelevant to the design.

The takeaways that drive the rest: (1) counting is the hard part and an exact per-topic counter is the rung-0 that dies on memory (3.8 GB and unbounded); (2) approximate counting collapses that to a fixed ~74 KB; (3) the read path is a cached 1.5 KB blob — ignore it; (4) freshness must come from a stream, and ranking must come from a baseline comparison, not the count itself.

3 · API design

GET /trending?window=1h&region=US&k=20 200 : { "as_of": "2026-06-21T18:30:05Z", "window": "1h", "region": "US", "trends": [ {"topic":"#Election","score":390.2,"volume":42000,"rank":1}, {"topic":"Game 7","score":120.7,"volume":31000,"rank":2}, ... ] } # score = surge, volume = raw count (shown separately) INGEST (internal, not user-facing) produce(tweet) -> tokenizer -> N topic-events -> Kafka(partition by topic) event = {topic, ts, region, weight} # weight lets you down-rank spam/bots

The read endpoint is a bare keyed GET (window × region × k) → a CDN/edge cache (P11) serves it with a few-seconds TTL, so the serving fleet barely sees traffic. Note the response separates score (surge — what we rank by) from volume (raw count — what we display): they are deliberately different, see §8.

4 · Counting — derive it, don’t assert “Count-Min Sketch”

The serve-time job is topic → approximate frequency in this window for the heavy hitters. Walk the ladder; each rung dies on a specific number.

Rung 0 — per-event increment in a database

For each topic in each tweet, UPDATE counts SET c = c+1 WHERE topic = ? AND bucket = ? (or a Redis INCR per topic). Dead simple; correct.

Bottleneck — two of them. (1) Write QPS & hot key: at ~105–106 increments/s, and with the single hottest topic taking a large fraction of them, that one counter row is a hot key (P2) — row-lock contention serializes the firehose. (2) It’s the wrong shape: you only need the top ~50, but you’re paying to durably maintain millions of counters you’ll never read.

Rung 1 — in-memory hashmap counter + periodic flush

Keep an exact map[topic]→count in RAM per stream-processor; increment is O(1) with no DB round-trip; flush aggregates to a store every few seconds. This is the natural first fix and it’s a real rung — at small/medium cardinality it’s exactly right.

Bottleneck — memory grows with cardinality. The map holds one entry per distinct topic ever seen in the window: ~5e7/day → 3.8 GB per node and unbounded (a spam wave of unique junk strings can OOM it). You’re spending gigabytes to track millions of one-hit topics so you can return 50. The mismatch — “I only want the heavy hitters” vs “I’m paying for the whole tail” — is what the next rung removes.

Rung 2 — approximate counting: Count-Min Sketch + a heavy-hitter summary

Two cooperating structures, both with memory independent of cardinality:

(a) Count-Min Sketch (CMS) answers “roughly how many times have I seen topic X?” in fixed space. It is a d × w grid of counters with d independent hash functions. To add topic X: hash it with each of the d functions and increment the one counter it picks in each row. To query X: read those same d counters and take the minimum. Collisions only ever add mass, so every row is an over-estimate; taking the min picks the least-polluted row. It never under-counts. Worked, with d=3, w=8:

add #WorldCup x900, #Election x400, and rare #t5 x4 bucket: 0 1 2 3 4 5 6 7 row0: [900, 0, 0, 4, 0, 5, 0, 400] row1: [400, 0, 900, 4, 0, 0, 0, 0 ] row2: [ 0, 400, 0, 4, 0, 904, 0, 0 ] <- #t5 collides w/ #WorldCup here query #WorldCup -> rows read [900, 900, 904] -> min = 900 (exact) query #Election -> rows read [400, 400, 400] -> min = 400 (exact) query #t5 -> rows read [ 4, 4, 904] -> min = 4 (exact!) in row2 the rare topic collided with #WorldCup and read 904, but rows 0 and 1 were clean, so MIN corrects it back to 4.

For the sized sketch (ε=0.001, δ=0.001 → w=2719, d=7) the over-count is bounded by ε·N with probability 1−δ. Heavy hitters dwarf that bound, so their estimates are essentially exact; the tail is noisy but we don’t care about the tail. Total: ~74 KB, constant no matter how many distinct topics appear.

(b) A heavy-hitter / stream-summary (Misra–Gries or Space-Saving) answers “which topics are the big ones?” — because a CMS tells you a count if you already know the key, but it doesn’t enumerate keys. Misra–Gries keeps m counters; a new item takes a free slot, increments an existing slot, or (if all m are full) decrements every counter by one (evicting any that hit zero). Worked, m=3:

stream: A A B C A B A D A A E A (N=12) final MG counters: { A:5, E:1 } exact: {A:7, B:2, C:1, D:1, E:1} guarantee: any topic with true freq > N/(m+1) = 3 is GUARANTEED retained. A (true 7) is kept; its counter 5 is a lower bound (true is 5..7).

So the pair is: Misra–Gries/Space-Saving gives you the candidate set of heavy hitters; the CMS gives you a sharp frequency estimate for each candidate. Together, fixed memory, no per-topic counter.

Bottleneck: a CMS counts everything since you zeroed it — it has no notion of time. “Trending” is about the recent window, so an all-time sketch makes evergreen topics permanent winners. That forces windowing/decay (§6). And the count it gives is still volume, not surge (§8).

5 · Top-K maintenance & the distributed merge

Given per-topic frequencies, how do you keep the top K without sorting millions of topics every refresh?

Rung 0 — sort everything on read

Collect all topic→count pairs and sort, take the first K. O(N log N) over millions of entries, per refresh.

Bottleneck: sorting tens of millions of topics every few seconds is wasted work — you re-sort a near-static tail to find a near-static head.

Rung 1 — a min-heap of size K

Maintain a bounded min-heap holding the current top-K. On each count update, if the value beats the heap’s minimum, push-pop; else skip. O(log K) per update, O(1) to read the head. Worked, K=3:

counts arrive: #a50 #b80 #c30 #d95 #e10 #f70 (K=3) ...fill heap [#a50,#b80,#c30] #d(95) > min(30) -> push, evict #c(30) #e(10) <= min(50) -> skip #f(70) > min(50) -> push, evict #a(50) final top-K = [#d:95, #b:80, #f:70]

(The Space-Saving summary from §4 already is a bounded top-m structure, so in practice the heavy-hitter summary and the top-K heap are the same idea sized to m ≈ several·K.)

Rung 2 — per-shard local top-K + scatter-gather merge (← the real distributed answer, P7)

One node can’t take the whole firehose, so partition the event stream by hash(topic) across S shards (P2 — every event for a topic lands on one shard, so its count is local and complete). Each shard keeps its own CMS + local top-K. A coordinator periodically gathers the shards’ local top-K lists and merges them (P7). But there is a classic trap:

Bottleneck — the merge can miss a true global heavy hitter. If each shard sends only its local top-K, a topic that is just-below-top on several shards but globally large gets dropped. Worked: true global #2 is #y = 18, spread as 9+9 across two shards but never a shard’s #1:

shardA: #x40 #y9 #z8 shardB: #x40 #p10 #q7 shardC: #x40 #y9 #r6 true global: #x=120, #y=18, #p=10, #z=8, #q=7, #r=6 merge per-shard TOP-1: {#x:120} <- #y MISSED merge per-shard TOP-2: {#x:120, #y:18, #p:10} <- #y recovered merge per-shard TOP-3: {#x:120, #y:18, #p:10}

Fix — over-fetch. Each shard returns its top K' = c·K (or its whole heavy-hitter summary), not just K; the coordinator sums per topic and takes the global top-K. Because partitioning is by topic, a shard’s count for a topic is complete, so summing the over-fetched candidates is exact for anything that made any shard’s K' list. This is the same “ask for more than you need, then merge” pattern as distributed search’s query-then-fetch (S7) and the global-top-N merge in P7.

Why partition-by-topic, not by ingest-node: if you sharded the firehose round-robin, every topic’s count would be smeared across all S shards and the merge could never be exact without shipping full per-shard counts. Hashing by topic makes each shard’s local count authoritative for its topics — the merge just concatenates disjoint key-spaces, and over-fetch only guards the items that straddle a shard’s K cut.

6 · Time windows & decay — “last 5 min” without rescanning the firehose

The board needs several rolling windows (5 min / 1 h / 24 h). Naively you’d re-aggregate the raw event log for each window on each refresh — re-reading the firehose repeatedly. Two ways out, used together.

Tumbling buckets + rollup (exact-ish, multi-window)

Aggregate events into small fixed tumbling windows — e.g. one CMS/heavy-hitter snapshot per 1-minute bucket. Any larger window is then a sum of buckets: “last 5 min” = sum the last 5 one-minute snapshots; the rolling update is “drop the oldest bucket, add the newest” — O(1), never a rescan. Worked (#Election per-minute counts):

per-min buckets: [120, 90, 300, 5000, 4200] (newest last) last-5-min total = 120+90+300+5000+4200 = 9,710 next minute: drop bucket[0]=120, add new bucket -> O(1), no raw rescan HIERARCHICAL ROLLUP (so 24h doesn't sum 1,440 one-min buckets): 60 x 1-min -> roll up into 1 x 1-hour cell 24 x 1-hour -> roll up into 1 x 1-day cell each window reads its OWN grain: 5-min from 1-min cells, 24h from 1-hour cells

This is a columnar/time-partitioned rollup store (P8) — the same pattern as a metrics TSDB (S6): pre-aggregate at write time into the grains you’ll query, so no query rescans raw events. Note the CMS is mergeable: summing the per-minute sketches cell-wise gives the window’s sketch for free (they share dimensions).

Exponential decay (a continuously sliding window, O(1) state)

For the “now” board you often want a smooth slide rather than discrete buckets. Keep one decaying score per heavy hitter and update it lazily: when an event arrives at time t, score = score · e^(−λ·Δt) + weight, where Δt is time since the last update and λ = ln2 / half_life. Old mass evaporates; recent mass dominates. Worked (half-life 10 min):

half-life = 600 s -> lambda = ln2/600 t=0s burst +1 -> score 1.000 t=1s +1 -> 1.999 t=2s +1 -> 2.997 (recent burst piles up) t=300s +1 -> 3.124 (5 min later: old mass half-decayed) t=900s +1 -> 2.562 (15 min: the early burst has mostly evaporated) => a fresh burst outweighs an old steady trickle, with ONE float per topic.

Why lazy decay matters: you never sweep all topics to age them — you apply the decay factor only when a topic is touched or read. That keeps per-event work O(1) and avoids an O(distinct-topics) tick.

The firehose is tokenized into topic-events and published to Kafka partitioned by hash(topic) so each topic’s counting is local to one shard (P2). Stream shards maintain approximate counts per 1-minute bucket (§4) and write rollups (§6). A coordinator merges the shards’ over-fetched top-K (§5), a surge scorer converts volume into a trending score against a batch-computed baseline (§8), and the tiny result is cached per window×region and fronted by the edge. The reader never touches the pipeline.

8 · The ranking signal — “trending” is surge, not volume

This is the part that separates a real trending board from a leaderboard. If you rank by raw count, evergreen terms (#love, good-morning greetings, a perennial sports club) win forever and nothing ever “trends.” Trending = anomalous acceleration vs the topic’s own baseline.

Maintain, per topic, a long-run baseline rate (an EWMA of its historical events/min, computed by the batch path). The score is how far the current window exceeds what we’d expect. Model the count as Poisson(expected): standard deviation ≈ √expected, so a z-score (observed − expected)/√expected measures surprise. Worked:

topic raw(5m) expected z-score ratio #love 26,000 25,000 6 1.04x <- huge volume, NOT surging #Election 4,000 100 390 40.0x <- breaking: 40x its baseline #goodmorning 4,200 4,000 3 1.05x <- diurnal, expected rank by RAW COUNT -> #love wins (boring, always there) rank by SURGE (z) -> #Election wins (it's actually trending)

So the board ranks by z-score / baseline ratio, displays raw volume alongside, and typically applies a volume floor (ignore topics below ~a few hundred events so a topic going 2→40 doesn’t show a 20× surge from noise). Variants: χ²/G-test of observed-vs-expected, or comparing the current decayed score to the same topic’s score one window ago (the discrete derivative). The mechanism is the same: deviation from a baseline, not the count.

9 · Scaling, failure & the cross-cutting checks

  • Partition key (the scatter-gather check): the count stream is partitioned by hash(topic) so each topic’s count is local and complete; the only cross-shard query is the final top-K merge, handled by over-fetch (§5). A single ultra-viral topic is still one hot partition (P2) — split it by hash(topic)+salt across a few sub-counters and sum at merge, the standard hot-key split.
  • Bounded failover of in-memory state: the CMS/heavy-hitter/decay state is in RAM. Don’t rebuild it by replaying all of history — checkpoint each shard’s sketch to durable store every bucket (it’s only ~74 KB) and, on failover, reload only the working window’s buckets (last few minutes / the current rollup grains). Recovery is ms, not a firehose replay (the same bounded-rebuild discipline as S12).
  • Backpressure (P16): a 10× event spike (exactly when trends matter most) must not topple the counters. Kafka decouples ingest from processing; if shards fall behind, you sample the stream (count 1-in-r and scale up) — approximate counting already tolerates sampling, so shedding load degrades accuracy gracefully instead of dropping the system.
  • Spam / bot manipulation: raw counts are gameable (a botnet hammering a hashtag). Defenses: per-user/per-IP dedup so one account can’t inflate a topic (count distinct users, not raw events — estimated with a HyperLogLog per topic, the cardinality cousin of CMS), event weight down-ranking low-reputation accounts, and the baseline/anomaly model itself (a botnet’s unnatural spike is detectable).
  • Scaling at 10×: ingest scales by adding Kafka partitions + stream shards (counting is embarrassingly partitionable by topic); serving never scales because the result is one cached blob. The thing that gets harder is the merge fan-out (more shards → more partial top-K lists) — bounded by hierarchical merging (shards → regional coordinators → global), the same tree-merge as P7.

10 · Tradeoffs & alternatives

Exact vs approximate. If the “top-K” is a leaderboard with few entities and money/fairness on the line (game high-scores, a sales bestseller list over thousands of SKUs), drop the sketches: a Redis sorted set (ZADD/ZINCRBY + ZREVRANGE) gives exact, totally-ordered top-K in O(log N) because cardinality is small enough to hold every member. Approximate counting earns its keep only when distinct cardinality is too large to count exactly (millions of topics) — that’s the dividing line between “leaderboard” and “trending firehose.”

Volume vs surge. A “most-viewed in 24h” board wants raw volume (no baseline); a “trending now” board wants surge. Same plumbing, different scorer — keep the scorer pluggable.

Lambda vs kappa. We used a batch baseline (long-run rate) + a streaming hot path. A pure-streaming (kappa) build computes the baseline as a slow-decay EWMA in the same stream job — simpler ops, slightly less stable baseline.

11 · Real-world example

Twitter trends · Kafka + Heron/Storm · approximate algorithms

Twitter’s trends pipeline ingests the tweet stream through Kafka and a stream processor (historically Storm, then Heron), extracts candidate entities, and ranks by an anomaly/surge model against a learned baseline with personalization and spam filtering — not by raw count, which is why niche breaking topics out-rank perennial ones. The approximate-counting toolkit here is industry-standard: Count-Min Sketch (Cormode–Muthukrishnan) for frequency in fixed memory, Space-Saving for heavy hitters, and HyperLogLog for distinct-user cardinality — all chosen because exact per-topic state is infeasible at firehose scale. The lesson mirrors S6 (metrics) and S19 (autocomplete): pre-aggregate into the grains you’ll query, and accept bounded approximation to make the memory constant.

12 · Interview follow-ups (pre-empted)

  • “Why not just GROUP BY topic ORDER BY count DESC LIMIT k?” At ~106 events/s you can’t durably increment per event, can’t hold an exact counter per distinct topic (~3.8 GB & unbounded), and re-sorting millions per refresh is wasted. CMS + heavy-hitters + per-shard top-K replaces all three with fixed memory.
  • “CMS over-counts — doesn’t that corrupt the ranking?” It only ever over-estimates, bounded by ε·N, and only for items that collide with heavy ones; the heavy hitters themselves are essentially exact because their true count dwarfs the bound. The min across d rows corrects most collisions (see the #t5 example). We rank heavy hitters, where the error is negligible.
  • “How does a topic on no single shard’s top-K still make the global top-K?” Over-fetch: each shard returns K′=c·K candidates (or its full summary), the coordinator sums per topic. Since partitioning is by topic, a shard’s count is complete, so the sum is exact for anything in any shard’s K′ list (the #y example).
  • “How do you do ‘last 5 minutes’ vs ‘last 24 hours’ cheaply?” Tumbling 1-min buckets summed into the window; hierarchical rollup (1-min→1h→1d) so a 24h query reads ~24 hour-cells, not 1,440 minute-cells. Mergeable sketches make the per-bucket sum free.
  • “What stops #love being #1 forever?” Rank by surge (z-score / ratio vs a per-topic baseline) plus a volume floor — not by raw count. #Election at 40× baseline beats #love at 1.04× even though #love has 6× the raw volume.
  • “A botnet is spamming a hashtag — how do you not trend it?” Count distinct users (HyperLogLog) not raw events, weight by account reputation, and lean on the anomaly model — an unnatural spike from few accounts is detectable and floored out.
  • “Leader/shard dies mid-window — do you lose the trend?” No: checkpoint the ~74 KB sketch per bucket; on failover reload only the working window’s buckets → ms recovery, not a firehose replay.
  • “Exactly-once counting?” Unaffordable and unnecessary — approximate counting already tolerates the occasional double/dropped event; we choose at-least-once ingest with sampling-friendly sketches, and the ranking is robust to small count errors by construction.

13 · Key talking points

  • Write-heavy, read-trivial (mirror of S19): all engineering is on counting; the result is a cached 1.5 KB blob.
  • Counting ladder: per-event DB increment (hot key) → exact in-RAM map (~3.8 GB, unbounded) → CMS + heavy-hitter summary (~74 KB, constant in cardinality).
  • CMS = d×w counters, add to one per row, query = min across rows (never under-counts); Misra–Gries/Space-Saving finds which topics are heavy.
  • Distributed top-K: partition by hash(topic), per-shard local top-K, scatter-gather merge with K′=c·K over-fetch (P7) to not miss a straddling heavy hitter.
  • Windows via tumbling 1-min buckets + hierarchical rollup (no rescans); “now” via lazy exponential decay (one float/topic).
  • Trending = surge, not volume: rank by z-score/ratio vs a per-topic baseline + a volume floor; count distinct users (HLL) to resist spam.
  • Boundary: this is detection/ranking — the search behind a trend is S7; the typeahead is S19; small-cardinality exact ranking is a Redis sorted set, not sketches.
