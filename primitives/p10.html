P10 · Consensus, leader election & fencing

What it is: getting distributed processes to agree, or to elect one leader, despite failures.
Reach for it when: you need mutual exclusion, a single writer, or a single cron firer.
The lever: Raft/Paxos — a majority quorum agrees; on leader failure a new one is elected. A lock is a leader-with-metadata. Watches notify on release without polling. Session/heartbeat TTL detects a dead holder.
New problem it hands you: the slow-but-alive holder — a GC pause or network blip means a process still "holds" the lock while another takes over. TTL tuning can't fix this; fencing tokens (monotonic, reject stale) solve correctness. Quorum adds latency; minority partitions can't progress.
Exemplified by: Distributed lock, Task scheduler, Kafka (controller).

1 · The problem it solves

Sometimes many nodes must agree on one thing — who is the leader, who holds the lock, what the current config is — even though nodes crash and networks drop messages. Consensus gets a group to agree on a value despite failures; leader election is consensus on “who’s in charge.” The trap is that the obvious cheap answers are subtly wrong, so the right way in is a ladder.

2 · How to elect one leader safely — the ladder

Rung 0 — a static/hardcoded leader

Designate one node the leader in config. Trivial — and fine until that node dies, at which point there is no automatic failover and the system stalls.

Bottleneck: no failover; a SPOF by construction.

Rung 1 — a lease in a single store (e.g. Redis SET key val NX PX 10000)

Candidates race to set a key with a TTL; whoever wins is leader and renews the lease by heartbeat. If it dies, the TTL expires and someone else wins. This looks done.

Two bottlenecks: (1) the store is itself a single point of failure, and you can’t naively replicate a lock (an async replica could grant the same lock twice on failover — split brain). (2) the slow-but-alive holder: a leader that GC-pauses past its TTL is presumed dead and replaced, then wakes up still believing it leads. TTL tuning cannot fix this — shorten it and a hiccup causes spurious failovers; lengthen it and recovery is slow.

Rung 2 — consensus-backed lease (etcd / ZooKeeper, Raft/Paxos) ← the answer

Put the lease in a replicated state machine that commits via majority quorum, so the “who is leader” decision survives a minority of node losses and can never be granted twice (any two majorities overlap). This is what etcd/ZK/Chubby exist to provide, so you never implement Paxos yourself.

Residual costs (accepted): every commit pays a majority round-trip (latency), a minority partition can’t make progress (CP by construction), and the slow-but-alive holder is still possible — consensus picks one leader but can’t stop a paused old one from acting. That last gap is closed by fencing tokens (§4), not by consensus.

3 · How consensus works — a Raft walkthrough

graph TD
  C[client] --> L["Leader (term 4)"]
  L -->|"append (4, idx7, x=9)"| F1[Follower A]
  L -->|"append (4, idx7, x=9)"| F2[Follower B]
  F1 -->|ack| L
  F2 -->|ack| L
  L -->|"committed once majority (3/5) ack"| C

Raft replicates an ordered log. Each entry is (term, index, command); the leader appends and ships entries to followers, and an entry commits the moment a majority has it — so any future majority overlaps the commit and can never lose it. Trace a 5-node cluster (majority = 3, tolerates 2 losses):

term 4, leader = N1 log: (4,5,set a) (4,6,set b) (4,7,set c) N1 replicates (4,7) to N2,N3 -> 3/5 have it -> COMMIT (4,7), reply to client N4,N5 lagging, only up to (4,6) -> fine, they catch up N1 crashes. Election for term 5: N2 has log through (4,7); N4 has only (4,6) Raft elects the candidate with the MOST up-to-date log -> N2 wins (not N4) -> committed entry (4,7) is preserved; N2 leads term 5

The election rule (most-up-to-date log wins) is what guarantees a committed entry is never dropped on failover — the key property the naive single-store lease lacks.

4 · The slow-but-alive holder & fencing tokens

Consensus elects exactly one leader, but it cannot stop a previously elected leader that paused (GC, VM migration, network blip) from waking up and acting as though it still leads. Trace it:

t0 Leader A holds lease, fencing token = 33 t1 A stalls (long GC pause) ... t2 lease TTL expires; cluster elects B, fencing token = 34 t3 B writes to the DB with token 34 -> DB records "highest seen = 34" t4 A wakes, still thinks it leads, writes with token 33 t5 DB sees 33 < 34 -> REJECTS the stale write <-- correctness saved

The fencing token is a monotonically increasing number handed out with the lock; the protected resource remembers the highest token it has accepted and rejects anything lower. TTL tuning can never make the lease alone safe — only the token does, because it makes a stale writer detectable at the resource regardless of timing.

5 · Applied — safe ID allocation & worker-ID leasing

This is what an id-generation service (and any coordinator) leans on for failover correctness:

  • Durable range allocation. A key-gen coordinator hands out counter ranges by a compare-and-swap on a consensus store (etcd/ZK) — CAS(next, 9_000_000 → 9_001_000). Because the new high-water mark is committed to a majority before the range is returned, a coordinator failover can never re-hand the same range to two servers. Contrast a counter on an async-replicated store: promoting a lagging replica re-issues already-used ids (collisions). Consensus is precisely the fix.
  • Worker-ID leasing. Snowflake-style ids embed a machine id; two nodes with the same id silently collide. Assign ids as ephemeral nodes (a ZK ephemeral znode or an etcd lease): a node leases worker id 7, heartbeats to keep it, and on death the lease lapses and id 7 is freed for reuse — no static config to get wrong.
  • Graceful degradation. Because servers hold a local leased range, they keep minting ids while the coordinator is briefly unavailable — coordination is needed only once per range, not per id (P16).

6 · Real-world example

Google Chubby / etcd

Google’s Chubby lock service (built on Paxos) is how much of Google does leader election — services don’t implement Paxos themselves, they grab a Chubby lock. etcd (built on Raft) plays the same role in Kubernetes: it stores cluster state and elects leaders via a quorum. Both expose sessions/leases with TTLs to detect dead holders and watches so clients react to changes without polling. The fencing-token lesson comes from Martin Kleppmann’s well-known critique of naive Redis locks: a lock alone is not safe without a fencing token.

7 · Applying it in a design

Don’t roll your own consensus — delegate to ZooKeeper/etcd/Chubby. Say: "single leader elected via etcd lease; the leader carries a fencing token so a stale leader can’t corrupt state." See Distributed lock.

8 · Pitfalls

  • Believing a lock TTL alone guarantees mutual exclusion (it doesn’t — fence).
  • Even-sized clusters (no clean majority); use odd numbers.
  • Putting high-throughput data through a consensus store (it’s for coordination, not bulk).
