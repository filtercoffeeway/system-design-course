11 · Notification System

Hard constraint: at-least-once is easy; exactly-once is effectively impossible — the channels (FCM/APNs/Twilio) are black boxes you can't introspect. Design for idempotent dedup, and pace every external provider or get throttled/banned.
Primitives exercised: P9 (idempotency), P5 (fan-out), P1 (priority/batching), P16 (backpressure), S1 (rate limit), S12 (scheduling).

Priority queues (critical vs marketing); idempotency keys; per-channel rate limits; scheduled/delayed delivery; per-channel preference + quiet-hours checks at send time; retry/DLQ.

1 · Requirements

Functional: a unified API any internal service can call to notify a user across channels (push, SMS, email, in-app), with priorities, scheduling, templating, user preferences/opt-outs, and delivery tracking. Non-functional:

Deliveryat-least-once, no duplicate effectidempotent dedup key (P9)
Isolationa 50M marketing blast can't delay a 2FA codepriority queues (§6)

Throughput~6k/s avg, campaign spikes 30k+/squeue-buffered, horizontally scaled workers

The defining reality: the hard parts are not storage or novel algorithms — they're reliability, dedup, and third-party management. The channels are external services you must integrate with, pace, and treat as unreliable black boxes. The system is mostly a reliable, idempotent, rate-controlled fan-out pipeline.

2 · Capacity estimation

500,000,000 notifications/day / 86,400 s ~ 5,800/s average campaign spike: 10,000,000 sent over 5 min = ~33,000/s (10x+ average -> must buffer) fan-out: one "campaign" event -> 50,000,000 individual sends (P5 fan-out) provider ceilings (illustrative): Twilio long-code SMS default ~1 msg/s per number -> bulk needs short codes / many numbers APNs / FCM: high but per-connection/per-project quotas -> pace per channel dedup state: key per (user, event, channel), 30-day TTL 500M/day x 30 = 15B keys x ~64 B ~ 1 TB -> Redis w/ TTL (or a Bloom filter front)

3 · API & data model

notify(user_id, event_type, payload, channels[], priority, idempotency_key, send_at?) -> notif_id getStatus(notif_id) -> {per_channel: {queued|sent|delivered|failed}} device_registry: user_id -> [ {channel, token/address, platform, active, last_seen} ] preferences: (user_id, channel, event_type) -> {opted_in, quiet_hours, frequency_cap} dedup: idempotency_key -> {state, sent_at} (TTL) outbox/schedule: time_bucket -> [notif_id] (for send_at, like S12)

4 · The hard constraint: delivery semantics (ladder)

You cannot achieve exactly-once delivery: once you hand a message to APNs/FCM/Twilio, you get an accepted ack, not a delivered-to-human confirmation, and the network can fail after a send but before you record it. So a retry might double-send. The only safe design is at-least-once + idempotent dedup.

Rung 0 — fire and forget

Call the provider, move on. Breaks on: any transient failure loses the notification silently — the 2FA code never arrives.

Rung 1 — at-least-once with retries

Persist the request, send, retry on transient errors with exponential backoff, dead-letter after N attempts. Breaks on: retries cause duplicates — the user gets the same "your driver arrived" push three times because the first two succeeded at the provider but the ack was lost.

Rung 2 — at-least-once + idempotency key (the answer)

The caller supplies (or the system derives) an idempotency key per (user, event, channel) (P9). Before sending, the worker atomically claims the key (SET NX); if it's already claimed/sent, skip. A retry re-presents the same key → deduped. Now retries are safe: at-least-once attempts, at-most-once effect.

worker: if not dedup.setnx(key, "sending", ttl): return // someone else has it / already sent ok = provider.send(msg) dedup.set(key, "sent" if ok else "failed") if not ok and attempts < N: requeue with backoff // same key -> still deduped
New bottleneck — the dedup store at 15B keys. A 1 TB Redis of keys is costly. Relief: TTL keys (most events only need a 24h–30d dedup window), and front the exact store with a Bloom filter (P9) — "definitely not seen" skips the lookup; "maybe seen" checks the store. The Bloom filter's false positives only cause an occasional extra lookup, never a wrong dedup.

6 · Priority & isolation

A single FIFO queue is fatal: a 50M-recipient marketing campaign sitting ahead of a 2FA code means the code arrives after the user gives up. Split into priority lanes — e.g. critical (2FA, security), transactional (trip updates, order shipped), bulk (marketing, digests) — with separate queues and worker pools so bulk can never starve critical. Workers drain higher lanes first (or dedicated pools per lane to guarantee headroom). Per-channel buffers apply P16 backpressure so a slow provider doesn't back up the whole pipeline; under overload, bulk is shed first.

7 · The fan-out path & provider pacing

A campaign is a P5 fan-out: one event expands to millions of per-user sends. The expansion happens off the critical path — the API enqueues a campaign job; a fan-out worker reads the audience and emits individual notification messages into the bulk lane. Each individual send then passes through:

  1. Preference + quiet-hours check (per channel, at send time — not once globally, because a user may allow push but not SMS, and quiet hours are per-timezone). A frequency cap ("max 3 marketing/day") is enforced here too.
  2. Per-channel rate limiter (S1 token bucket): pace sends to stay under each provider's ceiling. Exceeding APNs/FCM/Twilio limits gets you throttled or banned, so the limiter is a hard gate, not best-effort. Distinct buckets per provider/account.
  3. Channel adapter: translate to the provider's API (APNs payload, FCM message, Twilio SMS), send, record the result.

Channel-specific batching helps: APNs/FCM accept multicast/batch sends, so the worker batches many tokens per call (P1) to amortize round trips and stay efficient under the rate cap.

8 · Scheduling, device registry & feedback

Scheduling: future-dated notifications go to a time-bucketed outbox scanned per minute — the exact delayed-delivery pattern from S12 (don't full-scan a pending table; partition by due-time bucket). Device registry maps a user to their device tokens; tokens expire and rotate, so the system must process provider feedback (APNs/FCM return "unregistered" for dead tokens) and prune them, or you waste sends and hurt your sender reputation. Delivery receipts (where the channel provides them — SMS DLRs, email opens/bounces) feed a reconciliation loop that updates status and retries or dead-letters failures.

New bottleneck — debugging async sends. A notification that fires minutes later from a queue is a black hole for tracing. Carry W3C trace context + the idempotency key through the queue (the same async-context check as S12) so a delivery (or a missing one) stitches back to the originating request in your APM.

9 · Scaling & failure

At 10× (campaign spikes to 300k/s): the bulk lane absorbs the spike in the queue and drains at the providers' allowed rate — you can't send faster than the channels accept, so the queue is the shock absorber (and bulk latency stretching to minutes is acceptable; critical lanes are isolated). Shard dedup and rate-limit state by user/channel. The DLQ collects exhausted retries for inspection. The whole system degrades gracefully: shed bulk, protect transactional and critical.

10 · Tradeoffs & alternatives

Push (fast, free, but only to app-installed devices that may be offline) vs SMS (high reach/reliability, costs money, rate-limited) vs email (cheap, high volume, low urgency). Aggressive retries improve delivery but raise duplicate risk and cost — idempotency is what lets you retry freely. Real-time vs digest/batched delivery trades immediacy for volume control and user-annoyance (collapse 20 "someone liked your post" into one digest). Build-your-own channel adapters vs a delivery aggregator (e.g. a CPaaS) trades control for speed.

11 · Real-world example

Uber

Uber has described a real-time push platform handling on the order of hundreds of millions of messages a day across push/SMS/email, with exactly this shape: a unified API in front of channel-specific adapters (APNs, FCM, SMS providers), priority separation so operational messages (trip updates) beat promotional ones, idempotency to avoid duplicate trip alerts, and per-provider rate control. The platform also tracks delivery receipts and reconciles, and prunes dead device tokens from provider feedback. The defining lesson matches the constraint: the channels are third parties you must integrate with and pace, so the platform is mostly reliability, dedup, and provider management rather than novel storage.

12 · Interview follow-ups (Q&A)

Q: Why can't you guarantee exactly-once delivery? The providers ack acceptance, not human delivery, and a failure can occur after a successful send but before you record it — so any safe system retries, which can duplicate. You make the effect idempotent (dedup key) so duplicates don't matter, rather than chasing impossible exactly-once.

Q: A marketing campaign is delaying 2FA codes — fix? Priority lanes with isolated queues and worker pools: critical/transactional never share a queue with bulk, and bulk is shed first under load. The campaign drains at provider-allowed rate from its own lane.

Q: How do you avoid getting banned by Twilio/APNs? A per-channel/per-account token-bucket rate limiter (S1) that's a hard gate before every send, plus honoring provider feedback (back off on 429s, prune dead tokens). Bulk SMS uses short codes / many numbers because long codes are ~1/s.

Q: The dedup store is 1 TB — can you make it cheaper? TTL keys to the needed dedup window, and front the store with a Bloom filter so "definitely new" skips the exact lookup. False positives only cost an extra lookup, never a wrong dedup (P9).

Q: How do scheduled notifications work without scanning a huge table? Time-bucketed outbox scanned per minute (same as S12) — the scheduler reads only the current bucket's partition, never the whole pending set.

Q: Where are preferences and quiet hours enforced? Per channel, at send time — not once at the API — because consent and timezone-based quiet hours differ per channel and the audience may have been computed hours earlier (a user could opt out in between).

13 · Key talking points

  • Design for at-least-once + idempotent dedup, not exactly-once (P9); channels are unreliable black boxes.
  • Priority lanes isolate critical/transactional from bulk (P1, P16) so a campaign never delays a 2FA code.
  • Respect per-channel provider rate limits with token buckets (S1) — a hard gate, not best-effort.
  • Campaign = fan-out off the critical path (P5); batch multicast sends per channel.
  • Time-bucketed outbox for scheduling (S12); prune dead device tokens from provider feedback.
  • The queue is the shock absorber: you can't send faster than channels accept, so buffer and pace.
