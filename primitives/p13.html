P13 · Real-time connections & routing

What it is: maintaining persistent bidirectional channels to many clients and delivering a message to the right one.
Reach for it when: server-initiated push, live updates, presence.
The lever: WebSocket/SSE; stateful connection servers; a fast lookup user_id → server_id (Redis) updated on every connect/disconnect; per-conversation sequence numbers for ordering.
New problem it hands you: routing — which of 10k connection servers holds recipient B's socket? The lookup must be fast and consistent. Connection state is hard to scale and fail over; reconnection needs dedup.
Exemplified by: Chat, Notification.

1 · The problem it solves

Request/response over HTTP is client-initiated; it can’t push. For chat, live updates, and presence you need the server to send to the client the instant something happens, over a persistent connection — and then you need to find the right connection among millions.

2 · The delivery ladder

Rung 0 — short polling

Client asks “anything new?” every few seconds. Trivial, works over plain HTTP — but it’s nearly all wasted requests, and updates lag by the poll interval.

Bottleneck: at scale the empty polls dwarf real traffic; latency = poll interval.

Rung 1 — long polling / SSE

The server holds the request open until it has something (long-poll), or pushes a one-way event stream (SSE). Lower latency, fewer empty round-trips — but still one-directional and connection-churny.

Bottleneck: no efficient client→server push; reconnect overhead.

Rung 2 — WebSocket on a stateful gateway ← the answer

graph LR
  A[user A] <-->|WebSocket| GA[gateway 3]
  B[user B] <-->|WebSocket| GB[gateway 47]
  GA -->|"lookup: B -> gw47"| RT[(routing registry)]
  GA -->|forward| GB
  GB --> B

A WebSocket holds an open, bidirectional, stateful connection on a gateway server. To deliver A→B, A’s gateway looks up which gateway holds B’s socket via a fast registry (user_id → server_id in Redis), updated on every connect/disconnect; per-conversation sequence numbers give ordering. The cost this exposes is holding millions of live connections:

1,000,000 conns x ~10 KB socket buffers = ~10 GB RAM just for sockets 10,000,000 conns = ~100 GB -> spread over many gateway nodes => connections are state; the gateway tier scales horizontally, and a dead gateway forces all its clients to reconnect + re-register (the failover cost)

3 · The new bottleneck it hands you

Routing is the hard part: the registry must be fast, consistent, and updated on every churn event (millions of connects/disconnects). Connection state is hard to scale and to fail over — if a gateway dies, every client must reconnect and re-register. Reconnection needs dedup so resends don’t duplicate messages.

4 · Real-world example

Slack / Discord

Slack clients hold a WebSocket to a gateway tier; on login the client calls an endpoint that returns a WebSocket URL and the initial state, then receives events (messages, typing, presence) pushed over that socket. Discord runs a similar gateway that fans guild/channel events to the right connections, backed by an internal routing layer that knows which gateway node holds each session. In both, the defining engineering problem is presence and routing at scale — tracking who is online and which node holds their socket — not the message store itself. See Chat.

5 · Applying it in a design

"Persistent WebSockets on a stateful gateway tier, a Redis registry mapping user → gateway updated on connect/disconnect, and per-conversation sequence numbers for ordering." Call out reconnection + dedup.

6 · Pitfalls

  • Polling instead of persistent connections at scale.
  • Forgetting the registry must update on every disconnect (stale routes).
  • Treating global message ordering as required (per-conversation is enough).
