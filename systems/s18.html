18 · Web Crawler

Hard constraint: the URL frontier must be distributed (too big for one node), prioritized (crawl what matters first), and polite (never hammer one domain) all at once — a distributed priority queue with per-domain pacing.
Primitives exercised: P2 (partition by domain), P1 (batching), P4 (DNS cache), P17 (Bloom seen-set), P9 (idempotent re-fetch), P16 (backpressure).

URL frontier (priority by freshness/importance); politeness (robots.txt, per-domain rate); exact + near-duplicate detection (Bloom + SimHash/MinHash); mandatory DNS caching; recrawl scheduling via change-frequency backoff; trap avoidance.

1 · Requirements

Functional: starting from seed URLs, traverse the web graph — fetch a page, extract its links, store its content for indexing (S7), enqueue new links, and recrawl pages as they change. Non-functional:

The interesting tension: maximize fetch throughput while being polite to every individual site. Those pull in opposite directions — raw throughput wants to blast a domain, politeness forbids it — and reconciling them is what makes the frontier hard.

4 · The hard constraint: the URL frontier (ladder)

The frontier is a priority queue that's simultaneously too big for one node, must be ordered by importance/freshness, and must never release two URLs for the same domain too close together. Those three requirements fight each other.

Rung 0 — one in-memory FIFO queue (BFS)

Classic breadth-first: pop a URL, fetch, push its links. Breaks on: (a) it doesn't fit on one node at billions of URLs; (b) FIFO ignores importance — you waste days on junk before reaching valuable pages; (c) no politeness — links from one page mostly point to the same domain, so a FIFO hammers that host.

Rung 1 — a priority queue (importance/freshness first)

Order by a priority score (PageRank-ish importance, freshness, depth). Now you fetch valuable pages first. Breaks on: still hammers a domain — the highest-priority URLs at any moment are often clustered on one popular host, so a pure priority pop violates politeness.

Rung 2 — two-level frontier: priority front + per-domain back queues (the answer)

The Mercator design. A front set of priority queues decides which URLs are important. A back set of per-domain queues enforces politeness: each domain has its own FIFO, and a heap of (next-allowed-time, domain) releases a URL from a domain only once its politeness delay has elapsed. A fetcher is handed the domain whose next-allowed-time is soonest. This decouples "what to crawl" (priority) from "when we're allowed to" (politeness).

back queue selection (politeness heap): heap entries: (next_ok_time, domain) fetcher pops domain with smallest next_ok_time (wait if in the future) fetch one URL from that domain's queue push back (now + crawl_delay(domain), domain) -> guarantees >= crawl_delay between hits to any single host

Rung 3 — distributed frontier (shard by domain)

Partition the whole frontier across nodes by hash(domain) (P2) so all of a domain's URLs — and its politeness state — live on one shard (politeness is meaningless if two nodes independently crawl the same host). Within a shard, use the two-level structure. Persist queues (RocksDB/Kafka) so a node restart doesn't lose the frontier.

5 · Politeness in depth

Politeness is non-negotiable — crawl too aggressively and you DoS a site and get IP-banned. It has three parts: (1) robots.txt — fetch and cache each domain's rules; honor Disallow and Crawl-delay; (2) per-domain rate limit — typically at most one in-flight request per host and a minimum delay between requests (the politeness heap above enforces it); (3) crawl budget — cap how much of any one site you crawl per period so a giant site doesn't starve the rest. Identify yourself with a proper User-Agent and back off on 429/503 (P16).

6 · Deduplication: exact + near-duplicate

Exact URL dedup ("have we already queued this?"): canonicalize the URL (lowercase host, strip default ports, sort query params, drop fragments and tracking params) then check a Bloom filter (P17) — 1.2 GB for 1B URLs at ~0.8% false positives. A Bloom false positive means a genuinely-new URL is wrongly skipped (acceptable: you miss one page), never a wrong re-crawl, so the asymmetry is safe.

Near-duplicate content dedup: the web is full of mirrors, print-vs-web versions, and pages differing only in ads/timestamps. Exact hashing misses these. Use SimHash (Charikar): hash each page into a 64-bit fingerprint where similar pages get fingerprints within a small Hamming distance. Two pages are near-duplicates if their fingerprints differ in ≤ ~3 bits.

SimHash(page): 1. weight features (shingles/tokens) of the page 2. for each feature hash to 64 bits; sum +w for 1-bits, -w for 0-bits per position 3. fingerprint bit = 1 if its column sum > 0 else 0 near-dup if HammingDistance(fp_a, fp_b) <= 3 (out of 64) -> small edits flip few bits; boilerplate changes don't flip the fingerprint

(MinHash + LSH is the alternative when you want estimated Jaccard set-similarity rather than SimHash's cosine-style similarity.) Near-dup detection both avoids re-storing junk and helps the indexer pick a canonical page.

7 · DNS caching (P4)

New bottleneck — DNS. A naive crawler does one DNS lookup per fetch; at 30k fetches/s that's 30k lookups/s, which melts resolvers and adds latency to every fetch (DNS is often the slowest step). Because pages cluster heavily by domain, a local DNS cache (P4) gets a very high hit ratio — mandatory, not optional. Pre-resolve domains in the frontier ahead of fetching, and respect TTLs. Some crawlers run their own recursive resolver to avoid hammering public DNS.

8 · Traps, junk & the high-level pipeline

graph LR
  F[(frontier: priority front + per-domain back, sharded)] --> SEL[politeness selector]
  SEL --> DNS[DNS cache] --> FET[fetchers]
  FET --> PARSE[parse + extract links + canonicalize]
  PARSE --> DEDUP{URL seen? content near-dup?}
  DEDUP -->|new URL| F
  PARSE --> FP[SimHash fingerprint]
  PARSE --> STORE[(content store, 100 TB)]
  PARSE --> META[recrawl scheduler]

Crawler traps are infinite or near-infinite URL spaces — calendars (?date= forever), session-id query params that make every URL look new, deep faceted-search permutations, and self-referential redirect loops. Defenses: cap crawl depth and per-domain budget, canonicalize away session params, detect and break redirect cycles, and bound the URL length / path-segment repetition. Without these, a single trap can consume the whole frontier.

9 · Recrawl scheduling

The web changes at wildly different rates, so a fixed recrawl interval is wasteful (recrawling a static archive hourly) or stale (recrawling a news homepage monthly). Schedule recrawls by observed change frequency: track last_changed via ETag/Last-Modified and content fingerprint; if a page changed since last visit, shorten its interval; if not, lengthen it (exponential backoff toward a max). News homepages converge to minutes/hours; archives to weeks/months. This is a per-URL adaptive timer — the same change-frequency-driven scheduling Googlebot calls crawl-budget management. Conditional GETs (If-None-Match/If-Modified-Since) make an unchanged recrawl nearly free (a 304, no body).

10 · Scaling, fault tolerance & alternatives

Fetchers are stateless and scale horizontally; the frontier shards by domain. Idempotent re-fetch (P9): a URL may be fetched more than once after a crash — storing by canonical URL makes re-storage harmless. Persist the frontier and seen-set so a node restart resumes rather than re-crawling the web. Tradeoffs: breadth-first is simple but ignores importance; priority crawling fetches valuable pages first at the cost of frontier complexity. Aggressive recrawl keeps the index fresh but wastes bandwidth on stable pages. Bloom-filter dedup is memory-cheap but admits rare false negatives in coverage (a skipped page) — pair with an exact store if completeness matters. Politeness throughput vs total throughput is the eternal tension: more domains in flight, not more requests per domain, is how you go faster.

11 · Real-world example

Googlebot / Mercator

The architecture traces to the Mercator crawler (Heydon & Najork) and the original Google crawler (Brin & Page): a prioritized URL frontier, per-host politeness queues, a seen-URL set, content fingerprinting to avoid duplicates, and an aggressive DNS cache. The two-level frontier (front priority queues + back per-domain queues with a politeness heap) is Mercator's signature. Modern Googlebot adds crawl-budget management (how much to crawl a site without overloading it, balancing crawl-rate limit and crawl demand) and change-frequency-driven recrawl scheduling. The defining challenge is unchanged: a frontier that is simultaneously huge, ordered, and polite.

12 · Interview follow-ups (Q&A)

Q: How do you crawl fast but stay polite? Parallelize across domains, not within a domain. The two-level frontier separates priority (front queues) from politeness (per-domain back queues + a next-allowed-time heap), so thousands of domains are in flight at once while each host sees at most one request per crawl-delay.

Q: Why shard the frontier by domain specifically? Politeness state is per-domain; if two nodes independently held the same domain's URLs they'd both crawl it and violate the rate limit. Hashing by domain keeps all of a host's URLs and its pacing state on one shard (P2).

Q: How do you detect that two pages are the same content at different URLs? SimHash each page to a 64-bit fingerprint where similar content lands within a few bits' Hamming distance; treat ≤ ~3 bits as a near-duplicate. Exact URL/content hashing can't catch mirrors and boilerplate variants; SimHash/MinHash can.

Q: Why is DNS a problem and how do you fix it? One lookup per fetch at 30k/s melts resolvers and adds latency. Pages cluster by domain, so a local DNS cache (P4) with TTL respect gets a very high hit ratio; pre-resolve frontier domains ahead of fetching.

Q: What's a crawler trap and how do you survive one? An effectively-infinite URL space (calendars, session-id params, faceted search, redirect loops). Defend with depth + per-domain budget caps, query-param canonicalization (drop session ids), redirect-cycle detection, and URL-length/repetition limits — otherwise one site can swallow the whole frontier.

Q: How do you decide when to recrawl? Adaptively, by observed change frequency: shorten the interval for pages that change, lengthen it (exponential backoff) for stable ones, and use conditional GETs so an unchanged recrawl returns a cheap 304. News → minutes; archives → weeks.

Q: A Bloom-filter false positive skips a real page — acceptable? Usually yes: missing one page out of billions is fine, and the Bloom filter saves ~20× the memory of an exact set. If full coverage matters, back the Bloom with an exact store checked only on Bloom hits.

13 · Key talking points

  • The wall is the frontier: distributed + prioritized + polite at once → two-level (priority front + per-domain back) sharded by domain (P2).
  • Parallelize across domains, not within one; a next-allowed-time heap enforces crawl-delay.
  • Dedup is two-layered: exact URL via Bloom filter (P17) + near-duplicate content via SimHash/MinHash.
  • DNS caching is mandatory (P4) — pages cluster by domain so hit ratio is high.
  • Recrawl by change frequency with conditional GETs; respect robots.txt and crawl budget.
  • Survive traps with depth/budget caps, URL canonicalization, and redirect-loop detection.
