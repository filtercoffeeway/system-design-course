P1 · Batching / async buffering

What it is: absorbing bursts and amortizing per-item overhead by grouping work and decoupling producer from consumer in time.
Reach for it when: write throughput is too high, or a downstream is slower or less available than the upstream.
The lever: client/agent-side batching; an in-memory or durable buffer (queue/log) between stages; pre-aggregation at the source before the wire.
New problem it hands you: added latency (a batch waits to fill); lost or weakened ordering; the buffer becomes state you must size, persist, and back-pressure (see P16).
Exemplified by: Rate limiter, Kafka, Datadog, Notification.

1 · The problem it solves

Every unit of work carries fixed overhead that has nothing to do with the work itself: a network round trip, TLS amortization, a syscall, a disk seek, a lock acquisition, an index update. When you do work one item at a time, that fixed cost dominates and throughput collapses long before CPU or disk is saturated. A service that can write 1 MB in a single request might manage only a few thousand 100-byte writes per second if each is its own request — the bottleneck is the per-operation tax, not the bytes.

Batching attacks the fixed cost directly: pay it once for many items. Async buffering attacks a second, related problem — a fast producer pointed at a slow or flaky consumer. Put a buffer between them and the producer no longer waits; the buffer absorbs bursts and smooths the rate the consumer actually sees. The two ideas almost always travel together: you buffer items, then flush them in batches.

2 · How it works

A batching writer accumulates items in memory and flushes when one of two triggers fires: a size trigger (the batch reaches N items or B bytes) or a time trigger (the oldest item has waited T milliseconds). The time trigger is what bounds latency — without it, a low-traffic key could sit in the buffer forever waiting for the batch to fill.

Three increasingly powerful forms, in order of how much they reduce load:

  1. Batching — group N items into one request. Reduces request count N×.
  2. Buffering / decoupling — add a durable queue or log so producer and consumer run at independent rates and the consumer can fail without backing up the producer (the whole reason Kafka exists).
  3. Pre-aggregation at the source — don't just group the items, collapse them. If you're shipping a counter, increment locally and send the sum; if you're shipping latencies, build a histogram/sketch locally and send the sketch. This turns O(events) network traffic into O(unique series), often a 100–1000× reduction. Aggregation is the highest-leverage form and the one to reach for on any high-write ingestion path.

The tuning knobs always express the same tension: bigger batch / longer timer = higher throughput and efficiency but higher latency and more data at risk in memory on a crash; smaller / shorter = lower latency, lower throughput, more overhead.

4 · The new bottleneck it hands you

Latency floor. Items now wait for the batch. A 200 ms flush timer adds up to 200 ms before data even leaves the producer. Fine for metrics; maybe not for a synchronous user request.

Durability gap. Anything in an in-memory buffer is lost if the process crashes before flush. You either accept the loss (metrics, logs — usually fine) or make the buffer durable (a WAL or a log like Kafka), which reintroduces disk cost — see P17.

Ordering. Batching across keys, retries, and parallel flushers weakens global ordering. You usually recover only per-key ordering (the model Kafka commits to).

The buffer is now state you must bound. If the consumer stalls, an unbounded buffer grows until it OOMs or the lag is useless. This is exactly where batching forces you into P16 backpressure: block, drop, or sample once the buffer is full.

5 · Real-world example

Datadog

Datadog's metrics pipeline is the textbook case of pre-aggregation at the source, applied at two layers before a single point ever reaches the backend.

Layer 1 — in the client (DogStatsD client-side aggregation). Your app emits metrics like statsd.increment("checkout.success") thousands of times a second. Rather than send a UDP packet per call, the DogStatsD client aggregates locally: counters are summed, gauges keep the last value, histograms/distributions are accumulated — then flushed on a short interval (the Go client defaults to ~2s, chosen to evenly divide the Agent's interval and avoid aliasing). A million increment calls become one number on the wire.

Layer 2 — in the Agent. The DogStatsD server embedded in the Datadog Agent receives those partial aggregates from every process on the host and aggregates again over a 10-second flush interval. All points for a unique (metric, tag-set) seen between 10:00:00 and 10:00:10 collapse into one datapoint timestamped 10:00:00, then get submitted to intake.

The clever bit — mergeable percentiles with DDSketch. Percentiles (p95, p99) can't be averaged across hosts, so historically you had to ship raw samples. Datadog instead builds a DDSketch at the source — a quantile sketch with a relative-error guarantee (e.g. 1%: a true p99 of 1000 ms returns within 990–1010 ms) that is fully mergeable. Per-host sketches combine on the backend into a global sketch without ever shipping the underlying samples. That is pre-aggregation taken to its logical end: you ship a fixed-size summary, not the data.

The payoff lands on the system's real hard constraint — cardinality. Aggregating at the source means network and backend cost scale with the number of unique series, not the raw event rate.

Kafka producer

The Kafka producer batches records per partition in memory and flushes on batch.size (bytes) or linger.ms (time) — the canonical size-or-time pattern. Setting linger.ms=5 tells the producer "wait up to 5 ms to let a batch fill before sending," trading a few ms of latency for far fewer, larger requests and much higher throughput. Combined with compression (which compresses the whole batch, not per record) this is how a single producer sustains hundreds of thousands of messages per second.

6 · Applying it in a design

Reach for batching the moment you see a high-write path: telemetry, logs, clickstream, analytics events, notifications. The sentence that signals you understand the tradeoff: "I'll aggregate at the source so cost scales with unique series, not event volume — accepting a flush-interval latency and the loss of any in-memory buffer on crash, which is fine for metrics." Then immediately name the follow-on: a durable buffer (Kafka) if you can't lose data, and backpressure for when the consumer falls behind.

7 · Pitfalls

  • No time trigger. Size-only batching starves low-traffic keys — they wait indefinitely. Always pair size with a timer.
  • Unbounded buffer. A buffer with no max size just converts a throughput problem into an OOM. Bound it and decide the drop/block policy up front.
  • Batching non-mergeable data. Pre-aggregation only works if your data has a combine operator (sum, max, last, or a mergeable sketch). Raw percentiles don't — that's why DDSketch exists.
  • Hiding partial failure. When one record in a batch fails, what happens to the other 999? Define whether a batch is all-or-nothing or partially committable.
