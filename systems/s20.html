20 · Authentication & Authorization

Hard constraint: verify identity on every request with low latency and no shared bottleneck, while keeping credentials un-stealable and sessions revocable. The central tension is where session state lives — a server-side store (revocable but a lookup per request) vs a stateless token (free to verify but hard to revoke).
Primitives exercised: P3 (session replication/consistency), P4 (session cache), S1 (login rate-limiting), S3 (distributed session store), P9 (idempotent flows).

Password storage; session vs token (JWT) ladder; refresh-token rotation + revocation; OAuth/OIDC delegated auth; MFA; rate-limiting login attempts. "Design login/auth for a website" — one of the most commonly asked real-interview questions.

1 · Requirements

Functional: register and log a user in; verify identity on each subsequent request (authentication); decide what they may do (authorization); support logout, "log out everywhere," password reset, MFA, and third-party login. Non-functional:

Two ideas separate a strong answer from a memorized one: login is rare but verification is constant (you authenticate once, then verify a token on every one of thousands of later requests), and the security comes from making the right operation slow (hashing) while keeping the hot path fast.

4 · Password storage (ladder)

Assume the database will be breached. The design must ensure that even then, attackers can't recover users' passwords. So storage is derived against an attacker who has your whole user table.

Rung 0 — plaintext

Store the password as-is. Breaks instantly: one breach reveals every password, and users reuse passwords across sites. Catastrophic.

Rung 1 — a fast hash (MD5/SHA-256)

Store hash(password). Breaks on: (a) rainbow tables — precomputed hash→password dictionaries crack common passwords instantly; (b) fast hashes are the problem — a GPU does billions of SHA-256/s, so brute force is trivial; (c) identical passwords produce identical hashes (visible in the table).

Rung 2 — salted hash

Store (salt, hash(salt + password)) with a unique random salt per user. Salts defeat rainbow tables (the attacker must crack each user separately) and hide identical passwords. Breaks on: the hash is still fast — per-user brute force is still billions/s on a GPU.

Rung 3 — a slow, tunable KDF: bcrypt / scrypt / Argon2 (the answer)

Use a password hashing function with a cost factor that makes each hash deliberately expensive. bcrypt's cost c runs 2c key-expansion rounds; at cost 12 that's ~250 ms per hash — negligible for one login, but it caps an attacker to ~4 guesses/s/core instead of billions. As hardware improves you raise the cost. Argon2id additionally is memory-hard (each hash needs lots of RAM), which neutralizes GPU/ASIC parallelism that bcrypt doesn't.

store: argon2id(password, salt, m=64MB, t=3, p=1) -> {salt, params, hash} login: recompute with stored params, constant-time compare tune: pick cost so a single hash ~ 200-300 ms on YOUR hardware -> 1 login = 250 ms (fine); 1M-guess attack = ~70 hours/core (not seconds)

5 · Where session state lives (the core ladder)

After login, every later request must prove "I am this logged-in user." Where you keep that proof — on the server or in the token — is the central tradeoff between revocability and statelessness.

Rung 1 — server-side session, sticky to one server

On login, create a random session_id, store session_id → user in that server's memory, send the id as a cookie. Breaks on: the session lives on one server, so a load balancer must pin the user there (sticky sessions) — which breaks horizontal scaling and loses sessions when that server dies.

Rung 2 — server-side session in a shared store (revocable, the safe default)

Move sessions to a shared, replicated store (Redis / S3-style KV) that any app server can read. Verification is a single ~1 ms cache lookup (P4). Revocation is trivial — delete the row and the session is instantly dead, so "log out everywhere" and forced logout on compromise are easy. Breaks on: a store lookup on every request — at very high RPS the session store becomes a shared hot dependency and a latency floor.

Rung 3 — stateless token (JWT): no lookup at all

Issue a signed JWT containing the claims (user_id, scopes, exp). The app server verifies the signature with the issuer's public key — no store lookup, fully stateless, any server can verify offline. Perfect for latency and scale. Breaks on revocation: a JWT is valid until it expires; you can't un-issue it. A stolen token works until exp, and "log out everywhere" is impossible without adding state back.

Rung 4 — short-lived access JWT + refresh token (the production synthesis)

Combine the two: a short-lived access JWT (e.g. 15 min, stateless, verified with no lookup on the hot path) plus a long-lived refresh token (e.g. 30 days, opaque, stored server-side and thus revocable). The access token's short TTL bounds the damage window; the refresh token is the revocable handle.

hot path (most requests): verify access JWT signature locally -> ~0 lookup every 15 min: client calls refresh(refresh_token) -> new access + ROTATED refresh revoke: delete the refresh token; access dies within <= 15 min automatically

6 · Refresh-token rotation & revocation

A long-lived refresh token is a juicy theft target, so it must rotate: each refresh issues a new refresh token and invalidates the old one. This enables reuse detection — if an old (already-rotated) refresh token is ever presented again, that means someone has a stolen copy, so the system revokes the entire token family (logs the user out everywhere) and forces re-login.

refresh(rt_old): if rt_old is unknown or already-used: // theft signal revoke entire family(user, device) ; require re-login else: issue (access_new, rt_new) ; mark rt_old used, link rt_new -> family

For the rare need to kill an access JWT before its TTL (e.g. account compromise), keep a small revocation/deny-list of token ids (jti) checked on the hot path — this reintroduces a lookup, but only for the short access-TTL window, so the list stays tiny. The honest tradeoff: pure statelessness and instant revocation are fundamentally at odds; short TTL + refresh is how production splits the difference.

7 · Login abuse, MFA & high-level architecture

graph LR
  U[user] --> LB[load balancer]
  LB --> APP[stateless app servers]
  APP -->|login: verify argon2| UDB[(user store: salted argon2 hashes)]
  APP -->|issue/rotate| RT[(refresh-token store: revocable, S3/Redis)]
  APP -->|verify access JWT signature| KEYS[(public keys / JWKS)]
  APP -->|rate limit login| RL[S1 rate limiter]

Rate-limiting login (S1) is essential against credential stuffing (attackers replaying breached email/password pairs): limit per-account (e.g. 5 fails / 15 min → backoff/lockout), per-IP, and globally (anomaly detection on a spike of failures). Add CAPTCHA after N failures and exponential backoff. MFA (TOTP, WebAuthn/passkeys, SMS as a weak last resort) means even a correct password isn't enough — the dominant defense against stuffing, since the attacker lacks the second factor. Constant-time comparison of hashes/tokens avoids timing side-channels, and the login error must be generic ("invalid email or password") so it doesn't reveal which emails are registered (account enumeration).

8 · OAuth 2.0 / OIDC: delegated auth

"Log in with Google" is delegated authentication: your app never sees the user's Google password. The Authorization Code flow with PKCE is the modern standard:

  1. App redirects the user to the provider's /authorize with a code_challenge (the hash of a secret code_verifier the app keeps).
  2. User authenticates at the provider and consents; provider redirects back with a one-time authorization code.
  3. App exchanges the code + code_verifier at /token for tokens. PKCE proves the app redeeming the code is the same one that started the flow, defeating code interception.

The distinction: OAuth 2.0 is about authorization (granting an app scoped access to resources); OIDC layers authentication on top via an ID token (a JWT asserting "this is who the user is"). Use OIDC's ID token to establish identity, OAuth access tokens to call APIs on the user's behalf. The redirect dance exists precisely so the password stays with the identity provider.

9 · Authorization (authz) & scaling

Authentication says who; authorization says what they may do. Models escalate by complexity: RBAC (roles → permissions; simple, coarse) → ABAC (policies over attributes; flexible) → ReBAC (relationship graphs, e.g. Google Zanzibar: "is user X an editor of doc Y?" resolved over a globally-replicated permission graph) for fine-grained, shared-resource permissions at scale. Put coarse scopes in the access token (no lookup) and resolve fine-grained checks against an authz service. Scaling auth is mostly about keeping verification stateless (JWT signature, public keys distributed via JWKS) so app servers add linearly, while the few stateful pieces (refresh store, revocation list) stay small and cacheable (P4, P3).

10 · Tradeoffs & alternatives

Stateless JWT (fast, scalable, hard to revoke) vs server sessions (instantly revocable, a lookup per request) — short-TTL access + revocable refresh is the standard compromise. Cookies (with HttpOnly + Secure + SameSite, automatically sent, XSS-resistant for storage) vs Authorization: Bearer headers (explicit, needed for cross-origin APIs, but token storage in JS is XSS-exposed). Passkeys/WebAuthn are increasingly replacing passwords entirely (phishing-resistant public-key auth, nothing crackable to steal). SMS MFA is better than nothing but vulnerable to SIM-swap; prefer TOTP/WebAuthn.

11 · Real-world example

Auth0 / Okta · Google

Managed identity providers (Auth0, Okta, AWS Cognito) implement exactly this shape: argon2/bcrypt password hashing, short-lived access JWTs verified statelessly via published JWKS public keys, revocable rotating refresh tokens, OAuth2/OIDC flows with PKCE, MFA, and login-anomaly rate limiting. Google's Zanzibar is the canonical large-scale authorization system — a globally-distributed, consistent permission graph (ReBAC) answering billions of "can user X do Y on object Z?" checks with low latency, the model behind Google Docs sharing. The recurring lesson matches the constraint: keep verification stateless and cheap, keep the revocable state small, and make the credential-storage operation deliberately slow.

12 · Interview follow-ups (Q&A)

Q: Why is a slow hash good? Login happens once and can afford ~250 ms; an attacker with a stolen DB must hash billions of guesses, so a slow, memory-hard KDF (Argon2/bcrypt) turns a seconds-long crack into years. The cost factor is raised as hardware improves.

Q: JWT vs server-side sessions — which and why? JWT for stateless, lookup-free verification at scale; server sessions for instant revocation. The tension is real, so production uses a short-lived access JWT + revocable refresh token: stateless on the hot path, revocable via the refresh handle, with damage bounded by the short access TTL.

Q: How do you "log out everywhere" with JWTs? You can't revoke the access JWTs directly, but you delete all the user's refresh tokens (so no new access tokens issue) and the existing ones expire within the short TTL. For immediate kill, add the token ids to a small deny-list checked on the hot path.

Q: A refresh token is stolen — how do you detect it? Rotate refresh tokens on every use; if an already-used (rotated) token reappears, that's a theft signal — revoke the whole token family and force re-login.

Q: How do you stop credential stuffing? Rate-limit logins per account/IP/global (S1), add CAPTCHA and backoff after failures, and require MFA — the breached password alone is then useless. Generic error messages prevent account enumeration.

Q: What does "Log in with Google" actually prove, and where's the password? The user authenticates at Google; your app only receives an authorization code it exchanges (with PKCE) for an OIDC ID token asserting identity. The password never leaves Google — that's the point of the redirect flow.

Q: Where do you store the token client-side? Prefer an HttpOnly+Secure+SameSite cookie (not readable by JS, so XSS can't exfiltrate it; pair with CSRF protection). Bearer tokens in JS-accessible storage are exposed to XSS.

13 · Key talking points

  • Login is rare (can be slow); verification is constant (must be ~free) — design around that asymmetry.
  • Password storage ladder: plaintext → fast hash → salted → Argon2/bcrypt with a cost factor (slow + memory-hard); assume the DB will leak.
  • Session-state ladder: sticky → shared store (revocable) → stateless JWT (no lookup, hard to revoke) → short access JWT + revocable refresh.
  • Refresh-token rotation + reuse detection revokes a stolen family; short access TTL bounds the window.
  • Rate-limit + MFA (S1) defeat credential stuffing; constant-time compare + generic errors close side channels.
  • OAuth2/OIDC + PKCE keeps the password with the IdP; authz scales via stateless scopes + a permission service (RBAC→ReBAC/Zanzibar).
