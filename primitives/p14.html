P14 · Saga & compensation

What it is: a multi-step workflow across services where each step has a compensating undo, used when a single distributed transaction is too expensive.
Reach for it when: a business operation spans systems you can't wrap in one ACID transaction (debit → credit → notify).
The lever: orchestrated or choreographed steps; each records progress; failure triggers compensating actions; combine with P9 for safe retries.
New problem it hands you: intermediate states are visible (no isolation); compensations may themselves fail; you need reconciliation as the final net (2PC is the alternative and rarely worth it).
Exemplified by: Payment, order/checkout flows.

1 · The problem it solves

A business operation often spans several services — reserve inventory, charge the card, book the shipment — that you can’t wrap in one ACID transaction (they’re different databases, some external). A saga runs the steps as a sequence, and if a later step fails, runs compensating actions to undo the earlier ones.

2 · The ladder to a multi-service transaction

Rung 0 — one ACID transaction

If all the data lives in one database, wrap the steps in a single transaction and you’re done — atomicity for free. Always prefer this when it’s possible.

Bottleneck: the steps span different services/DBs (and external APIs), so no single transaction can cover them.

Rung 1 — two-phase commit (2PC)

A coordinator asks every participant to prepare, then commit only if all voted yes. Gives atomicity across services — but every participant must hold locks through the whole round trip, and if the coordinator dies after prepare, participants are blocked indefinitely.

Bottleneck: locks held across services kill throughput; coordinator failure blocks resources. Rarely scales.

Rung 2 — saga (compensations) ← the answer

graph LR
  S1[reserve] --> S2[charge] --> S3[ship]
  S3 -.->|fail| C2[refund]
  C2 --> C1[release reservation]

Run the steps as a sequence with no global lock; if a later step fails, run a compensating action to undo each completed step. Orchestration uses a central coordinator; choreography has each service emit events the next reacts to. Each step records progress (so a crashed saga resumes) and is idempotent (P9). Worked, an order that fails at shipping:

reserve inventory OK (compensation ready: release) charge card OK (compensation ready: refund) book shipment FAIL -> run compensations in reverse: refund card OK release inventory OK -> system back to a consistent state, no locks ever held across services

3 · The new bottleneck it hands you

Sagas give up isolation: intermediate states are visible (a charge exists before the shipment is booked), so other readers can see partial results. Some actions have no clean undo (you can refund a charge, but you can’t un-send an email). Compensations can themselves fail, so you still need reconciliation as the final safety net. The alternative, two-phase commit, blocks resources and rarely scales — which is why sagas exist.

4 · Real-world example

Uber Cadence / Temporal

The saga concept dates to Garcia-Molina & Salem (1987), but its modern form is durable workflow engines. Uber built Cadence (now continued as Temporal) precisely to run long-lived, multi-step business workflows reliably: the engine persists every step’s state so a process can crash and resume exactly where it left off, and developers write compensation logic for rollback. Trip booking, payments, and driver onboarding are modeled as orchestrated sagas on top of it. The engine turns "remember where we are and what to undo" into infrastructure. See Payment and Task scheduler.

5 · Applying it in a design

For multi-service operations: "orchestrated saga with a compensating action per step, idempotent steps, and nightly reconciliation — rather than a distributed transaction."

6 · Pitfalls

  • Designing only the happy path; the compensations are the hard part.
  • Assuming compensations always succeed.
  • Reaching for 2PC when a saga would do.
