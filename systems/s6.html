6 · Distributed Metrics Logging & Aggregation (Datadog)

Hard constraint: cardinality — the number of unique tag combinations, not the point rate, drives cost and scalability.
Primitives exercised: P1, P6, P7, P8, P5, P16, P17, P9, P12, P2.

Push vs pull collection; agent-side batching + pre-aggregation; mergeable sketches; Gorilla compression; inverted index on tag space; columnar TSDB; logs index-vs-archive; head- vs tail-based trace sampling; the three pillars correlated on trace context; scatter-gather MPP with pushdown.

1 · Requirements

Functional. Ingest the three pillars of observability — metrics (numeric time series), logs (high-cardinality text events), and traces (per-request span trees) — from millions of hosts; let users query and aggregate them ad hoc ("p99 latency of checkout by region, last 4h"); search logs; inspect traces; build dashboards; fire threshold and anomaly alerts; correlate across the three (a metric spike → the traces behind it → the logs of those traces); and retain data from high-resolution recent to cheap long-term.

The three pillars have very different shapes — that difference is the design:

PillarUnitSizeCardinalityHow it's stored
Metric(series, ts, value)~16 B/pointLow (bounded tag sets)Pre-aggregated, always indexed → columnar TSDB
LogTimestamped event~1 KB/event (~62×)High (free text, ids)Archive all, index selectively
TraceTree of spans~500 B/span, 10–50/traceVery high (per-request)Sample, then store like logs

Non-functional. Ingest must never block the app and must shed gracefully under load; recent queries are interactive (sub-second to seconds); recent data is durable while old data can be lossy-rolled-up; ingest is at-least-once with idempotent points (a re-sent flush must not double-count). The cost axis to bound everywhere is cardinality (§9).

2 · Capacity estimation

Numbers first — they justify every later choice, and they explain why the three pillars get three different storage strategies.

METRICS 2,000,000 hosts × 200 metrics/host = 4×10^8 active series each series: 1 point / 10s = 4×10^7 points/sec THE 1000× TRAP: 1 UDP packet per metric call ≈ 4×10^10 calls/s → aggregate at source (§7) storage naive 16 B/pt ≈ 55 TB/day raw → Gorilla ~1.37 B/pt ≈ 4.5 TB/day ≈ 1.6 PB/yr (~12×) LOGS 2,000,000 hosts × ~5 logs/s = 1×10^7 logs/s × ~1 KB/log ≈ 10 GB/s ≈ 864 TB/day raw (a log ≈ 62× a metric point!) → can't index all → archive 100%, index ~5% (decouple ingest from index, §10) TRACES 1,000,000 req/s × ~20 spans/trace = 2×10^7 spans/s × ~500 B/span ≈ 10 GB/s ≈ 864 TB/day raw → sample: head 1% → 2×10^5 spans/s stored; tail buffer = 2×10^7×10s×500B ≈ 100 GB (§11) CARDINALITY (the actual cost driver, all three pillars) 10 tags × 10 values each = 10^10 potential series one high-card tag (user_id, ~10^7 users) turns a 1k-series metric into 10^10

Three takeaways drive the rest. (1) The metric event rate is ~1000× the point rate, so you aggregate before the wire (P1). (2) Logs and traces are huge and high-cardinality — orders of magnitude more bytes than metrics — so you cannot index them wholesale; you archive cheaply and index/sample selectively. (3) What blows up cost across all three is cardinality, not raw point rate (§9).

3 · API design

Four surfaces: three fire-and-forget submit paths (one per pillar, always 202 — the client never blocks on the backend) and a query/search path.

POST /api/v1/series (metrics; agent → intake) { "series":[{ "metric":"request.latency", "type":"distribution", "interval":10, "tags":["service:checkout","region:eu","host:i-09f"], "points":[[1718000000, ]] }] } → 202 POST /api/v1/logs (logs; batched) { "ddsource":"nginx", "service":"checkout", "hostname":"i-09f", "trace_id":"4bf9…4736", // ← carries trace context for correlation "message":"127.0.0.1 - GET /checkout 200 1234" } → 202 POST /api/v1/traces (spans; batched per trace) { "spans":[{ "trace_id":"4bf9…4736", "span_id":"00f0…02b7", "parent_id":"…", "name":"checkout.handler", "service":"checkout", "start":…, "duration":…, "tags":{…} }] } → 202 GET /api/v1/query ?q = p99:request.latency{service:checkout} by {region} &from=now-4h GET /api/v1/logs/search ?q = service:checkout status:error &from=now-15m GET /api/v1/trace/{trace_id}

The metric query language carries the whole shape of the system: an aggregator (p99), a metric, a tag filter (resolved through the inverted index), and a group-by (the scatter-gather grouping key). The log search reads like a search engine query (facets + full-text), because that is exactly what it is (§10).

4 · Data model & store selection (metrics)

A metric point is logically (series_key, timestamp, value), where series_key = metric name + sorted tag set. The honest way to pick a store is start with SQL and escalate only when it breaks — not "NoSQL because time-series." Walk the ladder, anchored to 4×107 points/s from §2. (Logs and traces land on different stores — §10, §11.)

Rung 0 — one SQL table (right default at small scale)

points(series_key, ts, value), composite B-tree index on (series_key, ts). Fetching one series over a time range is a contiguous range scan on that index — exactly what a B-tree is good at. Fine for thousands of series.

Bottleneck: at 4×107 inserts/s the index has no insert locality — points for thousands of distinct series interleave, so every insert lands at a random place in the B-tree. Once the index exceeds RAM, each insert is a random disk seek; you're also storing an 8-byte float in a ~40-byte row. Volume (1.6 PB/yr) and random-write amplification kill it together.

Rung 1 — time-partition + per-series clustering (helps writes & retention)

Cut the table into immutable time-windowed partitions (e.g. 2-hour blocks), and within a block cluster rows by series. Old blocks become read-only and tier off cheaply; deletes become a partition drop. Right shape — but still a row store.

Bottleneck: a query like "p99 by region over 4h" reads millions of rows to touch one column (the value). A row store pulls whole rows off disk, so you pay I/O for tags and timestamps you've already filtered on; random per-series writes still thrash the page cache. Two bottlenecks: write amplification and scan I/O.

Rung 2 — LSM-backed engine for the write side (P17)

Buffer writes in an in-memory memtable + write-ahead log, flush sorted runs sequentially — an LSM tree (log-structured merge: writes go to RAM and an append-only log, then merge into sorted files in the background, turning random writes into sequential ones; P17). Absorbs 4×107 sequential writes/s. But it doesn't fix scan I/O — you still read rows to touch one column.

Rung 3 — columnar TSDB (← pick; P8)

Store each series as its own column of values in an immutable, time-partitioned block, compressed with Gorilla (delta-of-delta on timestamps, XOR on floats — § below). A scan now touches only the value column, decode is cheap, and the data is ~12× smaller. The memtable still absorbs writes (rung 2's LSM idea), but flushed blocks are columnar. This is what a purpose-built TSDB (Prometheus TSDB, Gorilla, M3, InfluxDB) is — and we arrived here because the access pattern is scan-few-columns-over-many-rows, the exact case a row store dies on.

RungGood untilBreaks on
One SQL tablethousands of series, index fits RAMRandom-seek inserts; 1.6 PB/yr on one node
+ time-partitionretention & drops easyRow store: scan I/O reads whole rows; write amp
LSM engineabsorbs sequential writesStill scans rows to touch one column
Columnar TSDB (← pick)scan-optimized, 12× smaller(accepted) immutable blocks → rewrite to update

Two layouts: point/scan values vs the tag index

One partition key can't serve both access patterns, so the store is split (P2):

VALUE BLOCKS partition by series_id (+ time window) ← "give me series X's points, 4h" series_id → [Gorilla-compressed (ts,value) column], immutable per 2h block TAG INDEX partition by tag ← "which series match this filter?" "service:checkout" → postings [12, 47, 88, 91, 140, 205] (sorted series_ids) "region:eu" → postings [3, 47, 91, 140, 301]

A query resolves the filter against the tag-partitioned inverted index first (intersect postings, §8), then fans out to the value blocks of just the matching series. The tag lookup does not scatter across every series shard — only the value scan does, and that fan-out is expected (P7). That split is the answer to "which partition does this query hit?"

One agent collects all three pillars and pushes to intake (§6). A durable Kafka log decouples ingest from storage and fans into three pipelines: metrics are aggregated, rolled up, indexed by tag, and written to a columnar TSDB; logs are parsed and split between a search index (the few you choose to index) and a cheap archive (everything); traces are assembled by trace_id, sampled, and stored. Queries resolve through the relevant index, scatter across shards, aggregate partials, and gather. The pillars are stitched back together at query time by shared trace_id + tags (§12).

6 · Collection model — push vs pull

Before the data is aggregated, you choose how it's collected. This is a classic interview probe, and the answer is derived from where your targets live, not from taste.

Push — the client/agent decides when to send

The app emits to a local agent (DogStatsD, the OpenTelemetry SDK) and the agent ships outbound to intake. Logs and traces are inherently push — they're events as they happen, not pollable state. Pros: works for ephemeral/short-lived targets (serverless functions, batch jobs, a request handler that lives 50 ms — there's nothing to scrape later); firewall/NAT-friendly (outbound only); and it crosses trust boundaries — a SaaS vendor can't reach into a customer's private network, but the customer's agent can push out. Cons: the server can be flooded (no rate control — you need backpressure, §16); target discovery is implicit; and a silent target is ambiguous — is it healthy-but-quiet or dead?

Pull — the server scrapes targets on a schedule

The Prometheus model: the server discovers targets via service discovery and scrapes each one's /metrics endpoint every N seconds. Pros: the server controls the scrape interval and cardinality; a failed scrape is an unambiguous "target down" signal (the synthetic up metric flips to 0); config is centralized. Cons: the server needs network reachability to every target (hard across NAT/firewalls and impossible across a SaaS trust boundary); and it can't see short-lived jobs that finish between scrapes — you bolt on a Pushgateway, which is admitting push was needed.

push vs pull → when each wins

Pull (Prometheus): you own the network, targets are stable and discoverable (long-running services in your cluster) → server-controlled, down-detection for free → breaks on ephemeral jobs & SaaS trust boundaries.  Push (Datadog): targets are ephemeral, client-side, or across a trust boundary → universal coverage → server must rate- limit/backpressure, and "is it down?" needs a heartbeat.

This is precisely why a SaaS observability platform like Datadog is push at the edge: it cannot scrape into millions of customers' private networks, so the agent pushes out. Prometheus pulls because it lives inside the cluster it watches. (You can bridge: Datadog and the OTel collector can scrape Prometheus endpoints locally on the host, then push upstream — pull on the LAN, push across the WAN.)

7 · The metrics write path — aggregate at the source (the ladder)

"Ingest metrics from millions of hosts" sounds like a throughput problem. It's really an aggregation-placement problem: the cheapest point is the one you never send. Derive it rung by rung.

Rung 0 — one packet per metric call

Every statsd.increment() sends a UDP packet. At ~1000 emissions per aggregated point that's 4×1010 packets/s across the fleet — the network stack and intake melt long before storage is reached.

Bottleneck: per-call network + syscall overhead, not data volume.

Rung 1 — aggregate in-process (client)

The DogStatsD client aggregates in the app process — counters summed, gauges last-write-wins, distributions accumulated — and flushes every ~2 s (P1 batching). A million calls/s collapse to one value per metric per flush.

Bottleneck: N processes per host each open a connection and ship a partial — the host still emits many small payloads, and per-process partials can't be merged for percentiles.

Rung 2 — host agent aggregates across processes

The Agent's embedded DogStatsD server aggregates across all processes on the host over a 10 s flush interval, then compresses and batches one stream to intake. Now the host emits ~points/s, not calls/s — the 1000× reduction is real, one connection per host.

Rung 3 — mergeable sketches for percentiles (DDSketch)

Sums and counts merge trivially across hosts; percentiles do not — a global p99 isn't the average of per-host p99s. The naive fix ships raw samples centrally, re-introducing the volume you just removed. Instead each host builds a DDSketch: a fixed-size quantile sketch that buckets values on a logarithmic scale, so it guarantees a relative error (1% accuracy → a true p99 of 100 ms reads back in [99, 101]) and is fully mergeable with no added error. For 1% accuracy over 1 ms–100 s:

gamma = (1+0.01)/(1−0.01) = 1.0202 bucket i covers ~ gamma^i buckets to span 1ms…100s (10^5 range) = ln(10^5)/ln(1.0202) ≈ 576 buckets → a few hundred counts, FIXED regardless of how many samples flow through

Per-host sketches merge on the backend into a global sketch, so a distribution metric costs a bounded amount no matter the sample count.

After the wire — buffer, dedup, roll up, index

Intake appends to a durable Kafka log (S2): it absorbs bursts, decouples ingest from storage, and gives at-least-once replay. Rollup workers then dedup each point by (series_id, timestamp) — last-write-wins makes a re-delivered flush idempotent (P9), so at-least-once delivery doesn't double-count; roll up to 1-min/1-hour aggregates (P5 write-time rollup) for fast dashboards; append to the in-memory head block (memtable + WAL); and index the tags. When intake or rollup lags, Kafka holds the backlog; if it keeps growing the Agent must shed — sample, drop, slow down — rather than fall over (P16 backpressure).

rung → good until → breaks on

Per-call packets: tiny fleets → 4×1010/s melts the network.  Client aggregation: collapses calls per process → many small per-process streams.  Agent (10 s): one stream/host at point-rate → percentiles still don't merge.  DDSketch: bounded, mergeable percentiles → ~1% relative error (accepted).

8 · The metrics read path — scatter-gather with pushdown

Take p99:request.latency{service:checkout AND region:eu} by {region}, last 4h:

  1. Resolve the filter against the inverted index (P6). Intersect the two sorted posting lists by a merge walk:
    service:checkout : [12, 47, 88, 91, 140, 205] region:eu : [ 3, 47, 91, 140, 301] AND (intersect) : [47, 91, 140] ← only these 3 series get read
    Cardinality is precisely the size of this index — it's why high-card tags are the cost.
  2. Scatter to the value shards owning series 47/91/140 across the 4h window (P7).
  3. Partial-aggregate at the shard (pushdown): each shard decodes its Gorilla blocks, applies the time filter, and emits a partial — sum counters, merge DDSketches — not raw points. Shipping summaries instead of points keeps the coordinator cheap.
  4. Gather & merge partials by the by {region} key, finalize p99 from the merged sketch.
Bottleneck — tail latency. A query that fans out to 100 shards fails its budget if any one is slow: with each shard 1% likely slow, P(≥1 slow) = 1−0.99100 ≈ 63% (P12). Mitigate with hedged requests above p95 and keeping recent blocks in RAM. Older data reads from the cold object-store tier at coarser rolled-up resolution.

9 · The hard constraint, in depth: cardinality

A metric like request.latency isn't one series — it's one series per unique tag combination {service, host, endpoint, status, region, version, …}. Ten tags with ten values each is 1010 potential series. The point rate may be modest, but each new series needs an index entry, its own compressed block, and memory in the write path. High-cardinality tags (user_id, request_id, full URL) are the classic way to turn a cheap metric into a multi-million-series monster — attaching a 107-user user_id to a 1k-series metric yields 1010 series.

So "contain cardinality" is itself a ladder:

RungGood untilBreaks on
Index every tag combolow-cardinality metricsCombinatorial index blow-up; SQL tag cross-product
Split metrics vs logsmost telemetryNeeds two engines (pre-agg TSDB + archived logs)
Tag budgets / drop-or-aliasalways-on guardrail(accepted) loses the offending dimension

This is why the design splits metrics (low-cardinality, pre-aggregated, always indexed) from logs/traces (high-cardinality, archived, indexed/sampled selectively). It's also why "just put it in a SQL table with a tag column" fails: the index on the tag cross-product is the cost, and it grows combinatorially. The same lens explains the logs (§10) and traces (§11) strategies — both refuse to index/keep everything.

10 · Logs pipeline, in depth

A log is a timestamped event — often a raw text line, ideally structured JSON. At ~1 KB it's ~62× a metric point, and its fields (request id, user id, full URL, stack trace) are unbounded cardinality. At 864 TB/day you cannot index it all; the whole logs design is about what you choose to make searchable.

Ingestion & parsing

The agent tails files, journald, container stdout, or receives syslog /HTTP, batches, and pushes (§6). A parsing stage turns text into structured fields with grok patterns — named regexes that extract fields. For an nginx line:

raw: 127.0.0.1 - - [21/Jun/2026:10:00:00] "GET /checkout HTTP/1.1" 200 1234 grok: %{IP:client} … "%{WORD:method} %{URIPATH:path} …" %{NUMBER:status} %{NUMBER:bytes} out: { client:"127.0.0.1", method:"GET", path:"/checkout", status:200, bytes:1234 }

Then enrich (add host/service/env tags, geo-IP), filter (drop noise), and route. Structured fields are what make logs facetable — you can group by status or path only because parsing pulled them out.

The index-vs-archive ladder

The core decision is what to index, derived against the 864 TB/day.

RungWhat it doesBreaks on
Index everything (ES-style)full inverted index on every field + full-textIndex ≈ raw size → 864 TB/day of index; cost explodes
Index facets + full-text onlyindex chosen fields; full-text on messageStill indexing 100% of ingest; most logs never searched
Decouple ingest from index (← pick)archive 100% cheaply; index a chosen subset live(accepted) un-indexed logs need rehydration to search

Rung 2 is "Logging without Limits": every log is written to a cheap, compressed object-store archive (columnar, partitioned by time — P8), while index filters decide which logs also enter the hot search index. You pay the expensive index cost only for the ~5% you'll actually query interactively; the rest sits in archive at ~1/20th the price. To investigate an old or rare incident you rehydrate — pull the matching window back out of archive and index it on demand.

Search

The hot index is an inverted index — the same structure as a search engine (S7, P6): facet terms (status:error) map to posting lists of log ids; full-text tokens map to the logs containing them; a query intersects/unions postings, then fetches and ranks. It's sharded by time so a "last 15 min" query hits only recent shards — and, like metrics, fans out scatter-gather with the slowest shard setting the tail (P12).

Log-based metrics & patterns

Two cost-savers fall out. Log-based metrics: generate a metric (e.g. count(status:error) by service) from the log stream before archiving, so you keep a cheap, permanent time series even after the raw logs age out — logs feeding the metrics pipeline. Exclusion filters / sampling: for high-volume low-value logs (health checks, 200s), keep 1-in-N or drop entirely from the index while still archiving. Both apply the cardinality lesson: don't keep at full fidelity what you won't query.

11 · Distributed tracing, in depth

A trace is the tree of work for one request; each node is a span {trace_id, span_id, parent_id, name, service, start, duration, tags}. Spans for one trace are emitted by many services on many hosts, arrive out of order, and must be assembled by trace_id.

Context propagation (W3C traceparent)

For services to share a trace_id, each hop forwards a traceparent header. Its 55-char layout is exact:

traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01 ┌┘ └──────── trace-id (16 B) ────────┘ └─ span-id (8B) ┘ └┘ version flags flags = 01 → "sampled" bit SET; the whole trace shares this one decision downstream trace-id space = 2^128 → collision-free at any realistic rate

The caller's span_id becomes the callee's parent_id; the sampled flag is propagated so every service makes the same keep/drop choice — otherwise you'd store half a trace.

The sampling ladder — head vs tail

At 2×107 spans/s (864 TB/day) you can't keep everything. When you decide what to keep is the whole game.

Rung 0 keep 100% → 864 TB/day. Unaffordable. Forces a decision. Rung 1 HEAD sampling, fixed rate → decide at the root, propagate the flag. Stateless, cheap. breaks on: you decide BEFORE you know the outcome — errors & slow traces get dropped. Rung 2 HEAD, adaptive/per-service → rate-limit per endpoint, boost rare routes. Still outcome-blind. Rung 3 TAIL sampling (← want) → buffer ALL spans of a trace until it completes, THEN decide on the whole trace: keep if error / p99-slow / rare. Catches exactly the interesting ones. costs: (a) buffer = 2×10^7 × ~10s × 500B ≈ 100 GB across collectors; (b) ALL spans of one trace must reach the SAME collector → route by hash(trace_id)%N; (c) a decision-wait delay before the trace is stored. Rung 4 HYBRID (production norm) → low head baseline (e.g. 10–20% for healthy-traffic stats) + tail rules that always keep errors/slow/rare.

Head-based sampling is stateless and free but blind: the keep/drop bit is set when the root span starts, before any error has happened, so it can't preferentially keep failures. Tail-based sampling holds every in-flight trace's spans in a collector until a decision_wait timer expires, then judges the complete trace — which is why it can "always keep traces that errored." Its price is real: gigabytes of in-memory buffer, and a load-balancing constraint — since the decision needs all of a trace's spans, they must be routed to one collector by hash(trace_id) % N (consistent hashing, P2). Most shops run the hybrid: cheap head baseline for representative volume, tail rules for the traces that matter.

Trace-based metrics — compute before you sample

Key trick: derive the RED metrics (Rate, Errors, Duration) from 100% of spans before sampling, then sample only what you store. So your dashboards and alerts see exact request/error/latency numbers even though you keep a tiny fraction of full traces — the same "aggregate the cheap summary, archive/sample the expensive raw" move as metrics (§7) and logs (§10).

Storage & the service map

Sampled traces are stored like logs — high-cardinality events indexed by trace_id, service, resource, and tags, time-partitioned and tiered. From the span parent/child edges plus service tags you derive a service dependency map (who calls whom, with latency/error rates on each edge) — computed by aggregating span relationships, again from the full stream, not the sampled store.

12 · Correlation — unifying the three pillars

Three pipelines are only an observability platform if you can pivot between them. They unify on shared identity: a consistent tag set (service, env, version, host) on everything, plus the trace_id propagated into logs and spans. The investigation flow becomes: a metric spike on a dashboard → pivot to the traces in that window with the same tags (often via an exemplar — a metric bucket carries a sample trace_id) → open a slow/errored trace → jump to the logs emitted during its spans (they carry the same trace_id). This is why context propagation and a unified tagging scheme are foundational, not cosmetic: without a shared key, the three stores are three islands. It's also the answer to the cross-cutting "async work is a debugging black hole" check — carry trace context through every queue and deferred job so a side effect firing minutes later still stitches back to the originating request.

13 · Alerting — deriving when you need a stream processor

"Fire alerts on thresholds" sounds like it demands a real-time stream engine (Flink/Kafka Streams). It usually doesn't. Derive the rung from the alert condition.

Rung 0 — evaluate rules by querying the store

An evaluator runs each rule (avg(cpu) by region > 0.9 for 5m) on a schedule against the hot TSDB — the Prometheus model. The query path (§8) already exists, so an alert is a saved query plus a for-duration. Correct and cheap for threshold alerts at ~10 s–1 min latency.

Bottleneck: re-running thousands of windowed aggregations every tick re-scans the same recent blocks; and because the Agent flushes on ~10 s and the network adds lag, points arrive after their event-time, so a query-time window silently mixes late data and computes on arrival-time, not event-time.

Rung 1 — evaluate against the rollups you already compute

The pipeline already produces write-time 1-min rollups (§7, P5). Point the evaluator at those: far fewer rows per tick, and the rollup boundary gives a stable window. Reuses existing machinery, no new system.

Bottleneck: fixed rollup granularities can't express a sliding event-time window, late-data correction, sessionization, multi-stream joins, or anomaly/derivative conditions.

Rung 2 — stateful stream processor on the Kafka buffer (the Flink-class rung)

Now it's hard to avoid: a stream processor consumes the durable Kafka buffer already in the diagram and maintains keyed, event-time windowed state with watermarks (a moving "event-time is now at least T" marker that lets it close a window while tolerating bounded lateness), evaluating conditions in-stream — sliding p99 windows, late-data handling, anomaly detection at the full 4×107 points/s. Its cost is the operational weight of managed state and checkpointing. It need not be Flink — Kafka Streams, Spark Structured Streaming, or a streaming database (Materialize, RisingWave) sit on this same rung.

rung → good until → breaks on

Query-the-store: threshold alerts, moderate rules → expensive re-scan + no event-time correctness.  Evaluate-the-rollups: fixed-window aggregates → no sliding/event-time /stateful conditions.  Stream processor: event-time windowed + anomaly alerting at volume → managed state, checkpointing, ops weight.

So a stream processor becomes mandatory only once the spec includes event-time- correct windowed or anomaly alerting at high volume. The storage and threshold half ships fine without one — exactly the requirement to name out loud before adding it.

14 · Scaling & bottlenecks

At 10× the design stresses on axes that are mostly not point rate. Cardinality grows the inverted index and per-series memory (metrics), the log index, and trace volume — guarded by tag budgets, the metrics/logs split, index-vs-archive, and trace sampling. Query/search fan-out widens, so the tail worsens (63% → near-certain a slow shard appears) — bound it with hedging and tighter per-query shard counts (P12). Storage grows linearly — relieved by rolling up harder, archiving logs, sampling traces, and tiering to object storage sooner (P8). The ingest tier scales horizontally because Kafka partitions and the metric/log workers are stateless; the two stateful pieces are the metric head block (bounded by WAL replay of only the active ~2 h window) and the trace tail-sampling buffer (bounded by decision_wait and sharded by trace_id).

15 · Tradeoffs & alternatives

If you needed arbitrary high-cardinality metric slicing (per-user metrics), you'd stop pre-aggregating and move toward a columnar analytics engine over raw events — far more flexible, far more expensive, slower. If you needed exact percentiles rather than 1%-relative, you'd ship more data. Write-time vs read-time rollup is a dial: precompute fixed granularities (fast reads, more storage) vs store fine points and aggregate on query (flexible, slower, cheaper). Pull vs push (§6): pull gives free down-detection and server-side control but can't cross trust boundaries; push is universal but needs backpressure. Head vs tail sampling (§11): head is cheap and stateless but outcome-blind; tail keeps the interesting traces but costs buffer + routing. The whole platform is a set of dials between cost, query flexibility, resolution, and fidelity.

16 · Real-world example

Datadog

The two-tier source aggregation (DogStatsD client → Agent, 10 s flush, DDSketch for distributions) is exactly how Datadog keeps metric cost proportional to unique series instead of raw event volume, and it's push-based because a SaaS can't scrape into customer networks. Logs use "Logging without Limits": ingestion is decoupled from indexing, so customers archive everything cheaply and pay the search-index cost only for the logs they choose — the productized form of the cardinality constraint. Traces use ingestion-time sampling with retention filters (a head baseline plus tail-style rules that keep errors and high-latency traces). Percentile metrics use DDSketch (1%-relative-error, fully mergeable, open-sourced and published at VLDB 2019).

Other stacks make the same moves under different names: Prometheus pulls and stores locally in its own columnar TSDB and warns hard about high-cardinality labels; Facebook Gorilla is the source of the compression scheme (16 B → ~1.37 B/point, ~12×, 96% of timestamps to a single bit via delta-of-delta); OpenTelemetry standardizes traceparent propagation and the head/tail sampling collectors; Elasticsearch is the canonical index-everything logs engine the "without Limits" model reacts against; M3 (Uber) and Mimir/Thanos scale the Prometheus model with object-storage tiering.

17 · Interview follow-ups (Q&A)

Push or pull for metrics — which and why? Derive from where targets live. Pull (Prometheus) when you own the network and targets are stable + discoverable: server controls interval/cardinality and gets down-detection free. Push (Datadog/StatsD) when targets are ephemeral (serverless), client-side, or across a trust boundary — a SaaS can't scrape into customer networks, so the agent pushes out. Logs/traces are inherently push (events). Common hybrid: scrape Prometheus endpoints locally, push upstream.

Which partition does service=checkout AND region=eu hit? The tag-partitioned inverted index, not the series shards. Intersect the two posting lists to get matching series_ids, then the value scan fans out only to those series. A series_id-partitioned design would force the tag query to scatter every partition — which is why the store is two layouts (§4).

An ingest node holding the in-memory head block crashes — recovery time? Bound it: each ingest shard's head block is the active ~2 h window backed by a WAL; on failover the new owner replays only that head block's WAL (~108 points/shard), not the 1.6 PB/yr history. Flushed columnar blocks are already durable and immutable.

Kafka is at-least-once — won't a re-delivered flush double-count? Points are idempotent on (series_id, timestamp) with last-write-wins, and counters are aggregated at the source per flush — so re-applying a flush replaces that interval's value rather than adding (P9). At-least-once + idempotent point = effectively-once storage.

How do you compute a global p99 from per-host data? You can't average per-host p99s. Each host ships a mergeable DDSketch; the backend merges them with no added error and reads the quantile off the merged sketch (§7).

You can't store every log — how do you keep them useful? Decouple ingest from index: archive 100% to cheap object storage, index only the ~5% you'll search live, and rehydrate older windows on demand. Generate log-based metrics (error counts) before archiving so you keep a permanent cheap signal even after raw logs age out (§10).

Head or tail sampling for traces? Head = decide at the root, stateless and cheap, but blind to outcome (drops errors). Tail = buffer all spans of a trace until it completes, then keep errors/slow/rare — needs ~100 GB of buffer and routing all of a trace's spans to one collector via hash(trace_id)%N. Production answer is hybrid: head baseline + tail rules for what matters (§11).

If traces are sampled, how are trace dashboards accurate? Compute RED metrics (rate/errors/duration) and the service map from 100% of spans before sampling; sample only what you store. Aggregate signals are exact; only full-trace retrieval is sampled.

How do you pivot from a metric spike to a root cause? Shared identity: consistent service/env/version tags everywhere plus a propagated trace_id. Click a spike → exemplar trace_id → the trace → its logs (same trace_id). Without a shared key the three stores are islands (§12).

A team ships a metric tagged with user_id and cardinality explodes — what happens, what do you do? The index and write-path memory blow up first (cost is the index, not the points). Guard with a tag budget: cap series per metric, drop/alias the offending tag, and route that dimension to logs (archived, selectively indexed) instead of metrics.

An alert rule takes longer to evaluate than its interval — overlap? Single-flight per rule: skip the next tick while the prior evaluation is in flight (or queue at most one), so slow rules don't pile up and stampede the query path.

18 · Key talking points

  • Lead with the hard constraint: "the cost driver is cardinality — unique tag combinations — not point rate," and it governs all three pillars.
  • Three pillars, three strategies: metrics pre-aggregate + always index; logs archive all, index selectively; traces sample, then store.
  • Push at the edge because a SaaS can't pull into customer networks; pull (Prometheus) when you own the network and want free down-detection (§6).
  • Aggregate metrics at the source in two tiers; ship summaries not events (P1) — the 1000× reduction — and use mergeable DDSketch for percentiles.
  • Derive the metric store: SQL → time-partition → LSM → columnar TSDB with Gorilla (16 B → ~1.37 B/pt, ~12×; P8/P17), plus two layouts (values by series_id + inverted index by tag).
  • Logs: "Logging without Limits" — decouple ingest from index, rehydrate on demand; log-based metrics keep a cheap permanent signal.
  • Traces: head vs tail sampling (hybrid in practice); compute RED metrics from 100% of spans before sampling; propagate W3C traceparent.
  • Correlate the pillars on shared trace_id + tags; query is scatter-gather with pushdown, slowest shard sets the tail (P7, P12).
  • At-least-once ingest + idempotent points; Kafka buffer + backpressure (P9, P16).
  • Derive alerting by rung: query-store → evaluate-rollups → stream processor (forced only by event-time/anomaly alerting at volume).
