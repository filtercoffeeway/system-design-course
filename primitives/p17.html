P17 · Storage durability & write internals

What it is: how a single storage node doesn't lose data on crash, and the fundamental read-vs-write structure tradeoff.
Reach for it when: designing or reasoning about inside any database or durable store.
The lever: WAL (write-ahead log) — append intent before the mutation, replay on recovery (universal crash recovery). LSM-tree (writes sequential/fast, reads merge levels, compaction is the hidden cost) vs B-tree (predictable reads, random-I/O writes at scale). Erasure coding vs replication (erasure stores less for the same durability, costs reconstruction). Bloom filters to skip disk for absent keys.
New problem it hands you: LSM → read amplification + compaction load; B-tree → write amplification; WAL → fsync cost on the hot path; erasure coding → CPU + latency to reconstruct on read.
Exemplified by: KV store, RDBMS, Object store, Kafka (the log).

1 · The problem it solves

Underneath every database is the same pair of questions: how do you not lose data when the process crashes mid-write, and do you optimize the on-disk structure for writes or for reads? Getting this right is what separates a toy store from a durable one.

2 · How it works

WAL (write-ahead log): before mutating the main structure, append the intent to a sequential log and fsync it. On crash you replay the log — the universal recovery primitive. LSM-tree: buffer writes in memory (memtable), flush sorted runs (SSTables) to disk, and merge them in the background (compaction). Writes are sequential and fast; reads may touch several levels. B-tree: update pages in place — predictable reads, but random-I/O writes at scale. Bloom filters let a read skip an SSTable that definitely doesn’t contain a key.

graph TD
  W[write] --> WAL[(WAL: sequential)]
  W --> MT[memtable]
  MT -->|flush| L0[SSTable L0]
  L0 -->|compaction| L1[SSTable L1]
  R[read] --> BF{bloom filter} --> L0 & L1

3 · The new bottleneck it hands you

LSM → read amplification + compaction load (a read may check memtable + several SSTables; compaction competes for I/O). B-tree → write amplification (in-place page updates cause random I/O). WAL → fsync cost on the hot path (group-commit amortizes it). Erasure coding (for object stores) stores less than 3× replication for the same durability but costs CPU + latency to reconstruct on read.

4 · Real-world example

RocksDB / LevelDB

Google’s LevelDB and Facebook’s RocksDB are the LSM engines under a huge swath of modern infrastructure (it’s the storage engine inside CockroachDB, TiKV, Kafka Streams state stores, and many others). They embody the pattern exactly: a WAL for durability, a memtable for fast writes, leveled SSTable compaction, and bloom filters to avoid disk reads for absent keys. The recurring operational theme is the LSM tradeoff — tuning compaction to balance write throughput, read latency, and space amplification. Contrast with Postgres/MySQL-InnoDB, which use B-trees + WAL for read-predictable OLTP. See KV store and RDBMS.

5 · Applying it in a design

Pick the structure from the workload: "write-heavy ingest → LSM (sequential writes, accept compaction); read-heavy with range scans → B-tree; WAL either way for crash recovery."

6 · Pitfalls

  • Choosing a B-tree store for a write-saturated workload.
  • Ignoring compaction’s I/O cost when sizing an LSM store.
  • Skipping fsync and calling it "durable."
